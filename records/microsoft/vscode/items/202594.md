# Issue #202594 — Middle click on Linux can only paste from copied text, not from the Primary buffer

[https://github.com/microsoft/vscode/issues/202594](https://github.com/microsoft/vscode/issues/202594)

## Summary

Still reproducible from source on current main: when `editor.selectionClipboard` is false, `LinuxSelectionClipboardPastePreventer` blocks ALL middle-click paste (including from the OS primary selection buffer), and when true, VS Code's own selection writes can overwrite external primary-buffer content. The underlying design couples reading-from-primary with writing-to-primary, leaving no configuration that preserves standard Linux middle-click paste without VS Code overwriting the buffer.

## Change summary

Middle-click in the editor on Linux does not paste from the X11/Wayland primary selection buffer — it either pastes nothing (with `selectionClipboard: false`) or the VS Code-managed selection (which may differ from the OS primary buffer).

## Best solution

Decouple the read and write sides of the selection clipboard: allow middle-click paste to read the OS primary buffer even when VS Code does not write editor selections to it. The fix would likely live in `src/vs/workbench/contrib/codeEditor/electron-browser/selectionClipboard.ts` (the `LinuxSelectionClipboardPastePreventer` should not block paste when the OS primary buffer has external content) and possibly the `viewController.ts` dispatch path. Related issues https://github.com/microsoft/vscode/issues/167266 and https://github.com/microsoft/vscode/issues/130208 cover adjacent primary-buffer problems.
