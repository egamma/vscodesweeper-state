# Issue #311268 — Accessibility: unable to read chat messages in GitHub Copilot when todo list items are visible

[https://github.com/microsoft/vscode/issues/311268](https://github.com/microsoft/vscode/issues/311268)

## Summary

The todo list widget's WorkbenchList (with its own ARIA list role and IListService registration) likely interferes with NVDA's virtual buffer navigation of the chat message tree. No accessibility fix has been committed since the report. Routing to the accessibility and chat areas for investigation.

## Change summary

Screen reader (NVDA) cannot announce chat messages when the todo list widget is visible; clearing todos restores message readability.

## Best solution

Investigate how the WorkbenchList inside ChatTodoListWidget (src/vs/workbench/contrib/chat/browser/widget/chatContentParts/chatTodoListWidget.ts) interacts with the chat message tree's accessibility. The nested `role="list"` container and IListService registration in the input area may confuse NVDA's virtual buffer. Consider whether the todo list should use a simpler DOM structure without a full WorkbenchList, or ensure the ARIA roles and focus management don't break screen reader navigation of the parent chat list.

## Evidence

- **todo widget architecture:** ChatTodoListWidget (src/vs/workbench/contrib/chat/browser/widget/chatContentParts/chatTodoListWidget.ts) creates a WorkbenchList<IChatTodo> inside the chat input area, with role="list" on its container and role="listitem" on each todo element.
- **chat message list:** The chat messages use a WorkbenchObjectTree (src/vs/workbench/contrib/chat/browser/widget/chatListWidget.ts line 364) with its own ChatAccessibilityProvider returning role="list" and role="listitem".
- **IListService conflict potential:** Both lists register with IListService (src/vs/platform/list/browser/listService.ts); the todo list's focus handler (line 92) sets it as lastFocusedList, which may shift context keys away from the chat message tree.
- **no fix since report:** git log since 2026-04-19 shows no accessibility-related changes to the chat widget or todo list widget. Only functional changes (clearTodoList, hide widget) were committed.
- **reporter workaround:** Clicking 'Clear All Todos' removes the todo widget from the DOM (display: none), which restores screen reader navigation of chat messages — consistent with a DOM/ARIA conflict.
