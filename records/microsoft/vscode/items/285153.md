# Issue #285153 — Network proxy setting not inherit when start vscode from desktop launcher bar icon on Ubuntu.

[https://github.com/microsoft/vscode/issues/285153](https://github.com/microsoft/vscode/issues/285153)

## Summary

The report lacks detail on how the proxy env vars are defined — VS Code already resolves shell environment on Linux when launched from a desktop file (src/vs/platform/shell/node/shellEnv.ts), but GNOME system proxy settings configured via the Settings UI are stored in dconf/gsettings and are not necessarily exported as shell environment variables. Needs clarification on whether the vars are set in shell profiles or only via GNOME's network settings UI.

## Change summary

When VS Code is launched from a Linux desktop launcher, the integrated terminal does not have http_proxy/https_proxy environment variables that are present when launching from a terminal.

## Best solution

Clarify with the reporter how the proxy env vars are configured. If they're in shell profiles (.bashrc/.profile), VS Code's resolveShellEnv should pick them up — a potential shell resolution bug. If they're only set via GNOME's network settings UI (dconf/gsettings), this is expected because VS Code doesn't read GNOME proxy settings as env vars — the reporter should use VS Code's `http.proxy` setting or export the vars in their shell profile. The `terminal.integrated.inheritEnv` setting may also be relevant.
