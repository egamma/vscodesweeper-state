# Issue #96041 — unable to input chinese character

[https://github.com/microsoft/vscode/issues/96041](https://github.com/microsoft/vscode/issues/96041)

## Summary

Long-standing confirmed bug: IME input (Chinese/Japanese/Korean/Vietnamese) does not work in the VS Code snap package on Linux. The snap launcher already includes ibus/fcitx integration (electron-launch lines 263–272, libibus-1.0-5 staged), but the containerization boundary still prevents proper GTK IM module communication with the host. The issue is confirmed, labeled help-wanted, has 21 upvotes, and remains valid — keeping open.

## Change summary

Chinese (and other CJK) character input via IME does not work when VS Code is installed as a snap package on Linux; the .deb and .tar.gz distributions work correctly.

## Best solution

The snap electron-launch script (resources/linux/snap/electron-launch) and snapcraft.yaml already stage libibus and set up GTK IM module caching, but the containerized GTK toolkit still cannot reliably connect to the host's ibus/fcitx daemon. A fix likely requires either upgrading to core22/core24 (which may improve GTK IM module compatibility), adding the proper desktop/desktop-legacy snap interfaces, or setting environment variables (GTK_IM_MODULE, XMODIFIERS, QT_IM_MODULE) to bridge the host IME into the snap. The `--ozone-platform=x11` forced in line 279 also prevents native Wayland text-input-v3 IME support. This is partially upstream (snap platform limitation) but actionable in VS Code's snap packaging layer.
