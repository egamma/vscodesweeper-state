# Issue #151463 — Strange behavior of files.exclude in workspace with deep nested folders

[https://github.com/microsoft/vscode/issues/151463](https://github.com/microsoft/vscode/issues/151463)

## Summary

Source-reproducible edge case in multi-root workspaces: the explorer's "show excluded folders when they contain open editors" logic does not account for files being accessible through a more specific workspace root, causing excluded folders to appear in a parent root. The `explorer.autoRevealExclude` setting (added Nov 2022) provides a workaround but doesn't fix the underlying filter logic.

## Change summary

In a multi-root workspace where a subfolder is both excluded via `files.exclude` in a parent root and exposed as its own workspace root, opening files from the child root causes the excluded folder to appear in the parent root's explorer tree.

## Best solution

In `FilesFilter.isVisible()` (src/vs/workbench/contrib/files/browser/views/explorerViewer.ts, ~line 1414), when deciding whether to reveal an excluded folder because it contains an open editor, check whether the editor's file is more directly reachable through another workspace root (via `findClosestRoot`). If a more specific root owns the file, skip the reveal in the broader root. This keeps the existing "show opened excluded files" behavior for single-root cases while fixing the multi-root overlap.

## Evidence

- **source trace:** In src/vs/workbench/contrib/files/browser/views/explorerViewer.ts lines 1411–1418, `isVisible()` reveals excluded folders when any visible editor has a resource under the excluded stat, gated only by `stat.root === findClosestRoot(stat.resource)`. This gate checks the excluded folder's own root, not the editor file's closest root — so when a file at `packages/module-a/test1.txt` is opened via Module-A workspace root, the `packages` folder in Root still passes the gate and becomes visible.
- **workaround:** The `explorer.autoRevealExclude` setting (commit fbaacfb921a, Nov 2022) can suppress auto-reveal into excluded paths, but it does not fix the filter's reveal-on-open behavior — the excluded folder still appears when the user manually opens a file from the other workspace root.
