# Issue #231128 — font ligatures that span several textmate scopes assume the font settings of last character (last component of the composite symbol)

[https://github.com/microsoft/vscode/issues/231128](https://github.com/microsoft/vscode/issues/231128)

## Summary

Keep open: the report is still a plausible editor font-rendering bug, and I did not find a concrete fix or duplicate. Current rendering still builds spans from token styling metadata and does not appear to preserve or break on TextMate scope boundaries for ligature shaping.

## Change summary

Prevent font ligatures from visually spanning syntax scope boundaries when the combined glyph would hide or miscolor part of the source text.

## Best solution

Investigate in the editor font-rendering path whether ligature shaping should be blocked across token/scope boundaries, likely around `src/vs/editor/common/viewLayout/viewLineRenderer.ts`; if accepted, add a focused rendering test for adjacent differently scoped tokens such as Python `]#` and HTML `/>`.

- **token rendering:** `src/vs/editor/common/viewLayout/viewLineRenderer.ts` converts line tokens into `LinePart`s using token class names and later emits adjacent `<span class="...">` parts; the inspected code has splitting for overflow, whitespace, decorations, RTL, and large tokens, but not for TextMate scope boundaries as requested.
- **metadata limitation:** `src/vs/editor/common/encodedTokenAttributes.ts` and `src/vs/editor/common/tokens/lineTokens.ts` expose rendered token class names from foreground/font-style metadata, not original TextMate scope identities, so a scope-boundary-specific fix would need deliberate design rather than an already-present switch.
- **existing test behavior:** `src/vs/editor/test/common/viewLayout/viewLineRenderer.test.ts` has an `issue-21476` test documenting that with `fontLigatures: true`, large tokens are split only at spaces, which is adjacent to but does not cover the reported cross-scope ligature case.
