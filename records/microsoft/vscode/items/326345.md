# Issue #326345 — SCM Changes view does not restore repository/group collapsed state after reload

[https://github.com/microsoft/vscode/issues/326345](https://github.com/microsoft/vscode/issues/326345)

## Summary

Still reproduces on current main. The AsyncDataTree viewState only persists expanded node IDs; on reload, collapsed SCM repositories/groups fall through to their defaultCollapseState (PreserveOrExpanded) and silently reopen. Well-analyzed regression from the AsyncDataTree migration.

## Change summary

SCM Changes view loses user-chosen collapsed state for repositories and resource groups across reload because the persisted viewState only records expanded IDs, not collapsed ones.

## Best solution

The viewState format in AsyncDataTree needs to also persist collapsed node IDs (or a 'collapsed' list), so that on restore, nodes explicitly collapsed by the user are distinguished from genuinely new nodes that should use the default. The fix likely touches `getViewState()` and `asTreeElement()` in `src/vs/base/browser/ui/tree/asyncDataTree.ts`, plus possibly the SCM consumer in `scmViewPane.ts`. This is a tree-infrastructure change that affects all AsyncDataTree consumers using `collapseByDefault: false`, so it needs careful design.
