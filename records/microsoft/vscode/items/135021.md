# Issue #135021 — Extensions keeping the extension host busy lead to disconnection

[https://github.com/microsoft/vscode/issues/135021](https://github.com/microsoft/vscode/issues/135021)

## Summary

Current main contains a concrete IPC/persistent-protocol fix that avoids treating a busy event loop as a dead remote connection, which matches the failure mode discussed in this issue. Because the fix is not linked to this issue directly and there was partial flakiness reported afterward, this is a completed-close proposal rather than a triage-reason label.

## Change summary

Remote extension-host or extension work could keep the host busy long enough for VS Code to show false remote disconnection/reconnection prompts.

- **Proposed comment:**

  > This looks resolved by the persistent IPC timeout changes, especially https://github.com/microsoft/vscode/commit/45c70c29e572de3e9ca24d993f0616fd31f05029, which added event-loop load detection and avoids trashing the socket when a timeout may be caused by a busy process failing to read messages. That matches the reported failure mode here where extension or tokenizer work made the remote extension host appear disconnected even though the connection was still usable. I think this can be closed as completed; if the same false reconnect prompts still happen on current VS Code, please open a fresh issue with updated logs and repro steps.

## Evidence

- **root cause narrowed in thread:** A maintainer summarized the likely cause as anything keeping the remote extension host busy long enough that it cannot respond to the local window; repro comments cite large files and tokenizer/extension work such as Rainbow CSV and ccls.
- **fix commit:** Commit 45c70c29e572de3e9ca24d993f0616fd31f05029 (https://github.com/microsoft/vscode/commit/45c70c29e572de3e9ca24d993f0616fd31f05029) added LoadEstimator to src/vs/base/parts/ipc/common/ipc.net.ts and skips trashing the socket on timeout when high event-loop load is detected.
- **current code:** Current src/vs/base/parts/ipc/common/ipc.net.ts checks _loadEstimator.hasHighLoad() before firing socket timeout, with the comment that the timeout might be caused by the event loop being busy and failing to read messages.
- **later protocol changes:** Commit 3fb9624b29ef3878abef01c3a817a9b489b9c1fc further changed remote IPC timeout behavior to rely on unacknowledged messages, and current main keeps explicit reconnection handling in src/vs/platform/remote/common/remoteAgentConnection.ts.
