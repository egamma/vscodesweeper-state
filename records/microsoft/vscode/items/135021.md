# Issue #135021 — Extensions keeping the extension host busy lead to disconnection

[https://github.com/microsoft/vscode/issues/135021](https://github.com/microsoft/vscode/issues/135021)

## Summary

The core issue persists on main: when the remote extension host's event loop is blocked (e.g., by tokenization of large files), it cannot respond to protocol keepalives within the 20-second TimeoutTime, triggering a false disconnection. The LoadEstimator suppression only applies to the local side, not the remote.

## Change summary

Extensions keeping the remote extension host event loop busy (e.g., tokenizing large files) cause spurious disconnection dialogs because protocol keepalive acknowledgements cannot be sent while the EH is blocked.

## Best solution

The remote side needs either a LoadEstimator-equivalent that suppresses timeout signals when the remote EH reports high load, or the protocol acknowledgement/keepalive handling on the remote needs to run on a separate thread (or worker) so that a blocked extension host doesn't starve protocol-level responses. The relevant code is in `src/vs/base/parts/ipc/common/ipc.net.ts` (PersistentProtocol, _recvAckCheck, _keepAliveTimeoutCheck) and the remote server entry points.

- **local-only mitigation:** LoadEstimator.hasHighLoad() at line 1154 and 1201 suppresses timeout only when the LOCAL event loop is lagging — it cannot detect that the REMOTE side is busy.
- **prior mitigations:** Commit f148bc6ecab (2019-10-31) doubled timeouts; commit b041f460ce6 (2021-01-07) delayed the disconnection dialog to 40-50s. These reduce frequency but don't eliminate the root cause.
