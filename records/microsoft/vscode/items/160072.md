# Issue #160072 — Guarantee that TextDocument.version doesn't skip versions

[https://github.com/microsoft/vscode/issues/160072](https://github.com/microsoft/vscode/issues/160072)

## Summary

Keep open as an API contract discussion: current documentation still only promises that TextDocument.version strictly increases after changes, while current implementation suggests +1 for normal content-change events but does not support the stronger reopen guarantee requested in a follow-up.

## Change summary

The request is to document that TextDocument.version increments by exactly 1 per change and possibly continues monotonically across close/reopen during an LSP session.

## Best solution

Editor API owners should decide whether VS Code is willing to make the +1-per-observed-content-change behavior part of the stable API contract, and separately document that close/reopen may restore a previous version id rather than globally continuing to increment.

## Evidence

- **current API docs:** src/vscode-dts/vscode.d.ts:138 still says the document version "will strictly increase after each change" but does not guarantee no skipped versions or continuity across reopened documents.
- **current implementation:** src/vs/editor/common/model/textModel.ts:782 increments _versionId by 1 in _increaseVersionId, but src/vs/editor/common/model/textModel.ts:787 also exposes _overwriteVersionId for restore paths.
- **reopen behavior:** src/vs/editor/common/services/modelService.ts:337 restores disposedModelData.versionId when content matches; src/vs/editor/test/common/services/modelService.test.ts:363 asserts a recreated model gets the prior version id.
