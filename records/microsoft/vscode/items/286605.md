# Issue #286605 — Source Control panel – Staged file status

[https://github.com/microsoft/vscode/issues/286605](https://github.com/microsoft/vscode/issues/286605)

## Summary

Confirmed source-reproducible bug: the GitDecorationProvider maintains one FileDecoration per URI in a Map, and the working tree status overwrites the index status when a file exists in both groups, causing the SCM Staged Changes badge to show 'M' instead of 'A'.

## Change summary

A staged new file incorrectly shows the Modified badge/icon in the Staged Changes panel when the file also has unstaged modifications.

## Best solution

The fix is architecturally non-trivial because FileDecorationProvider is URI-keyed (one decoration per URI) but the SCM tree shows the same URI in multiple resource groups needing different badges. Options: (1) make the SCM tree render the Resource's own `resourceDecoration` badge instead of querying the global FileDecorationProvider, (2) always provide SCM resource icons (set `useIcons = true`) so the correct per-resource SVG is shown and badges are suppressed, or (3) register separate decoration providers with context-aware URI schemes. Likely files: `extensions/git/src/decorationProvider.ts` and possibly `src/vs/workbench/contrib/scm/browser/scmViewPane.ts`.

## Evidence

- **decoration overwrite:** extensions/git/src/decorationProvider.ts lines 121-133: onDidRunGitStatus() processes indexGroup first (line 124) then workingTreeGroup (line 126) into the same Map; collectDecorationData uses bucket.set(r.original.toString(), decoration) at line 141, so the last group processed wins for a given URI.
- **default uses badges not icons:** extensions/git/src/repository.ts line 3002: `const useIcons = !config.get<boolean>('decorations.enabled', true)` — with the default setting, useIcons is false, so no SVG iconPath is provided to SourceControlResourceDecorations, and the SCM tree falls back to FileDecorationProvider badges (scmViewPane.ts line 640: `badges: !icon`).
- **status parsing is correct:** extensions/git/src/repository.ts line 3073: raw.x='A' correctly maps to Status.INDEX_ADDED in the indexGroup — the parsing is fine; the bug is only in the decoration provider's single-map-per-URI design.
