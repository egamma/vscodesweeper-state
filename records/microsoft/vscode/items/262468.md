# Issue #262468 — Ctrl+C not working to end a process in external cmd.exe terminal

[https://github.com/microsoft/vscode/issues/262468](https://github.com/microsoft/vscode/issues/262468)

## Summary

Verified duplicate of #205417 — both report Ctrl+C failing in external cmd.exe terminals launched from VS Code on Windows, traced to the same root cause: `detached: true` in `WindowsExternalTerminalService.spawnTerminal()` causing `CREATE_NEW_PROCESS_GROUP` which suppresses `CTRL_C_EVENT`.

## Change summary

Ctrl+C does not terminate running processes in external cmd.exe terminals launched from VS Code on Windows, while Ctrl+Break still works.

## Best solution

Consolidate on https://github.com/microsoft/vscode/issues/205417 — both issues share the same root cause (`detached: true` on the `openTerminal` spawn in `src/vs/platform/externalTerminal/node/externalTerminalService.ts` line 71). The fix is to either remove `detached: true` or apply the same Windows Terminal (`wt`) fallback already used in `runInTerminal`. This report's additional evidence (PowerShell Ctrl+Break conflict, multiple keyboard confirmations) should carry over.

## Evidence

- **root cause in source:** Line 71 of `src/vs/platform/externalTerminal/node/externalTerminalService.ts`: `spawner.spawn(command, cmdArgs, { cwd, env, detached: true })` — on Windows, Node.js maps `detached: true` to `CREATE_NEW_PROCESS_GROUP`, which suppresses `CTRL_C_EVENT` delivery to the child process tree.
- **introducing commit:** Commit 9af00bdd834 ('fix: external terminal not registering Ctrl+C with .NET Core (#201020)') added `detached: true` to the `openTerminal` path while fixing the `runInTerminal` path — the `openTerminal` path was not given the `wt` fallback.
- **canonical issue:** https://github.com/microsoft/vscode/issues/205417 is open with labels `bug`, `terminal-external`, filed Feb 2024, and commenters there confirmed the same `detached: true` root cause and workaround (changing `detached:!0` in compiled `main.js`).
- **commenter cross-reference:** Comment 11 in this issue explicitly references #205417 and the `detached` workaround discussed there, confirming the issues are the same.
- **runInTerminal already fixed:** The `runInTerminal` method (line 106–110) already has a `wt` fallback that avoids `start /wait` — this same pattern was not applied to `openTerminal`.
