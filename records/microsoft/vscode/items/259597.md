# Issue #259597 — VS Code Fails to Launch

[https://github.com/microsoft/vscode/issues/259597](https://github.com/microsoft/vscode/issues/259597)

## Summary

Still a valid open bug. The Electron-level FSLogix workaround shipped in 1.103.0 (via https://github.com/microsoft/vscode/issues/251448) did not fully resolve the problem — two independent commenters confirm it persists on 1.104.1 and 1.107.1, and it also affects non-FSLogix writable-volume technologies (Omnissa AppVol). This is the sole remaining open issue tracking this class of VDI launch failure.

## Change summary

VS Code fails to launch on Windows VDI environments using profile-virtualization technologies (FSLogix, AppVol writable volumes) due to ENOENT errors when creating user data and extension directories.

## Best solution

Investigate why the Electron stat-operation workaround from 1.103 is insufficient — the ENOENT on `mkdir .vscode/extensions` and the `EntryNotFound` writing `storage.json` suggest the issue is broader than the stat-race that commit 1a1212a8bc9 addressed. Likely needs a more robust directory-creation fallback or retry in the main process startup path (`src/vs/code/` or `src/vs/platform/environment/`). The `upstream` label from the parent issue may still apply if this requires further Electron/Node changes.
