# Issue #315649 — IME candidate window frozen at fixed coordinate on Wayland — `zwp_text_input_v3.set_cursor_rectangle` never updates to follow Monaco caret (fcitx5, Ubuntu 26.04, VSCode 1.119.0)

[https://github.com/microsoft/vscode/issues/315649](https://github.com/microsoft/vscode/issues/315649)

## Summary

The root cause is in Chromium's Ozone/Wayland backend: VS Code's editor does position the textarea at the cursor (textAreaEditContext.ts lines 768–812) and the native EditContext path does call updateSelectionBounds/updateControlBounds (nativeEditContext.ts lines 572–574), but Chromium's text-input-v3 implementation does not translate those DOM/EditContext position updates into fresh zwp_text_input_v3.set_cursor_rectangle calls. This is an upstream Chromium/Electron issue.

## Change summary

IME candidate window is anchored at a fixed position on Wayland instead of following the Monaco caret, because Chromium never updates set_cursor_rectangle when the textarea/EditContext bounds change.

## Best solution

Track the upstream Chromium bug (likely in components/exo or ozone/wayland text-input). VS Code already positions its hidden textarea and updates EditContext bounds correctly; the fix needs to land in Chromium's Wayland text-input-v3 backend to observe element position changes and emit updated set_cursor_rectangle calls. Consider filing or linking to a crbug and tagging as upstream/Electron.
