# Issue #314663 — Copying symlink, then original file creates symlink

[https://github.com/microsoft/vscode/issues/314663](https://github.com/microsoft/vscode/issues/314663)

## Summary

Well-reported bug with clear reproduction steps: after copying a symlink in the explorer, copying the original (non-symlink) file to the same target incorrectly produces a symlink until the window is reloaded. The low-level copy logic (Promises.copy with SymlinkSupport.stat) looks correct — it does a fresh lstat each time — so the root cause likely lives in a caching layer above (explorer model, file service resolve, or the remote stat proxy). Needs area-owner investigation.

## Change summary

Copying a regular file in the explorer creates a symlink instead of a regular copy when a symlink pointing to the same file was previously copied, until the window is reloaded.

## Best solution

Investigate the explorer model's stat caching or the file service resolve path to find where stale isSymbolicLink state persists across copy operations. The disk-level copy (src/vs/base/node/pfs.ts Promises.copy) always does a fresh lstat, so the bug is likely in a layer above — possibly the ExplorerItem model (src/vs/workbench/contrib/files/common/explorerModel.ts) or the file service's resolve/stat caching for remote connections. The 'reload fixes it' pattern confirms in-memory state corruption.

## Evidence

- **copy preserves symlinks by design:** src/vs/platform/files/node/diskFileSystemProvider.ts line 711 calls Promises.copy with preserveSymlinks: true, which is correct for copying symlinks but should not affect copying non-symlink files.
- **fresh lstat on each copy:** src/vs/base/node/pfs.ts line 264-274: SymlinkSupport.stat always does a fresh fs.promises.lstat, so the raw copy path correctly distinguishes symlinks from regular files.
- **explorer model caches isSymbolicLink:** src/vs/workbench/contrib/files/common/explorerModel.ts line 104,145,259: ExplorerItem stores _isSymbolicLink and updates it from disk stat. A stale value here could propagate incorrect behavior.
- **reload fixes it:** Reporter confirms that 'Developer: Reload Window' resolves the issue, strongly indicating stale in-memory state rather than a file system bug.
