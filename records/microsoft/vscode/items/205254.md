# Issue #205254 — parentTerminal doesn't always work 

[https://github.com/microsoft/vscode/issues/205254](https://github.com/microsoft/vscode/issues/205254)

## Summary

Confirmed bug: `_deserializeParentTerminal` only looks up string-keyed entries in `_extHostTerminals`, but after `$acceptTerminalOpened` the terminal's identifier becomes numeric — so the map lookup fails for any terminal that finished creating. An open draft PR (#312550) already targets this.

## Change summary

The `parentTerminal` option in `createTerminal` is silently ignored when the parent terminal was not created in the same synchronous call, because its identifier has already transitioned from a string UUID to a numeric ID.

## Best solution

Review and land https://github.com/microsoft/vscode/pull/312550, which should fix `_deserializeParentTerminal` in `mainThreadTerminalService.ts` to use `_getTerminalInstance` (handles both string and numeric identifiers) instead of only checking `_extHostTerminals`.

## Evidence

- **source trace:** In src/vs/workbench/api/browser/mainThreadTerminalService.ts:181, `_deserializeParentTerminal` calls `this._extHostTerminals.get(location.parentTerminal.toString())` — but `_extHostTerminals` is keyed by the original string UUID. After `$acceptTerminalOpened` (extHostTerminalService.ts:636), the terminal's `_id` becomes numeric, so the serialized identifier sent from the ext host is the numeric ID converted to string, which doesn't match any map key.
- **existing _getTerminalInstance:** The same file already has `_getTerminalInstance` (line 132) which handles both cases: string → `_extHostTerminals.get(id)`, number → `terminalService.getInstanceFromId(id)`. Using this in `_deserializeParentTerminal` would fix the lookup.
- **open PR:** https://github.com/microsoft/vscode/pull/312550 (draft, opened 2026-04-25) targets this exact issue.
- **community confirmation:** Multiple commenters (2024–2026) confirm the bug: parentTerminal works only when both terminals are created synchronously, and fails once the parent has been 'accepted' internally.
