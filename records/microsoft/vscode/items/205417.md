# Issue #205417 — Launching External Terminal Changes Ctrl+C Handling Of Programs

[https://github.com/microsoft/vscode/issues/205417](https://github.com/microsoft/vscode/issues/205417)

## Summary

Still reproduces on main: `spawnTerminal` uses `start /wait` with `detached: true`, which breaks Ctrl+C in the spawned external terminal. The sibling method `runInTerminal` was fixed in 9af00bdd834 to prefer Windows Terminal, but the same fix was never applied to `spawnTerminal`.

## Change summary

Opening an external terminal via Ctrl+Shift+C on Windows causes Ctrl+C to stop working in programs running inside that terminal.

## Best solution

Apply the same Windows Terminal preference logic from `runInTerminal` (lines 106–110) to `spawnTerminal` in `src/vs/platform/externalTerminal/node/externalTerminalService.ts`, so the Ctrl+Shift+C path also avoids the `start /wait` command that eats Ctrl+C signals.

## Evidence

- **buggy code path:** src/vs/platform/externalTerminal/node/externalTerminalService.ts line 71: `spawner.spawn(command, cmdArgs, { cwd, env, detached: true })` with cmdArgs using `start /wait` (line 56). On Windows, `detached: true` sets CREATE_NEW_PROCESS_GROUP which disables Ctrl+C by default, and `start` also eats interrupt signals.
- **partial fix exists:** Commit 9af00bdd834 (https://github.com/microsoft/vscode/commit/9af00bdd834a4e9dc2f199251cdeeb4e9700f2f2) fixed the same Ctrl+C issue in `runInTerminal` by preferring Windows Terminal over `start`, but `spawnTerminal` was not updated — it even gained `detached: true` in that same commit.
- **commenter confirmation:** Comment 6 by kazza-dev confirms that setting `detached` to false in the compiled source fixes the issue.
