# Issue #297084 — Reveal In File Explorer Shows Wrong File

[https://github.com/microsoft/vscode/issues/297084](https://github.com/microsoft/vscode/issues/297084)

## Summary

Real bug, source-reproducible on current main: when the context menu is open, the explorer tree loses DOM focus, causing `getMultiSelectedResources` to skip the explorer context and fall back to the active editor's file. An open community PR (https://github.com/microsoft/vscode/pull/309250) addresses this — keeping open pending PR review.

## Change summary

Using the 'Reveal in File Explorer' keyboard shortcut while a context menu is open on a different file in the Explorer reveals the active editor's file instead of the right-clicked file.

## Best solution

Review and land https://github.com/microsoft/vscode/pull/309250, which adds a fallback check for the explorer's internal focus/selection state in `getMultiSelectedResources` (in `src/vs/workbench/contrib/files/browser/files.ts`) before falling through to the active editor resource.

The function falls through to `getResourceForCommand` at line 169, which returns the active editor's resource.
- **open PR:** https://github.com/microsoft/vscode/pull/309250 is an open community PR that adds an additional check for the explorer's internal focus/selection state before falling back to the active editor. 12 additions, 0 deletions, modifies `src/vs/workbench/contrib/files/browser/files.ts`.
- **keybinding context:** The `revealFileInOS` command (line 32–42) is registered with `when: EditorContextKeys.focus.toNegated()` and receives a `resource` argument, but when triggered via keyboard shortcut while a context menu is open, the `resource` argument is not populated from the context menu target.
