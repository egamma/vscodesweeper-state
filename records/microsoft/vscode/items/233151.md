# Issue #233151 — [WSL2] Use relative paths when linking to files

[https://github.com/microsoft/vscode/issues/233151](https://github.com/microsoft/vscode/issues/233151)

## Summary

Keep this open as an actionable editor link bug. Current main has relative `file://./...` handling, but it is explicitly limited to local `file` models, which leaves WSL/remote `vscode-remote` editors uncovered.

## Change summary

Relative editor links such as `file://./bar.ts` should open workspace files when the current editor is in WSL2.

## Best solution

Extend the relative file URI handling in `src/vs/editor/contrib/links/browser/links.ts` to resolve against remote editor model URIs, including WSL `vscode-remote` resources, and add regression coverage for `file://./bar.ts` from a remote-backed model.

## Evidence

- **current code:** `src/vs/editor/contrib/links/browser/links.ts` only enters the relative file URI rewrite when `modelUri.scheme === Schemas.file`, then joins the relative path against the current model URI.
- **remote model gap:** WSL resources are represented with the `vscode-remote` scheme; `src/vs/base/common/network.ts` defines `Schemas.vscodeRemote = 'vscode-remote'`, and remote file system registration uses that scheme in `src/vs/workbench/services/remote/common/remoteFileSystemProviderClient.ts`.
- **prior local fix:** Commit `5af7860ca23d38281f1f6d5b229b4668d2f13e24` / https://github.com/microsoft/vscode/pull/181475 adjusted relative file link handling, but the current guard still excludes WSL remote editor models.
