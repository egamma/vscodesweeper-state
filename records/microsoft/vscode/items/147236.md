# Issue #147236 — Bad symlinks  display

[https://github.com/microsoft/vscode/issues/147236](https://github.com/microsoft/vscode/issues/147236)

## Summary

Canonical bug — three later duplicates (https://github.com/microsoft/vscode/issues/233659, https://github.com/microsoft/vscode/issues/259209, https://github.com/microsoft/vscode/issues/262663) were all closed pointing here. No fix has been committed on main; the symlink decoration can go stale when a symlink is replaced by a regular directory.

## Change summary

Explorer continues showing the symlink decoration on a folder after the symlink is deleted and replaced with a regular directory of the same name, even after manual refresh.

## Best solution

The file-change handler in `explorerService.ts` (lines 65–101) may not trigger a full re-stat when a symlink is replaced by a same-name directory (DELETE+ADD in a single batch where the ADDED check sees the child already exists). Even the manual `refresh()` path re-resolves via `fileService.resolve()` — investigate whether the file service returns stale `isSymbolicLink` metadata. The fix likely touches `src/vs/workbench/contrib/files/browser/explorerService.ts` (file-change handling or refresh path) and potentially the platform file service stat layer on Windows.
