# Issue #184720 — `editor.action.inlineSuggest.commit` doesn't commit when suggestWidget visible

[https://github.com/microsoft/vscode/issues/184720](https://github.com/microsoft/vscode/issues/184720)

## Summary

Still reproduces on main: when the suggest widget is visible and `_shouldShowOnSuggestConflict` is false (the default), the inline completions model returns no state, so `accept()` silently exits even though the command is invoked.

## Change summary

The `editor.action.inlineSuggest.commit` command does nothing when the suggest widget is also visible, even when explicitly bound by the user.

## Best solution

When `accept()` is explicitly invoked (not via default Tab keybinding), it should commit the inline suggestion regardless of whether the suggest widget is visible. The fix likely belongs in `inlineCompletionsModel.ts` — either `accept()` should bypass the `_shouldShowOnSuggestConflict` guard, or the command should dismiss the suggest widget first.

## Evidence

- **source trace:** In src/vs/editor/contrib/inlineCompletions/browser/model/inlineCompletionsModel.ts line 660: when `_shouldShowOnSuggestConflict` is false (default) and a suggest item is selected, state can return undefined (line 665), causing accept() at line 916 to return early.
- **command precondition:** The command's precondition (commands.ts:193) only checks `inlineSuggestionVisible OR inlineEditVisible`, but the context key may still be true even though the model state is suppressed.
- **default keybinding guards:** The default Tab keybinding (commands.ts:213) explicitly has `SuggestContext.Visible.toNegated()`, confirming this is a known conflict area — but a user's custom binding bypasses the when clause, invokes the command, and gets a no-op.
- **commenter workaround:** Comment from 2025-08-02 confirms adding `when: inlineSuggestionVisible && !editorReadonly` to the custom keybinding works, but this doesn't address the underlying issue that the command itself fails silently.
