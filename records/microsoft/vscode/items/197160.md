# Issue #197160 — Git - Auto Stash handles submodules incorrectly

[https://github.com/microsoft/vscode/issues/197160](https://github.com/microsoft/vscode/issues/197160)

## Summary

Valid bug with milestone 'On Deck' — must keep open per scheduling rules. The maybeAutoStash fallback (git < 2.27.0) does a blanket stash/pop that includes submodule dirty state, and even the git-native --autostash path (git >= 2.27.0) may mishandle submodule checkout changes during sync. The bug is still present on main.

## Change summary

Git sync with Auto Stash enabled incorrectly stashes and pops submodule state, causing spurious 'No stash entries found' errors or misapplied changes when a submodule is checked out to a different commit.

## Best solution

The maybeAutoStash method in extensions/git/src/repository.ts should exclude submodule-only changes from the stash decision, or the sync/pull flow should handle submodule dirty state separately. The fix likely involves filtering submodule entries from workingTreeGroup.resourceStates in the shouldAutoStash check, and possibly passing --recurse-submodules=no or similar to the stash commands.
