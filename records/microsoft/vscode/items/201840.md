# Issue #201840 — Changing selection of just after text insertion messes up the undo stack [jsx]

[https://github.com/microsoft/vscode/issues/201840](https://github.com/microsoft/vscode/issues/201840)

## Summary

Keep open: the report has a concrete linked-editing/undo corruption scenario, and I did not find a fixing commit or canonical duplicate that covers this exact selection-after-edit path.

## Change summary

Changing the selection while linked editing is active after an insertion can leave extra tag text after undo.

## Best solution

Add a focused regression test under `src/vs/editor/contrib/linkedEditing/test/browser/linkedEditing.test.ts` for the reported selection-change plus undo flow, then fix the linked-editing undo grouping or cancel linked editing before undo if that is the intended behavior.

## Evidence

- **issue scope:** The reporter narrowed https://github.com/microsoft/vscode/issues/201840 to linked editing mode and provided a simpler repro using JSX text `<div>Status</div>` followed by undo while linked editing is active.
- **current implementation:** `src/vs/editor/contrib/linkedEditing/browser/linkedEditing.ts` still handles linked-edit synchronization by calling `popUndoStop()` and then `executeEdits('linkedEditing', edits)` in `_syncRanges`, which is directly in the undo stack path implicated by the report.
- **command behavior:** `cancelLinkedEditingInput` is registered in `src/vs/editor/contrib/linkedEditing/browser/linkedEditing.ts` with Escape keybindings only; I did not find an out-of-the-box undo pre-handler equivalent to the reporter's workaround.
- **test coverage:** `src/vs/editor/contrib/linkedEditing/test/browser/linkedEditing.test.ts` covers simple linked-editing undo/redo and delete undo cases, but I did not find a regression test for changing selection after an insertion before undo.
