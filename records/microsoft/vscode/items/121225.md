# Issue #121225 — QuickPick.onDidChangeActive fires before QuickPick.onDidChangeValue on a value change

[https://github.com/microsoft/vscode/issues/121225](https://github.com/microsoft/vscode/issues/121225)

## Summary

Still reproduces on current main: in `doSetValue`, `ui.list.filter()` fires focus-change events (triggering `onDidChangeActive`) synchronously before `onDidChangeValueEmitter.fire()` runs, so consumers see active-item changes before the value change that caused them.

## Change summary

QuickPick.onDidChangeActive fires before QuickPick.onDidChangeValue when the input value changes, making it impossible for extensions to correlate the active-item change with the value that caused it.

## Best solution

In `QuickInputPick.doSetValue` (src/vs/platform/quickinput/browser/quickInput.ts), wrap the `ui.list.filter()` + `trySelectFirst()` calls in `_focusEventBufferer.bufferEvents()` (the same pattern already used in the `update()` method at line 1090) and ensure `onDidChangeValueEmitter.fire()` runs before the buffered focus events flush — or fire `onDidChangeValue` before filtering. The related issue https://github.com/microsoft/vscode/issues/135971 (triple-fire of onDidChangeActive) was closed but via verification, not via a commit fixing this ordering; the root cause here remains.

## Evidence

- **source trace:** src/vs/platform/quickinput/browser/quickInput.ts lines 594–607: `doSetValue` calls `this.ui.list.filter()` (line 601) which synchronously fires list focus changes → `onDidChangeActiveEmitter.fire()` (line 927), then only afterward fires `onDidChangeValueEmitter.fire()` (line 606).
- **buffering not applied:** The `_focusEventBufferer` is used in the `update()` method (line 1090) to buffer focus events during item/filter updates, but `doSetValue` calls `filter()` outside any buffering, allowing focus events to escape before value-change.

- **maintainer comment:** TylerLeonhardt confirmed the root cause in July 2021: `ui.list.onDidChangeFocus` fires before `ui.inputBox.onDidChange`.
