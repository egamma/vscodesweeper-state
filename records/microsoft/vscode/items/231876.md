# Issue #231876 — Multiline cursor using ALT-Up and Down on number pad causes character to display after releasing keys

[https://github.com/microsoft/vscode/issues/231876](https://github.com/microsoft/vscode/issues/231876)

## Summary

Windows Alt+Numpad input method conflict: Alt+numpad arrows trigger the system-level Alt-code character entry (a Windows input method that produces a Unicode character when Alt is released after pressing numpad digits/keys). VS Code's keyboard handling does not use KeyboardEvent.location to distinguish numpad from regular arrow keys, so there is no guard against the OS-level text input event that fires on Alt release. This is an upstream platform behavior.

## Change summary

Using Alt+Up/Down on the numpad to create multiline cursors inserts stray characters on key release, because Windows interprets Alt+numpad as an Alt-code character entry.

## Best solution

The root cause is the Windows Alt-code input method (Alt + numpad = character). A potential fix in VS Code would be to detect and suppress text input events that originate from an Alt+numpad sequence upon Alt release, likely in the text input handling layer (editContext/textArea). However, this may require Electron/Chromium cooperation to reliably distinguish Alt-code text entry from legitimate composition. Alternatively, VS Code could register secondary keybindings that explicitly include numpad arrow variants, though the deeper issue is the OS-generated text input on release.

## Evidence

- **keybinding registration:** src/vs/editor/contrib/multicursor/browser/multicursor.ts registers Alt+UpArrow/DownArrow at line 55 and 106; the keyCode does not distinguish numpad from standard arrows (both map to VK_UP/VK_DOWN = 38/40).
- **no location check:** src/vs/base/browser/keyboardEvent.ts does not use KeyboardEvent.location to differentiate numpad keys from standard keys for arrow key dispatch.
- **Windows Alt-code behavior:** Windows interprets Alt held + any numpad key sequence as an Alt-code character entry; upon Alt release, a WM_CHAR message is generated with the corresponding character, which Chromium/Electron forwards as a text input event.
- **platform-specific:** Reporter confirms this only happens on Windows with numpad keys, not with dedicated arrow keys — consistent with the Windows Alt-code input method.
