# Issue #313526 — "Close Other Repositories" is non-functional for worktrees — they reappear immediately

[https://github.com/microsoft/vscode/issues/313526](https://github.com/microsoft/vscode/issues/313526)

## Summary

Confirmed bug: checkForWorktrees() in extensions/git/src/model.ts re-scans and re-opens worktrees on every git-status event without filtering the closed-repositories set, silently undoing the user's Close action. An open PR (#313612) by a contributor already targets the fix — keeping open pending that PR's review and landing.

## Change summary

"Close Other Repositories" on worktree entries in Source Control is immediately undone because the worktree auto-discovery pass runs on every git-status event and bypasses the closed-repositories guard.

## Best solution

Land https://github.com/microsoft/vscode/pull/313612 (or an equivalent fix) that filters closed repositories out of the checkForWorktrees() scan in extensions/git/src/model.ts before queuing them for openRepository(). The PR also addresses the regex and trace-string issues in the same area.

## Evidence

- **source trace:** extensions/git/src/model.ts lines 806–828: checkForWorktrees() iterates repository.worktrees and calls eventuallyScanPossibleGitRepository() for each, with no check against _closedRepositoriesManager. This runs on every onDidRunGitStatus event (line 841–843).
- **closed-set guard:** The guard at model.ts:666 (`if (!openIfClosed && this._closedRepositoriesManager.isRepositoryClosed(…))`) only fires when openIfClosed is false, but the worktree scan path reaches openRepository() via eventuallyScanPossibleGitRepository() which does not pass openIfClosed=true — meaning the guard should fire, but the scan itself re-queues the path every status cycle, creating the close-then-reopen flicker.
- **open fix PR:** https://github.com/microsoft/vscode/pull/313612 by eamodio (CONTRIBUTOR) targets this issue with Fixes #313526 closing syntax; still open and unmerged.
- **community confirmation:** 23 upvotes, multiple independent reproductions across VS Code 1.118–1.128, with detailed git extension logs showing the close/reopen loop.
- **root cause comment:** Comment 10 by mpearson traces the interaction to the openRepository API change in commit efb6eeee9dd (#293681) and how openIfClosed=true from extension callers bypasses the closed-set guard.
