## Summary

Still plausible on current main: the recursive `fileService.resolve()` path (used by the explorer with `resolveSingleChildDescendants: true`) has no visited-path cycle detection, so symlink loops in a workspace can cause unbounded recursion and 100% CPU. The issue remains valid.

## Change summary

Symlink loops in a project directory (e.g. systemd build artifacts with a fake /sys) cause VS Code's main process to spin at 100% CPU due to unbounded directory traversal.

## Best solution

Add symlink-cycle detection (a visited-realpath set or depth cap) to the recursive directory resolution in `fileService.toFileStat()` and/or to the explorer's `fetchChildren` path, so that symlink loops are detected and traversal terminates gracefully. The fix likely touches `src/vs/platform/files/common/fileService.ts` (the `toFileStat` recursion) and possibly `src/vs/workbench/contrib/files/common/explorerModel.ts`.
