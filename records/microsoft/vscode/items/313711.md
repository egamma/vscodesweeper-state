# Issue #313711 — Arch with Niri compositor and gnome-keyring

[https://github.com/microsoft/vscode/issues/313711](https://github.com/microsoft/vscode/issues/313711)

## Summary

Verified duplicate of #187338 — the canonical upstream tracking issue for Linux keyring detection failing on non-standard/unrecognized desktop environments. Niri is another compositor where Chromium's os_crypt layer doesn't detect the running gnome-keyring.

## Change summary

VS Code shows 'OS keyring is not available for encryption' on Arch Linux with Niri compositor despite gnome-keyring running.

## Best solution

Consolidate on https://github.com/microsoft/vscode/issues/187338 which tracks the upstream Chromium/Electron limitation where keyring detection fails for unrecognized desktop environments. The fix lives in Electron/Chromium's safestorage API.

  > Thanks for the report. This is a known upstream limitation in Chromium/Electron's keyring detection — when the desktop environment isn't recognized (Niri in your case), the os_crypt layer doesn't probe for a running gnome-keyring. This is tracked in https://github.com/microsoft/vscode/issues/187338 which consolidates all reports of this class.
  > 
  > As a workaround, you can launch VS Code with `--password-store=gnome-libsecret` to force it to use gnome-keyring.
