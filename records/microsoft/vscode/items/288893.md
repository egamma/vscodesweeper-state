# Issue #288893 — [Linux] Orphaned zygote processes spam "Socket operation on non-socket" errors filling system logs after VS Code crash

[https://github.com/microsoft/vscode/issues/288893](https://github.com/microsoft/vscode/issues/288893)

## Summary

Confirmed upstream Chromium/Electron bug still unresolved. The orphaned zygote spinloop is caused by a Chromium zygote code path that doesn't handle ENOTSOCK — the Electron issue was closed as not-planned and the proposed Electron PR was not merged, so the fix needs to land in Chromium itself. Multiple users continue to report disk-filling log spam (up to 1.2 TB). Keeping open as an upstream tracker.

## Change summary

After a VS Code crash on Linux, orphaned Chromium zygote processes spin in a tight loop logging 'Socket operation on non-socket' errors, filling the system journal and exhausting disk space.

## Best solution

The fix belongs in Chromium's `content/zygote/zygote_linux.cc` — the zygote main loop must handle `ENOTSOCK` from `recvmsg()` as a fatal condition (like it handles `ECONNRESET`/EOF) and exit instead of looping. Track https://issues.chromium.org/issues/41377796 for the upstream fix. Once Chromium ships it, VS Code picks it up through an Electron upgrade. In the meantime this issue serves as VS Code's upstream tracker.

## Evidence

- **upstream root cause:** Bug is in Chromium's content/zygote/zygote_linux.cc:245 — recvmsg() returns ENOTSOCK when the IPC fd is replaced by a regular file, but the error handler just logs and returns to the main loop where ppoll() returns POLLIN immediately, creating a spinloop.
- **Chromium tracker:** https://issues.chromium.org/issues/41377796 is the Chromium-side bug tracking the fix.
- **Electron issue closed not-planned:** https://github.com/electron/electron/issues/49967 was closed as 'not_planned' — the proposed fix PR https://github.com/electron/electron/pull/49968 was closed without merging, indicating the fix should come from Chromium upstream.
- **VS Code PR closed:** https://github.com/microsoft/vscode/pull/307548 (workaround pointing desktop exec to wrapper script) was closed without merging — the workaround did not resolve the issue.
- **ongoing user impact:** Multiple users confirm severe disk exhaustion: 900 GB (comment 5), 1.2 TB (comment 8), 15 GB in minutes (comment 3). Reports span January–June 2026 across multiple VS Code versions.
- **cross-app confirmation:** Same bug reported in Signal Desktop (https://github.com/signalapp/Signal-Desktop/issues/7885) and ente, confirming this is a Chromium-level issue affecting all Electron apps on Linux.
