# Issue #81520 — ctrl + click "go to definition" clashes with ctrl + click "follow link"

[https://github.com/microsoft/vscode/issues/81520](https://github.com/microsoft/vscode/issues/81520)

## Summary

Keep open: current main still has separate ctrl/cmd-click handlers for document links and go-to-definition, and the issue discussion shows this affects multiple language servers beyond the original Python/Jedi case.

## Change summary

Ctrl/cmd-clicking text that is both a document link and a definition target can trigger both follow-link and go-to-definition behavior.

## Best solution

Resolve this in the editor by coordinating the follow-link gesture across `src/vs/editor/contrib/links/browser/links.ts` and `src/vs/editor/contrib/gotoSymbol/browser/link/goToDefinitionAtPosition.ts`, likely with a shared controller or explicit precedence/picker when both a link and a definition are available.

  > Thanks for the continued context. The likely fix belongs in the editor gesture/link handling rather than in an individual language extension.

## Evidence

- **current code:** `src/vs/editor/contrib/links/browser/links.ts` still creates its own `ClickLinkGesture` and opens a link from `onExecute` via `onEditorMouseUp` when a link occurrence exists.
- **current code:** `src/vs/editor/contrib/gotoSymbol/browser/link/goToDefinitionAtPosition.ts` also creates its own `ClickLinkGesture` and invokes `gotoDefinition` from `onExecute` when a definition provider has a result.
- **no coordinating implementation found:** Repository search found no `FollowLinkController`, `IFollowLinkHandler`, or `queueExclusiveHandler` implementation corresponding to the coordination designs proposed in the maintainer comments.
- **issue scope:** Later comments report the same conflict for Go/gopls and Swift/SourceKit-LSP, so the remaining problem is the editor-level gesture conflict, not only the original Jedi string-literal behavior.
