# Issue #246811 — Debug restart missing call to `onDidTerminateDebugSession`

[https://github.com/microsoft/vscode/issues/246811](https://github.com/microsoft/vscode/issues/246811)

## Summary

Source-reproducible race condition in the debug restart flow: when the adapter takes longer than 300ms to exit, `session.initialize()` calls `shutdown()` which clears `rawListeners` before the old adapter's exit event fires, silently dropping the `onDidTerminateDebugSession` notification to extensions.

## Change summary

When restarting a debug session for adapters that don't support `supportsRestartRequest`, the `onDidTerminateDebugSession` extension API event may not fire if the debug adapter takes longer than 300ms to shut down.

## Best solution

Fix the race between the 300ms restart timeout and adapter exit in `debugService.ts`/`debugSession.ts`. The root cause is that `session.initialize()` (line 348–351 of `debugSession.ts`) calls `shutdown()` when `this.raw` still exists, which clears the `rawListeners` (including the `onDidExitAdapter` listener at line 1350) without first firing `_onDidEndAdapter`. The fix needs to ensure `_onDidEndSession` fires for the old adapter before re-initializing the session, but must be careful not to also dispose the NEW listener set registered by `registerSessionListeners` in `launchOrAttachToSession` (called just before `initialize`). One approach: move the `registerSessionListeners` call in `launchOrAttachToSession` to after `session.initialize()`, and have `initialize()` call `onDidExitAdapter()` instead of `shutdown()` when `this.raw` exists.

## Evidence

- **race condition in restart flow:** In `src/vs/workbench/contrib/debug/browser/debugService.ts` lines 928–957, when `!supportsRestartRequest`, `session.terminate(true)` is called, then after a fixed 300ms `setTimeout`, `launchOrAttachToSession(session)` re-initializes the same session object.
- **shutdown clears listeners:** `session.initialize()` at `src/vs/workbench/contrib/debug/browser/debugSession.ts` line 348–350 checks `if (this.raw)` and calls `this.shutdown()`, which clears `rawListeners` (line 1491). The `onDidExitAdapter` listener (line 1350) is in `rawListeners`, so the old adapter's exit event is never received.
- **event chain broken:** Without `onDidExitAdapter` → no `_onDidEndAdapter.fire()` → the handler at `debugService.ts` line 748 never runs → `_onDidEndSession` never fires → `mainThreadDebugService.ts` line 63 (`$acceptDebugSessionTerminated`) never called → `onDidTerminateDebugSession` extension API event is silently dropped.
- **timing-dependent reproduction:** DanTup (CONTRIBUTOR) confirmed mock-debug always fires the event (adapter exits fast, within 300ms), while the Dart adapter doesn't — consistent with the race condition: adapters that exit quickly work, slow adapters lose the event.
- **related upstream issue:** https://github.com/Dart-Code/Dart-Code/issues/5481 — the downstream symptom is broken test result state because the Dart extension cleans up TestRun in `onDidTerminateDebugSession`.
