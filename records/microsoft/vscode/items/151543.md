# Issue #151543 — Read-only --extensions-dir break remote extension installation

[https://github.com/microsoft/vscode/issues/151543](https://github.com/microsoft/vscode/issues/151543)

## Summary

Still reproduces on main: the zip function preserves read-only file permissions from the local extensions directory, and when extracted on the remote the files remain read-only, causing writes to fail. No fix has been merged. The closed PR #242256 addressed this but was never landed.

## Change summary

When local extensions live in a read-only directory (e.g. Nix store), copying them to a remote fails because the zip/extract pipeline preserves read-only permissions, and VS Code then cannot write to the extracted files.

## Best solution

In `src/vs/base/node/zip.ts`, ensure extracted files get owner-writable permissions (e.g. OR the mode with 0o200) in `extractEntry`, or ensure the `zip` function in `extensionManagementService.ts` forces writable mode when adding files. The closed PR https://github.com/microsoft/vscode/pull/242256 took the approach of making files writable during zip creation — a similar or alternative approach (making them writable on extraction) should be landed.
