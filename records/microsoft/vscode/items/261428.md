# Issue #261428 — Memory Leak in File Search (Ctrl+P) with Large Projects (1M+ Files)

[https://github.com/microsoft/vscode/issues/261428](https://github.com/microsoft/vscode/issues/261428)

## Summary

The reporter responded to the info-needed request confirming the issue still reproduces in 1.103.2. The info-needed label is stale and should be removed. The underlying architecture in rawSearchService.ts accumulates all file matches in memory for sorted search (doSortedSearch collects an unbounded results array before sorting), which is legitimate memory pressure for 1M+ file projects. This should be routed to the search/quick-open area for evaluation.

## Change summary

File search (Ctrl+P) accumulates all matching file paths in memory without bound during sorted search, causing multi-GB memory growth and eventual OOM with very large projects (1M+ files).

## Best solution

The doSortedSearch path in src/vs/workbench/services/search/node/rawSearchService.ts collects all IRawFileMatch results into an unbounded array before sorting. For extremely large workspaces, this should either cap the accumulated results or stream/sort incrementally. The search/quick-open area owner should evaluate whether the 512-result maxResults limit can be pushed down into the engine to avoid collecting all results before sorting, or whether the cache (FileQueryCacheState) needs a size budget. The cancel fix in PR #261091 addressed a different issue (search view cancel) and does not resolve this.
