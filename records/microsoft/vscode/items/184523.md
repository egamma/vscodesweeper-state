# Issue #184523 — Problems are not always generated when running tasks

[https://github.com/microsoft/vscode/issues/184523](https://github.com/microsoft/vscode/issues/184523)

## Summary

Plausible race condition in terminal line data event processing for task problem matchers on Linux, but the report is over 3 years old with no community confirmation, and several related fixes have landed since — needs re-verification on the current build.

## Change summary

Task problem matchers intermittently miss output lines on Linux, producing fewer problems than expected.

## Best solution

Ask the reporter to verify on current VS Code (1.100+). If still reproducible, the likely culprit is the async gap in `LineDataEventAddon.activate()` (the `await _initializationPromise` before registering `onLineFeed`) or the 100ms `setTimeout` flush in `terminalTaskSystem.ts:1127`. A fix would ensure no line-feed events are lost between process start and listener registration.

## Evidence

- **race condition in lineDataEventAddon:** src/vs/workbench/contrib/terminal/browser/xterm/lineDataEventAddon.ts uses `await this._initializationPromise` before registering `onLineFeed`, creating a window where early output could be missed.
- **100ms flush hack:** src/vs/workbench/contrib/tasks/browser/terminalTaskSystem.ts line 1127 uses setTimeout(100) before disposing the problem matcher — may not be sufficient for all line events to fire.
- **related fix since report:** Commit e3516c12ae0 (2024-07-16) fixed a buffer namespace issue in lineDataEventAddon that could cause data loss on disposal.
