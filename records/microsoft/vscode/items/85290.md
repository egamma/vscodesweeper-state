# Issue #85290 — Debugging should handle identical sources in sourceReference better

[https://github.com/microsoft/vscode/issues/85290](https://github.com/microsoft/vscode/issues/85290)

## Summary

Still reproduces on main: `getUriFromSource` in `debugSource.ts` unconditionally creates a `debug:` scheme URI when `sourceReference > 0`, without comparing content to the on-disk file. This causes duplicate editors, split breakpoint lists, and a read-only view even when the source is identical. The issue is a real gap in the debug experience for hot-reload workflows (Flutter, etc.).

## Change summary

When a debug adapter returns a sourceReference with a path matching an on-disk file whose content is identical, VS Code opens a separate read-only editor instead of reusing the existing editable file, causing breakpoint confusion and a degraded hot-reload experience.

## Best solution

In the debug source resolution path (`src/vs/workbench/contrib/debug/common/debugSource.ts`, `getUriFromSource`), when `sourceReference > 0` and a valid filesystem `path` is also provided, compare the sourceReference content against the on-disk file content; if identical, resolve to the filesystem URI instead of the `debug:` URI. This keeps the user in the editable file with unified breakpoints. The `checksum` field from DAP could also be leveraged to avoid fetching content when checksums match.
