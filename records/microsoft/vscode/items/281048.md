# Issue #281048 — RHEL8 does not support code-1.106.3-1764110929 due to need of GLIBCXX_3.4.26 

[https://github.com/microsoft/vscode/issues/281048](https://github.com/microsoft/vscode/issues/281048)

## Summary

Real issue: the Electron 35 update introduced a GLIBCXX_3.4.26 dependency for aarch64/armv7hl RPM packages, which exceeds what RHEL 8 ships (GLIBCXX_3.4.25 via GCC 8.5). This is an upstream dependency from Electron/Chromium, not a VS Code source-level bug. x86_64 is unaffected.

## Change summary

VS Code 1.106+ RPM package for aarch64 cannot install on RHEL 8 because the Electron binary requires GLIBCXX_3.4.26, which RHEL 8's GCC 8.5 does not provide.

## Best solution

This is an Electron/Chromium upstream issue — the aarch64 binary now links against a newer libstdc++ symbol version than RHEL 8 provides. The VS Code team should decide whether RHEL 8 aarch64 remains a supported platform and, if so, coordinate with the Electron build to avoid the GLIBCXX_3.4.26 symbol, or document the platform as unsupported for arm64.

## Evidence

- **dep-lists.ts aarch64:** build/linux/rpm/dep-lists.ts lists 'libstdc++.so.6(GLIBCXX_3.4.26)(64bit)' in the aarch64 reference deps but NOT in x86_64, confirming the reporter's observation that this is aarch64-specific.
- **Electron 35 introduced it:** Commit https://github.com/microsoft/vscode/commit/12326f9906e5599f6502a2b7c15cf720c9f9b321 (chore: update to Electron 35, 2025-05-09) first added GLIBCXX_3.4.26 to the aarch64 and armv7hl dep lists.
- **RHEL 8 ships GCC 8.5:** RHEL 8's system GCC 8.5 provides libstdc++ with GLIBCXX up to 3.4.25. GLIBCXX_3.4.26 requires GCC 9.1+, which is not the default on RHEL 8.
- **reporter confirmation:** Reporter confirmed in a comment that this is specific to aarch64 and that version 1.100.3 (pre-Electron 35) was the last installable version.
