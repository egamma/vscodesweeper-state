# Issue #91132 — Cannot drag/drop file from VS Code to other apps due to invalid data object

[https://github.com/microsoft/vscode/issues/91132](https://github.com/microsoft/vscode/issues/91132)

## Summary

Team-authored issue (MEMBER) — keeping open. The core request for proper CF_HDROP/FileDrop support when dragging files from VS Code Explorer to native Windows apps (Visual Studio, Outlook, etc.) remains unresolved. VS Code now sets `text/uri-list` on drag (PR #181373, May 2023), but this doesn't satisfy native Windows apps expecting CF_HDROP format. The root limitation is in Electron/Chromium's IDataObject implementation.

## Change summary

Dragging files from VS Code Explorer to external Windows apps fails because the drag data object advertises CF_HDROP support via QueryGetData but returns DV_E_FORMATETC on GetData — an Electron/Chromium-level defect that prevents native file drop interop.

## Best solution

This needs Electron-level work to either properly implement CF_HDROP in the drag data object or stop advertising it. VS Code could also explore using Electron's `startDrag` API or clipboard `writeBuffer` to provide native file paths. The `text/uri-list` improvement from #181373 helped browser-to-browser drops but doesn't address native app interop. Track upstream Electron progress (https://github.com/electron/electron/issues/11838) and consider whether VS Code can work around it with Electron's native drag APIs.
