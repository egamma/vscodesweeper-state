# Issue #243578 — 'Next Result' in document should be higher priority than 'Next Result' in search panel

[https://github.com/microsoft/vscode/issues/243578](https://github.com/microsoft/vscode/issues/243578)

## Summary

This is a feature request for smarter keybinding resolution when the user binds both editor Find Next and search panel Focus Next Search Result to the same key. The current behavior — workbench-level commands winning over editor-level ones — is by design in the keybinding weight system, but the reporter's ask (context-dependent dispatch based on whether the find widget is active vs. search results exist) is a reasonable enhancement.

## Change summary

When the same key is bound to both 'Find Next' (editor) and 'Focus Next Search Result' (search panel), the search panel command always wins even when the editor has focus and the find widget is active, because WorkbenchContrib weight (200) beats EditorContrib weight (100).

## Best solution

The search action `search.action.focusNextSearchResult` (in `src/vs/workbench/contrib/search/browser/searchActionsNav.ts`) could add a `when` clause like `!editorFocus || !findWidgetVisible` so it yields to the editor's Find Next when the find widget is actively showing. Alternatively, the precondition `HasSearchResults` could be tightened to also require search view focus or visibility. This would let users safely bind both commands to the same key.
