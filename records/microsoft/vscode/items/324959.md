# Issue #324959 — Copilot Chat: All previous session history silently lost after upgrade (.json → .jsonl format migration not performed)

[https://github.com/microsoft/vscode/issues/324959](https://github.com/microsoft/vscode/issues/324959)

## Summary

The reporter lost chat session history after upgrading VS Code, but the root cause is unclear. The core ChatSessionStore in chatSessionStore.ts already reads both .json and .jsonl formats with fallback logic (lines 635-658). The loss may stem from workspace storage hash changes (orphaned buckets, as commenter #3 describes), the agent sessions cache not reflecting old sessions, or extension-side session storage. Needs investigation from the chat area owner to determine the exact failure path.

## Change summary

All previous Copilot Chat session history silently disappeared after upgrading VS Code, with no migration or recovery option.

## Best solution

The chat area owner should investigate whether the session loss is caused by orphaned workspace storage buckets (hash changes on upgrade), the agentSessions.model.cache not being re-indexed from on-disk files, or the Copilot extension's session reader ignoring legacy .json files. If the core ChatSessionStore's index (ChatIndexStorageKey in StorageService) is not rebuilt from orphaned storage directories, adding a scan/recovery mechanism would address this class of data loss.

## Evidence

- **core fallback exists:** src/vs/workbench/contrib/chat/common/model/chatSessionStore.ts lines 635-658: readSessionFromLocation reads .jsonl first, falls back to .json, so the core store already handles both formats.
- **index-driven discovery:** Sessions are discovered via ChatIndexStorageKey ('chat.ChatSessionStore.index') in StorageService, not by scanning the file system. If the index is lost or not rebuilt after a workspace hash change, on-disk sessions become orphaned.
- **workspace hash orphaning:** Comment #3 by omarkhan-incontrol describes the same symptom from a different angle: sessions exist under a previous workspace hash bucket but are invisible because the workspace ID changed.
- **agent sessions cache separate:** Comment #4 references agentSessions.model.cache (src/vs/workbench/contrib/chat/browser/agentSessions/agentSessionsModel.ts line 917), a separate storage-service cache for agent sessions that must also be populated correctly.
- **reproduction unclear:** The exact upgrade path and workspace configuration that triggers the loss is not documented precisely enough to reproduce. Reporter is on 1.127→1.128 on macOS, and sees partial old sessions but not recent ones.
