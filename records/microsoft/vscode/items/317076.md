# Issue #317076 — Visible editor does not refresh document links until tab switch or reopen

[https://github.com/microsoft/vscode/issues/317076](https://github.com/microsoft/vscode/issues/317076)

## Summary

Keep open and route to the editor/link-provider area: current main still appears to lack a supported document-link invalidation path for provider results that change without a text-model edit. This should be treated as an API/editor gap rather than closed as extension-caused, because any language-server-backed document link provider can hit the same stale-cache behavior.

## Change summary

Allow document links in an already visible editor to refresh when provider results change due to external state, without requiring a tab switch, reopen, or fake text edit.

## Best solution

Add or document a supported invalidation mechanism for document link providers, likely alongside `src/vs/editor/contrib/links/browser/links.ts` and the extension API bridge in `src/vs/workbench/api/*/extHostLanguageFeatures.ts` / `mainThreadLanguageFeatures.ts`, so extensions can signal that links for visible documents need recomputation.

  > Thanks for the report and the detailed reproduction. This looks like a real editor/API gap rather than something specific to one extension: document links are recomputed for model/editor lifecycle changes, but the current `DocumentLinkProvider` API does not expose a supported way for a provider to say that its link results changed because of external state. Routing this to the editor area to consider a document-link invalidation mechanism or clear guidance for language-server-backed providers.

## Evidence

- **current editor behavior:** `src/vs/editor/contrib/links/browser/links.ts` schedules recomputation on editor link configuration changes, model content changes, model changes, model language changes, and provider registry changes, but not on provider-owned external state changes.
- **extension API shape:** `src/vscode-dts/vscode.d.ts` defines `DocumentLinkProvider` with `provideDocumentLinks` and optional `resolveDocumentLink`, but no `onDidChangeDocumentLinks`-style event for invalidating existing results.
- **extension host bridge:** `src/vs/workbench/api/common/extHostLanguageFeatures.ts` and `src/vs/workbench/api/browser/mainThreadLanguageFeatures.ts` register document link providers and proxy provide/resolve/release calls, with no provider change event comparable to semantic tokens.
- **issue discussion:** The reporter and commenter describe the same stale visible-editor behavior and note that a fake zero-character `workspace.applyEdit` refreshes links, which is consistent with the editor only recomputing links after text-model/editor lifecycle events.
- **related context:** The report links https://github.com/microsoft/vscode-languageserver-node/issues/1492 as prior discussion of document-link refresh behavior, suggesting this is broader than the IBM Z Open Editor reproduction.
