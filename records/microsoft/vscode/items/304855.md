Team-authored bug report about local folders being unexpectedly added to the Sessions window after interacting with cloud sessions — must keep open per team-member policy. The issue describes a real UX problem where cloud session folder association and local folder list state get out of sync.

## Change summary

A local folder clone that was never explicitly added to Sessions appears in the session list after interacting with cloud sessions, and cloud sessions are associated with the wrong clone.

## Best solution

Investigate how cloud session folder resolution interacts with the local folder list in the Agents window — the root cause likely involves the session-to-folder matching logic incorrectly resolving a sibling clone when associating cloud sessions with local folders. The fix should ensure cloud sessions only appear under folders the user explicitly added, and that folder resolution doesn't auto-add unlisted folders. Likely lives in `src/vs/sessions/` folder resolution or session association code.
