# Issue #283649 — Extension tools should require confirmation by default if prepareInvocation is not implemented

[https://github.com/microsoft/vscode/issues/283649](https://github.com/microsoft/vscode/issues/283649)

## Summary

Team-authored issue with 'On Deck' milestone — hard keep-open. The default confirmation infrastructure exists (lines 1046–1059 of languageModelToolsService.ts), but `isToolEligibleForAutoApproval` still returns `true` by default (line 1359), so extension tools without `prepareInvocation` are auto-approved rather than requiring confirmation. The requested default-deny behavior for unimplemented `prepareInvocation` is not yet fully enforced.

## Change summary

Extension tools that do not implement `prepareInvocation` should require user confirmation by default instead of being auto-approved.

## Best solution

Change the default return value in `isToolEligibleForAutoApproval` (or add a separate check) so that extension-contributed tools lacking `prepareInvocation` fall into the confirmation path on lines 1046–1059. The fix is localized to `src/vs/workbench/contrib/chat/browser/tools/languageModelToolsService.ts` and possibly the ext-host side.
