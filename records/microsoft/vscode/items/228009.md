# Issue #228009 — Editor goes berserk and start scrolling on mouse movements

[https://github.com/microsoft/vscode/issues/228009](https://github.com/microsoft/vscode/issues/228009)

## Summary

Keep open: there are multiple independent KDE/Wayland reports and the symptoms match VS Code's editor drag/drop target behavior getting stuck, but there is no confirmed fix, duplicate, or enough proof that this is solely upstream.

## Change summary

The editor can enter a stuck drag/drop-like state where mouse movement scrolls the editor and a dotted insertion caret follows the cursor until VS Code is restarted or another drag is performed.

## Best solution

Route this through the editor drag/drop and scrollbar ownership and reduce it to a reproducible stuck-drag sequence, likely around editor drag/drop cleanup when KDE/Wayland loses or delays the expected drop/cancel/end event. If the investigation proves the missing event comes from KDE, Wayland, Electron, or Chromium, keep this as an upstream-tracking bug rather than closing it prematurely.

## Evidence

- **affected environment pattern:** The original report is on Manjaro Linux with extensions disabled, and later commenters repeatedly report KDE/Wayland environments including Arch, Fedora, NixOS, CachyOS, and Plasma 6.x.
- **workaround signal:** Several commenters report that dragging a file or VS Code file tab clears the dotted cursor/scroll-following state, which points to a stuck drag/drop interaction rather than ordinary scrollbar configuration.
- **matching code path:** src/vs/editor/contrib/dnd/browser/dnd.ts shows DragAndDropController rendering a dnd-target decoration and calling revealPosition(position, ScrollType.Immediate) while dragging; cleanup happens on mouse drop, cancel, blur, or disposal.
- **current-code inspection:** A local git search found no commit referencing https://github.com/microsoft/vscode/issues/228009, and the relevant editor drag/drop cleanup path still exists on current main.
- **upstream uncertainty:** The reporter later noted similar behavior in Chrome, Gimp, and KDE's task manager and linked https://bugs.kde.org/show_bug.cgi?id=498764, but other commenters still observe the most disruptive behavior in VS Code and tie it to editor drag/drop.
