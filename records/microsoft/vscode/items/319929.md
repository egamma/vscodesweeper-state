# Issue #319929 — Windows-style absolute paths still flagged as outside of workspace in GitHub Copilot

[https://github.com/microsoft/vscode/issues/319929](https://github.com/microsoft/vscode/issues/319929)

## Summary

Plausible bug: Windows absolute paths (with drive letter) passed by the model to the edit_file tool are incorrectly flagged as outside the workspace, despite the workspace being at the same root. The confirmation flow in `editFileToolUtils.tsx` normalizes paths and uses `extUriBiasedIgnorePathCase.isEqualOrParent` which should handle this — but the user is on Git Bash for Windows, which may introduce path representation mismatches between the workspace folder URI and the resolved file URI. Needs Windows reproduction by the area owner.

## Change summary

When the model provides a Windows-style absolute path (e.g. `C:\projects\app\foo\bar.txt`) for a file inside the workspace, Copilot incorrectly prompts that the file is outside the workspace.

## Best solution

Investigate on Windows with Git Bash whether the workspace folder URI and the model-provided path resolve to comparable URIs. The likely files are `extensions/copilot/src/extension/tools/node/editFileToolUtils.tsx` (the `makeUriConfirmationChecker` / `checkUri` flow) and `extensions/copilot/src/platform/prompts/common/promptPathRepresentationService.ts` (the `resolveFilePath` method). A Windows-specific unit test in `editFileToolUtils.spec.ts` exercising drive-letter paths against a drive-letter workspace folder would catch the regression.

- **related closed issues:** The reporter references https://github.com/microsoft/vscode/issues/293204 (closed: 'Malformed absolute paths in read_file and list_dir trigger external files prompt') and https://github.com/microsoft/vscode/issues/292424 (closed: 'Agent tools incorrectly treat non-existent workspace files as external').
