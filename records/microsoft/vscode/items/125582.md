# Issue #125582 — Restored custom editors never load if the extension is uninstalled

[https://github.com/microsoft/vscode/issues/125582](https://github.com/microsoft/vscode/issues/125582)

## Summary

Still reproduces on current main: when a custom editor's backing extension is uninstalled, the RevivalPool in webviewWorkbenchService.ts enqueues the webview indefinitely with no timeout or fallback, causing an infinite loading spinner. The team reached consensus on the fix approach (inline fallback message) but it was never implemented. The related issue #161008 was closed as a duplicate of this one, confirming this is the canonical tracker.

## Change summary

Restored custom editors spin forever with no recovery path when their contributing extension has been uninstalled or disabled.

## Best solution

Implement the agreed-upon fallback: when a custom editor's extension is not installed/enabled and the RevivalPool cannot find a reviver, show an inline message in the editor area (similar to the binary editor pattern) with a 'Reopen With...' action link. The fix touches `src/vs/workbench/contrib/webviewPanel/browser/webviewWorkbenchService.ts` (RevivalPool timeout/detection) and likely a new fallback rendering path in the custom editor contribution. The `help wanted` label is already present.
