# Issue #109083 — vscode.debug.startDebugging should support named automatic (dynamic) and global (user settings) debug configurations

[https://github.com/microsoft/vscode/issues/109083](https://github.com/microsoft/vscode/issues/109083)

## Summary

Still a valid, open gap on current main: `startDebugging` with a string name cannot resolve user-settings or dynamic launch configurations. Team contributors (weinand, isidorn) acknowledged the gap and explicitly chose to keep this open. With 5 upvotes and no milestone, it stays under discussion.

## Change summary

Allow `vscode.debug.startDebugging` to resolve named debug configurations from user settings (global launch) and dynamic configuration providers, not only from folder-scoped launch.json files.

## Best solution

Extend `$startDebugging` in `mainThreadDebugService.ts` so that when `folder` is `undefined` and `nameOrConfig` is a string, the lookup falls through to `UserLaunch` (and optionally queries dynamic config providers). The gap is in `getLaunch(undefined)` returning `undefined` and the subsequent name lookup requiring a non-null launch. The debug area team should decide the resolution order and whether dynamic configs are in scope.
