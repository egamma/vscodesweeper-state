# Issue #142182 — Consider reducing requirements for results provided by extensions

[https://github.com/microsoft/vscode/issues/142182](https://github.com/microsoft/vscode/issues/142182)

## Summary

Team-authored design discussion about allowing extensions to return plain data objects (without class instances like `new vscode.Range`) from provider APIs, reducing allocation overhead. Still under discussion with no resolution — hard keep-open applies.

## Change summary

Allow extension provider APIs to accept plain JSON-like data objects instead of requiring class instances (e.g. Range, Position), reducing allocations in the extension host.

## Best solution

Measure the actual performance gain from accepting plain data objects in high-volume providers (completions, diagnostics), then decide on the type-system approach (recursive `data<T>` vs LSP pass-through with lazy unwrapping). The work lives in the extension-host layer and touches the API type definitions plus the DTO conversion pipeline.
