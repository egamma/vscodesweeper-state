# Issue #166437 — Replace in files is buggy when multiple changes are made on same line of code

[https://github.com/microsoft/vscode/issues/166437](https://github.com/microsoft/vscode/issues/166437)

## Summary

Still reproduces on main. The bug is in the batch-replace flow: when multiple matches on the same line are replaced individually in sequence, each replacement uses stale range data from the original match objects, causing misplaced text. The issue is confirmed, labeled, and on the On Deck milestone — keeping open.

## Change summary

Replace-in-files corrupts text when replacing multiple search matches on the same line, because subsequent replacements use stale original ranges instead of updated positions.

## Best solution

Fix `batchReplace` in `searchResult.ts` (or `FileMatchImpl.replace` in `fileMatch.ts`) so that when multiple matches on the same line are replaced sequentially, the second (and later) replacements use the refreshed match ranges produced by `updatesMatchesForLineAfterReplace` rather than the stale ranges captured before any replacement occurred. One approach: when replacing a single match, look up the current match object from `_textMatches` by the original match's ID (or line/column) after each prior replacement, so the range passed to `replaceService.replace` is current. Alternatively, collect all matches for a file and apply them in a single `bulkEditorService.apply` call (as `applyEditsToPreview` already does correctly via `pushEditOperations`).
