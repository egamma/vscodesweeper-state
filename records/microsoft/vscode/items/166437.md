# Issue #166437 — Replace in files is buggy when multiple changes are made on same line of code

[https://github.com/microsoft/vscode/issues/166437](https://github.com/microsoft/vscode/issues/166437)

## Summary

Still reproduces on current main: when multiple matches on the same line are individually selected for replacement, the second replacement uses a stale range because the queued match object is not refreshed after the first replacement shifts the line content. The bug is confirmed, canonical, and on the On Deck milestone — keeping open.

## Change summary

Replacing multiple selected search matches on the same line corrupts the file because the second replacement uses the original (now-stale) character range.

## Best solution

In `FileMatch.replace()` (src/vs/workbench/contrib/search/browser/searchTreeModel/fileMatch.ts), after the first replacement and the call to `updatesMatchesForLineAfterReplace`, the next queued replacement should look up the refreshed match (by identity or pattern) from `_textMatches` instead of using the original stale `toReplace` object's range. Alternatively, `batchReplace` in `searchResult.ts` could group matches by file and delegate to the file-level replace-all path (which builds all edits at once via `createEdits`) rather than replacing individual matches sequentially.
