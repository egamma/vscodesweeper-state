# Issue #320389 — VSCode seems it does not respect my profiles when starting some extensions

[https://github.com/microsoft/vscode/issues/320389](https://github.com/microsoft/vscode/issues/320389)

## Summary

The report describes a specific workflow where connecting to WSL from a non-Default profile causes VS Code to switch to the Default profile, breaking the connection if WSL isn't installed there. The remote explorer code in core doesn't reference profiles at all, which is consistent with the claim, but the interaction between profiles and remote authority window opening needs maintainer judgment on whether this is as-designed or a gap. Keeping open for the user-profiles area owner to assess.

## Change summary

When initiating a remote WSL connection from a non-Default profile, VS Code switches to the Default profile instead of preserving the active profile.

## Best solution

The `openWindow` calls in the remote connection flow (and possibly in the WSL extension itself) may need to pass `forceProfile` set to the current profile rather than defaulting. The user-profiles area owner should decide whether remote windows should inherit the launching profile or always use Default, then implement accordingly.
