# Issue #320573 — Orphaned crashpad-handler child process survives update and pegs a CPU core indefinitely (Windows)

[https://github.com/microsoft/vscode/issues/320573](https://github.com/microsoft/vscode/issues/320573)

## Summary

Well-documented report of an orphaned crashpad-handler process surviving a Windows update and pegging a CPU core for weeks. The root cause spans Electron's process lifecycle (no job-object containment for the crashpad child) and VS Code's inno-updater (no orphan cleanup after rename). Not reproducible deterministically, but the evidence is credible and the recent COM-surrogate cleanup (commit c2a42605f78) shows the team already addresses similar orphan-process gaps in the update path. Routing to install-update + windows for triage.

## Change summary

After a Windows in-place update, the crashpad-handler child process spawned by the old Code.exe can become orphaned, consuming ~100% of one CPU core indefinitely.

## Best solution

Extend the update/GC path (likely in `src/vs/platform/update/electron-main/updateService.win32.ts` or the inno-updater) to enumerate and terminate orphaned child processes whose image matches `old_Code.exe` or `old_*` in the install directory — similar to the COM-surrogate cleanup in commit c2a42605f78. Longer-term, placing Electron helper children in a Windows Job Object with `JOB_OBJECT_LIMIT_KILL_ON_JOB_CLOSE` would prevent all such orphans, but that is an upstream Electron change.

## Evidence

- **credible report:** Reporter provides full command line, process start time, CPU measurements, and on-disk artifacts (old_Code.exe pinned). Process ran 29 days at ~100% of one core.
- **related issue:** https://github.com/microsoft/vscode/issues/308524 tracks the same rename mechanism (opposite failure: rename fails because child locks the file). This issue is the complementary case where the rename succeeds but the child isn't terminated.
- **analogous fix on main:** Commit c2a42605f78 ('Terminate COM surrogate process before update/gc', May 2026) addresses a similar orphan-process pattern during the Windows update path, showing the team is actively closing these gaps.
- **update service code:** src/vs/platform/update/electron-main/updateService.win32.ts spawns inno_updater for GC but does not enumerate/kill orphaned child processes whose image was renamed to old_Code.exe.
- **distinct from prior crashpad issues:** https://github.com/microsoft/vscode/issues/127392 was a dump-generation storm (100K files); this is a stuck-spinning orphan with only 6 reports on disk — different root cause.
