# Issue #143317 — Some Unicode characters collide in VS Code Explorer view but not on file system

[https://github.com/microsoft/vscode/issues/143317](https://github.com/microsoft/vscode/issues/143317)

## Summary

Still present on main: `getPlatformAwareName` uses JavaScript `.toLowerCase()` to key the children map on case-insensitive file systems, but JS folds U+212A (Kelvin sign) to 'k' while NTFS does not — causing a child-map collision that hides one of two legitimately distinct directories. Author is a MEMBER, so keeping open.

## Change summary

Explorer view silently hides a directory when its name differs from a sibling only by a Unicode character that JS `.toLowerCase()` folds to the same codepoint, even though the file system treats them as distinct.

## Best solution

The root cause is in `ExplorerItem.getPlatformAwareName` (`src/vs/workbench/contrib/files/common/explorerModel.ts:415-416`), which uses JS `.toLowerCase()` as a case-folding proxy for the file system. A proper fix would either delegate case-folding to the file-system provider or use a lighter normalization (e.g. ASCII-only lowercasing) that doesn't collapse Unicode compatibility equivalents the OS keeps separate. The fix is localized but needs careful validation across platforms.
