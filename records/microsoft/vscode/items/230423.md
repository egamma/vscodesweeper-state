# Issue #230423 — git: Staging a selection stages entire block

[https://github.com/microsoft/vscode/issues/230423](https://github.com/microsoft/vscode/issues/230423)

## Summary

Still present on main: the gutter 'Stage Selection' button stages the entire block because a click on the button collapses the editor selection before the action runner reads the toolbar context, causing it to fall back to the full-hunk mapping.

## Change summary

Clicking the gutter 'Stage Selection' button in the diff editor stages the entire hunk instead of only the selected lines.

## Best solution

Snapshot the selection-filtered mapping at render time (or on mousedown) rather than lazily reading `_item` in the `ActionRunnerWithContext` callback. The fix is localized to `src/vs/editor/browser/widget/diffEditor/features/gutterFeature.ts` — the `actionRunner` lambda (around line 262) should capture the current `DiffGutterItem` eagerly so a subsequent selection change doesn't invalidate it.

## Evidence

- **race condition in source:** In src/vs/editor/browser/widget/diffEditor/features/gutterFeature.ts line 262-270, the ActionRunnerWithContext reads `this._item.read(undefined)` at click time. But the click collapses the selection, causing `_selectedDiffs` (line 77) to return `emptyArr`, which makes `gutterItems` (line 125) revert to full-hunk items. By the time the action runner fires, `_item` holds the full hunk mapping.

- **workaround confirmed:** Comments #3 and #4 confirm that right-click → 'Stage Selected Ranges' (git.stageSelectedRanges) works because it captures selections from textEditor.selections before the context menu replaces them.
