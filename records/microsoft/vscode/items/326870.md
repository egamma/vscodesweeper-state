# Issue #326870 — Copilot Chat startup force-opens historical repositories outside the current workspace

[https://github.com/microsoft/vscode/issues/326870](https://github.com/microsoft/vscode/issues/326870)

## Summary

Valid, well-documented bug: Copilot's GitServiceImpl._getRepository unconditionally calls gitAPI.openRepository for URIs outside the current workspace when forceOpen is true, and this path is triggered during session initialization for historical session repositories. The issue is source-reproducible on current main — no workspace-boundary guard exists in the forceOpen path.

## Change summary

During Copilot Chat startup, historical session metadata triggers gitService.getRepository(uri, true) for repositories outside the current workspace, which force-opens them via the Git API, bypassing user settings like git.autoRepositoryDetection and git.openRepositoryInParentFolders.

## Best solution

Add a workspace-boundary guard in GitServiceImpl._getRepository (extensions/copilot/src/platform/git/vscode-node/gitServiceImpl.ts) so that the forceOpen/openRepository fallback only fires when the URI belongs to a current workspace folder (or no workspace is open). The reporter's suggested fix — checking workspace.getWorkspaceFolder(uri) — is a reasonable starting point. Callers in session initialization paths (copilotCLIChatSessionsContribution.ts, chatSessionRepositoryTracker.ts, folderRepositoryManagerImpl.ts) should also be audited to ensure they don't pass forceOpen=true for out-of-workspace URIs.
