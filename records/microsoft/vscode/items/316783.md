# Issue #316783 — vscode_listCodeUsages reports 0 usages to Copilot if a user invoked find all references is invoked

[https://github.com/microsoft/vscode/issues/316783](https://github.com/microsoft/vscode/issues/316783)

## Summary

Team-authored issue reporting a real interaction problem: the usages tool (`vscode_listCodeUsages`) and user-initiated Find All References share the same `provideReferences` call path with no caller context, so extensions that cancel a prior request (like C/C++) cause one side to silently get 0 results. No fix on main; keeping open per team-member rule.

## Change summary

When Copilot's usages tool and a user both invoke Find All References concurrently, extensions that cancel the previous request cause one invocation to silently return 0 results, misleading Copilot or the user.

## Best solution

Add a caller-context field (e.g. a `source` or `sender` discriminator) to `ReferenceContext` in `src/vs/editor/common/languages.ts` and thread it through the extension host protocol so extensions can distinguish tool-initiated from UI-initiated reference requests. This also affects the progress notification (Bug3) — tool-initiated requests should suppress the user-facing cancel progress UI. The usages tool in `src/vs/workbench/contrib/chat/browser/tools/usagesTool.ts` would pass the new context. This touches the LSP/extension-API surface, so it needs design review.
