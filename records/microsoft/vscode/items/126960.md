# Issue #126960 — Revisit the use of `ITextEditorService` in preferences component

[https://github.com/microsoft/vscode/issues/126960](https://github.com/microsoft/vscode/issues/126960)

## Summary

The debt remains on current main: `ITextEditorService.createTextEditor` is still called in the preferences component (3 call sites), and typed inputs (`SideBySideEditorInput`, `PreferencesEditorInput`, `KeybindingsEditorInput`) are still created directly and passed to `openEditor`. The issue also carries the 'On Deck' milestone, which is a scheduling decision that defers to maintainer judgment.

## Change summary

Migrate the preferences/settings component away from creating typed editor inputs via `ITextEditorService.createTextEditor`, using untyped inputs and the editor resolver API instead.

## Best solution

Refactor the 3 remaining `createTextEditor` call sites in `src/vs/workbench/services/preferences/browser/preferencesService.ts` (lines ~441, 442, 486) and the direct typed-input `openEditor` calls (e.g. `KeybindingsEditorInput`, `PreferencesEditorInput`, `SideBySideEditorInput`) to use untyped editor inputs routed through the editor resolver. This aligns with the broader editor architecture goal of keeping typed input creation internal to editor components.
