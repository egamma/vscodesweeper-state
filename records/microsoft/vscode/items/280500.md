# Issue #280500 — Unable to drag an editor into a split editor

[https://github.com/microsoft/vscode/issues/280500](https://github.com/microsoft/vscode/issues/280500)

## Summary

Already triaged appropriately as a feature request in Backlog. The behavior (file path inserted as text when dropping from OS file explorer onto diff editor content area with Shift) is technically as-designed — on Windows, the split modifier is Alt, and dropping onto the editor text area invokes the content drop handler, not the group-level file opener.

## Change summary

When dragging a file from Windows Explorer into a diff editor panel and pressing Shift, the file path is inserted as text rather than opening the file in that panel.

## Best solution

Improve the editor drop target so that external file drops onto the content area of a diff editor prefer opening the file (or splitting) over inserting the path as text, especially when a modifier key is held. The fix would likely touch `src/vs/workbench/browser/parts/editor/editorDropTarget.ts` and potentially the editor's content drop controller in `src/vs/editor/contrib/dropOrPasteInto/browser/dropIntoEditorController.ts`.

## Evidence

- **split modifier on Windows:** src/vs/workbench/browser/parts/editor/editorDropTarget.ts line 382: `isToggleSplitOperation` uses `e.altKey` on non-Mac, `e.shiftKey` on Mac — so Shift on Windows does NOT trigger the split overlay.
- **content drop vs group drop:** The editor group drop target (editorDropTarget.ts) handles external file drops to open files, but only when the drop lands on the group overlay. Drops that land on the editor text content area are handled by the editor's own drop handler, which inserts text.
