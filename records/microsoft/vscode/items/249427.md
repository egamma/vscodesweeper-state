# Issue #249427 — Extension signature verification is failing with error code 137

[https://github.com/microsoft/vscode/issues/249427](https://github.com/microsoft/vscode/issues/249427)

## Summary

Team-authored issue documenting an upstream .NET runtime GC problem in vsce-sign on memory-constrained Linux systems. No programmatic fix has landed in VS Code; the root cause lives in the .NET CoreCLR GC heap initialization. Keeping open per team-authored rule and because a proper fix (e.g. setting DOTNET_GCHeapHardLimit when spawning vsce-sign, or falling back to a Node-based verifier) has not been implemented.

## Change summary

Extension signature verification fails with error code 137 on Linux machines with virtual memory limits because vsce-sign's .NET runtime requests an excessive mmap allocation.

## Best solution

Set DOTNET_GCHeapHardLimit in the environment when spawning vsce-sign (in `src/vs/platform/extensionManagement/node/extensionSignatureVerificationService.ts`), or implement a fallback verification path that doesn't depend on the .NET-based binary. The upstream .NET issue (dotnet/runtime#79612) describes the root cause but is unlikely to be fixed for this scenario.
