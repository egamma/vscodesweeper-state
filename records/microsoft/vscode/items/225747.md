# Issue #225747 — Invalid line width

[https://github.com/microsoft/vscode/issues/225747](https://github.com/microsoft/vscode/issues/225747)

## Summary

Keep open for font-rendering ownership rather than closing: the evidence points to Chromium/font shaping and kerning behavior, not a VS Code editor logic regression.

## Change summary

The reporter expects equal character counts to produce equal rendered line widths for text containing wide Latin letters and playing-card suit glyphs.

## Best solution

Leave this with `font-rendering` unless a maintainer decides to close as by-design; any actual fix would likely need to come from the font or Chromium shaping/rendering behavior, with VS Code only documenting or working around it if there is a clear editor impact such as broken column selection.

## Evidence

- **maintainer comment:** alexdima stated that this is under font-family control and that VS Code rendering is based on Chromium, which respects kerning rules defined in the font.
- **reporter follow-up:** The reporter confirmed the issue does not occur with Consolas and is font-specific with Noto Sans Mono and Droid Sans Mono in the browser.
- **code inspection:** src/vs/editor/browser/config/fontMeasurements.ts measures whether a configured font is monospace from sampled glyph widths; src/vs/editor/browser/viewParts/viewLines/viewLineOptions.ts only enables monospace optimizations when that measured fontInfo is monospace.
- **unicode rendering path:** src/vs/editor/browser/viewParts/viewLines/viewLine.ts restricts the fast monospace rendered line path to basic ASCII, so the suit-symbol sample is not simply treated as a guaranteed fixed-width ASCII line.
