# Issue #94562 — Trim Final Newlines integrates poorly with EditorConfig for Windows files

[https://github.com/microsoft/vscode/issues/94562](https://github.com/microsoft/vscode/issues/94562)

## Summary

Valid feature request with 25 upvotes in Backlog: better interplay between VS Code's built-in trim-final-newlines save participant and EditorConfig's per-file-type `insert_final_newline` setting. The core save participants (`src/vs/workbench/contrib/codeEditor/browser/saveParticipants.ts`) read `files.trimFinalNewlines` from configuration but don't coordinate with EditorConfig's per-file preferences. Keeping open as a backlog item with community traction.

## Change summary

Make VS Code's Trim Final Newlines feature respect EditorConfig's per-file-type `insert_final_newline = false` setting, so files configured to have no trailing newline are properly trimmed.

## Best solution

The EditorConfig extension already overrides VS Code settings per file via configuration overrides. The real gap is likely that EditorConfig's `insert_final_newline = false` doesn't imply `trimFinalNewlines = true` for those file types. The fix would either live in the EditorConfig extension (setting `files.trimFinalNewlines` per resource when `insert_final_newline = false`) or in core's save participant logic to coordinate the two settings. Given EditorConfig is a marketplace extension, the most natural path is for it to set both settings together — making this partly an `*extension-candidate` concern. However, since EditorConfig is effectively a semi-official extension and the interplay is subtle, keeping this open for a maintainer decision is appropriate.
