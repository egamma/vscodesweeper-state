# Issue #171033 — Trim Trailing Whitespace does not trim whitespace before cursor on autoSave

[https://github.com/microsoft/vscode/issues/171033](https://github.com/microsoft/vscode/issues/171033)

## Summary

Current main still has explicit behavior that skips trimming the active cursor line during auto save, so this is not fixed and should not be swept closed. It should stay with the editor-input area for a maintainer decision on whether the intentional cursor-preserving behavior should change.

## Change summary

With files.trimTrailingWhitespace and autoSave afterDelay, trailing whitespace remains on disk for the line containing the active cursor.

## Best solution

Have the editor-input owners decide whether TrimWhitespaceParticipant should support trimming the saved contents without moving/changing the editor cursor line, or document the existing auto-save exception as designed.

## Evidence

- **current implementation:** src/vs/workbench/contrib/codeEditor/browser/saveParticipants.ts collects editor selections as cursor positions only when context.reason === SaveReason.AUTO before calling trimTrailingWhitespace, specifically to avoid cursor jumps.
- **trim behavior:** src/vs/editor/common/commands/trimTrailingWhitespaceCommand.ts documents that trimming ignores lines on which cursors are sitting; when the cursor is at the end of the line it continues without edits for that line.
- **tests:** src/vs/editor/test/browser/commands/trimTrailingWhitespaceCommand.test.ts includes coverage showing cursor positions limit or skip trailing-whitespace deletion on those lines.
- **issue discussion:** The reporter explicitly wants the saved file trimmed while preserving the editor cursor/content state; another commenter notes this would make editor contents differ from disk after save.
