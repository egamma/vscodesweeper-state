# Issue #207806 — settings files.exclude does not recognize .vscode folder

[https://github.com/microsoft/vscode/issues/207806](https://github.com/microsoft/vscode/issues/207806)

## Summary

This is a duplicate of #869. The reporter expects `false` values in `files.exclude` to act as whitelists overriding broader patterns like `*.*`, but by design `false` only disables that specific pattern — it does not negate matches from other patterns. The canonical request for "exclude all except..." semantics is tracked in #869.

## Change summary

Setting a pattern to `false` in `files.exclude` should override broader patterns and force-include the matching item.

## Best solution

Consolidate on https://github.com/microsoft/vscode/issues/869, which tracks adding negation/whitelist support to glob expressions in `files.exclude`. Implementing that feature would resolve this report.

## Evidence

- **as-designed behavior:** In `src/vs/base/common/glob.ts` line 749, `parseExpressionPattern` returns `NULL` when `value === false`, meaning the pattern is simply dropped from the expression — it does not create a negation or whitelist entry.
- **expression evaluation:** In `src/vs/base/common/glob.ts` lines 632-638, `parsedExpression` returns the first matching pattern. Since `false`-valued patterns are filtered out, `*.*: true` matches `.vscode` (the `*` before dot matches zero chars) with no override possible.
- **canonical issue:** https://github.com/microsoft/vscode/issues/869 is open since 2016 and tracks exactly this: `false` values in `files.exclude` should act as overrides to broader exclude patterns, enabling 'exclude all except...' workflows.
- **same underlying request:** Both issues describe the same gap: a broad exclude pattern (`**/*` or `*.*`) set to `true` cannot be selectively overridden by setting specific paths to `false`.
