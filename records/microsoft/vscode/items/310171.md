# Issue #310171 — File links in terminal behave incorrectly when the file contains unicode characters

[https://github.com/microsoft/vscode/issues/310171](https://github.com/microsoft/vscode/issues/310171)

## Summary

Still present on main: the terminal link opener passes the raw column number from the link suffix directly as `startColumn` to the editor with no code-point-to-UTF-16 conversion, so when a line contains characters outside the BMP (like emoji), the cursor lands on the wrong position.

## Change summary

Following a terminal file link (e.g. `file.py:2:8`) positions the cursor incorrectly when the target line contains non-BMP unicode characters like emoji.

## Best solution

In `TerminalLocalFileLinkOpener.open()`, after resolving the file, read the target line and convert the column number from code-point offset to UTF-16 code-unit offset before passing it as `startColumn`. This requires a design decision on the assumed source column convention (code points vs bytes vs display width) — most modern tools use code points, so that's a reasonable default. The fix lives in `src/vs/workbench/contrib/terminalContrib/links/browser/terminalLinkOpeners.ts`.
