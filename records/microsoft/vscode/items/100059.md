# Issue #100059 — Layout: waterfall for restoring default views

[https://github.com/microsoft/vscode/issues/100059](https://github.com/microsoft/vscode/issues/100059)

## Summary

The waterfall still exists on current main: sidebar, panel, and auxiliary bar each unconditionally await restoreDefaultViewsPromise regardless of whether any default view targets that location. The merged PR #100058 partially improved the situation but the core issue remains.

## Change summary

Layout restore creates an unnecessary waterfall where panel/auxiliary bar block on default-view restoration even when no default view targets those locations.

## Best solution

Refactor the restore logic in `src/vs/workbench/browser/layout.ts` so each location only awaits the default-views promise when a default view actually targets it — e.g. split the promise per ViewContainerLocation or check the defaults list before awaiting.
