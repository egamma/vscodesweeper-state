# Issue #100059 — Layout: waterfall for restoring default views

[https://github.com/microsoft/vscode/issues/100059](https://github.com/microsoft/vscode/issues/100059)

## Summary

The waterfall pattern still exists on current main: sidebar, panel, and auxiliary bar restore each await the full restoreDefaultViewsPromise even when no default view targets their location. PR #100058 partially addressed the async restore but the core inefficiency described in this issue remains.

## Change summary

Sidebar/panel/auxiliary bar restore unnecessarily waits for all default views to resolve, even when no default view targets that specific location.

## Best solution

Split restoreDefaultViewsPromise per location (sidebar, panel, auxiliary bar) so each part only awaits the subset of default views relevant to it, eliminating the cross-location waterfall in src/vs/workbench/browser/layout.ts around line 1065–1180.
