# Issue #323322 — Dragging a file from VSCode Explorer pane into a browser fails to open it

[https://github.com/microsoft/vscode/issues/323322](https://github.com/microsoft/vscode/issues/323322)

## Summary

Verified duplicate: this is a specific case of the long-standing canonical issue tracking native file drag-and-drop integration with external applications. The root cause is an Electron/Chromium limitation — VS Code sets `text/uri-list` with a `file:///` URI, but browsers block `file://` navigation from what they treat as a web-origin drop, producing `about:blank#blocked`. The canonical issue already tracks requesting native file data transfer support from Electron.

## Change summary

Dragging a file from VS Code's Explorer pane into a browser tab navigates to `about:blank#blocked` instead of opening the file, unlike dragging from Windows Explorer.

## Best solution

Consolidate on https://github.com/microsoft/vscode/issues/164, which tracks native file data transfer support in drag-and-drop across applications. The fix requires Electron to expose native OS drag data (tracked upstream at https://github.com/electron/electron/issues/2853); VS Code's current `fillEditorsDragData` in `src/vs/workbench/browser/dnd.ts` sets web-level DataTransfer types that browsers treat as web-originated drops subject to `file://` security restrictions.

Windows Explorer provides native OS file drag data that browsers handle differently.
- **prior duplicate:** https://github.com/microsoft/vscode/issues/98629 reported the same `about:blank#blocked` behavior and was closed as duplicate with `workbench-dnd` label.
