# Issue #230160 — [macOS] Custom terminal not found on first launch

[https://github.com/microsoft/vscode/issues/230160](https://github.com/microsoft/vscode/issues/230160)

## Summary

Still reproducible on main: when an extension-contributed terminal profile is set as default in workspace settings, the first terminal created after launch may fall back to the system shell because the extension hasn't activated and registered its profile provider yet. The race condition in the profile resolution path is still present.

## Change summary

Custom extension-contributed terminal profile set as workspace default is not used for the first terminal created after VS Code launch on macOS.

## Best solution

In `terminalService.ts` `createTerminal`, when the resolved default profile name matches a contributed profile that hasn't loaded yet, await extension activation (or a contributed-profiles-ready signal) before falling back to system shell. The fix likely involves `src/vs/workbench/contrib/terminal/browser/terminalService.ts` and possibly `terminalProfileService.ts`.

## Evidence

- **race condition in source:** In src/vs/workbench/contrib/terminal/browser/terminalService.ts line ~980, `createTerminal` only awaits `profilesReady` when `availableProfiles.length === 0`. If system profiles are already detected but extension-contributed profiles haven't loaded, the code proceeds without waiting.
- **contributed profile lookup:** In src/vs/workbench/contrib/terminal/browser/terminalProfileService.ts line 296, `getContributedDefaultProfile` reads `this.contributedProfiles` which may be empty if the extension hasn't activated yet — returning undefined and causing fallback to system shell.
- **related fix for different issue:** Commit 477129db8d3 (fix #200558) addressed custom pty implementations not awaiting profiles ready, but did not fix the general case of extension-contributed default profiles on first launch.
