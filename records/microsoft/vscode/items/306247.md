# Issue #306247 — Feature Request: UI or wildcard/grouping support for `chat.tools.urls.autoApprove` to reduce settings.json clutter

[https://github.com/microsoft/vscode/issues/306247](https://github.com/microsoft/vscode/issues/306247)

## Summary

The core requests (wildcard/glob support and domain-level grouping in the approval flow) are already implemented on main since late 2025. The setting supports `https://*.example.com` and path wildcards, and the confirmation UI offers approval at domain/subdomain/path levels — the central problem described is solved.

## Change summary

Request for wildcard patterns, domain-level grouping, and a dedicated UI to reduce clutter in `chat.tools.urls.autoApprove`.

## Best solution

Close as already addressed: glob matching via `testUrlMatchesGlob` and the `extractUrlPatterns`-driven approval flow already let users approve at domain or wildcard level. Users can manually consolidate old entries into patterns like `https://laravel.com/*`. A minor polish item for auto-dedup of redundant child entries could be filed separately if desired.

- **Proposed comment:**

  > Thanks for the detailed write-up! The features you describe — wildcard/glob support and domain-level approval — are actually already available in current VS Code. The `chat.tools.urls.autoApprove` setting supports glob patterns like:
  > 
  > - `"https://laravel.com/*": true` — approve all paths under laravel.com
  > - `"https://*.supabase.com": true` — approve all subdomains of supabase.com
  > 
  > And when you approve a URL through the confirmation dialog, it now offers options at different specificity levels (exact URL, domain, wildcard subdomain).
  > 
  > To clean up your existing entries, you can replace your many individual URLs with a few domain-level patterns. For example, replace all the laravel.com entries with a single `"https://laravel.com/*": true`.
  > 
  > The auto-deduplication idea (not storing child paths when a parent is already approved) is a nice polish — feel free to file a separate issue for that if you'd like. Closing this as the core requests are addressed.
