# Issue #303448 — Search: .gitignore global wildcard followed by negation (ignore then unignore) fails to index files

[https://github.com/microsoft/vscode/issues/303448](https://github.com/microsoft/vscode/issues/303448)

## Summary

Valid report: VS Code's text search (Ctrl+Shift+F) delegates .gitignore handling to ripgrep, which diverges from git on certain negation patterns. The same class of issue was previously reported as #107944 and closed as wont-fix in 2020 citing ripgrep issue BurntSushi/ripgrep#778. Keeping open because the team has since invested in gitignore negation fixes in VS Code's own IgnoreFile class (commit fc0b83a6e55), and the stance may have evolved.

## Change summary

Text search ignores files that should be un-ignored when a .gitignore uses a global wildcard pattern (e.g. `**Makefile`) followed by a negation (e.g. `!/Makefile`).

## Best solution

The root cause is in ripgrep's gitignore parser, which diverges from git on negation handling (BurntSushi/ripgrep#778). The `search` area owner should decide whether to upstream a fix to ripgrep, work around it in VS Code's search layer, or accept the divergence. VS Code's own `IgnoreFile` class (src/vs/workbench/services/search/common/ignoreFile.ts) appears to handle this pattern correctly, so only the ripgrep-backed text search path is affected.

## Evidence

- **upstream root cause:** Text search spawns ripgrep without `--no-ignore` (src/vs/workbench/services/search/node/ripgrepTextSearchEngine.ts:448-454), so ripgrep's built-in gitignore parser handles `.gitignore` negation. Ripgrep is known to diverge from git on negation patterns.
- **prior art:** https://github.com/microsoft/vscode/issues/107944 reported the same class of issue (gitignore negation not respected in search) and was closed as wont-fix in 2020, citing https://github.com/BurntSushi/ripgrep/issues/778 as the upstream cause.
- **VS Code IgnoreFile class handles it:** Code tracing of src/vs/workbench/services/search/common/ignoreFile.ts shows the pattern `**Makefile` / `!/Makefile` is correctly handled: the negation line is separated into `fileIncludeLines`, converted to glob `/Makefile`, and checked before returning an ignore verdict. The web worker file search path is not affected.
- **related fix on main:** Commit fc0b83a6e55 (2026-05-09) fixed a different gitignore negation scenario (child .gitignore negation overriding parent rules) in the IgnoreFile class. This shows ongoing investment in gitignore fidelity, but does not address the ripgrep text search path.
