# Issue #261604 — Debugger not attaching automatically anymore with `Javascript Debug Terminal`

[https://github.com/microsoft/vscode/issues/261604](https://github.com/microsoft/vscode/issues/261604)

## Summary

The JavaScript Debug Terminal's auto-attach mechanism (bootloader injection via NODE_OPTIONS) is owned by the ms-vscode.js-debug extension, not VS Code core. Multiple users confirm the issue persists, and there's a related open issue in vscode-js-debug. Keeping open as upstream — the fix likely belongs in microsoft/vscode-js-debug.

## Change summary

The JavaScript Debug Terminal no longer automatically attaches the debugger to Node.js processes; it behaves like a normal terminal.

## Best solution

This should be investigated and fixed in the microsoft/vscode-js-debug extension, which owns the bootloader injection and NODE_OPTIONS mechanism for the JavaScript Debug Terminal. The open issue https://github.com/microsoft/vscode-js-debug/issues/2296 may track overlapping work. A maintainer should confirm whether this is a duplicate of that issue or a distinct NODE_OPTIONS-handling bug in js-debug.

## Evidence

- **extension ownership:** The JavaScript Debug Terminal, bootloader.js, and NODE_OPTIONS injection are all implemented in the ms-vscode.js-debug extension (microsoft/vscode-js-debug repo), not in VS Code core. No relevant code exists under src/vs/workbench/contrib/debug/ for this mechanism.
- **multiple confirmations:** At least 4 independent users (floydspace, median-dxz, ace0527, Eptagone) confirm the debugger fails to attach in the JavaScript Debug Terminal across Windows and macOS.
- **related upstream issue:** https://github.com/microsoft/vscode-js-debug/issues/2296 is open in the js-debug repo about step debugging not working in VS Code versions after 1.103.
- **partial workaround:** Comment by Cellule identified that user-set NODE_OPTIONS in shell profiles can overwrite the debug terminal's injected value; appending $NODE_OPTIONS fixes their case. Another user (median-dxz) reports a different variant where even manual --require doesn't work.
