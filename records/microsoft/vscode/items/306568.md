# Issue #306568 — runSubagent ignores "agents" config by omitting "agentName" parameter

[https://github.com/microsoft/vscode/issues/306568](https://github.com/microsoft/vscode/issues/306568)

## Summary

Valid feature request for stricter enforcement of the agentName parameter in runSubagent when agent configurations restrict available agents. The code already makes agentName required when GeneralPurposeAgentEnabled is true, and has a defensive fallback when models omit it, but there's no enforcement tied to the per-file `agents` config.

## Change summary

When a model omits the agentName parameter in runSubagent and the user's .agent.md restricts agents via `agents = [...]`, the sub-agent falls through to the parent agent profile, potentially ignoring disable-model-invocation constraints.

## Best solution

When the parent agent's prompt file restricts `agents`, either require `agentName` in the tool schema dynamically, or validate at invocation time that the omitted-agentName fallback still respects the configured agent restrictions (e.g., reject the call or default to a safe listed agent). The relevant code is in `src/vs/workbench/contrib/chat/common/tools/builtinTools/runSubagentTool.ts` around lines 112–126 and the invoke fallback at lines 180–242.
