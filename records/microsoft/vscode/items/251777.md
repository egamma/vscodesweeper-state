# Issue #251777 — Out of memory: Cannot allocate Wasm memory for new instance (after update from 1.100.3 to 1.101)

[https://github.com/microsoft/vscode/issues/251777](https://github.com/microsoft/vscode/issues/251777)

## Summary

Confirmed upstream issue in Node.js/V8: the wasm trap handler allocates large virtual memory guard regions (~10 GB) that fail on systems with limited RLIMIT_AS (commonly HPC clusters capped at 8 GB). A team member (deepak1556) is actively tracking the upstream fix. The Node.js main-branch fix (nodejs/node#62132) has merged, but the v24.x backport (nodejs/node#64338) remains open and has not yet shipped in a Node.js release VS Code consumes. No VS Code-side auto-detection workaround has been committed yet. This should stay open until the upstream fix lands in VS Code's Node.js dependency.

## Change summary

VS Code Remote SSH server crashes with 'Cannot allocate Wasm memory' on systems with RLIMIT_AS capped below ~10 GB, due to Node.js 22+ wasm trap handler guard region allocations.

## Best solution

Wait for the Node.js v24.x backport (nodejs/node#64338) to merge and ship, then update VS Code's bundled Node.js. Alternatively, implement the server-side auto-detection workaround deepak1556 proposed in comment 12 — detect insufficient RLIMIT_AS at server startup and automatically pass --disable-wasm-trap-handler to Node.js, so users on constrained systems don't need manual .bashrc changes.

## Evidence

- **root cause confirmed:** deepak1556 confirmed the root cause: RLIMIT_AS capped at 8 GB on affected systems is below the ~10 GB V8 wasm guard region requirement. See comment 10 and comment 12.
- **upstream fix (main):** https://github.com/nodejs/node/pull/62132 merged into Node.js main — disables wasm trap handler when virtual memory is insufficient.
- **upstream backport (open):** https://github.com/nodejs/node/pull/64338 is the v24.x backport, still open as of July 2026.
- **VS Code Node.js version:** VS Code main currently uses Node.js 24.17.0 (.nvmrc), which does not yet include the fix from the open backport PR.
- **no VS Code workaround committed:** No commits on VS Code main add auto-detection of RLIMIT_AS or --disable-wasm-trap-handler to the server startup, despite deepak1556's stated intent in comment 12.
- **workaround available:** Users can set NODE_OPTIONS='--disable-wasm-trap-handler' in their remote environment (.bashrc) as a manual workaround, confirmed working by multiple commenters.
