# Issue #200586 — vscode lock-up for ~30s if database is locked with `OSCrypt tried libsecret but couldnt initialize`

[https://github.com/microsoft/vscode/issues/200586](https://github.com/microsoft/vscode/issues/200586)

## Summary

Genuine upstream issue: the ~30s lockup occurs in Chromium's key_storage_linux.cc which synchronously blocks when libsecret can't initialize. VS Code's EncryptionMainService delegates to Electron's safeStorage API and has no control over this blocking behavior. Already correctly labeled as upstream/electron.

## Change summary

VS Code locks up for ~30 seconds on Linux when the secret database (e.g. KeePassXC) is locked, because Chromium's OSCrypt synchronously blocks trying to initialize libsecret.

## Best solution

This needs a fix in Chromium/Electron — either making the libsecret initialization asynchronous or adding a shorter timeout. VS Code could potentially mitigate by deferring encryption checks off the critical startup path, but the real fix belongs upstream. Track alongside https://github.com/microsoft/vscode/issues/187338 which covers the broader Linux keyring detection problem.
