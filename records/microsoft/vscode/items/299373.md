# Issue #299373 — Source Control fails to track real-time changes on non-system drives (Windows/Linux) since v1.109.5

[https://github.com/microsoft/vscode/issues/299373](https://github.com/microsoft/vscode/issues/299373)

## Summary

Valid bug with multiple confirmations across 1.109.5–1.111.0, likely caused by the workspace-trust changes to the git extension that landed in 1.109. The diagnostic logs point to a URI-mapping failure in GitFileSystemProvider for non-system drive paths. The issue remains open on current main — trust-related repository handling was enhanced (ab8b9fe8eb5) but no fix specifically addresses the file-watching gap on secondary drives. Keeping open for the git area owner.

## Change summary

Git Source Control stops detecting real-time file changes on non-system drives (e.g. D:\) since v1.109.5; changes only appear after a window reload.

## Best solution

Investigate whether the workspace-trust resource-trust flow (commits around e4e61e5ffaf, 1bf610daabe in Jan 2026) inadvertently prevents file-system watchers from registering on secondary-drive repositories. The `createFileSystemWatcher` call in `extensions/git/src/repository.ts:920` relies on the repository being fully opened and trusted; if the trust dialog is skipped or the path normalisation fails for drive-letter URIs (`git:/d%3A/...`), the watcher never fires. A targeted fix in repository discovery or URI normalisation for non-C: drives would resolve this and likely help https://github.com/microsoft/vscode/issues/302846 as well.
