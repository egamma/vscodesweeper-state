# Issue #278865 — When Copilot creates a new file, I get this error: [error] [GitServiceImpl][getRepositoryFetchUrls] Failed to read remotes from .git/config: Error: ENOENT: no such file or directory, stat {{path_to_file}}

[https://github.com/microsoft/vscode/issues/278865](https://github.com/microsoft/vscode/issues/278865)

## Summary

Team-authored issue, still reproduces on current main. The ENOENT error in getRepositoryFetchUrls is caused by calling vscode.workspace.fs.stat() on a URI for a file that doesn't exist yet (newly created by the agent). The error is logged at error level but the underlying issue may also block downstream callers. Keeping open for the area owner.

## Change summary

When Copilot's agent creates a new file, GitServiceImpl.getRepositoryFetchUrls throws ENOENT because it stats the file URI before it exists on disk, logging a noisy error and potentially disrupting callers.

## Best solution

In `extensions/copilot/src/platform/git/vscode-node/gitServiceImpl.ts` around line 214, handle the case where the URI points to a not-yet-created file — either catch ENOENT from the stat call gracefully (log at trace/warn instead of error), or skip the stat entirely and go straight to directory-based repository root lookup. The hanging issue reported by the author and referenced in https://github.com/microsoft/vscode/issues/286626 should be investigated separately to determine if a caller is failing to handle the `undefined` return.
