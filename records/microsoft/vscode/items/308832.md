# Issue #308832 — VS Code on Linux opens a random high TCP listener bound to 0.0.0.0 (not loopback-only) when a workspace/project is opened; listener traced to node.mojom.NodeService

[https://github.com/microsoft/vscode/issues/308832](https://github.com/microsoft/vscode/issues/308832)

## Summary

Security-sensitive report — must be triaged by a human. The code in localProcessExtensionHost.ts unconditionally passes `--inspect-port=0` and `--experimental-network-inspection` to the extension host utility process. When the inspector activates, `--inspect-port=0` without an explicit host may default to wildcard binding (`0.0.0.0`) depending on Node.js/Electron version, making the inspector reachable from the LAN. The explicit `127.0.0.1` host is only set when a debug port is explicitly configured (line 261–265), not in the default `--inspect-port=0` path (line 268).

## Change summary

VS Code's extension host process may bind an inspector/debug listener to 0.0.0.0 instead of 127.0.0.1, making it network-reachable without user opt-in.

## Best solution

In `src/vs/workbench/services/extensions/electron-browser/localProcessExtensionHost.ts` line 268, change `--inspect-port=0` to include the loopback host (e.g. `--inspect-port=127.0.0.1:0`) to ensure the inspector, if activated, never binds to a wildcard address. A security review should confirm whether `--experimental-network-inspection` contributes to the binding behavior and whether equivalent hardening is needed in sharedProcess.ts.
