# Issue #167969 — Tasks use Windows instead of WSL env on WSL

[https://github.com/microsoft/vscode/issues/167969](https://github.com/microsoft/vscode/issues/167969)

## Summary

Team-authored issue; the bug appears still present on main — the electron-browser ConfigurationResolverService passes the local shell environment (via shellEnvironmentService.getShellEnv()) for ${env:...} resolution, which is the Windows host env even when connected to WSL. No fix commit found.

## Change summary

Tasks resolve ${env:...} variables using the Windows host environment instead of the WSL environment when connected to a WSL remote.

## Best solution

The variable resolution path for tasks connected to a WSL remote needs to source environment variables from the remote (WSL) side rather than the local Windows shell. The fix likely involves the configurationResolverService or the task system's variable resolution passing the remote environment when a remote is active, probably in src/vs/workbench/services/configurationResolver/electron-browser/configurationResolverService.ts or the task system's ITaskSystemInfo.resolveVariables path.
