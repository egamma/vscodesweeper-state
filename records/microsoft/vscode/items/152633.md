# Issue #152633 — Renaming a folder that is locked by an application locks up explorer for ~1 minute and eventually fails with EPERM

[https://github.com/microsoft/vscode/issues/152633](https://github.com/microsoft/vscode/issues/152633)

## Summary

Still reproduces on main: the `renameWithRetry` function in `src/vs/base/node/pfs.ts` retries EPERM/EACCES/EBUSY errors for up to 60 seconds before failing, causing a perceived hang in the explorer when a folder is locked by another application. The early-abort check (line 543–555) only fires when the *target* exists and is not a file — it doesn't help when the source folder is locked and the target doesn't exist yet.

## Change summary

Renaming a folder locked by another application in the explorer hangs for ~60 seconds before showing an EPERM error, instead of failing quickly.

## Best solution

Improve the retry logic in `renameWithRetry` (`src/vs/base/node/pfs.ts`) to detect that the *source* is a directory (not just the target) and skip retries for folder renames that fail with EPERM/EBUSY. Alternatively, surface the error to the user immediately on the first failure when the source is a directory, since AV-locking (the original retry rationale from issue #43813) applies to files, not locked directories. Consider also whether VS Code's own file watcher holds a handle that could be temporarily released before the rename.
