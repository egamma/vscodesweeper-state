# Issue #322584 — Extension host frozen ~14s by synchronous require() of undici llhttp on Copilot's first HTTPS request (starves other extensions / language clients at startup)

[https://github.com/microsoft/vscode/issues/322584](https://github.com/microsoft/vscode/issues/322584)

## Summary

Legitimate perf bug filed by a team member: the built-in Copilot extension's first HTTPS request triggers a synchronous `require()` of undici's llhttp parser on the extension-host thread, starving all other extensions for ~14s. No fix exists on current main. Keeping open per team-author rule.

## Change summary

The extension host freezes for ~14s during Copilot's first HTTPS request because undici lazily loads its HTTP/1 parser via synchronous `require()` inside a TLS callback, blocking all other extensions.

## Best solution

Pre-warm undici's llhttp module eagerly (e.g. via a top-level or early async `import()`) in the Copilot extension's networking layer (`extensions/copilot/src/platform/networking/node/nodeFetchFetcher.ts`) so the lazy synchronous `require()` inside `connectH1` is a no-op by the time the first real HTTPS request fires. Alternatively, trigger a throwaway fetch or `new undici.Agent()` connection early in activation to force the load before other extensions are competing for the thread.
