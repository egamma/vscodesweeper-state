# Issue #308807 — vscode.git mapped the files.watcherExclude files in include category

[https://github.com/microsoft/vscode/issues/308807](https://github.com/microsoft/vscode/issues/308807)

## Summary

The behavior is intentional: for non-recursive extension watchers inside a workspace, `files.watcherExclude` patterns are deliberately placed in the `includes` list to prevent duplicate events with the standard workspace watcher. The recursive watcher correctly uses them as excludes.

## Change summary

Reporter observed that `files.watcherExclude` patterns appear in the `includes` field for non-recursive file watcher requests from vscode.git.

## Best solution

Close as designed. The two log entries show the correct dual behavior: recursive watchers use the patterns as `excludes`, while non-recursive watchers inside the workspace use them as `includes` to receive only events for files excluded from the main workspace watcher, preventing duplicates (see code comments at `src/vs/workbench/api/common/extHostFileSystemEventService.ts` lines 193–228 and https://github.com/microsoft/vscode/issues/148245).

  > Thanks for the detailed report. The behavior you're seeing is actually intentional:
  > 
  > - **Recursive watchers** (your first log entry): `files.watcherExclude` patterns are used as `excludes` — reducing overhead by skipping those paths.
  > - **Non-recursive watchers** (your second log entry): the same patterns are placed in `includes` by design. This is because the standard workspace watcher already covers everything *except* excluded paths. A non-recursive extension watcher inside the workspace intentionally listens *only* for events in excluded areas (to fill the gap without producing duplicates). See the [design discussion](https://github.com/microsoft/vscode/issues/148245).
