# Issue #315271 — Deleting git worktrees closes the git repository, which cannot be reopened without closing the folder

[https://github.com/microsoft/vscode/issues/315271](https://github.com/microsoft/vscode/issues/315271)

## Summary

Legitimate bug in worktree deletion flow — already assigned to the git extension maintainer (lszomoru). The issue is traceable in the source: after `deleteWorktree` calls `worktree?.dispose()`, subsequent git operations on the main repository can hit `NotAGitRepository` errors (e.g. from file-watcher-triggered status checks on stale worktree paths), setting `RepositoryState.Disposed` and closing the main repository via the `disappearListener`.

## Change summary

Deleting a git worktree from VS Code causes the main repository to be closed and added to `closedRepositories`, preventing reopening without reloading the workspace.

## Best solution

Fix the worktree deletion flow so the main repository is not disposed when a child worktree disappears — likely guard in `extensions/git/src/repository.ts` around line 2728 to not set `RepositoryState.Disposed` when the `NotAGitRepository` error originates from a worktree path that was just deleted, or ensure the DotGitWatcher ignores transient worktree-deletion events.
