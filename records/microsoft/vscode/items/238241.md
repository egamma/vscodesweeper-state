# Issue #238241 — Test Run completion takes very long on many tests (after `RunHandler` is completed)

[https://github.com/microsoft/vscode/issues/238241](https://github.com/microsoft/vscode/issues/238241)

## Summary

Still reproduces on main: each test state transition fires an individual change event through the tree projection without batching, causing O(n × transitions × depth) DOM updates for large test suites. Prior perf fixes addressed markTaskComplete and log rendering but not the per-item state-change path during a run.

## Change summary

After a RunHandler completes reporting ~5k test results, VS Code spends significant time propagating individual state-change events through the tree UI, making it appear the run is still in progress.

## Best solution

Batch or throttle `TestResult.changeEmitter` firings during active runs so that the tree projection (`treeProjection.ts`) and decorations (`testingDecorations.ts`) process state updates in bulk on an animation frame rather than individually per test item. The `fireUpdateAndRefresh` method in `testResult.ts` is the source; the tree projection's `onTestChanged` handler at line ~141 is where batching would have most impact.

- **multiple consumers:** At least 7 listeners subscribe to onTestChanged (treeProjection, listProjection, testingDecorations ×2, testingExplorerView, testingOutputPeek ×2), each processing every event individually.
