[https://github.com/microsoft/vscode/issues/313209](https://github.com/microsoft/vscode/issues/313209)

## Summary

The report is internally inconsistent — it claims camelCase naming breaks CodeLens but the workaround is a restart, suggesting a timing/initialization issue instead. No reproduction steps identify which extension or server name triggers this, and no other users have confirmed.

## Change summary

MCP server CodeLens actions (Start/Stop/Restart) fail to appear in mcp.json after installing an MCP server via the extension gallery.

## Best solution

Ask the reporter to clarify: which extension was installed, what server name was auto-generated, and whether the issue persists after a restart or only before one. The CodeLens provider in `src/vs/workbench/contrib/mcp/browser/mcpLanguageFeatures.ts` matches servers by `definition.label` against JSON keys (line 194) — if the generated key doesn't match the label, CodeLens won't appear. But if restart fixes it, the root cause is likely a race in registry population.
