# Issue #257247 — Extended git conflict markers are not handled correctly

[https://github.com/microsoft/vscode/issues/257247](https://github.com/microsoft/vscode/issues/257247)

## Summary

Confirmed on current main: the merge-conflict extension's parser uses strict equality for the splitter marker (`line.text === '======='`), so extended 8-character conflict markers produced by git's ORT merge engine are not recognized. The start/end markers use `startsWith` and would match, but the splitter breaks parsing.

## Change summary

VS Code fails to detect and resolve git merge conflicts when conflict markers are 8+ characters long (extended markers from git ORT merge engine).

## Best solution

Change the splitter check in `extensions/merge-conflict/src/mergeConflictParser.ts` from exact equality (`===`) to a `startsWith` check (or a regex matching 7+ `=` characters not followed by more text, to avoid the false positive from #95040). The `mergeMarkersController.ts` `blockToRemoveStartLinePrefix` already uses `startsWith` so the merge editor side likely works, but should be audited for the same pattern.

- **intentional restriction from 2021:** Commit 2678e4b848a ("Only match exactly 7 equals (fixes #95040)") changed `startsWith` to `===` to avoid false positives from lines starting with `=======` in normal content. The fix was correct for that case but breaks extended markers.
