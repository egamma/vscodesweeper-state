# Issue #257247 — Extended git conflict markers are not handled correctly

[https://github.com/microsoft/vscode/issues/257247](https://github.com/microsoft/vscode/issues/257247)

## Summary

Confirmed in source: the merge-conflict extension's splitter check uses strict equality (`===`) against exactly 7 `=` characters, so extended 8-character conflict markers produced by git's ORT merge engine are not recognized. The 3-way merge editor also hardcodes 7-character markers. No fix has landed on main.

## Change summary

Git's ORT merge engine can produce 8-character conflict markers (e.g. `<<<<<<<<`, `========`, `>>>>>>>>`) in certain rename-conflict scenarios, but VS Code only recognizes 7-character markers, causing conflicts to not be displayed or resolved properly.

## Best solution

Fix the splitter check in `extensions/merge-conflict/src/mergeConflictParser.ts` (line 62) to use `startsWith` instead of strict equality, matching how the start/end markers are already handled. Also update the hardcoded markers in `src/vs/workbench/contrib/mergeEditor/browser/mergeMarkers/mergeMarkersController.ts` to handle variable-length markers. Both changes are small and localized.
