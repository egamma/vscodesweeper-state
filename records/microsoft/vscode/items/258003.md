# Issue #258003 — "Save As" dialog box sometimes corrupts the file extension 

[https://github.com/microsoft/vscode/issues/258003](https://github.com/microsoft/vscode/issues/258003)

## Summary

This is an upstream issue in the Windows Common File Dialog (or Electron's bridging of it): when compound extensions like `.prompt.md` are registered as file filters, switching to a different file type only strips the last extension segment (`.md`), leaving the `.prompt` prefix behind. VS Code correctly passes these extensions to `electron.dialog.showSaveDialog` — the replacement logic lives in the OS dialog. Keeping open as upstream, consistent with bpasero's assessment.

## Change summary

When cycling through 'Save as type' entries in the Windows native Save dialog, switching away from a compound-extension type (e.g. `.prompt.md`) leaves a stale prefix in the filename (e.g. `Untitled-1.prompt.conf` instead of `Untitled-1.conf`).

## Best solution

File an upstream Electron bug (or confirm an existing one) for incorrect compound-extension replacement in the Windows Common File Dialog bridge. As a potential VS Code-side workaround, the filter-building code in `abstractFileDialogService.ts` could avoid registering compound extensions (like `prompt.md`) in the save-dialog filter list, though that would reduce fidelity. The cleaner fix belongs in Electron or the Windows dialog layer.
