# Issue #321394 — Copy/paste to the same location does not work on WSL

[https://github.com/microsoft/vscode/issues/321394](https://github.com/microsoft/vscode/issues/321394)

## Summary

Team-authored bug report — keeping open. Copy/paste a file to the same location in WSL fails because the clipboard stages files via a local `remote-clipboard` path, and the paste logic in `findValidPasteFileTarget` fails to reconcile the staged local URI with the remote WSL target URI that already exists, producing the error instead of incrementing the filename.

## Change summary

Copy/pasting a file to the same directory in a WSL remote window errors instead of creating an incremented copy.

## Best solution

The paste path in `pasteFileHandler` (src/vs/workbench/contrib/files/browser/fileActions.ts ~line 1188–1219) needs to handle the case where clipboard resources come from a staged local path (`remote-clipboard`) but the target is a remote URI — the `isEqual` check at line 1199 likely never matches across schemes, so the same-location fallback to `element.parent` is skipped, and the subsequent `fileService.stat` or copy fails against the staged path. The fix should detect same-origin paste in the remote clipboard scenario and apply `incrementalNaming` correctly.

## Evidence

- **paste logic:** src/vs/workbench/contrib/files/browser/fileActions.ts line 1199: `uriIdentityService.extUri.isEqual(element.resource, fileToPaste)` compares the remote WSL URI against the local clipboard-staged URI — these will never match across schemes, so the code falls through to treating the target directory as the element itself rather than its parent.
- **clipboard staging:** The `readResources` in src/vs/workbench/services/clipboard/electron-browser/clipboardService.ts returns URIs serialized from the native clipboard buffer; in a WSL remote window the staged clipboard path is a local Windows path under `remote-clipboard`, not the original `vscode-remote://wsl+…` URI.
