# Issue #324392 — Shell integration: executeCommand().read() stream and onDidEndTerminalShellExecution both permanently silent (zero data) for certain multi-line / subshell command shapes

[https://github.com/microsoft/vscode/issues/324392](https://github.com/microsoft/vscode/issues/324392)

## Summary

Detailed, well-diagnosed report of the known OSC-633 multi-line/subshell parsing cluster. The reporter adds unique diagnostic value — corrupted `commandLine.value` containing stdout concatenated into the command — which strongly implicates the parser state machine. Three related open issues exist (#250764, #254724, #316556) but none is a clear canonical that fully absorbs this; routing to the terminal-shell-integration area so the maintainer can consolidate the cluster.

## Change summary

The `executeCommand().read()` stream delivers zero data and `onDidEndTerminalShellExecution` never fires for commands containing embedded newlines or `( )` subshells, because the OSC-633 parser misparses command/output boundaries for these shapes.

## Best solution

Fix the OSC-633 sequence parser in `src/vs/platform/terminal/common/xterm/shellIntegrationAddon.ts` to correctly handle command-line boundaries when the shell emits sequences around multi-line or subshell constructs. The corrupted `commandLine.value` evidence in this report (stderr appearing in the command field) is the best diagnostic lead — the parser's `handleCommandStart`/`handleCommandExecuted` transition likely consumes output text as command-line text. Once fixed, consolidate the cluster: #250764, #254724, #316556, and this issue all stem from the same root cause.
