# Issue #123934 — Git - panel incorrectly shows case-sensitive renames on case-insensitive FS

[https://github.com/microsoft/vscode/issues/123934](https://github.com/microsoft/vscode/issues/123934)

## Summary

The bug describes incorrect status letters (both showing 'D') when a case-only renamed file is modified on a case-insensitive filesystem. The defect likely lives in the decoration provider's URI-keyed lookup or in the SCM resource resolution, where case-insensitive FS semantics cause the wrong decoration to be applied. No fix found on main; keeping open. Hard keep-open rule does not apply but this appears to still be a real, unresolved bug — however the report lacks enough detail to confirm reproduction on current builds, and no commenter has confirmed it.

## Change summary

On case-insensitive filesystems, after `git mv makefile Makefile` followed by modifying the file, the SCM panel shows both entries as 'D' (deleted) instead of 'A' (added) for the new name and 'D' for the old.

## Best solution

Investigate whether the decoration provider in `extensions/git/src/decorationProvider.ts` (the `collectDecorationData` Map keyed by URI string) or the platform's decoration service normalizes URIs in a way that collapses case-only-different paths. Likely shares root cause with https://github.com/microsoft/vscode/issues/103444 (gutters not showing on case-changed filenames, milestone 'On Deck', labeled `uri`). May be worth consolidating under that issue or a new tracking issue for case-insensitive FS URI handling in git integration.

On a case-insensitive FS, when git reports a case-rename as add+delete, both paths map to the same file but produce distinct URI strings — the last write to the Map may overwrite the correct decoration.
