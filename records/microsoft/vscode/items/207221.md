# Issue #207221 — can't access module `original-fs` in remote machine

[https://github.com/microsoft/vscode/issues/207221](https://github.com/microsoft/vscode/issues/207221)

## Summary

Keep this open for now, but the actionable gap is not clear: `original-fs` is Electron-specific and the remote extension host is a Node process, while normal `fs` should be the cross-environment API to test. The reporter should provide a minimal extension repro showing why `fs` is insufficient in a remote extension host.

## Change summary

An extension cannot require Electron's `original-fs` module when running in a remote extension host.

## Best solution

Clarify whether there is a real VS Code extension-host API gap after switching the extension from `original-fs` to `fs`; if `fs` works, no core change is needed, and if it does not, the extensions/remote owners need a minimal repro to decide whether a compatibility shim is warranted.

## Evidence

- **maintainer comment:** In https://github.com/microsoft/vscode/issues/207221#issuecomment-1986819335, bpasero noted that `original-fs` likely would not be available in remote extension hosts because they run in a pure Node.js environment, and suggested extensions should get the original filesystem through `fs` instead.
- **remote extension host code:** src/vs/server/node/extensionHostConnection.ts starts the remote extension host with `cp.fork(FileAccess.asFileUri('bootstrap-fork').fsPath, ...)` and sets `VSCODE_ESM_ENTRYPOINT: 'vs/workbench/api/node/extensionHostProcess'`, which supports the comment that this is a Node-side remote extension host, not an Electron renderer/main process.
- **module mapping code:** src/bootstrap-esm.ts maps the specifier `fs` to `node:original-fs` only when `process.env['ELECTRON_RUN_AS_NODE']` or `process.versions['electron']` is present; it does not provide a general `original-fs` module shim for remote Node extension hosts.
- **require interceptor:** src/vs/workbench/api/common/extHostRequireInterceptor.ts registers extension-host module shims for `vscode`, renamed internal modules, and `open`/`opn` in remote cases, but there is no `original-fs` interceptor.
