# Issue #242003 — Extensions cannot contribute to scm/resourceState/context if the SCM provider is in another extension host

[https://github.com/microsoft/vscode/issues/242003](https://github.com/microsoft/vscode/issues/242003)

## Summary

Valid feature request: when an extension in a different extension host contributes to `scm/resourceState/context`, the passed SourceControlResourceState is an unresolved stub rather than a usable object. No fix or related work found on current main. Keeping open in the Backlog for the SCM area owner to evaluate.

## Change summary

Extensions in a different extension host than the SCM provider receive an invalid/unresolved SourceControlResourceState when their scm/resourceState/context menu command is invoked.

## Best solution

The SCM extension API marshalling layer (extHostSCM / mainThreadSCM) should reconstruct a minimal valid SourceControlResourceState (at least the resourceUri) when the command target crosses extension-host boundaries, so non-SCM extensions can meaningfully act on SCM context menu items. The SCM area owner should evaluate whether passing the raw URI or a fully hydrated state object is the right contract.
