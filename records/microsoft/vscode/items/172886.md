# Issue #172886 — The Hosts file is not applied in the network proxy.

[https://github.com/microsoft/vscode/issues/172886](https://github.com/microsoft/vscode/issues/172886)

## Summary

The report conflates two issues — /etc/hosts not being honored for the SOCKS proxy hostname, and ALL_PROXY not being read. The proxy hostname is passed to Node.js's `net.Socket.connect()` (via the `socks` library), which uses `dns.lookup` and should respect /etc/hosts. The `.local` TLD may trigger mDNS instead of /etc/hosts on Linux/WSL, making this potentially an OS-level configuration issue rather than a VS Code bug. The report is from VS Code 1.74 (Jan 2023) with no recent confirmation; requesting verification on current builds.

## Change summary

SOCKS proxy connections fail when the proxy hostname is defined only in /etc/hosts; additionally, the ALL_PROXY environment variable is not read by VS Code's proxy resolver.

## Best solution

Ask the reporter to verify on the latest VS Code (1.96+) and provide details on whether the `.local` hostname resolves correctly from Node.js outside VS Code (e.g. `node -e "require('dns').lookup('Proxy.local', console.log)"`). If confirmed, the fix belongs in `@vscode/proxy-agent` (the `resolveProxy` path in `node_modules/@vscode/proxy-agent/out/index.js`) or upstream in the `socks` package. The ALL_PROXY question is a separate feature request for `@vscode/proxy-agent`'s env-var reading logic (line 62 of index.js only checks HTTP_PROXY/HTTPS_PROXY).
