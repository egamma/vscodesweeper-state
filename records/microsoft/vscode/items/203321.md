# Issue #203321 — Git - rescan workspace after connection to the workspace folder restored

[https://github.com/microsoft/vscode/issues/203321](https://github.com/microsoft/vscode/issues/203321)

## Summary

Still valid on main: the git extension's initial repository scan (`scanWorkspaceFolders`) runs only at startup and there is no command or mechanism to re-trigger it after a network drive reconnection. The filesystem watcher may not reliably fire `.git` change events for sub-repositories after a network restore, leaving sub-repos undiscovered until a window reload.

## Change summary

After a network drive disconnects and reconnects, the git extension does not rescan for sub-repositories — only the root repository is re-detected, requiring a window reload to restore the full list.

## Best solution

Add a 'Rescan Workspace for Repositories' command (or trigger `scanWorkspaceFolders` automatically when a workspace folder becomes available again) in `extensions/git/src/model.ts`. The related issue https://github.com/microsoft/vscode/issues/320749 asks for similar re-scan capability for worktrees — both could share a workspace-level rescan mechanism.

## Evidence

- **no rescan mechanism:** `scanWorkspaceFolders()` in extensions/git/src/model.ts (line 361) is only called from `doInitialScan()` (line 327), which runs once at extension activation. There is no command or event handler to re-trigger it.
- **filesystem watcher limitations:** The watcher at model.ts line 301-307 filters for `.git` path changes on URIs not already tracked, but after a network disconnect, previously-tracked repositories are disposed, and the watcher may not fire reliably for all sub-repository `.git` directories on reconnection.
- **git.refresh is per-repo:** The `git.refresh` command (commands.ts line 807) refreshes a single already-opened repository, not a workspace-wide scan for new/restored repositories.
- **related issue:** https://github.com/microsoft/vscode/issues/320749 requests similar re-scan for worktrees and explicitly references this issue; both share the need for a workspace-level repository re-discovery path.
