# Issue #273181 — Poor graph rendering in source control graph, unusable with >2 branches

[https://github.com/microsoft/vscode/issues/273181](https://github.com/microsoft/vscode/issues/273181)

## Summary

Still valid on current main: the SCM graph swimlane layout algorithm in scmHistory.ts appends new parent lanes without compaction, causing merge commits across multiple branches to produce unnecessarily wide and loopy graphs. No changes to the layout algorithm have been made since the report. Keeping open as a valid polish request under discussion.

## Change summary

SCM source control graph renders unnecessarily wide and loopy visualizations when viewing repositories with 2+ long-running branches that merge regularly.

## Best solution

Improve the swimlane layout algorithm in `src/vs/workbench/contrib/scm/browser/scmHistory.ts` (`toISCMHistoryItemViewModelArray`) to reuse freed swimlane columns when a branch merges back, and consider chronological commit ordering to reduce visual sprawl. The Git Graph extension's compact rendering is a good reference for the expected behavior.
