# Issue #166390 — Sluggish scrolling in search view when there are many results in one file

[https://github.com/microsoft/vscode/issues/166390](https://github.com/microsoft/vscode/issues/166390)

## Summary

Still a valid, confirmed performance bug. The search view's tree rendering path has received incremental improvements (memoized previews, debounced refreshes) but the core scroll-blocking issue — rendering many match items in a single file without frame-budget optimization — remains unaddressed on current main.

## Change summary

Scrolling the search results view becomes sluggish/blocking when a single file contains many matches, because rendering highlights and labels for each visible item during scroll can take ~200ms without any scroll-frame optimization.

## Best solution

Optimize the search tree's scroll rendering path: either defer match-row rendering to the next animation frame during fast scrolling (as the reporter suggests), batch DOM updates, or implement a virtualization improvement that avoids creating/disposing tree items at high frequency. The likely files are the search match renderer in `searchResultsView.ts` and the underlying list/tree scroll handling in `src/vs/base/browser/ui/list/listView.ts`.
