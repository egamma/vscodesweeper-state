# Issue #304214 — "Open with Code" still missing from context menu in latest version

[https://github.com/microsoft/vscode/issues/304214](https://github.com/microsoft/vscode/issues/304214)

## Summary

Still a valid bug on current main. The fix in commit 20a1637cef5 (shipped in 1.112.0, closing the earlier #299964) only repairs the folder context menu for the narrow case where classic Win10 context menu is forced AND a legacy file context menu already exists. Users on Windows 11 using the modern context menu — such as this reporter — remain affected, because `ShouldRepairFolderContextMenu()` requires `IsWindows10ContextMenuForced()` to be true. The reporter's log confirms this gap: `isWindows10ContextMenuForced=false`, `shouldRepairFolderContextMenu=false`.

## Change summary

On Windows 11 with the modern context menu, the 'Open with Code' Explorer context menu entry disappears after a background auto-update and does not come back.

## Best solution

The repair logic in `build/win32/code.iss` (`ShouldRepairFolderContextMenu`) needs to be broadened to also handle the modern (AppX-based) Win11 context menu path during background updates — e.g. re-register the AppX package when it was previously installed but is now missing. The current fix only covers the legacy registry-based path when the classic context menu is forced.

- **fix scope of #300752:** Commit 20a1637cef5 (`fix: repair folder context menu when only legacy file context menu is present`) shipped in 1.112.0 but its `ShouldRepairFolderContextMenu()` function at build/win32/code.iss:1533-1545 requires `IsWindows10ContextMenuForced()`, which is false for modern Win11 users.
- **prior closed issue:** https://github.com/microsoft/vscode/issues/299964 was closed as verified with this fix, but the verification covered only the forced-classic scenario (per maintainer comment: 'This is specific to classic context menu for folders').

- **AppX install path:** build/win32/code.iss:108-109 installs the AppX when `ShouldUseWindows11ContextMenu()` is true, but during background updates the `addcontextmenufolders` task is not selected (defaults to unchecked), and there is no equivalent repair logic for the AppX-based context menu path.
