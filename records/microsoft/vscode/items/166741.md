# Issue #166741 — Sticky scroll messes up accurate markdown scroll sync

[https://github.com/microsoft/vscode/issues/166741](https://github.com/microsoft/vscode/issues/166741)

## Summary

Keep open as a valid backlog bug: the markdown preview scroll sync still depends on editor visible ranges/reveal behavior that sticky scroll can occlude, and there is no concrete fixing commit in current main.

## Change summary

Markdown preview-to-editor scroll sync becomes inaccurate when editor sticky scroll is enabled because the editor does not reveal the preview's top line until later.

## Best solution

The editor/sticky-scroll implementation should expose or account for occluded sticky-scroll height in visible range or reveal calculations so markdown preview sync can target the true top visible source line without extension-specific workarounds.

## Evidence

- **member analysis:** In https://github.com/microsoft/vscode/issues/166741#issuecomment-1329102097, alexdima says the markdown side is doing the correct thing and the editor core does not know sticky scroll is hiding lines because sticky scroll is implemented as an editor contribution.
- **current markdown code:** extensions/markdown-language-features/src/preview/topmostLineMonitor.ts still derives sync from editor.visibleRanges[0].start, so it has no explicit sticky-scroll occlusion adjustment.
- **current reveal code:** extensions/markdown-language-features/src/preview/scrolling.ts still calls editor.revealRange(..., vscode.TextEditorRevealType.AtTop) for preview-to-editor sync.
- **current editor code:** src/vs/editor/browser/viewParts/viewLines/viewLines.ts still factors editor.stickyScroll.maxLineCount into reveal padding, which matches the issue's suspected interaction with sticky scroll reveal behavior.
- **no fixing commit found:** Read-only git history searches for issue 166741 and relevant markdown/sticky-scroll files did not find a commit that clearly fixes this reported sync problem.
