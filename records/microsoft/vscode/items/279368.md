# Issue #279368 — GitHub Copilot chat tab doesn't honor `Alt+Left` "Back" to restore tab after close.

[https://github.com/microsoft/vscode/issues/279368](https://github.com/microsoft/vscode/issues/279368)

## Summary

Still reproduces on main. Chat editor inputs are removed from the navigation history stack on disposal because they lack `toUntyped()` and their URI scheme isn't in the 'durable' scheme list, so both Go Back and Reopen Closed Editor fail for chat tabs.

## Change summary

After closing a Copilot chat tab, Alt+Left (Go Back) does not restore it, unlike normal editor tabs.

## Best solution

Implement `toUntyped()` on `ChatEditorInput` (returning a resource-based untyped input) so that the navigation stack can store a durable, disposal-surviving entry. This would fix both Go Back (Alt+Left) and Reopen Closed Editor (Ctrl+Shift+T) for chat tabs. The fix belongs in `src/vs/workbench/contrib/chat/browser/widgetHosts/editor/chatEditorInput.ts`.

## Evidence

- **missing toUntyped():** `ChatEditorInput` at `src/vs/workbench/contrib/chat/browser/widgetHosts/editor/chatEditorInput.ts` does not override `toUntyped()` (base returns `undefined`). This means `handleEditorCloseEventInReopen()` at historyService.ts:690 bails out, so chat tabs never enter `recentlyClosedEditors`.
- **nav stack disposal removal:** In `EditorNavigationStack.addOrReplace()` at historyService.ts:1798, typed `EditorInput` entries get an `onEditorDispose` listener that calls `this.remove(editor)`. Because chat's URI scheme (`vscodeChatEditor`) isn't in the 'valid resource editor input scheme' list (historyService.ts:2098–2103), the entry stays as a typed input and is removed on dispose — so Go Back can't navigate to a closed chat.
