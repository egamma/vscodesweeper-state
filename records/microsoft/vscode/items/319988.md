# Issue #319988 — [Linux] Workbench system-ui / sans-serif (and explicitly-named non-Nerd fonts) render as an installed Nerd Font; terminal glyph fallback also broken — Electron 39

[https://github.com/microsoft/vscode/issues/319988](https://github.com/microsoft/vscode/issues/319988)

## Summary

Confirmed upstream Chromium/Electron font-resolution bug on Linux — Chromium ignores fontconfig aliases and resolves system-ui/sans-serif to broad-coverage installed fonts (Nerd Fonts, FreeMono, Adwaita Mono) instead of the correct sans-serif. Multiple commenters confirm it affects all Chromium-based apps, not just VS Code. Related to but broader than the existing #206202. Worth keeping open as an upstream tracking issue with proper labels.

## Change summary

On Linux with broad-coverage fonts installed (e.g. Nerd Fonts), the workbench UI and terminal fallback render in the wrong font because Chromium's font resolver disagrees with fontconfig.

## Best solution

Route to the linux/font-rendering area owners. The root cause is in Chromium's font matching on Linux (ignoring fontconfig aliases); VS Code could potentially mitigate by adding a `workbench.fontFamily` setting (tracked in https://github.com/microsoft/vscode/issues/519) or by adjusting the default font stack in `src/vs/base/browser/fonts.ts` to include more explicit Linux sans-serif families. Upstream Electron/Chromium bugs should be filed or linked. This is related to https://github.com/microsoft/vscode/issues/206202 but covers a wider symptom set (terminal fallback, Nerd Font hijacking, not just charset edits).

## Evidence

- **upstream confirmation:** Comment 6 by ioquatix confirms the issue affects all Chromium/Electron apps on Arch Linux, not just VS Code — changing the GNOME interface font to 'Noto Sans' fixes it across all apps.
- **fontconfig vs chromium:** fc-match resolves system-ui/sans-serif correctly (Noto Sans, Adwaita Sans), but Chromium's internal font matcher picks wrong fonts (Nerd Font, FreeMono, Adwaita Mono). The bug is in Chromium's font resolution layer.
- **multiple confirmations:** 5+ users across Fedora/Bluefin, Arch Linux, openSUSE Tumbleweed, KDE and GNOME, all on recent Electron/Chromium versions.
- **font stack code:** VS Code's Linux font stack is defined in src/vs/base/browser/fonts.ts as 'system-ui, "Ubuntu", "Droid Sans", sans-serif' — no changes to this file in recent history; the regression comes from Electron/Chromium updates.
- **related issue:** https://github.com/microsoft/vscode/issues/206202 tracks fontconfig charset edits not applying in VS Code (same root cause: Chromium ignoring fontconfig), labeled bug+upstream+linux+font-rendering.
- **related issues in context:** https://github.com/microsoft/vscode/issues/320369 (unable to use Consolas since last update) and https://github.com/microsoft/vscode/issues/322478 (visible space glyphs with Yutapon font since 1.123) appear to be the same underlying Chromium font resolution problem.
