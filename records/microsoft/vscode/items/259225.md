# Issue #259225 — Add leading --purpose arg to processes to more easily identify them in task manager / htop etc

[https://github.com/microsoft/vscode/issues/259225](https://github.com/microsoft/vscode/issues/259225)

## Summary

Legitimate feature request to improve process identifiability in system tools. VS Code already uses `--type=` for its own child processes (extensionHost, ptyHost, fileWatcher, agentHost), but Electron/Chromium-managed renderer and utility processes remain opaque. The `under-discussion` label is appropriate — the scope of what VS Code can control vs. what is upstream needs maintainer judgment.

## Change summary

Add a human-readable `--purpose=` argument to VS Code-spawned processes so users can identify them in htop / Task Manager.

## Best solution

For VS Code's own forked processes, a `--purpose=` or similar leading argument is straightforward — these already carry `--type=` (see `nodePtyHostStarter.ts`, `watcherClient.ts`, `nodeAgentHostStarter.ts`, `extensionHostConnection.ts`). For Electron-managed renderers and utility processes, VS Code would need Electron support or a creative workaround (e.g. setting `process.title`). A maintainer should decide how far to go and whether to file an upstream Electron request for renderer-level naming.
