# Issue #319129 — Copilot Sessions right-pane (Files) disappears immediately on first message when workspace root is a non-repo multi-root folder

[https://github.com/microsoft/vscode/issues/319129](https://github.com/microsoft/vscode/issues/319129)

## Summary

The bug is source-reproducible on main: when a Copilot Sessions session has no git repository, the provider falls back to `URI.parse('unknown:///')` as the workspace folder root (line 1289 of copilotChatSessionsProvider.ts), which causes ENOPRO errors when the file system tries to watch/read files at that URI, crashing the Files pane.

## Change summary

The right-side Files pane in Copilot Sessions disappears when the session's working directory is a non-git-repo folder in a multi-root workspace, due to an `unknown:///` fallback URI being used as the workspace root.

## Best solution

When `_extractRepositoryFromMetadata` returns no `repoUri`, the session workspace should either use the actual `file:///` working directory URI from the session metadata (not a synthetic `unknown:///`), or the Files pane rendering should gracefully handle a missing/unresolvable workspace folder without clearing itself. The fix likely touches `copilotChatSessionsProvider.ts` around line 1289 and possibly the `WorkspaceFolderManagementContribution`.

- **Fix prompt:**

  > The Copilot Sessions Files pane disappears because `copilotChatSessionsProvider.ts` line 1289 uses `URI.parse('unknown:///')` as a fallback workspace folder root when `_extractRepositoryFromMetadata` returns no `repoUri`. This `unknown:///` URI propagates to `WorkspaceFolderManagementContribution` which calls `workspaceEditingService.addFolders` with it, causing ENOPRO errors from the file system provider (no provider registered for `unknown:` scheme), which crashes the Files pane rendering. The fix should either: (1) use the session's actual `workingDirectory` file URI from metadata as the folder root when no git repo is detected (the metadata's `workingDirectory` key already has a valid `file:///` path), or (2) guard the workspace folder path in `WorkspaceFolderManagementContribution.getActiveSessionFolderData` to return `undefined` when the folder root scheme is `unknown`, preventing the bad URI from being added as a workspace folder. Do NOT change the `unknown:///` fallback behavior for the `gitRepository` field (that's used elsewhere for status display), only for the folder/workspace resolution path. Validate with the existing test infrastructure in copilotChatSessionsProvider.test.ts — add a case where a session with no `repositoryPath` but a valid `workingDirectory` in metadata produces a workspace with a `file:///` folder root, not `unknown:///`.
