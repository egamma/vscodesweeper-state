# Issue #202373 — Cursor position is incorrect in selection when cursorStyle is "Block"

[https://github.com/microsoft/vscode/issues/202373](https://github.com/microsoft/vscode/issues/202373)

## Summary

Keep open and route to the core editor area; current main still appears to render the block cursor from the selection active position without a special visual adjustment for non-empty left-to-right selections.

## Change summary

With `editor.cursorStyle: block`, a left-to-right selection can show the block cursor on the character after the selected text while the extension API active position describes the selection endpoint.

## Best solution

Have the editor owner decide whether block cursor rendering for non-empty selections should visually cover the last selected character while preserving `Selection.getPosition()` API semantics and existing right-to-left selection behavior.

## Evidence

- **current code:** `src/vs/editor/browser/viewParts/viewCursors/viewCursors.ts` renders cursor positions from `e.selections[i].getPosition()` in `onCursorStateChanged`, so the cursor follows `Selection.positionLineNumber/positionColumn`.
- **block cursor rendering:** `src/vs/editor/browser/viewParts/viewCursors/viewCursor.ts` computes the block cursor range as `Range(position.lineNumber, position.column, position.lineNumber, position.column + nextGrapheme.length)`, which paints the grapheme at the active position.
- **selection semantics:** `src/vs/editor/common/core/selection.ts` documents `getPosition()` as `positionLineNumber`/`positionColumn`; for a left-to-right range this is the range end, while the report expects the block cursor to cover the last selected character.
