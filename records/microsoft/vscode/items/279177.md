# Issue #279177 — Possible memory leak in debounce

[https://github.com/microsoft/vscode/issues/279177](https://github.com/microsoft/vscode/issues/279177)

## Summary

Still reproduces on main: the `debounce` decorator in `decorators.ts` stores a `setTimeout` handle and accumulated result on the instance but never clears them on disposal, so disposed `ChildProcessMonitor` instances (and any other `Disposable` using `@debounce`) are retained by the pending timer's closure. No fix has landed.

## Change summary

The `@debounce` decorator leaks references to disposed instances because it stores a timer and result on `this` without clearing them when the object is disposed.

## Best solution

The `debounce` (and `throttle`) decorator should integrate with the `Disposable` lifecycle: detect when the decorated class is a `Disposable`, register cleanup that clears the timer (`clearTimeout`) and nulls out the stored result on disposal. Alternatively, the decorator could be refactored to return an `IDisposable` or the specific call sites (like `ChildProcessMonitor`) could manually clear their debounce state. The fix is localized to `src/vs/base/common/decorators.ts` but touches a widely-used utility, so careful testing of all ~23 call sites is warranted.
