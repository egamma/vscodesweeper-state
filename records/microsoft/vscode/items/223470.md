# Issue #223470 — Scroll momentum transferred between different scrollables

[https://github.com/microsoft/vscode/issues/223470](https://github.com/microsoft/vscode/issues/223470)

## Summary

Still reproduces on main: VS Code's scrollable elements have no mechanism to 'claim' or stop momentum scroll events when the cursor moves between them. No fix has been committed for this cross-scrollable momentum transfer issue — it specifically affects Magic Mouse and similar devices that generate momentum events even after the finger lifts.

## Change summary

Vertical scroll momentum from the editor is transferred to the horizontal tab bar (or other scrollable elements) when the cursor moves while momentum events are still firing, causing unintended horizontal scrolling.

## Best solution

Implement momentum scroll locking in the scrollable element infrastructure: when a scroll gesture starts on one scrollable, suppress momentum-phase wheel events that arrive on a different scrollable. This could use the WheelEvent phase information (available in Electron/Chromium) or a time/velocity heuristic to detect momentum events crossing element boundaries. The fix belongs in `src/vs/base/browser/ui/scrollbar/scrollableElement.ts`.
