# Issue #286663 — repeated Remote Extension Host core dump 

[https://github.com/microsoft/vscode/issues/286663](https://github.com/microsoft/vscode/issues/286663)

## Summary

The crash is in libuv's `uv__io_poll` (bundled with Node.js), not in VS Code's own code — an upstream issue in Node.js/libuv's epoll handling in containerized Linux environments. Routing to the `remote` and `server` areas for visibility.

## Change summary

The Remote Extension Host process repeatedly crashes (SIGABRT) due to an assertion failure in libuv's epoll event-loop code when running in K8S containers on Oracle Linux kernels.

## Best solution

Track upstream in Node.js/libuv — the assertion failures are in `deps/uv/src/unix/linux.c` (`uv__io_poll`), where `epoll_ctl` returns unexpected error codes in certain container+kernel combinations. VS Code should consider bumping its server-side Node.js version once a libuv fix lands upstream, and could document `UV_USE_IO_URING=0` as a partial workaround in the meantime. The related libuv issue https://github.com/libuv/libuv/issues/5130 covers a similar abort-on-epoll_ctl race.

## Evidence

- **crash location:** All stack traces terminate in `uv__io_poll` at `deps/uv/src/unix/linux.c` (lines 1307 and 1430 in Node v22.21.1's bundled libuv), not in any VS Code JavaScript/TypeScript code.
- **assertion details:** The reporter identified two distinct assertion failures: one in io_uring_enter handling (line 1307) and one checking `assert(errno == EEXIST)` after `epoll_ctl` (line 1430). Both call `abort()` from within libuv.
- **environment-specific:** Triggered by K8S pods on Oracle Linux 8/9 with UEK kernel 5.15.0 and RHCK kernel 5.14.0; started after OL9 kernel upgrade in late Nov/Dec 2025.
- **multiple VS Code versions:** Reproduced across VS Code 1.108.2 through 1.122.0, confirming the issue is not a VS Code regression but an environment/upstream interaction.
- **related libuv issue:** https://github.com/libuv/libuv/issues/5130 (closed) covers a similar `abort()` on unexpected epoll_ctl errors racing with concurrent close() in libuv on Linux.
- **second reporter:** Comment by @arogozhnikov confirms the same crash pattern in containers, noting io_uring is not even permitted (EPERM), so the root cause is in the epoll fallback path.
