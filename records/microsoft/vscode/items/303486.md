# Issue #303486 — Vision not enabled

[https://github.com/microsoft/vscode/issues/303486](https://github.com/microsoft/vscode/issues/303486)

## Summary

Real bug with two layers: (1) the Copilot API incorrectly reports vision support for certain org/actor configurations, causing a 400 error, and (2) the Copilot extension doesn't gracefully handle that 400 and loops instead. Already assigned to justschen; a related copilot-api issue is open. The GA vision commit (f170fd1f23d, July 2026) may have changed the landscape, but the looping-on-error behavior for orgs without vision likely persists.

## Change summary

When vision is not enabled for an organization, Copilot sends vision requests that return a 400 error, then loops uncontrollably instead of gracefully degrading.

## Best solution

Two fixes needed: (1) upstream in copilot-api, the `/models` endpoint should accurately report `supports.vision` for all actor types (tracked in https://github.com/github/copilot-api/issues/34514); (2) in the extension, the 400 'vision is not enabled' error should be caught and handled gracefully — either strip images and retry once, or surface a clear error to the user instead of looping. The error-handling fix likely lives in `extensions/copilot/src/extension/prompt/node/chatMLFetcher.ts` around the vision request paths.
