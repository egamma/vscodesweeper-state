# Issue #273381 — Git - repository with directory junction does not mark changed files inside that junction as modified without reloading extensions

[https://github.com/microsoft/vscode/issues/273381](https://github.com/microsoft/vscode/issues/273381)

## Summary

Kept open: the issue carries the 'On Deck' milestone, which is a scheduling decision that defers to maintainer judgment. The bug describes a real gap — the file watcher (likely @parcel/watcher) does not follow NTFS directory junctions, so the Git extension's continuous file-change detection misses modifications inside them. A related but distinct issue (https://github.com/microsoft/vscode/issues/144780) tracks junction-based repository auto-detection; both are on 'On Deck'.

## Change summary

Git SCM decorations ('M' marker) do not appear for files modified inside an NTFS directory junction until VS Code is reloaded, because the file watcher does not traverse junctions.

## Best solution

The root cause is likely in the recursive file watcher (`src/vs/platform/files/node/watcher/parcel/parcelWatcher.ts`) not following directory junctions on Windows. A fix would ensure the watcher resolves or follows junctions so that changes inside them emit file-change events to the Git extension. This is related to but distinct from https://github.com/microsoft/vscode/issues/144780 (junction repo auto-detection); both may benefit from a unified junction-awareness pass in the file watcher layer.
