# Issue #117554 — When processing setBreakpoints response treat undefined source.path the same as undefined source

[https://github.com/microsoft/vscode/issues/117554](https://github.com/microsoft/vscode/issues/117554)

## Summary

Still present on main: an empty `source: {}` in a setBreakpoints response is not treated the same as no source, causing the breakpoint URI to resolve incorrectly. The check at debugModel.ts:1056 tests `this.data.source` truthiness but doesn't guard against an empty object with no `path`.

## Change summary

When a debug adapter responds with an empty `source: {}` (no path) in a setBreakpoints response, breakpoints display incorrectly because the empty source object is treated as a valid source rather than as absent.

## Best solution

Guard the `source` check in `Breakpoint.get uri()` (debugModel.ts ~line 1056) to also verify `this.data.source.path || this.data.source.sourceReference`, treating an empty source object the same as no source. The `toBreakpointSessionData` function could also normalize empty sources to `undefined` at the conversion boundary.
