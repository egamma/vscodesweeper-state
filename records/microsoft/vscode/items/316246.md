# Issue #316246 — Mouse drag stops working in tmux session when the terminal window is moved to a New Window

[https://github.com/microsoft/vscode/issues/316246](https://github.com/microsoft/vscode/issues/316246)

## Summary

Still reproduces on main: when a terminal is moved to an auxiliary window, xterm.js registers mouse drag/up listeners on the original document (`this._document`), which is never updated after a window move — so tmux mouse drag events in the new window are silently lost.

## Change summary

Mouse drag (e.g. tmux pane resize) stops working in the terminal after moving it to a New Window.

## Best solution

The root cause spans VS Code and xterm.js. In VS Code, `xtermTerminal.ts` hardcodes `documentOverride: layoutService.mainContainer.ownerDocument` at construction time (line 230). In xterm.js `CoreBrowserTerminal.ts`, `this._document` is set once during `open()` and never updated when the element moves to a new document (lines 402–414 update `_coreBrowserService.window` but not `_document`). The fix likely belongs in xterm.js — updating `_document` alongside `_coreBrowserService.window` when the element's ownerDocument changes — but VS Code may also need to stop hardcoding `documentOverride` to the main window. File an upstream issue on xtermjs/xterm.js for the `_document` staleness.

## Evidence

- **stale document reference:** In `node_modules/@xterm/xterm/src/browser/CoreBrowserTerminal.ts`, `this._document` is set at line 411–413 during initial `open()`. When the element moves to a new window (line 402–408), only `_coreBrowserService.window` is updated — `_document` stays stale.
- **mouse listeners on wrong document:** Lines 783–786 of `CoreBrowserTerminal.ts` register `mouseup`/`mousemove` (drag) listeners on `this._document!` — after a window move, these fire on the old (main) window's document, so mouse events in the auxiliary window are never captured.
- **hardcoded documentOverride:** `src/vs/workbench/contrib/terminal/browser/xterm/xtermTerminal.ts:230` sets `documentOverride: layoutService.mainContainer.ownerDocument`, permanently binding xterm to the main window's document.
- **reporter confirmation:** Reporter confirmed in comment 5 that docking the window back into the main VS Code window restores mouse functionality, consistent with the stale-document theory.
