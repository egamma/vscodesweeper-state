# Issue #307176 — TUI `Shift+Enter` causes SEND request instead of newline

[https://github.com/microsoft/vscode/issues/307176](https://github.com/microsoft/vscode/issues/307176)

## Summary

VS Code already supports the kitty keyboard protocol (which explicitly enables Shift+Enter differentiation), but more information is needed to determine whether the TUI app (`codex`) opts into the protocol or whether VS Code is mishandling the opt-in.

## Change summary

Pressing Shift+Enter in the `codex` TUI running inside VS Code's terminal sends the input instead of inserting a newline, unlike in Alacritty or iTerm2.

## Best solution

Determine whether `codex` opts into the kitty keyboard protocol via `CSI > flags u`. If it does and VS Code still sends `\r`, that's a real terminal-input bug in xterm.js or the keybinding layer. If it doesn't opt in, this is a codex-side issue — the existing `terminal.integrated.enableKittyKeyboardProtocol` setting (default `true`) already provides the infrastructure, and the app needs to request it.

## Evidence

- **kitty protocol setting:** src/vs/workbench/contrib/terminal/common/terminalConfiguration.ts registers `terminal.integrated.enableKittyKeyboardProtocol` with default `true` and description: 'This can, for example, enable Shift+Enter to be handled by the program.'
- **protocol is opt-in:** The kitty keyboard protocol requires the TUI app to send CSI > flags u to opt in; without opt-in, Shift+Enter and Enter both produce \r per VT standard.
- **related fix:** https://github.com/microsoft/vscode/issues/304765 fixed a bug where kitty sequences leaked to non-opted-in apps; verified and shipped in Insiders.
- **codex-side issue:** Reporter links https://github.com/openai/codex/issues/16445 — suggesting the problem may be that codex doesn't opt into the protocol.
