# Issue #140633 — Language detection is ignoring file extension for untitled files

[https://github.com/microsoft/vscode/issues/140633](https://github.com/microsoft/vscode/issues/140633)

## Summary

The described symptom (ML detection overriding the extension-derived language) is likely no longer reproducible on current main thanks to the `quickGuess` short-circuit in `detectLanguage`, but the underlying design gap remains: untitled files with a file extension in their URI path never set `hasLanguageSetExplicitly`, relying solely on `quickGuess` to prevent overrides.

## Change summary

ML-based language detection overrides the language correctly derived from the file extension of an untitled file's URI path.

## Best solution

The `UntitledTextEditorModel` should treat a language derived from a file extension in the resource URI as explicitly set (or at minimum mark it so auto-detection won't override it). The fix belongs in `src/vs/workbench/services/untitled/common/untitledTextEditorModel.ts` — after `createTextEditorModel` resolves the language from the URI path, set `_languageChangeSource` to prevent ML override. Alternatively, accept that `quickGuess` in `LanguageDetectionWorkerClient.detectLanguage` already prevents the symptom and close as effectively resolved.

## Evidence

- **quickGuess mechanism:** src/vs/workbench/services/languageDetection/browser/languageDetectionWorkerServiceImpl.ts line 275: `_guessLanguageIdByUri(resource)` short-circuits ML detection by returning the extension-based language for URIs with known extensions, including untitled scheme (falls into `default` case using `resource.path`).
- **hasLanguageSetExplicitly gap:** src/vs/workbench/common/editor/textEditorModel.ts line 142: `doAutoDetectLanguage` skips detection when `hasLanguageSetExplicitly` is true, but for untitled files created via URI with extension, this flag is never set because the language was applied at model creation time (no change event fires).
- **related fix:** Commit 7a82e67c8fe (Sep 2022) improved the `hasLanguageSetExplicitly` mechanism by listening for lower-level language changes, but doesn't cover the initial-creation case for untitled files.
- **URI parsing works:** src/vs/editor/common/services/languagesAssociations.ts line 155-156: untitled scheme falls into `default` case using `resource.path`, and `basename()` correctly extracts the filename with extension.
