# Issue #234437 — Explorerer becomes empty when `package.json` is presented

[https://github.com/microsoft/vscode/issues/234437](https://github.com/microsoft/vscode/issues/234437)

## Summary

Still reproduces on current main: a file nesting child pattern with more than one `*` (e.g. `sentry.*.config.*`) causes `SufTrie.add` to throw, which propagates up through `fetchChildren` and blanks the explorer. No fix has been committed.

## Change summary

Explorer becomes empty when a `fileNesting.patterns` child pattern contains more than one literal `*` wildcard.

## Best solution

Guard the `parentFinder.add` call in `ExplorerFileNestingTrie.nest()` (or the SufTrie itself) so that child-pattern results containing multiple `*` are silently skipped rather than throwing. A unit test with `sentry.*.config.*` as a child pattern should confirm it no longer throws.

## Evidence

- **root cause in source:** src/vs/workbench/contrib/files/common/explorerFileNestingTrie.ts line 171-172: `SufTrie.add` throws `Error('Unexpected star in SufTrie key')` when the last character of the key is `*`. A child pattern like `sentry.*.config.*` produces a literal `sentry.*.config.*` string (no `$(capture)` substitution), which is then added to the `parentFinder` PreTrie in `nest()` at line 71. The PreTrie consumes `sentry.` character-by-character, hits the first `*`, and delegates `*.config.*` to SufTrie. SufTrie processes from the tail — the last char is `*`, triggering the throw.
- **error propagation:** The throw at line 172 propagates through `nest()` → `fetchChildren()` (explorerModel.ts line 350). Since `fetchChildren` is async, the rejected promise results in an empty explorer view.
