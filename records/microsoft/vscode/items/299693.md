[https://github.com/microsoft/vscode/issues/299693](https://github.com/microsoft/vscode/issues/299693)

## Summary

This is a question about the release tagging process. The 1.110 tag is indeed missing the .0 suffix, but all subsequent releases (1.111.0 onward) returned to the standard three-part format. This is a one-off tagging irregularity, not a VS Code product issue.

## Change summary

The git tag for 1.110 was created as `1.110` instead of the usual `1.110.0` three-part semver format.

## Best solution

The anomaly was a one-time occurrence already corrected in subsequent releases. If automation depends on three-part tags, consumers can treat `1.110` as equivalent to `1.110.0`. Pushing an additional `1.110.0` tag pointing to the same commit would be the ideal resolution for downstream automation, but that's a release-engineering decision.
