# Issue #316888 — GitHub Copilot conversation compaction corrupts Markdown export

[https://github.com/microsoft/vscode/issues/316888](https://github.com/microsoft/vscode/issues/316888)

## Summary

Real bug: when conversation compaction occurs mid-response, the compaction notice (emitted as a `progressTask` or `markdownContent` part) is concatenated directly with the subsequent LLM response text in the Markdown export, because `partsToRepr()` treats these as non-block inline segments joined without any separator.

## Change summary

The 'Copy All' Markdown export concatenates the compaction notice text directly with the following response content (e.g. 'Compacted conversationYes, that was…') instead of separating them.

## Best solution

The compaction notice emitted at `extensions/copilot/src/extension/intents/node/agentIntent.ts` lines 807 and 936/944 should either use a `systemNotification` part (which `partsToRepr()` already treats as a block with `\n\n` separation) or append a trailing `\n\n` to the markdown text. The core `partsToRepr()` in `src/vs/workbench/contrib/chat/common/model/chatModel.ts` could also be hardened to insert a newline between `progressTask` parts and subsequent content. The `chat-summarization` area label applies.

## Evidence

- **root cause in partsToRepr():** In `src/vs/workbench/contrib/chat/common/model/chatModel.ts` lines 656–662, `markdownContent`, `progressTask`, and `progressTaskSerialized` are all treated as non-block segments (`{ text: part.content.value }` without `isBlock: true`). Non-block segments are joined with empty string at line 672 (`currentBlockSegments.join('')`), so 'Compacted conversation' + 'Yes, …' becomes 'Compacted conversationYes, …'.
- **compaction notice emission:** At `extensions/copilot/src/extension/intents/node/agentIntent.ts` line 807, the extension reports `new ChatResponseProgressPart2(l10n.t('Compacted conversation'), async () => l10n.t('Compacted conversation'))`. Because it has a task callback, the API converts it to a `progressTask` kind (via `extHostTypeConverters.ts` ChatTask.from). The same pattern at lines 936 and 944.
- **systemNotification is block-separated:** In contrast, `systemNotification` parts at line 629–630 are marked `isBlock: true`, and blocks are joined with `\n\n` at line 690. If the compaction notice used `systemNotification` instead of `progressTask`, it would be properly separated.
- **stringifyItem joins without prefix:** `stringifyItem()` in `src/vs/workbench/contrib/chat/browser/actions/chatActions.ts` line 1536 prepends the username but relies entirely on `response.toString()` for content separation. The concatenation bug is in the model layer, not the copy action.
- **no existing fix:** git log since 2026-05-01 on both the copilot extension and chat core shows no commit addressing this copy/export concatenation issue.
