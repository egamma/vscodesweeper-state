# Issue #316928 — Merge conflict dialog shows inverted perspective in deleted file scenario

[https://github.com/microsoft/vscode/issues/316928](https://github.com/microsoft/vscode/issues/316928)

## Summary

The dialog correctly reflects Git's own terminology. During a rebase, Git swaps 'ours' and 'theirs' — 'ours' is the branch being rebased onto (upstream), and 'theirs' is your local commits being replayed. VS Code faithfully surfaces Git's DELETED_BY_US/DELETED_BY_THEM status without reinterpretation.

## Change summary

User requests inverting 'ours/theirs' wording in the merge conflict dialog during rebase, believing VS Code shows it backwards.

## Best solution

Keep current behavior — VS Code mirrors Git's status output faithfully. Reinterpreting 'ours/theirs' based on the operation type (merge vs rebase) would be inconsistent with Git's own messaging and could confuse users who understand Git's semantics. The Git documentation explicitly states this inversion during rebase.

- **Proposed comment:**

  > Thanks for the detailed report. This is actually working as designed — it's a well-known aspect of Git's rebase semantics that can be counterintuitive.
  > 
  > During a **rebase**, Git internally swaps the meaning of "ours" and "theirs":
  > - **"ours"** = the branch you're rebasing *onto* (the upstream/remote)
  > - **"theirs"** = *your* commits being replayed on top
  > 
  > This happens because `git rebase` first checks out the target branch (making it "ours") and then cherry-picks your commits onto it (making them "theirs"). VS Code faithfully reflects Git's own `DELETED_BY_US` status without reinterpretation.
  > 
  > If VS Code tried to invert the wording based on whether you're merging vs. rebasing, it would diverge from what Git itself reports and could confuse users who are familiar with Git's terminology. The [Git rebase documentation](https://git-scm.com/docs/git-rebase) explains this inversion.
  > 
  > Closing as by-design, but I understand the confusion — it trips up many developers!
