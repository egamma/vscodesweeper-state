# Issue #183092 — Support 'textEdit.newText' in 'resolveSupport' of 'CompletionClientCapabilities'

[https://github.com/microsoft/vscode/issues/183092](https://github.com/microsoft/vscode/issues/183092)

## Summary

Team-authored feature request still under active discussion with no resolution. The current code warns but still allows insertText changes during resolve (marked '(fishy) async insertText' in extHostLanguageFeatures.ts). No design decision has been reached.

## Change summary

Allow language servers to declare `textEdit.newText` in `resolveSupport` so they can lazily compute expensive completion text without the current deprecation warning.

## Best solution

The team needs to decide whether to formally support lazy-resolving `textEdit.newText` (adding it to the declared `resolveSupport` properties) or commit to blocking it and require extensions to use `additionalTextEdits` or another mechanism. The telemetry data from PR #184994 should inform this decision. The relevant code is in `src/vs/workbench/api/common/extHostLanguageFeatures.ts` around line 1238–1259.
