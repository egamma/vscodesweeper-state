# Issue #319113 — Safety Report: AI Coding Extensions Lack Enforceable Guardrails — AWS Account Destroyed

[https://github.com/microsoft/vscode/issues/319113](https://github.com/microsoft/vscode/issues/319113)

## Summary

A broad safety report requesting platform-level enforcement guardrails for AI coding agents (hard gates, persistent violation state, blast radius limits, mandatory dry-run). The described incident stems from extension-level agent behavior (Terraform deployment to wrong AWS account), and the failed mechanisms (system prompts, workspace rules, MCP resources) are extension features, not VS Code core. However, some requested capabilities (session boundary enforcement, persistent state across relogins, authorization taxonomy) touch VS Code's platform-level agent security surface, which is an active area of investment. Keeping open for maintainer judgment on which parts, if any, belong in core vs. extensions.

## Change summary

Request for enforceable (non-prompt-based) guardrails in VS Code's agent platform: hard gates blocking destructive operations, persistent violation state surviving session resets, mandatory dry-run for infrastructure changes, and blast radius limits per conversational turn.

## Best solution

Route to the agent security area for triage. Some asks (persistent rule enforcement across session boundaries, approval taxonomy for destructive operations) overlap with the ongoing agent sandboxing and trust/consent work. Others (Terraform-specific dry-run, infrastructure blast radius) are extension-level concerns better addressed by the specific AI extension (e.g. GitHub Copilot). Related open issues like https://github.com/microsoft/vscode/issues/298393 (unauthorized file deletion) and https://github.com/microsoft/vscode/issues/324136 (granular terminal auto-approve) cover narrower slices of the same space. A maintainer should decide which concrete sub-requests are actionable at the platform level and whether to consolidate with existing tracking issues.
