# Issue #142208 — Dark Editor Mouse Pointer - Again

[https://github.com/microsoft/vscode/issues/142208](https://github.com/microsoft/vscode/issues/142208)

## Summary

Keep open as an under-discussion editor-rendering feature request rather than closing: the current native cursor behavior is intentional, but the remaining ask is for an opt-in setting to restore/customize the darker text pointer.

## Change summary

Add a setting or other supported path to choose the editor mouse text pointer appearance on dark themes.

## Best solution

Editor rendering maintainers should decide whether VS Code should expose a pointer-style override for editor text areas, or explicitly close this as designed if the project wants to rely only on OS-provided cursor accessibility settings.

## Evidence

- **maintainer context:** alexdima explained that VS Code intentionally stopped using the custom macOS text cursor because Catalina and newer provide a native bordered cursor and the custom cursor overrode user accessibility cursor settings; see https://github.com/microsoft/vscode/issues/142208#issuecomment-1030184230.
- **source history:** Commit 56718a15f025a59bfe9a25529212d7f8220e41dd removed the dark-theme macOS custom cursor CSS from src/vs/base/browser/ui/mouseCursor/mouseCursor.css to fix https://github.com/microsoft/vscode/issues/140490.
- **current code:** Current main still defines .monaco-mouse-cursor-text as plain cursor: text in src/vs/base/browser/ui/mouseCursor/mouseCursor.css, with editor surfaces applying that class from files such as src/vs/editor/browser/viewParts/viewLines/viewLines.ts and src/vs/editor/browser/controller/editContext/textArea/textAreaEditContext.ts.
- **discussion state:** The reporter accepted the rationale if VS Code is bound to the system cursor, but a later Ubuntu 22.04 comment reports a similar dark-theme pointer problem outside macOS, so the issue is not purely resolved by the macOS accessibility explanation.
