# Issue #325767 — Webview service-worker cache is not invalidated when an extension updates in place: webview permanently blank, survives reboot

[https://github.com/microsoft/vscode/issues/325767](https://github.com/microsoft/vscode/issues/325767)

## Summary

Well-documented bug report about the webview service-worker Cache API retaining stale resource entries after an extension updates in place, causing permanently blank webview panels that survive restarts and reboots. No existing issue tracks this specific cache-invalidation gap. Routing to the webview area for investigation.

## Change summary

When an extension auto-updates while VS Code is running, the webview service-worker cache on disk retains entries pointing to the now-deleted old extension directory, causing all asset loads to fail with net::ERR_FAILED and leaving the webview permanently blank.

## Best solution

The service worker's resource cache (`vscode-resource-cache-N` in `src/vs/workbench/contrib/webview/browser/pre/service-worker.js`) needs an invalidation or re-keying strategy that accounts for extension directory changes. Options include: keying cache entries by extension version so stale entries are never matched, evicting entries whose backing files no longer exist, or clearing the resource cache on extension update events. Related but distinct from https://github.com/microsoft/vscode/issues/320928 (unbounded cache growth). The webview area owner should investigate the interplay between `localResourceRoots`, the Cache API entries, and the extension lifecycle.

## Evidence

- **service worker cache architecture:** src/vs/workbench/contrib/webview/browser/pre/service-worker.js uses the Cache API (`caches.open('vscode-resource-cache-5')`) keyed by request URL. The VERSION constant (currently 5) controls the cache name but does not incorporate extension version — a stale entry for a deleted extension path remains valid until the SW VERSION bumps.
- **no cache invalidation on extension update:** No code in src/vs/workbench/contrib/webview/ clears or invalidates Cache API entries when an extension is updated or removed. The etag-based revalidation (ifNoneMatch) only works when the backing file still exists; if the old extension directory is deleted, the file service returns Failed, but the cached stale response may already have been matched.
- **related cache issue:** https://github.com/microsoft/vscode/issues/320928 tracks unbounded cache growth in the same cache — a related but distinct problem (growth vs. staleness).
- **cross-extension reports:** Reporter cites the same blank-webview symptom across unrelated extensions (anthropics/claude-code#13130, openai/codex#32388, #15975, #28280), supporting that this is a VS Code platform issue rather than extension-specific.
- **recent SW work:** Commit aca2ea52b7b (Apr 2026) improved service worker freshness on first reload after a VS Code update, but does not address extension-level cache invalidation.
