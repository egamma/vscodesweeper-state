## Summary

The request to show persistent inline breakpoint candidates (like Chrome DevTools) remains unaddressed on main. The fundamental blocker — candidate locations require an active debug adapter session — still exists in the code. Keeping open as a valid polish item in the Backlog.

## Change summary

Inline breakpoint decorations shift text when toggled and are only available during active debug sessions; request is for persistent, stable inline breakpoint hints like Chrome DevTools.

## Best solution

A maintainer decision is needed on whether to pursue this, given the technical constraint that candidate breakpoint locations come from the debug adapter and are unavailable outside a session. If pursued, a possible path is caching known candidate locations or using language-service heuristics, but that requires design input from the debug team.
