# Issue #293403 — GitHub Copilot breaks custom prompt link when editing

[https://github.com/microsoft/vscode/issues/293403](https://github.com/microsoft/vscode/issues/293403)

## Summary

Real bug in the chat widget's edit-previous-request flow for custom prompt references. When editing a previously submitted prompt that used `/foo`, the expanded instruction text (`Follow instructions in [foo.prompt.md](vscode-userdata:...)`) is placed back in the input instead of the original `/foo` syntax, breaking the prompt reference on resubmission. Related to a cluster of known issues around editing custom prompts (#281766 was closed as fixed but the reporter contends the custom-prompt aspect remains broken, and #317543 was filed in May 2026 against v1.121 referencing this same family of bugs). Routing to the chat area for investigation.

## Change summary

Editing a previously submitted chat prompt that uses a custom prompt (`/foo`) breaks the custom prompt link, replacing it with raw expanded markdown text the LLM cannot access.

## Best solution

The edit-previous-request flow in `src/vs/workbench/contrib/chat/browser/widget/chatWidget.ts` (`clickedRequest` method, around line 1790) should restore the original `/foo` syntax rather than the expanded prompt text. The `ChatRequestSlashPromptPart` stores the original `/name` form in its `text` and `name` properties — the reconstruction should use these to set the input value and re-register the prompt part as a dynamic variable so the parser recognizes it on resubmission. This may also require the Copilot extension side (which generates the `Follow instructions in ...` text) to not overwrite the stored request message. This bug is part of the same family as #317543 (open) and the reopened concerns on #281766.
