# Issue #200479 — macOS path bar "Open in Terminal" opens VSCode instead of Terminal

[https://github.com/microsoft/vscode/issues/200479](https://github.com/microsoft/vscode/issues/200479)

## Summary

This is a macOS Launch Services issue, not a VS Code bug. VS Code legitimately registers as a `public.folder` handler (line 229 of `build/lib/electron.ts`) so users can open folders in it, but macOS's path bar "Open in Terminal" action uses Launch Services to resolve the handler, and when VS Code becomes the default for `public.folder`, it intercepts that action. The same problem affects other apps (CotEditor filed an identical report). Keeping open as upstream/under-discussion since it's unclear whether VS Code could mitigate without losing folder-opening support.

## Change summary

macOS Finder path bar's "Open in Terminal" launches VS Code instead of Terminal.app when VS Code is the default handler for `public.folder`.

## Best solution

This is a macOS Launch Services issue. VS Code registers `public.folder` with role `Editor` in its bundle document types (`build/lib/electron.ts:229`), which is necessary for drag-and-drop and "Open With" folder support. The root cause is macOS resolving the path bar's "Open in Terminal" through Launch Services rather than directly invoking Terminal.app. The workaround (resetting default terminal via iTerm2 or `duti`) is the practical fix. VS Code could theoretically change the role from `Editor` to `Viewer` for `public.folder`, but the impact on other folder-opening flows would need investigation.
