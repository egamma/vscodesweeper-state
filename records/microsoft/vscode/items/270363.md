# Issue #270363 — hashChange event does not trigger in session when we initialize a git repo

[https://github.com/microsoft/vscode/issues/270363](https://github.com/microsoft/vscode/issues/270363)

## Summary

Team-authored bug with milestone 'On Deck'. The initial PR (d65085a01eb) improved the situation by reacting to new repository additions, but per the reporter's Nov 7 comment the first commit after `git init` still doesn't fire hashChange — only the second does. The issue remains valid and partially unfixed.

## Change summary

The `hashChange` telemetry event does not fire on the first commit after initializing a git repo in a session; it only fires starting from the second commit.

## Best solution

The `headCommitHashObs` observable in `scmAdapter.ts` derives from `historyProvider.historyItemRef.revision`. When a repo is freshly initialized and the first commit is made, the observable goes from `undefined` to a hash — but `runOnChange` may not trigger for this initial transition. The fix likely needs to handle the undefined→value transition in `editSourceTrackingImpl.ts` lines 95–102, or ensure the `runOnChange` helper fires for the first non-undefined value after the repo appears.
