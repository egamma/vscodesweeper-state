# Issue #118893 — Using key commands to scroll to top or bottom of document does not incorporate editor.paddingTop and editor.paddingButtom

[https://github.com/microsoft/vscode/issues/118893](https://github.com/microsoft/vscode/issues/118893)

## Summary

Keep open: the current cursor top/bottom command path still uses normal cursor reveal behavior, which can align the first or last line to the viewport edge rather than preserving configured editor padding. There is no concrete duplicate or fixing commit in the provided discussion.

## Change summary

Keyboard commands for moving to the start or end of the buffer should respect `editor.padding.top` and `editor.padding.bottom` the same way mouse scrolling does.

## Best solution

Handle `cursorTop`/`cursorBottom` in the editor command or reveal path so the final scroll position honors configured padding at document boundaries, likely around `src/vs/editor/browser/coreCommands.ts` and `src/vs/editor/browser/viewParts/viewLines/viewLines.ts`.

## Evidence

- **current labels:** The issue is already labeled `bug` and `editor-commands`, and has no milestone or hard keep-open blocker beyond needing normal maintainer triage.
- **command path:** `src/vs/editor/browser/coreCommands.ts` implements `cursorTop` and `cursorBottom` by moving the cursor to the buffer boundary and then calling `viewModel.revealAllCursors(args.source, true)`.
- **reveal behavior:** `src/vs/editor/common/viewModel/viewModelImpl.ts` routes `revealAllCursors` through `VerticalRevealType.Simple`; `src/vs/editor/browser/viewParts/viewLines/viewLines.ts` then computes minimum scrolling for the cursor box rather than a boundary-specific scroll position that preserves document padding.
- **padding model:** `src/vs/editor/common/viewLayout/linesLayout.ts` includes padding in line vertical offsets and total height, so a normal reveal of line 1 can produce a scrollTop equal to the top padding, making the first line sit at the top of the viewport.
