# Issue #238210 — Inconsistent line copy-paste on Chromebook

[https://github.com/microsoft/vscode/issues/238210](https://github.com/microsoft/vscode/issues/238210)

## Summary

Still reproducible on main: whole-line paste relies on `vscode-editor-data` custom MIME metadata surviving the clipboard round-trip. On ChromeOS (and possibly other Linux Wayland/web-based clipboard stacks), the custom MIME type is stripped, and the in-memory fallback can fail if the system clipboard modifies the text (e.g., newline normalization), causing the line to paste inline rather than on a new line.

## Change summary

Copying a full line (empty selection) and pasting inserts text inline instead of on a new line above the cursor, on ChromeOS and some Linux configurations.

## Best solution

Make the in-memory clipboard metadata fallback more resilient — e.g., normalize line endings before comparison in `InMemoryClipboardMetadataManager.get()`, or store a trimmed/canonical form of the copied text as the match key. The fix would be localized to `src/vs/editor/browser/controller/editContext/clipboardUtils.ts`.

## Evidence

- **mechanism:** Whole-line paste depends on `ClipboardStoredMetadata.isFromEmptySelection` being present. This metadata is stored as a custom `vscode-editor-data` MIME type and also cached in `InMemoryClipboardMetadataManager` (src/vs/editor/browser/controller/editContext/clipboardUtils.ts:79-100).
- **root cause:** ChromeOS/Linux may strip custom MIME types from the clipboard. The in-memory fallback at line 93 does an exact string comparison (`lastCopiedValue === pastedText`); if the system clipboard normalizes newlines (\r\n → \n or adds/strips trailing newline), the match fails and metadata is lost.
- **Firefox precedent:** A Firefox workaround already exists at line 26 (`textToCopy.replace(/\r\n/g, '\n')`) for exactly this class of newline mismatch, but only on the write side and only for Firefox.
- **no fix found:** git log on clipboardUtils.ts since 2024 shows no commit addressing this platform-specific metadata loss; the bug remains on current main.
- **confirmed by multiple users:** Comment from el1s7 (2025-06-19) confirms the issue persists on VS Code 1.100.3; tobiaschristensen also reported on macOS 15.1.
