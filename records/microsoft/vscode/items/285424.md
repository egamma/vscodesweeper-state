# Issue #285424 — VS Code shortcuts break after switching from Ukrainian to English layout when returning from Electron apps (Ubuntu 24.04)

[https://github.com/microsoft/vscode/issues/285424](https://github.com/microsoft/vscode/issues/285424)

## Summary

Legitimate Linux keyboard layout dispatch bug with multiple independent confirmations across distros (Ubuntu, Fedora). The `code` dispatch mode misidentifies key positions after switching layouts between Electron apps. Needs triage by the keybindings/keyboard-layout area owner — the root cause likely involves how Electron/native-keymap reports layout changes on focus.

## Change summary

On Linux with multiple keyboard layouts, Ctrl-based shortcuts use the wrong layout's key mapping after switching apps, making shortcuts like Ctrl+/ fire as Ctrl+. until another app is focused with the correct layout active.

## Best solution

Investigate whether the `native-keymap` module (or Electron's Chromium layer) fails to re-detect the active keyboard layout on window focus on Linux. The fix likely belongs in `src/vs/platform/keyboardLayout/electron-main/keyboardLayoutMainService.ts` or upstream in `native-keymap`. Issue https://github.com/microsoft/vscode/issues/287040 describes the identical symptom and should be consolidated with this one. Issue https://github.com/microsoft/vscode/issues/127932 (Backlog, keyboard-layout + linux + wayland) may share the same root cause.

## Evidence

- **multiple confirmations:** Four independent users confirmed the same Ctrl+/ → Ctrl+. symptom across Ubuntu 24.04, Ubuntu 25.10, and Fedora 43 with Ukrainian, Russian, and Portuguese layouts.
- **workaround confirms dispatch mode:** Switching `keyboard.dispatch` to `keyCode` resolves the issue (comment by digitr4ce), pointing to a defect in the default `code` dispatch path.
- **related issue:** https://github.com/microsoft/vscode/issues/287040 describes the identical symptom (Ctrl+/ → Ctrl+. on Linux with Russian layout) and cross-references this issue.
- **keyboard layout service:** Layout change detection lives in `src/vs/platform/keyboardLayout/electron-main/keyboardLayoutMainService.ts`, which delegates to `native-keymap.onDidChangeKeyboardLayout`. The bug suggests this event does not fire (or fires with stale data) when the app regains focus after a layout switch in another Electron app.
- **broader canonical:** https://github.com/microsoft/vscode/issues/127932 (Backlog, labels: bug, linux, keyboard-layout, wayland) tracks custom keyboard layout issues on Wayland and may share the same root cause.
