# Issue #316133 — Custom subagent model allowlist/order is ignored; subagents run on Claude Sonnet 4.6 even when Sonnet is not in the configured model list

[https://github.com/microsoft/vscode/issues/316133](https://github.com/microsoft/vscode/issues/316133)

## Summary

Credible bug report: the user's custom subagent model allowlist is bypassed and a non-listed model (Claude Sonnet 4.6) is selected, which is neither on the configured list nor the parent session's model. The core `resolveSubagentModel` logic in `runSubagentTool.ts` iterates the configured list correctly, but when no qualified-name match is found it falls back to the parent model — the fact that a third model appears suggests either a name-matching gap or a downstream override in the Copilot extension's automode/router layer. Needs investigation by the subagents team.

## Change summary

Custom subagent model resolution ignores the agent's configured model allowlist and selects a model not present in the list or the parent session.

## Best solution

Investigate two failure paths: (1) in `src/vs/workbench/contrib/chat/common/tools/builtinTools/runSubagentTool.ts`, verify that the qualified names written in `.agent.md` files (e.g. `GPT-5.5 (copilot)`) are correctly resolved by `lookupLanguageModelByQualifiedName` — a mismatch would cause the loop to silently fall through to the parent model; (2) downstream in the Copilot extension's automode/router (`extensions/copilot/src/platform/endpoint/node/automodeService.ts`), check whether the resolved model ID is being overridden after it leaves the core tool. The fix should ensure the runtime either honors the configured list or surfaces a clear error when no listed model is available.
