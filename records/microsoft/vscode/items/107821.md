# Issue #107821 — Git conflict resolution "accept both changes" deletes lines not related with the conflict

[https://github.com/microsoft/vscode/issues/107821](https://github.com/microsoft/vscode/issues/107821)

## Summary

Still reproduces on main: a race condition in the merge-conflict extension's CodeLens causes stale conflict ranges to be applied when the user clicks successive accept actions before the CodeLens refreshes, deleting unrelated lines. No fix has been committed.

## Change summary

Clicking merge conflict CodeLens actions (Accept Current/Incoming/Both) in quick succession causes stale line ranges to be applied, deleting lines unrelated to the conflict.

## Best solution

In `extensions/merge-conflict/src/commandHandler.ts`, when the command is invoked via CodeLens (`'known-conflict'` path), re-validate the conflict against the current document state before applying the edit — either re-scan to find the matching conflict at current line positions, or verify the expected conflict markers (<<<<<<, =======, >>>>>>) still exist at the stored range. The `applied` flag in `documentMergeConflict.ts` already prevents double-application of the same object but does not guard against stale sibling conflicts.

## Evidence

- **stale CodeLens args:** In `extensions/merge-conflict/src/codelensProvider.ts` lines 63–94, conflict objects (with their ranges) are captured at CodeLens creation time and passed as command arguments (`'known-conflict', conflict`). After a prior accept edits the document, these ranges become stale.
- **no re-validation on accept:** In `extensions/merge-conflict/src/commandHandler.ts` lines 228–256, the `accept()` method takes the stale conflict directly from args (line 234) and calls `conflict.commitEdit()` without re-scanning the document.
- **applied flag is per-object only:** In `extensions/merge-conflict/src/documentMergeConflict.ts` lines 57–60, the `applied` flag prevents the same conflict object from being applied twice, but does not protect against a different stale conflict object whose ranges shifted.
- **multiple confirmations:** Comments by sbuser, zxch3n (with GIF), Brocklobsta, porceCodes, Gruensil, connorjclark, and dmsteck all confirm the race condition across versions 1.48–1.85+.
- **workaround identified:** Comment by Brocklobsta: disabling `merge-conflict.codeLens.enabled` avoids the issue (keyboard shortcuts use cursor-based lookup instead of stale CodeLens args). Also resolving conflicts bottom-to-top avoids line-shift issues.
- **no fix in git history:** git log on `extensions/merge-conflict/src/commandHandler.ts` and `codelensProvider.ts` shows no commits addressing this race condition; the last substantive change was #168598 (CodeLens flashing fix) in 2022.
- **related issue #191038:** https://github.com/microsoft/vscode/issues/191038 was closed as a duplicate of this issue, confirming it is the canonical tracker.
- **related issue #123721:** https://github.com/microsoft/vscode/issues/123721 is a separate open bug about empty-line handling in accept, rooted in the `isNewlineOnly` check in `documentMergeConflict.ts` lines 90–93.
