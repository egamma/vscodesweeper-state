# Issue #325711 — [Bug] Claude agent SDK crashes with "Error during execution" in Copilot Chat (VS Code 1.127.0, Electron 42 / Node 24)

[https://github.com/microsoft/vscode/issues/325711](https://github.com/microsoft/vscode/issues/325711)

## Summary

The Claude agent SDK path crashes because the customization harness tries to read agent definitions via `claude-code:/agents/...` URIs before (or without) a FileSystemProvider being registered for the `claude-code:` scheme. The bug is in the built-in Copilot extension's provider registration or in the harness not guarding against missing providers. Routing to the claude-agent / copilot area owners.

## Change summary

Using Claude agent delegation in Copilot Chat crashes with ENOPRO because no FileSystemProvider is registered for the `claude-code:` URI scheme when the customization harness reads agent definitions.

## Best solution

Either the Copilot extension needs to ensure the `claude-code:` FileSystemProvider is registered before the customization harness reads agent definitions, or `customizationHarnessService.getCustomAgents` / `promptsServiceImpl.parseNew` should gracefully handle missing providers instead of letting the ENOPRO error propagate as a crash. The fix likely lives in `extensions/copilot/src/extension/chatSessions/claude/` (provider registration) and/or `src/vs/workbench/contrib/chat/browser/aiCustomization/customizationHarnessService.ts` (resilient error handling).

- **scheme definition:** The `claude-code` scheme is defined in extensions/copilot/src/extension/chatSessions/claude/common/claudeSessionUri.ts:9 (`export const scheme = 'claude-code'`).
- **code path on main:** src/vs/workbench/contrib/chat/common/customizationHarnessService.ts:568-577 shows getCustomAgents falls through to promptsService when no harness itemProvider is found, which then tries to read the URI without verifying the scheme's provider exists.
