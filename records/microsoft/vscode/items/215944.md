# Issue #215944 — Breakpoint classes have no property for triggered breakpoints

[https://github.com/microsoft/vscode/issues/215944](https://github.com/microsoft/vscode/issues/215944)

## Summary

Valid API gap: triggered breakpoints have a `triggeredBy` property in the internal debug model (`IBreakpoint`, `IFunctionBreakpoint`, `IDataBreakpoint` in `src/vs/workbench/contrib/debug/common/debug.ts`) but it is never surfaced on the extension API `Breakpoint` classes in `src/vscode-dts/vscode.d.ts` or `src/vs/workbench/api/common/extHostTypes.ts`. The request for API parity is reasonable; keeping open for the debug/API area owners.

## Change summary

Expose the `triggeredBy` property on the extension API `Breakpoint` class so extensions can distinguish triggered breakpoints from unconditional ones.

## Best solution

Add a `triggeredBy` (or `waitFor`) readonly property to the `Breakpoint` base class in the extension API (`src/vscode-dts/vscode.d.ts`), wire it through the DTO layer in `extHostDebugService.ts` and `extHost.protocol.ts`, and include it in the `$acceptBreakpointsDelta` delta flow. This is consistent with how `condition`, `hitCondition`, and `logMessage` are already exposed.
