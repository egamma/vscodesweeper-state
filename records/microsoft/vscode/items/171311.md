# Issue #171311 — vscode.TextEditor.edit breaks undo/redo history

[https://github.com/microsoft/vscode/issues/171311](https://github.com/microsoft/vscode/issues/171311)

## Summary

Keep open: this is an extension API undo/redo behavior report with a linked older repro, and current main has coverage for basic undo-stop handling but no concrete fixing commit or reporter confirmation for the reported extension workflow.

## Change summary

Extension authors need `vscode.TextEditor.edit` with `undoStopBefore` and `undoStopAfter` to preserve intended undo/redo grouping.

## Best solution

Confirm the linked `https://github.com/microsoft/vscode/issues/38535` scenario against current main and add or update an API test that covers the interleaved extension-edit workflow; close as completed only if a concrete fix commit can be cited.

## Evidence

- **linked prior report:** The issue explicitly says it is the same problem as `https://github.com/microsoft/vscode/issues/38535`, so the central claim is about the existing `TextEditor.edit` undo-stop API behavior rather than a new feature request.
- **current implementation:** `src/vs/workbench/api/common/extHostTextEditor.ts` preserves `undoStopBefore` and `undoStopAfter` in `IEditData` and forwards them through `$tryApplyEdits`; `src/vs/workbench/api/browser/mainThreadEditor.ts` calls `pushUndoStop()` before and after `executeEdits()` when those options are true.
- **current tests:** `src/vs/workbench/api/test/browser/mainThreadEditors.test.ts` contains tests named `applyWorkspaceEdit can control undo/redo stack 1/2` that assert two direct `$tryApplyEdits` calls can merge when both undo stops are false and split when `undoStopBefore` is true.
- **no close-grade fix proof:** The undo/redo tests were added in commit `65b22997d7addf8d34400207bc8a3d7c4cf07f70`, but that commit is a test conversion/coverage change, not a clear fixing commit for this issue's reported VS Code 1.73 extension scenario.
