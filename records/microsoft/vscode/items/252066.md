# Issue #252066 — Environment Variables are not inherited when running with Ozone platform (for Wayland)

[https://github.com/microsoft/vscode/issues/252066](https://github.com/microsoft/vscode/issues/252066)

## Summary

Team members (connor4312, deepak1556) are actively engaged and have requested logs. The bug is partially improved per comment 13 (1.103.1/Electron update fixed LC_CTYPE) but SSH key inheritance still fails when not launched from terminal with Ozone flags. Keeping open for continued investigation.

## Change summary

Environment variables (PATH for fnm/node, LC_CTYPE, SSH_AUTH_SOCK) are not inherited by VS Code's main process when launched with Ozone/Wayland flags from a desktop launcher, while the integrated terminal resolves them correctly.

## Best solution

Investigate whether the Ozone platform flag changes how Electron/Chromium inherits the process environment from the desktop session (e.g. DBus activation vs direct exec), and whether `doResolveUnixShellEnv` in `src/vs/platform/shell/node/shellEnv.ts` correctly resolves all variables in that launch context. The partial fix in Electron 36 (1.103.1) suggests this may be partly upstream — confirm whether remaining SSH_AUTH_SOCK issues are a VS Code shell-env-resolution gap or an Electron/Chromium Ozone-backend issue.
