# Issue #234565 — Git commands do not work if the profile folder name does not contain ASCI characters

[https://github.com/microsoft/vscode/issues/234565](https://github.com/microsoft/vscode/issues/234565)

## Summary

Still reproduces on main: the git extension forces LC_ALL=en_US.UTF-8 and LANG=en_US.UTF-8 when spawning git processes (git.ts:692-693), which causes Git for Windows (MSYS2) SSH to misinterpret non-ASCII characters in the Windows user profile path. The bug is valid and the problematic code path remains unchanged.

## Change summary

Git push/pull/clone via SSH fails when the Windows user profile folder contains non-ASCII characters (e.g. Cyrillic) because VS Code's forced UTF-8 locale makes Git for Windows SSH misinterpret the HOME path.

## Best solution

The forced locale in `extensions/git/src/git.ts` spawn() is needed for parsing git's English output, but it breaks HOME path resolution on non-UTF-8 Windows systems. A fix would need to either use `-c` git config for language instead of environment variables, or selectively avoid overriding locale on Windows when the HOME path contains non-ASCII characters. Related issue https://github.com/microsoft/vscode/issues/189924 reports the same root cause (forced LC_ALL) in dev-containers — both should be addressed together.

## Evidence

- **source trace:** extensions/git/src/git.ts line 689-694: spawn() forces LANGUAGE=en, LC_ALL=en_US.UTF-8, LANG=en_US.UTF-8 via process.env override. This is still present on current main.
- **root cause:** Git for Windows uses MSYS2 which translates Windows paths to POSIX paths. When LC_ALL is forced to en_US.UTF-8 but the Windows HOME path bytes are in a legacy codepage (e.g. CP1251 for Cyrillic), MSYS2 misinterprets the path, producing the garbled /c/Users/\335\344\343\340\360 seen in the error.
- **related issue:** https://github.com/microsoft/vscode/issues/189924 reports the same forced-locale problem in dev-containers; both share the root cause of unconditionally overriding locale variables.
- **no fix found:** git log on extensions/git/src/git.ts shows no changes to the locale-forcing code since the issue was filed.
