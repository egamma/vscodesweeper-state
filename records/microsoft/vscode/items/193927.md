# Issue #193927 — Repeated searches leaves orphaned, unkillable rg processes

[https://github.com/microsoft/vscode/issues/193927](https://github.com/microsoft/vscode/issues/193927)

## Summary

The orphaned unkillable rg process issue remains open with no concrete fix on main. Multiple reporters confirm it across platforms with different triggers (network drives on PATH, large workspaces, specific extensions). The reporter's deep investigation traced one root cause to ripgrep's compression binary search on network-mounted paths (linked to upstream BurntSushi/ripgrep#2111). The `search.ripgrep.maxThreads` setting is a workaround, not a fix. Keeping open as it represents a real, multi-faceted process lifecycle issue.

## Change summary

Repeated VS Code searches can leave orphaned, unkillable rg (ripgrep) processes that survive workspace/window close and cannot be terminated without a reboot.

## Best solution

Investigate process lifecycle management in `src/vs/workbench/services/search/node/ripgrepTextSearchEngine.ts` and `ripgrepFileSearch.ts` — the `cancel()` path calls `rgProc.kill()` but there may be cases where spawned rg processes become detached (especially when rapidly cancelled and respawned). Consider adding process group kills, timeouts on process cleanup, or tracking spawned PIDs for forced cleanup on window close. The upstream ripgrep compression-binary search issue (BurntSushi/ripgrep#2111) is a contributing factor for users with network drives on PATH.
