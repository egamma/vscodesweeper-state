# Issue #193927 — Repeated searches leaves orphaned, unkillable rg processes

[https://github.com/microsoft/vscode/issues/193927](https://github.com/microsoft/vscode/issues/193927)

## Summary

The core issue — orphaned rg processes surviving VS Code shutdown — is still present on main. The `search.ripgrep.maxThreads` setting reduces occurrence but doesn't fix the underlying race condition in process lifecycle management. Multiple independent users confirm the problem across Windows and macOS, with contributing factors including network drives on PATH, rapid successive searches, and large workspaces. Keeping open as a real, community-validated bug in the search infrastructure.

## Change summary

Repeated searches or rapid search invocations can leave orphaned, unkillable ripgrep (rg) processes that persist after VS Code closes, consuming CPU indefinitely.

## Best solution

Harden the ripgrep process lifecycle in `src/vs/workbench/services/search/node/ripgrepTextSearchEngine.ts` and `ripgrepFileSearch.ts`: ensure cancellation of a prior search kills the spawned rg process before spawning a new one (debounce or serialize spawns), and register a process-exit handler that force-kills all child rg processes. The `maxThreads` workaround helps but doesn't address the orphaning race. Consider also passing `--no-search-zip` to rg to avoid the decompression-binary PATH probing that Comment 11 identified as an amplifying factor (see https://github.com/BurntSushi/ripgrep/issues/2111).

## Evidence

- **process kill code:** src/vs/workbench/services/search/node/ripgrepTextSearchEngine.ts lines 94-100 call `rgProc?.kill()` on cancellation, but rapid successive searches can race: a new search spawns before the prior rg process is fully killed, and if the CancellationToken fires after `rgProc` is reassigned or nulled, the old process leaks.
- **maxThreads workaround:** Commit 00369907846 added `search.ripgrep.maxThreads` setting, which reduces concurrency and masks the issue but doesn't fix the orphaning race condition.
- **cross-platform confirmation:** Multiple independent users confirm on Windows (Comments 1, 6, 13, 14, 19, 24) and macOS (Comments 7, 10, 20), ruling out a single-platform fluke.
- **ripgrep upstream factor:** Comment 11 traces an amplifying factor to ripgrep probing PATH for decompression binaries (https://github.com/BurntSushi/ripgrep/issues/2111), which stalls on network drives — but the orphaning itself is a VS Code process-management issue.
- **community interest:** 9 upvotes and active discussion from 10+ distinct users over 2 years indicates ongoing impact.
