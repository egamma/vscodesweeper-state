# Issue #318893 — removeDangerousEnvVariables gap - Strongest argument. Every other spawn path in VS Code strips NODE_OPTIONS/LD_PRELOAD/DEBUG. MCP is the sole omission. This is objectively a bug, not a design decision.

[https://github.com/microsoft/vscode/issues/318893](https://github.com/microsoft/vscode/issues/318893)

## Summary

The issue is already labeled `*as-designed` by a maintainer — the MCP spawn path intentionally passes through user-specified env vars since MCP servers are user-configured external processes in trusted workspaces, unlike VS Code's internal infrastructure spawns. Keeping open only because a maintainer should confirm the label sticks given the security framing.

## Change summary

MCP server spawn path does not call `removeDangerousEnvVariables()` before spawning child processes, unlike extension host and utility process paths.

## Best solution

The `*as-designed` label already applied reflects the design difference: MCP servers are user-authored trusted-workspace processes where the user explicitly controls the `env` block. If the team reconsiders, a targeted call to `removeDangerousEnvVariables()` at line ~83 in `extHostMcpNode.ts` (after env assembly, before spawn) would close the gap — but stripping `NODE_OPTIONS` would break legitimate MCP debugging workflows.
