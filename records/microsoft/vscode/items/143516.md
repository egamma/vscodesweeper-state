# Issue #143516 — Git - $GIT_INDEX_FILE is set after newly starting VS Code as a git editor

[https://github.com/microsoft/vscode/issues/143516](https://github.com/microsoft/vscode/issues/143516)

## Summary

Still reproduces on main: when VS Code is launched as a git editor, `GIT_INDEX_FILE` from the parent git process is inherited into `process.env` and never stripped. Since CLI-launched instances skip shell-env resolution (line 49 of shellEnv.ts), the polluted env persists into terminals and the built-in git extension.

## Change summary

When VS Code is started as a git editor (`code -nw`), the `GIT_INDEX_FILE` env var leaks into VS Code's environment, causing the git extension to show incorrect status (e.g. files appear untracked).

## Best solution

Strip git-internal environment variables (`GIT_INDEX_FILE`, `GIT_DIR`, `GIT_WORK_TREE`, `GIT_AUTHOR_*`, `GIT_COMMITTER_*`, `GIT_PREFIX`, etc.) from the inherited process environment early in the main process startup or when constructing `userEnv`. The fix likely belongs in `src/vs/code/electron-main/app.ts` or `src/vs/platform/shell/node/shellEnv.ts`, similar to how `XDG_RUNTIME_DIR` and `ELECTRON_*` vars are already stripped.

- **CLI skip path:** shellEnv.ts line 49: when launched from CLI (VSCODE_CLI set), shell env resolution is skipped entirely, so the inherited process.env (including GIT_INDEX_FILE) passes through unmodified.
