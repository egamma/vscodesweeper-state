# Issue #227452 — [scss] inbuild sass/scss extension shows error that is none

[https://github.com/microsoft/vscode/issues/227452](https://github.com/microsoft/vscode/issues/227452)

## Summary

Still reproduces on current main: the SCSS parser in vscode-css-languageservice parses mixin bodies with `_parseRuleSetDeclaration`, which cannot handle keyframe selectors like `0% { ... }` inside a mixin definition. The `@include` at the keyframe level is supported, but the mixin body content is not context-aware.

## Change summary

SCSS extension produces a false `css-rcurlyexpected` error when a mixin body contains keyframe-style percentage selectors (`0%`, `100%`) that are intended for use inside `@keyframes`.

## Best solution

The fix belongs in microsoft/vscode-css-languageservice's SCSS parser — `_parseMixinDeclaration` should allow keyframe selector syntax in mixin bodies (or suppress validation errors for ambiguous mixin content). Alternatively, the mixin body parser (`_parseRuleSetDeclaration`) could be extended to try `_tryParseKeyframeSelector()` as a fallback, similar to how `_parseMixinReferenceBodyStatement` already does.

## Evidence

- **source trace:** In `extensions/css-language-features/server/node_modules/vscode-css-languageservice/lib/esm/parser/scssParser.js`, `_parseMixinDeclaration()` (line 482) parses the mixin body with `_parseRuleSetDeclaration` which does not handle percentage keyframe selectors like `0% { ... }`.
- **parser limitation:** `_parseRuleSetDeclaration` falls through to `_tryParseRuleset(true)` then `_parseDeclaration()` — neither handles `0% { transform: ... }` as valid syntax outside a `@keyframes` context, producing `RightCurlyExpected` error.
- **contrast with include body:** The parser already has `_parseMixinReferenceBodyStatement()` (line 617) which tries `_tryParseKeyframeSelector()` for `@include ... { }` content blocks — but the mixin *definition* body does not have this fallback.
- **upstream dependency:** The fix needs to be in `microsoft/vscode-css-languageservice` (currently pinned at `^7.0.0-next.1` in the CSS language features server).
