# Issue #319211 — On m5 Macbook Pro, will not start from shell

[https://github.com/microsoft/vscode/issues/319211](https://github.com/microsoft/vscode/issues/319211)

## Summary

The user symlinked the VS Code Electron binary directly to ~/.local/bin, which breaks Electron's helper-app bundle resolution on macOS. The supported way to launch VS Code from the terminal is via the shell script wrapper installed by the 'Install code command in PATH' command palette action, which correctly resolves the .app bundle path.

## Change summary

VS Code crashes on macOS when launched from a symlink to the Electron binary, because Electron cannot find its helper apps outside the .app bundle context.

## Best solution

Direct the user to use the supported CLI launcher: Command Palette → 'Shell Command: Install code command in PATH', which installs a proper shell script (resources/darwin/bin/code.sh) that resolves symlinks back to the .app bundle before launching. Symlinking the raw binary is not supported because Electron requires its helper apps to be co-located in the bundle.

  > Thanks for the report. The `Unable to find helper app` crash happens because you've symlinked the VS Code Electron binary directly to `~/.local/bin`. Electron requires its helper apps (GPU process, renderer, network service) to be co-located inside the `.app` bundle — a bare symlink to the binary breaks that resolution.
  > 
  > The supported way to get a `code` command on macOS is:
  > 
  > 1. Open VS Code
  > 2. Open the Command Palette (⇧⌘P)
  > 3. Run **Shell Command: Install 'code' command in PATH**
  > 
  > This installs a shell script wrapper that correctly resolves symlinks back to the `.app` bundle before launching. If you need the command in `~/.local/bin` specifically, you can symlink the *script* it installs (typically at `/usr/local/bin/code`) rather than the Electron binary itself.
  > 
  > The fact that it worked before may be due to a change in Electron's helper-app lookup between VS Code versions. Closing as a support question — please reopen if the official shell command doesn't resolve it.

## Evidence

- **root cause:** The FATAL error 'Unable to find helper app' at electron/shell/app/electron_main_delegate_mac.mm:72 occurs because the user symlinked the Electron binary directly — Electron's GPU, renderer, and network helper processes must be found relative to the .app bundle, which a bare binary symlink breaks.
- **supported CLI script:** resources/darwin/bin/code.sh is the official macOS CLI launcher. It resolves symlinks back to the .app bundle via app_realpath() (lines 15–24) and runs Electron with ELECTRON_RUN_AS_NODE=1, avoiding the helper-app resolution problem entirely.
- **user confirmation:** The user states 'The Code executable is symlinked to my ~/.local/bin directory' — this is the unsupported path. Launching from Dock works because macOS uses the full bundle.
