# Issue #325845 — Agent host: corrupt (7-byte) pasted-image attachment is persisted without validation and reproducibly crashes the renderer (SIGTRAP in image decode) on session open

[https://github.com/microsoft/vscode/issues/325845](https://github.com/microsoft/vscode/issues/325845)

## Summary

Confirmed in source: the agent host attachment write path (`_writeAndRewrite`) persists bytes with no size or magic-byte validation, and the render path feeds those bytes to Chromium's image decoder, which can SIGTRAP on corrupt data. Both gaps are present on current main. The Chromium-level crash (Rust image decoder panic) is partially upstream, but VS Code's write-path validation gap and the render-path's unguarded image decode are its own defects.

## Change summary

Corrupt or truncated pasted-image attachments are persisted without validation and crash the renderer on session open.

## Best solution

Two-layer fix: (1) in `src/vs/platform/agentHost/node/agentService.ts` `_writeAndRewrite`, validate that bytes are non-empty and plausibly large enough for the declared content type before persisting; (2) in `src/vs/workbench/contrib/chat/browser/chatImageUtils.ts` and the `renderPreviewImage` flow in `chatAttachmentWidgets.ts`, wrap the image-decode paths in a try/catch and surface a broken-image placeholder instead of feeding potentially corrupt bytes to Chromium's native decoder. The upstream Chromium SIGTRAP should also be filed, but VS Code should not depend on Chromium handling corrupt images gracefully.
