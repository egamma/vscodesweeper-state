# Issue #265301 — When WSL is closed / crashes, all open instances of VSCode have to be closed completely - reload window and attempting to reconnect fails

[https://github.com/microsoft/vscode/issues/265301](https://github.com/microsoft/vscode/issues/265301)

## Summary

Still valid on main: when WSL is force-killed, VS Code's remote connection enters a static permanent-failure state that persists across reload attempts. The reconnection infrastructure cannot recover because the remote server process no longer exists and a fresh server launch isn't triggered on reload.

## Change summary

After WSL crashes, VS Code's reconnect/reload options fail to restore the remote session — only a full restart or DevTools refresh works.

## Best solution

On reload after a permanent connection failure to a WSL remote, VS Code should be able to spawn a fresh remote server instance rather than attempting to reconnect to the dead one. The fix likely spans the remote connection layer (`src/vs/platform/remote/common/remoteAgentConnection.ts` permanent-failure handling) and the WSL extension's authority resolver. The reconnection grace time feature (commit 6cc2564bf9d) helps with transient drops but doesn't address full WSL process death.
