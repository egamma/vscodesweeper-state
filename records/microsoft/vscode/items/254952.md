# Issue #254952 — Slow calls to getRepositoryFetchUrls for larger repos for codebase search

[https://github.com/microsoft/vscode/issues/254952](https://github.com/microsoft/vscode/issues/254952)

## Summary

Milestone 'On Deck' is a scheduling decision, so deferring to maintainer judgment. Multiple commits have already landed targeting this exact issue, substantially addressing the original report — a maintainer should evaluate whether the remaining work justifies keeping this open.

## Change summary

In repos with 100k+ indexable files, per-file calls to getRepositoryFetchUrls (via isCopilotIgnored) are too slow because each call spawns a process to find the repository root.

## Best solution

Evaluate whether the three landed fixes (parallel isIgnored checks, skipping expensive RepoContextImpl creation, reducing redundant repository URL fetches) have resolved the issue for large repos. If so, close as completed. If edge cases remain (e.g. repos not yet discovered by the git extension hitting the getRepositoryRoot fallback path), the remaining slow path is in `extensions/copilot/src/platform/git/vscode-node/gitServiceImpl.ts` lines 213–239 where it falls back to spawning git to find the root — caching or batch-resolving repository roots for sibling files (as lszomoru suggested) would address the residual cost.
