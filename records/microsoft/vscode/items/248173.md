# Issue #248173 — F4 breaks in search view after deleting result

[https://github.com/microsoft/vscode/issues/248173](https://github.com/microsoft/vscode/issues/248173)

## Summary

Team-authored bug that is still reproducible on main. PR #248565 only patched the delete action's focus restoration path (catching errors in `getLastNodeFromSameType`), but the real problem — F4 (`selectNextMatch`) operating on a stale/deleted tree element — remains unguarded. The reporter re-confirmed the bug in August 2025.

## Change summary

Pressing F4 (Focus Next Search Result) after deleting a search result throws an error because `selectNextMatch` navigates from the stale deleted element.

## Best solution

Guard `selectNextMatch` (and `selectPreviousMatch`) in `src/vs/workbench/contrib/search/browser/searchView.ts` against stale/deleted selections — e.g. fall back to navigating from the tree root when `this.tree.navigate(selected)` or `this.tree.isCollapsed(selected)` throws because the selected element is no longer in the tree. Alternatively, ensure the delete action in `searchActionsRemoveReplace.ts` clears the tree selection when the focused element is removed so F4 starts from a clean state.
