# Issue #137256 — Incorrect URI casing passed to extensions on Windows after file rename

[https://github.com/microsoft/vscode/issues/137256](https://github.com/microsoft/vscode/issues/137256)

## Summary

Team-authored issue (MEMBER) describing a long-standing design tension around URI casing on Windows after file rename. The core behavior (first-appearance-defines-canonical-form) hasn't changed on main; the issue remains a valid discussion about whether VS Code should re-emit didClose/didOpen or expose better utilities for case-insensitive URI comparison to extensions.

## Change summary

On Windows, after renaming a file to change only its casing, VS Code continues passing the old-cased URI to extensions instead of updating it.

## Best solution

Either (a) emit didClose/didOpen when the canonical URI's casing diverges from the filesystem after a rename on case-insensitive systems, or (b) expose an `IExtUri`-like comparison/key utility in the stable extension API so extensions can reliably normalize URIs. The `isCaseSensitive` option on `registerFileSystemProvider` partially helps but doesn't solve the stale-URI problem for built-in filesystem providers. The fix likely lives in `src/vs/workbench/services/textfile/` (preferred-resource propagation to extension host) or `src/vs/workbench/api/` (LSP document sync).
