# Issue #185548 — Invalid mouse focus when releasing the cursor after a text selection

[https://github.com/microsoft/vscode/issues/185548](https://github.com/microsoft/vscode/issues/185548)

## Summary

This appears fixed on current main by a concrete Sticky Scroll change after the report: normal Sticky Scroll navigation now runs from a DOM click on the Sticky Scroll widget instead of an editor mouse-up over the overlay. That matches the reported failure mode where releasing a text selection over Sticky Scroll triggered navigation/focus incorrectly.

## Change summary

Selecting text in the search control could release the mouse over editor Sticky Scroll and trigger Sticky Scroll mouse-up navigation/focus unexpectedly.

## Best solution

Treat this as completed by commit a29a3782af70d01cedacf77e850e451066128f7f, which moved normal Sticky Scroll activation to a click listener on the widget; no triage-reason label is needed.

## Evidence

- **current code:** src/vs/editor/contrib/stickyScroll/browser/stickyScrollController.ts currently registers normal Sticky Scroll navigation with dom.EventType.CLICK on the stickyScrollWidgetDomNode, so a mouse release that started outside Sticky Scroll should not invoke this handler.
- **fix commit:** Commit a29a3782af70d01cedacf77e850e451066128f7f ("Shift hover on sticky scroll shows the last line of scope (#189939)") removed the prior this._editor.onMouseUp handler for Sticky Scroll normal clicks and replaced it with dom.addStandardDisposableListener(stickyScrollWidgetDomNode, dom.EventType.CLICK, ...).
- **discussion:** The only issue comment says this is specific to Sticky Scroll and suggests changing Sticky Scroll handling from mouseUp to click, referencing https://github.com/microsoft/vscode/issues/181200 and https://github.com/microsoft/vscode/issues/174205; the current code has made that exact event-model change.
