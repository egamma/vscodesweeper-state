# Issue #251396 — Fails to access OS keyring on Windows when signing in, even after clean reinstall and extensive troubleshooting

[https://github.com/microsoft/vscode/issues/251396](https://github.com/microsoft/vscode/issues/251396)

## Summary

The failure is in Chromium's os_crypt_win.cc DPAPI layer (error 0x8009000B / 0x80090345), not in VS Code's own code. VS Code correctly detects the encryption failure and surfaces the notification. This should be classified as upstream (Chromium/Electron) and routed to the secret-storage area for visibility, but there's no VS Code code fix to make here.

## Change summary

VS Code fails to access the OS keyring on certain Windows machines due to Chromium's os_crypt DPAPI encryption failing with 'Key not valid for use in specified state' and delegation errors.

## Best solution

Classify as upstream (Chromium/Electron os_crypt_win.cc). The DPAPI errors (0x8009000B, 0x80090345) indicate a Windows system-level cryptographic state problem — likely a machine trust/delegation misconfiguration or corrupted DPAPI keys. The existing Linux canonical issue (https://github.com/microsoft/vscode/issues/187338) is Linux-specific and not a duplicate. This Windows variant should stay open with `upstream` + `secret-storage` labels. If the Electron team has a bug tracker entry for os_crypt_win failures, linking it would help.
