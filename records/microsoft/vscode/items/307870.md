# Issue #307870 — VS Code NodeService binds to common proxy ports on Windows Insider

[https://github.com/microsoft/vscode/issues/307870](https://github.com/microsoft/vscode/issues/307870)

## Summary

Well-documented report, but the root cause is upstream: `node.mojom.NodeService` is an Electron/Chromium utility process, and the spurious port bindings only occur on Windows Insider Dev Channel builds — pointing to a Windows networking stack regression around SO_REUSEADDR semantics. VS Code's own code does not explicitly bind to any of the listed ports. Keeping open and classifying as upstream.

## Change summary

VS Code's Electron NodeService utility process binds to well-known proxy ports (7890, 7897, 9090, etc.) on Windows Insider Dev Channel builds, hijacking system proxy traffic.

## Best solution

Route to the Electron or Chromium project as a platform interaction issue with Windows Insider networking changes. The `node.mojom.NodeService` process and its socket behavior are Electron internals, not controlled by VS Code's TypeScript layer. If this persists into stable Windows builds, a workaround in VS Code's utility process launch flags may be warranted, but currently this only affects Insider Preview builds.

## Evidence

- **no VS Code port binds:** Searched the VS Code source (`src/`) for the specific ports 7890, 7897, 8317, 9090, 9999, 25666 — none are explicitly bound by VS Code code. The only network-related flag is `--inspect-port=0` in `src/vs/workbench/services/extensions/electron-browser/localProcessExtensionHost.ts:268`, which requests an ephemeral port, not a fixed one.
- **Electron utility process:** `node.mojom.NodeService` is an Electron/Chromium utility subprocess type, not a VS Code-controlled Node.js process. Its socket behavior is governed by Electron and Chromium's networking stack.
- **Windows Insider only:** Reporter confirms the issue does NOT reproduce on stable Windows 11 builds with the same VS Code version and proxy setup, isolating the cause to Windows Insider Dev Channel networking changes.
- **reporter's own analysis:** Reporter's 'Possible Cause' section attributes this to Windows Insider-specific SO_REUSEADDR semantics where a specific-address bind (127.0.0.1:port) takes priority over a wildcard bind (0.0.0.0:port).
- **Linux commenter:** A Linux commenter (FlyingFathead) reports a possibly related but distinct issue with random ephemeral port listeners, traced to `--experimental-network-inspection --inspect-port=0`, which is expected behavior for the Node.js inspector agent.
