# Issue #130486 — Stale semantic tokens across several files

[https://github.com/microsoft/vscode/issues/130486](https://github.com/microsoft/vscode/issues/130486)

## Summary

Keep open as an unresolved semantic-tokens/LSP invalidation design issue. Current VS Code has unscoped provider refresh events, but the issue discussion and current API still show no way for a server/provider to invalidate only specific affected URIs.

## Change summary

Language servers/providers need a better way to refresh semantic tokens and related pull features across dependent files without issuing broad, wasteful refreshes.

## Best solution

Resolve the API/LSP design around scoped invalidation for semantic tokens, likely in the semantic-tokens provider API and corresponding LSP client binding, so servers can report affected documents without forcing all visible documents for the provider to refresh.

## Evidence

- **current labels and milestone:** The issue is already classified as feature-request and semantic-tokens and is parked in Backlog, which does not force closure and fits an accepted-but-unscheduled design gap.
- **current API shape:** src/vscode-dts/vscode.d.ts still declares DocumentSemanticTokensProvider.onDidChangeSemanticTokens?: Event<void> and DocumentRangeSemanticTokensProvider.onDidChangeSemanticTokens?: Event<void>, so the public API does not carry affected URIs.
- **internal API shape:** src/vs/editor/common/languages.ts still models DocumentSemanticTokensProvider.onDidChange and DocumentRangeSemanticTokensProvider.onDidChange as Event<void>, matching an unscoped invalidation event.
- **main-thread bridge:** src/vs/workbench/api/browser/mainThreadLanguageFeatures.ts creates Emitter<void> for semantic-token events and fires undefined, so extension-host semantic token changes are not scoped by URI on current main.
- **editor refresh behavior:** src/vs/editor/contrib/semanticTokens/browser/documentSemanticTokens.ts fans provider change events out to model watchers, but each watcher only fetches when its model is attached to an editor; this matches the broad visible-document refresh approach discussed in the issue rather than a precise affected-URI solution.
- **recent discussion:** DanTup's 2025-12-18 comment reports that using /refresh for a similar Dart case can cause duplicate semantic-token requests when affected files are open but hidden, and says sending URIs would probably solve that case.
