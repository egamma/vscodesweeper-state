# Issue #130486 — Stale semantic tokens across several files

[https://github.com/microsoft/vscode/issues/130486](https://github.com/microsoft/vscode/issues/130486)

## Summary

The core request — allowing semantic token providers to specify which URIs are invalidated rather than broadcasting a blanket refresh — remains unimplemented. The API is still `Event<void>` with no URI support. Some mitigations exist (only re-fetching for visible/attached documents) but the design gap persists, as confirmed by DanTup's Dec 2025 comment showing double-requesting.

## Change summary

Allow semantic token providers to specify affected URIs when firing invalidation events, avoiding unnecessary re-computation for all visible files.

## Best solution

Extend the `onDidChangeSemanticTokens` event signature to `Event<void | Uri[]>` as proposed in the discussion (Comment 16), enabling providers to target specific files. This touches `src/vscode-dts/vscode.d.ts`, the ext host adapter in `src/vs/workbench/api/common/extHostLanguageFeatures.ts`, and the main thread handler. The LSP protocol side (workspace/semanticTokens/refresh with URIs) would need parallel work in vscode-languageserver-node.
