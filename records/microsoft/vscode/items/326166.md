# Issue #326166 — Plugins hooks on devcontainers can't call upon plugin location

[https://github.com/microsoft/vscode/issues/326166](https://github.com/microsoft/vscode/issues/326166)

## Summary

Plugin hooks that reference files within the plugin's installation directory fail in devcontainers/WSL because the host-side Windows path is passed verbatim to the container's shell. Closely related to #307478 (local variant of the same root cause) but adds the distinct remote/container path-translation dimension. Keeping open as a real bug in the agent-plugins area.

## Change summary

Agent plugin hooks referencing plugin-bundled files produce 'not found' errors in devcontainers because the host's Windows plugin path is not translated to the container filesystem.

## Best solution

The hook execution machinery needs to either (a) translate the plugin root path to the remote/container filesystem when running in a devcontainer/WSL remote, or (b) make plugin files available inside the container at a known location. This likely lives in the same area as https://github.com/microsoft/vscode/issues/307478 — resolving both would mean giving hooks a reliable, environment-aware `PLUGIN_ROOT` or equivalent. The fix probably touches the hook runner in the agent plugins service and the remote path resolution logic.
