# Issue #291269 — class names aren't (internal) API

[https://github.com/microsoft/vscode/issues/291269](https://github.com/microsoft/vscode/issues/291269)

## Summary

The debt is still present on main: `overlayManager.ts` continues to use a hardcoded list of CSS class names (`OVERLAY_DEFINITIONS`) to detect overlays, with no proper widget-level show/hide event API. The list has even grown since filing.

## Change summary

The integrated browser detects overlapping UI by scanning for hardcoded CSS class names, which are not stable API and can silently break.

## Best solution

Introduce an event or service-level mechanism (e.g. on editor widgets, context views, quick-input) that signals overlay show/hide/layout, so the browser overlay manager can subscribe instead of polling DOM class names. The work likely touches `src/vs/workbench/contrib/browserView/electron-browser/overlayManager.ts` and the relevant widget implementations.
