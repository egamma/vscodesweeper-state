# Issue #311819 — [Wayland] First keystroke lost after Alt+Tab with custom title bar

[https://github.com/microsoft/vscode/issues/311819](https://github.com/microsoft/vscode/issues/311819)

## Summary

Real bug on Wayland with custom title bar: when returning to VS Code via Alt+Tab, the Alt keydown+keyup delivered on refocus is treated as a 'clean alt press and release' by the menubar's `onModifierKeyToggled`, which steals focus to the menu bar and swallows the first keystroke. The `ModifierKeyEmitter` resets on blur but cannot suppress the synthetic Alt events Wayland/Electron deliver on window reactivation. Still present on current main — no fix found.

## Change summary

First keystroke is swallowed after Alt+Tab back to VS Code on Wayland when using the custom title bar, because the menubar intercepts the Alt key release as a menu-focus gesture.

## Best solution

The `ModifierKeyEmitter.resetKeyStatus()` fires on blur but the Alt keydown/keyup from the Alt+Tab sequence still reaches the menubar's `onModifierKeyToggled` after refocus. A fix would suppress Alt-focus behavior when the window has just regained focus (e.g. ignore Alt press/release within a short window after a blur→focus transition, or track that the Alt arrived as part of a window-switch). The fix likely touches `src/vs/base/browser/dom.ts` (ModifierKeyEmitter) and/or `src/vs/base/browser/ui/menu/menubar.ts` (onModifierKeyToggled). May also be partially upstream in Electron's Wayland ozone backend if the key events shouldn't be delivered at all.
