# Issue #280933 — Debugger has races and can show "Paused on Entry" shown for continued threads

[https://github.com/microsoft/vscode/issues/280933](https://github.com/microsoft/vscode/issues/280933)

## Summary

Still reproduces on current main: the race lives in `handleStop`'s use of `fetchThreads(stoppedDetails)` which applies stoppedDetails to the model as a side effect of the thread-ID-resolution promise — this side effect fires before the ThreadStatusScheduler's cancellation check, so a late-resolving fetchThreads can re-stop a thread that the continued handler already cleared.

## Change summary

Debugger UI can show threads as "Paused on Entry" even after continued events have been processed, due to a race in handleStop where fetchThreads applies stoppedDetails to the model before the cancellation guard runs.

## Best solution

Separate the thread-ID fetch from the stoppedDetails model update in `handleStop`: fetch threads without passing `stoppedDetails` to `fetchThreads`, then apply stoppedDetails inside the `operation` callback (after the cancellation check passes). This keeps the model update gated behind the ThreadStatusScheduler's cancellation logic, preventing a late-resolving fetch from overwriting a continued state.
