# Issue #208400 — Inconsistent behavior of Explorer when copying/moving file into folder with filename already present

[https://github.com/microsoft/vscode/issues/208400](https://github.com/microsoft/vscode/issues/208400)

## Summary

Confirmed bug still present on main: the Ctrl+X/Ctrl+V move-paste path skips conflict checking in findValidPasteFileTarget (allowOverwrite=true when pasteShouldMove) but then creates the ResourceFileEdit with overwrite=false (when incrementalNaming !== 'disabled'), causing the error instead of prompting. The DnD move path handles this correctly via error-catch-and-prompt. The copy paths (both DnD and paste) consistently apply incremental naming.

## Change summary

Explorer paste-move (Ctrl+X/Ctrl+V) of a file into a folder where the filename already exists shows an error instead of prompting to replace, unlike drag-and-drop move which correctly prompts.

## Best solution

Align the paste-move path in fileActions.ts with the DnD move path in explorerViewer.ts: either prompt for overwrite confirmation before the edit (like `askForOverwrite` does in the disabled-incremental-naming case), or set overwrite:true on the ResourceFileEdit to match the allowOverwrite flag passed to findValidPasteFileTarget. The DnD move pattern (try without overwrite → catch FILE_MOVE_CONFLICT → prompt → retry with overwrite:true) is the cleanest model to follow.
