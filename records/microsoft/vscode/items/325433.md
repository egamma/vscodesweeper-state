# Issue #325433 — Main process crash closes all windows: access violation in unloaded explorerframe.dll (BEX64, 0xc0000005), recurring since June 2026

[https://github.com/microsoft/vscode/issues/325433](https://github.com/microsoft/vscode/issues/325433)

## Summary

Well-analyzed upstream crash: the access violation occurs in native Windows shell code (`explorerframe.dll`) after COM unloads the DLL, likely triggered by a dangling callback from a native file dialog. VS Code's own code does not control DLL loading/unloading or COM library lifecycle — the root cause is in Electron's native dialog integration or in Windows itself. Keeping open for routing and upstream tracking.

## Change summary

The VS Code main process crashes with an access violation (0xc0000005) in unloaded explorerframe.dll roughly weekly when idle after using a native file dialog, killing all windows.

## Best solution

Route to the Electron layer or Windows platform. The crash is in native code after COM's `CoFreeUnusedLibrariesEx` unmaps `explorerframe.dll` while a callback/subclass pointer still references it. VS Code's dialog service (`src/vs/platform/dialogs/electron-main/dialogMainService.ts`) calls Electron's `dialog.showOpenDialog`/`showSaveDialog`, which uses `IFileDialog` COM interfaces. The fix likely needs to happen in Electron (preventing the dangling reference) or in Windows (the OS bug the reporter linked). Filing upstream at electron/electron with the WER signature and minidump would be the productive next step. In the meantime, there is no VS Code-side mitigation available.

## Evidence

- **crash in native code:** The faulting module is `explorerframe.dll_unloaded` (a Windows shell DLL), not any VS Code or Electron binary. The `_unloaded` suffix means code was called into a DLL that COM had already unmapped from the process.
- **COM lifecycle:** The reporter's timing analysis (crash ~10 min after last activity) matches COM's `CoFreeUnusedLibrariesEx` default delay, pointing to a dangling callback after COM unloads the file-dialog library.
- **VS Code dialog path:** VS Code's native dialogs go through `src/vs/platform/dialogs/electron-main/dialogMainService.ts` → Electron's `dialog.showOpenDialog`/`showSaveDialog` → Win32 `IFileDialog` COM interfaces → `explorerframe.dll`. VS Code has no control over the DLL lifecycle.
- **cross-app reports:** Reporter links https://learn.microsoft.com/en-us/answers/questions/2805633/explorer-crashes-due-to-explorerframe-dll showing similar crashes in other apps on current Win11 builds, supporting an OS-level root cause.
- **no VS Code fix found:** No references to `explorerframe` in the VS Code codebase or git history; the crash is below the Electron abstraction layer.
