# Issue #296012 — foldingImportsByDefault does not appear to work for C# usings

[https://github.com/microsoft/vscode/issues/296012](https://github.com/microsoft/vscode/issues/296012)

## Summary

The foldingImportsByDefault feature works correctly in VS Code core — it folds ranges tagged with FoldingRangeKind.Imports. For C# usings, the C# extension (csdevkit) must tag its folding ranges with this kind; this is an extension-side responsibility, not a VS Code bug.

## Change summary

C# using statements are not automatically folded when editor.foldingImportsByDefault is enabled.

## Best solution

File this against the C# extension (ms-dotnettools.csdevkit or ms-dotnettools.csharp) — their folding provider needs to return FoldingRangeKind.Imports for using-statement ranges. The VS Code core logic at src/vs/editor/contrib/folding/browser/folding.ts line 323-324 correctly collapses ranges of that kind; the built-in TypeScript extension already does this (extensions/typescript-language-features/src/languageFeatures/folding.ts line 76).

## Evidence

- **core logic correct:** src/vs/editor/contrib/folding/browser/folding.ts lines 323-324: when foldingImportsByDefault is true and model hasn't folded imports yet, it calls foldingRanges.setCollapsedAllOfType(FoldingRangeKind.Imports.value, true). This works for any extension that tags import ranges correctly.
- **TS extension works:** extensions/typescript-language-features/src/languageFeatures/folding.ts line 76 maps span.kind 'imports' to vscode.FoldingRangeKind.Imports — the built-in TS extension correctly supports this feature.
- **C# extension responsibility:** The C# extensions (csdevkit v2.11.4, csharp v2.120.3) must return FoldingRangeKind.Imports for using-statement folding ranges. If they return undefined or a different kind, the feature cannot detect them.
