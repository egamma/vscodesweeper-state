# Issue #322679 — Middle-click paste Primary Selection doesn't work after copying a file

[https://github.com/microsoft/vscode/issues/322679](https://github.com/microsoft/vscode/issues/322679)

## Summary

Keep open and route as a likely Linux clipboard regression from the recent Explorer native file-copy work; the report is actionable and has clear steps on a new release.

## Change summary

After copying a file in Explorer on Linux, middle-click paste in the editor inserts the copied file URI/path instead of the current primary selection text.

## Best solution

Verify the Linux clipboard formats after Explorer copy and keep `editor.selectionClipboard` isolated from file-copy clipboard data, likely around `NativeClipboardService.writeResources` and the Explorer `setToCopy` path.

## Evidence

- **report:** Issue https://github.com/microsoft/vscode/issues/322679 includes concrete steps, Linux OS details, VS Code 1.125.1, and confirms the problem occurs with extensions disabled.
- **selection clipboard implementation:** On Linux, `src/vs/workbench/contrib/codeEditor/electron-browser/selectionClipboard.ts` writes selected editor text to `clipboardService.writeText(..., 'selection')` and the paste action reads `clipboardService.readText('selection')`.
- **file copy path:** Explorer copy calls `ExplorerService.setToCopy` in `src/vs/workbench/contrib/files/browser/explorerService.ts`, which delegates to `clipboardService.writeResources`.
- **recent relevant change:** Commit f9070acd20a269fb07dd15389676bd6875b9db05, PR https://github.com/microsoft/vscode/pull/320685, changed Linux file-copy behavior to write native `text/uri-list` clipboard data in `src/vs/workbench/services/clipboard/electron-browser/clipboardService.ts`.
