# Issue #322679 — Middle-click paste Primary Selection doesn't work after copying a file

[https://github.com/microsoft/vscode/issues/322679](https://github.com/microsoft/vscode/issues/322679)

## Summary

Legitimate bug on Linux: copying a file in the Explorer calls writeResources which writes to the system clipboard via Electron's clipboard.writeBuffer, likely clearing or interfering with the X11 PRIMARY selection buffer. The issue is in the same family as #130208 and #261170 but has a distinct trigger (file copy in Explorer).

## Change summary

After copying a file with Ctrl+C in the Explorer, middle-click paste in the editor inserts the file path instead of the X11 Primary Selection content.

## Best solution

The fix likely needs to ensure that NativeClipboardService.writeResources() (which calls Electron's clipboard.writeBuffer with a custom format) does not clobber the X11 PRIMARY selection. This may require saving and restoring the selection clipboard around the writeBuffer call, or using Electron APIs that avoid touching PRIMARY. Related issues: https://github.com/microsoft/vscode/issues/130208, https://github.com/microsoft/vscode/issues/261170.

## Evidence

- **code path:** Explorer copy calls explorerService.setToCopy() → clipboardService.writeResources() → nativeHostService.writeClipboardBuffer('code/file-list', buffer) → Electron clipboard.writeBuffer(format, buffer). On Linux/X11, this may clear the PRIMARY selection.
- **related issues:** https://github.com/microsoft/vscode/issues/130208 (Ctrl+F overwrites PRIMARY), https://github.com/microsoft/vscode/issues/261170 (search highlighting updates PRIMARY) — same class of problem, different triggers.
- **selection clipboard code:** src/vs/workbench/contrib/codeEditor/electron-browser/selectionClipboard.ts manages the PRIMARY selection for text selections; it writes via clipboardService.writeText(text, 'selection'). The file-copy path writes without 'selection' type but may still interfere.
