# Issue #224933 — Sidebar does not remember original width when it is forced to narrow due to a smaller window

[https://github.com/microsoft/vscode/issues/224933](https://github.com/microsoft/vscode/issues/224933)

## Summary

Still reproducible on main: the sidebar's LayoutPriority.Low means it shrinks first when the window narrows, but the editor (LayoutPriority.High) absorbs space when the window grows — there is no preferred-size memory to restore the sidebar. A valid bug in the grid-widget layout system that has not been addressed.

## Change summary

Sidebar loses its user-set width when the window shrinks and then grows back, because the splitview has no mechanism to remember and restore a view's preferred size after constraint-driven shrinkage.

## Best solution

Add a 'preferred size' or 'unconstrained size' concept to the splitview/grid layout so that low-priority views (sidebar, auxiliary bar) can restore their user-set width when space becomes available again. Likely touches `src/vs/base/browser/ui/splitview/splitview.ts` (layout algorithm) and possibly `src/vs/workbench/browser/parts/sidebar/sidebarPart.ts`.
