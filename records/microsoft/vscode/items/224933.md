# Issue #224933 — Sidebar does not remember original width when it is forced to narrow due to a smaller window

[https://github.com/microsoft/vscode/issues/224933](https://github.com/microsoft/vscode/issues/224933)

## Summary

Still reproduces on main. The workbench grid uses proportionalLayout:false with sidebar at LayoutPriority.Low, so when the window shrinks, the sidebar loses size first. When the window grows back, the splitview's relayout gives extra space to the high-priority editor — there's no mechanism to remember the sidebar's pre-shrink 'preferred' size.

## Change summary

Sidebar does not restore its original width after being forced to shrink by a window resize and then the window is enlarged again.

## Best solution

Introduce a 'preferred size' or 'target size' concept in the splitview/grid for low-priority views, so that when space becomes available again after a forced shrink, the view can be restored toward its user-set width. The fix likely involves changes to `src/vs/base/browser/ui/splitview/splitview.ts` (resize/relayout logic) and possibly `src/vs/workbench/browser/layout.ts`.
