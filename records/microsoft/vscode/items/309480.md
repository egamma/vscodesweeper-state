# Issue #309480 — Chat in editor hides ask_questions interface when tab is hidden

[https://github.com/microsoft/vscode/issues/309480](https://github.com/microsoft/vscode/issues/309480)

## Summary

Confirmed bug: the question carousel UI rendered in the input part disappears when the chat editor tab loses and regains visibility, likely due to a lifecycle issue during list refresh on the visibility transition. Two independent reporters confirm the behavior. No fix found on main.

## Change summary

The ask-questions carousel UI disappears from the chat editor when the user switches to another tab and back, leaving only a progress message with no interactive question input.

## Best solution

Fix the carousel rendering lifecycle in `chatListRenderer.ts`'s `renderQuestionCarousel` path so that when `onDidChangeItems` triggers a list refresh after the tab becomes visible, the carousel either survives the re-render or is properly re-rendered into the input part. The issue likely lives in how `ChatListWidget.refresh()` (which calls `setChildren`) interacts with the `renderNoContent` placeholder and the input part's carousel tracking in `chatInputPart.ts`.

- **rendering architecture:** The carousel is rendered in the input part (src/vs/workbench/contrib/chat/browser/widget/input/chatInputPart.ts:4026) while a `renderNoContent` placeholder sits in the list. On tab re-show, `chatWidget.ts:1623` triggers `onDidChangeItems(true)` which calls `listWidget.refresh()` — the full tree `setChildren` may cause elements to be disposed/re-rendered, and the carousel placeholder/input-part lifecycle may not survive this cycle.
- **no fix on main:** git log --grep 309480 returns no commits; no related fix found in recent carousel or editor visibility changes.
- **progressive rendering paused:** src/vs/workbench/contrib/chat/browser/widget/chatListRenderer.ts:1787 — `doNextProgressiveRender` skips rendering when not visible, so if carousel content arrives while hidden it may not render until a refresh, which then may not re-invoke the carousel render path due to the `hasSameContent` guard.
