# Issue #226755 — How should bad/stale providers be handled?

[https://github.com/microsoft/vscode/issues/226755](https://github.com/microsoft/vscode/issues/226755)

## Summary

Open design question about how the search API should handle providers that never resolve — no implementation has landed, and the question remains valid for the search-api surface. Keeping open as an unresolved API design concern.

## Change summary

The search find widget can stall entirely when a TextSearchProvider never resolves its promise; the request is for more resilient handling (e.g. streaming partial results from other providers).

## Best solution

The search service (src/vs/workbench/services/search/common/searchService.ts) should adopt a pattern similar to suggest or references: race or stream results from providers independently so one hung provider doesn't block the entire find widget. This is an API-level design decision that needs maintainer input on the desired contract for TextSearchProvider timeout/cancellation behavior.
