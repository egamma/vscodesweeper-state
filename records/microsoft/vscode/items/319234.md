# Issue #319234 — Using a non-string for `contributes.commands[*].enablement` silently breaks extension commands

[https://github.com/microsoft/vscode/issues/319234](https://github.com/microsoft/vscode/issues/319234)

## Summary

Keep open as an actionable core robustness bug: current main still allows falsy non-string command `enablement` values to reach context-key parsing, where they can throw and abort the commands extension-point handler.

## Change summary

A malformed extension command contribution with non-string `enablement` can silently prevent commands from later extensions from being registered.

## Best solution

Harden the commands contribution validation in `src/vs/workbench/services/actions/common/menusExtensionPoint.ts` so every defined non-string `enablement` is rejected with an extension validation error, and add coverage that one invalid command contribution does not stop later extensions' commands from registering.

## Evidence

- **issue report:** https://github.com/microsoft/vscode/issues/319234 reports `TypeError: this._input.charCodeAt is not a function` after installing `pypa.hatch` v0.2.0, followed by missing command-registration errors for other extensions.
- **current validation gap:** `src/vs/workbench/services/actions/common/menusExtensionPoint.ts:810` checks `command.enablement && typeof command.enablement !== 'string'`, so falsy non-string values such as `false` are not rejected.
- **current parse path:** `src/vs/workbench/services/actions/common/menusExtensionPoint.ts:961` passes `enablement` directly to `ContextKeyExpr.deserialize(enablement)` when registering the command.
- **matching failure mode:** `src/vs/platform/contextkey/common/contextkey.ts:628-633` only special-cases `undefined`/`null` before parsing, and `src/vs/platform/contextkey/common/scanner.ts:187-188` stores the value as `_input`; later scanner code calls `_input.charCodeAt`, matching the reported TypeError for a boolean/object value.
- **handler abort behavior:** `src/vs/workbench/services/extensions/common/extensionsRegistry.ts:153-156` catches errors around the whole extension-point handler call, so an exception inside the commands handler can abort processing of the remaining command contributions.
