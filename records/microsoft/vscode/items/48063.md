# Issue #48063 — Search and replace across files corrupts files with CR line endings

[https://github.com/microsoft/vscode/issues/48063](https://github.com/microsoft/vscode/issues/48063)

## Summary

Still reproduces on main: the editor text buffer only supports LF and CRLF (`EndOfLineSequence` has no CR variant), so files with CR-only line endings get their line boundaries misreported, causing search-and-replace to corrupt file contents. The author is a MEMBER, so this stays open for maintainer judgment.

## Change summary

Search and replace across files corrupts files that use CR-only (\r) line endings by misidentifying line boundaries and applying replacements at wrong offsets.

## Best solution

The root cause is the lack of CR-only line ending support in the text buffer (tracked by https://github.com/microsoft/vscode/issues/35797). A minimal fix for search could detect CR-only files and either skip them with a warning or handle the CR character as a line separator in the search/replace pipeline. The match offset logic in `src/vs/workbench/contrib/search/browser/searchTreeModel/match.ts` only normalizes CRLF→LF (lines 96, 112), never bare CR. Alternatively, the broader fix in #35797 would resolve this systemically.

## Evidence

- **match.ts only normalizes CRLF:** src/vs/workbench/contrib/search/browser/searchTreeModel/match.ts lines 96 and 112 replace \r\n with \n but never handle bare \r, so CR-only line endings cause offset misalignment.
