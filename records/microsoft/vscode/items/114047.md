# Issue #114047 — Unexpected editor.selection in onDidChangeActiveTextEditor callback

[https://github.com/microsoft/vscode/issues/114047](https://github.com/microsoft/vscode/issues/114047)

## Summary

Keep open: this is a confirmed API timing gap that was intentionally moved to Backlog, and I did not find evidence in the checkout that the requested active-editor event ordering has been implemented. The current labels already route it to the API/editor ownership area.

## Change summary

Extension authors want `window.onDidChangeActiveTextEditor` to expose the restored selection and visible range immediately when switching back to an already-open editor.

## Best solution

If VS Code wants to guarantee this API behavior, the workbench/editor handoff should restore model state before the extension host receives the active-editor change, likely by adding an editor-side operation equivalent to the `setModelAndSelection` support suggested by the maintainer. Otherwise the API docs should explicitly describe that selection/visible ranges may be restored after the active-editor event and that consumers should listen to the dedicated selection/range events.

## Evidence

- **suggested implementation direction:** The same maintainer comment says this likely needs editor support such as `setModelAndSelection` so the workbench can set the model and restored selection together.
- **current status:** The issue remains open with milestone Backlog and labels `feature-request`, `api`, and `editor-core`; Backlog is a holding bin and does not trigger the milestone keep-open exception, but it supports treating this as accepted unscheduled work rather than stale cleanup.
- **related user impact:** A later commenter reported the same timing pattern for `visibleRanges`, where `onDidChangeActiveTextEditor` fires before restored view state is observable, reinforcing that the gap is about API event ordering rather than a single reporter setup.
