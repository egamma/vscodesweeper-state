# Issue #303481 — Crash when VS Code runs out of RAM on Linux but not on Windows

[https://github.com/microsoft/vscode/issues/303481](https://github.com/microsoft/vscode/issues/303481)

## Summary

The crash is caused by the Linux OOM killer (signal 15/SIGTERM) terminating VS Code's child processes when the system runs out of memory — the same OS-level behavior previously triaged as *as-designed in the predecessor issue. VS Code cannot prevent the kernel from selecting its processes for termination.

## Change summary

On Linux, when a Python process exhausts system RAM, the OS OOM killer terminates VS Code's processes (ptyHost, shared-process, extension host), crashing the editor — whereas on Windows the Python process alone is killed.

## Best solution

Keep the current behavior as designed. The logs show processes killed with code 15 (SIGTERM from Linux OOM killer) — this is an OS-level decision VS Code cannot override. Users can mitigate by configuring swap space, cgroup memory limits for their Python processes, or adjusting oom_score_adj. This matches the precedent set by the closed predecessor issue.

## Evidence

- **OS-level behavior:** On Linux the kernel OOM killer selects processes to terminate based on oom_score; on Windows, memory allocation fails with an error that applications can handle. This difference is at the OS/kernel level, not something VS Code can control.
