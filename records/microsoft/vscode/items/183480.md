# Issue #183480 — "Search files by name" is slow when using remote

[https://github.com/microsoft/vscode/issues/183480](https://github.com/microsoft/vscode/issues/183480)

## Summary

Still valid on current main: Quick Open (Cmd+P) over Remote SSH runs ripgrep --files on the remote, collects the full file list (434K+ entries in the reporter's case), and sends it back in 512-item batches via the extension host RPC channel — the fuzzy filtering happens on the local side after the transfer. No architectural change has been made to move the filtering to the remote side. 48 upvotes and active investigation by andreamah confirm this is a real, widely-felt issue.

## Change summary

Quick Open file search is disproportionately slow over Remote SSH because the full ripgrep --files result set is transferred from the remote extension host to the local renderer before fuzzy filtering.

## Best solution

Move the fuzzy file-name filtering to the remote extension host side so only the top ~512 matching results are sent over the wire, rather than the entire file listing. The key files are src/vs/workbench/services/search/common/fileSearchManager.ts (BATCH_SIZE=512, batching logic), src/vs/workbench/api/common/extHostSearch.ts ($handleFileMatch sends full batches), and src/vs/workbench/services/search/node/rawSearchService.ts (caching/scoring). The caching in rawSearchService.ts already demonstrates that scoring can happen on the search side — the gap is that the initial uncached search sends the full file list before scoring.
