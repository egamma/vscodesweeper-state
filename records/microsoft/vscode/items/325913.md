# Issue #325913 — Bash shell integration: PROMPT_COMMAND array entries from other profile scripts get misreported as the current command

[https://github.com/microsoft/vscode/issues/325913](https://github.com/microsoft/vscode/issues/325913)

## Summary

Confirmed on current main: the bash shell integration's DEBUG trap treats PROMPT_COMMAND array entries (appended by other profile scripts like systemd's) as user-entered commands, breaking command detection for the session. The bug is clearly present in the source and the reporter's patch is well-reasoned.

## Change summary

PROMPT_COMMAND array entries from other profile scripts (e.g. systemd) are misreported as the current command by the bash shell integration's DEBUG trap, permanently breaking command detection.

## Best solution

Adopt the reporter's approach: add a helper function that checks whether `$BASH_COMMAND` matches any entry in the (possibly array) `PROMPT_COMMAND`, and skip the preexec/command-detection path for those entries. The fix touches `shellIntegration-bash.sh` in three places: the `__vsc_preexec` filter (line 412), and the two `__vsc_preexec_only`/`__vsc_preexec_all` wrappers (lines 441, 448). The existing comment on line 482-483 acknowledges the array handling gap.

  > In src/vs/workbench/contrib/terminal/common/scripts/shellIntegration-bash.sh, the DEBUG trap's preexec logic misidentifies PROMPT_COMMAND array entries as user commands. The filter on line 412 (`if [[ ! $BASH_COMMAND == __vsc_prompt* ]]`) only matches VS Code's own `__vsc_prompt*` functions — but on bash 5.1+, other scripts (e.g. systemd's `80-systemd-osc-context.sh`) append their own functions to `PROMPT_COMMAND` as higher array indices. When bash executes those entries, the DEBUG trap fires with `$BASH_COMMAND` set to the foreign function name, which passes the filter and is treated as the user's command. This also consumes the `__vsc_in_command_execution` guard, so the real next command never gets its preexec.
