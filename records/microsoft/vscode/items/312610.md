# Issue #312610 — [critical] new GitHub Copilot sessions never appear in list and are irretrievable if Copilot enabled after workspace opened

[https://github.com/microsoft/vscode/issues/312610](https://github.com/microsoft/vscode/issues/312610)

## Summary

Plausible race-condition bug where sessions created after enabling Copilot post-workspace-load never appear in the sessions list, leading to data loss. The reporter identified a concrete trigger (per-workspace enablement after startup) and confirmed sessions exist on disk but are invisible in the UI. No maintainer response or fix found on main after 82 days; routing to chat-setup and multi-chat-sessions for investigation.

## Change summary

New Copilot chat sessions are never listed in the Sessions panel when Copilot is enabled after the workspace is already open, making them irretrievable once another session is selected.

## Best solution

Investigate the session-list initialization path in the chat sessions service to ensure sessions created after a deferred Copilot activation are properly registered and visible. The race condition likely lives at the boundary between Copilot extension activation (chat-setup) and the session storage/listing code (`src/vs/workbench/contrib/chat/browser/chatSessions/chatSessions.contribution.ts` or `src/vs/workbench/contrib/chat/common/model/chatSessionStore.ts`). The related per-workspace enablement persistence bug (https://github.com/microsoft/vscode/issues/311898) compounds the problem because users must re-enable Copilot every session, hitting this race every time.
