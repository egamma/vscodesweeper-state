# Issue #240539 — \ path separators are interpreted as escape sequences when HOME is set in bash terminal

[https://github.com/microsoft/vscode/issues/240539](https://github.com/microsoft/vscode/issues/240539)

## Summary

Confirmed bug: the env-var encoding in `_encodeColons` only escapes `:` but not `\`, so `echo -e` in the bash shell integration script interprets backslashes in Windows paths as escape sequences. An open PR (#240614) exists to fix this but has not merged.

## Change summary

Backslash path separators in environment variable values (e.g. `HOME=C:\Users\…`) are corrupted by `echo -e` in the bash shell integration script because backslashes are not escaped during encoding.

## Best solution

Land the fix in https://github.com/microsoft/vscode/pull/240614 (or an equivalent): escape `\` to `\\` in `_encodeColons` (rename to `_encodeValue`) so that `echo -e` round-trips correctly. The fix touches `src/vs/platform/terminal/common/environmentVariableCollection.ts` (the encoder) and possibly `src/vs/workbench/contrib/terminal/common/scripts/shellIntegration-bash.sh` (the decoder). The open PR should be reviewed and merged.

## Evidence

- **buggy encoder:** src/vs/platform/terminal/common/environmentVariableCollection.ts line 109-111: `_encodeColons` only does `value.replaceAll(':', '\x3a')` — backslashes are not escaped, so `C:\Users\abrahan` becomes `C:\x3a...` with `\a` interpreted as BEL by `echo -e`.
- **buggy decoder:** src/vs/workbench/contrib/terminal/common/scripts/shellIntegration-bash.sh lines 76/85/94 use `echo -e` to decode `\x3a` back to `:`, but this also interprets any unescaped backslash sequences in the original value.
- **open fix PR:** https://github.com/microsoft/vscode/pull/240614 — 'Fix missing encoding of backslash in terminal shell integration environment variable values' — is still open and unmerged.
- **introducing commit:** Reporter identified the bug was introduced in commit 689c9126defdc7564b8deafb6f1d6481b1bcda30 which added colon-only escaping.
