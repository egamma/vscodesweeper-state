# Issue #171238 — Regression: debugger cannot attach to a child process

[https://github.com/microsoft/vscode/issues/171238](https://github.com/microsoft/vscode/issues/171238)

## Summary

The race condition is still present on current main: `debugSession.ts` line 399 cancels child launch if the parent is Inactive, and PR #166137's await on disconnect makes the parent terminate before the child's `startDebugging` request arrives. No fix has been committed. Related issue #131442 tracks the broader parent/child lifecycle debt.

## Change summary

When a debug adapter (e.g. debugpy) sends a `startDebugging` reverse request for a child process and the parent session terminates, VS Code rejects the child launch because the parent is already Inactive.

## Best solution

Fix the race in `debugSession.ts` and/or `rawDebugSession.ts` so that a pending `startDebugging` reverse request keeps the parent session alive (or allows child launch even with an Inactive parent). This overlaps with the lifecycle cleanup tracked in https://github.com/microsoft/vscode/issues/131442 — resolving both together would be ideal.

## Evidence

- **race condition in source:** src/vs/workbench/contrib/debug/browser/debugSession.ts line 399: `if (this.parentSession && this.parentSession.state === State.Inactive) { throw canceled(); }` — this guard rejects child launches when the parent has already shut down.
- **causal commit:** https://github.com/microsoft/vscode/commit/d9a70c4236a8bde4fe7435df3ecdd333c7bc939c (PR #166137) added an await on the disconnect response, causing the parent to fully terminate before the child's startDebugging request is processed.
- **startDebugging handler:** src/vs/workbench/contrib/debug/browser/rawDebugSession.ts lines 699-722: the `startDebugging` reverse request handler calls `this.dbgr.startDebugging(config, this.sessionId)` — if the parent session identified by `this.sessionId` is already Inactive by then, the child session creation fails.
- **maintainer confirmed:** Comment by @roblourens (MEMBER) on 2023-01-13 confirmed the reproduction and identified the race: the parent is dead by the time VS Code gets the startDebugging call for the child.
- **related lifecycle debt:** https://github.com/microsoft/vscode/issues/131442 is an open debt issue to remove lifecycle handling heuristics around parent/child sessions, which encompasses this problem.
