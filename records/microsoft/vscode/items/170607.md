# Issue #170607 — set decoration's opacity is not working

[https://github.com/microsoft/vscode/issues/170607](https://github.com/microsoft/vscode/issues/170607)

## Summary

Still reproduces on main: the `editorUnnecessaryCode.opacity` CSS rule applied via the `squiggly-inline-unnecessary` inline class cascades onto user-defined decorations (including `before`/`after` pseudo-elements) on the same range, and there is no mechanism for a decoration to opt out of this inherited opacity.

## Change summary

Decoration `opacity: "1"` on a TextEditorDecorationType does not override the editor's fading of unused/unnecessary code, causing custom decorations to appear dimmed on unused code ranges.

## Best solution

The fix should ensure that decoration render options (particularly `before`/`after` pseudo-elements) are not affected by the unnecessary-code opacity. Options include isolating the unnecessary opacity to text content only (not pseudo-element decorations), or applying the user-set opacity to override the inherited one on decoration elements. The fix likely lives in how the `squiggly-inline-unnecessary` CSS rule interacts with decoration rendering in `src/vs/editor/browser/widget/codeEditor/codeEditorWidget.ts` and possibly `src/vs/editor/common/services/markerDecorationsService.ts`.
