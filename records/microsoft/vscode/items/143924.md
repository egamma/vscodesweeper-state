# Issue #143924 — Delay starting the web worker extension host until activating first extension

[https://github.com/microsoft/vscode/issues/143924](https://github.com/microsoft/vscode/issues/143924)

## Summary

Team-authored (alexdima, MEMBER) with milestone 'On Deck' — hard keep-open. The lazy-start mechanism (`LazyAutoStart` / `LazyCreateExtensionHostManager`) exists on desktop but the browser extension service still eagerly starts the web worker extension host; the request appears partially unresolved.

## Change summary

Delay spawning the web worker extension host until the first extension running in it actually needs to activate, avoiding unnecessary overhead when no web-worker extensions are used.

## Best solution

Extend the browser `ExtensionService.createExtensionHost` to use `ExtensionHostStartup.LazyAutoStart` for the `LocalWebWorker` kind (mirroring the native service's lazy path), so the web worker is not started until activation is needed. The change would be in `src/vs/workbench/services/extensions/browser/extensionService.ts`.
