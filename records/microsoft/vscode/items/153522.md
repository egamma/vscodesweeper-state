# Issue #153522 — Replacing with capture groups writes the first match's values to all matches in that line

[https://github.com/microsoft/vscode/issues/153522](https://github.com/microsoft/vscode/issues/153522)

## Summary

Still reproduces on current main: when a regex with lookaheads matches multiple times on a line, the surrounding-content fallback in getReplaceString always exec's from position 0, so all matches on that line get the first match's capture groups.

## Change summary

Search and Replace with regex capture groups writes the first match's captured values to all matches on the same line when lookaheads are involved.

## Best solution

Fix `getReplaceString` in `src/vs/workbench/services/search/common/replace.ts` to accept the match offset within the surrounding text, so that when the fallback path passes the full line, exec can skip to the correct match position (or use `match.index` from the already-known match location to validate the correct match is used).

  > Bug: `ReplacePattern.getReplaceString` in `src/vs/workbench/services/search/common/replace.ts` always runs `this._regExp.exec(text)` from position 0 (line 62-63). When the caller in `src/vs/workbench/contrib/search/browser/searchTreeModel/match.ts` falls through to the surrounding-content path (line 105-108, for lookahead patterns), it passes the full line as `text`. For the second (or later) match on that line, exec still finds the first match, so the wrong capture groups are substituted. Fix approach: `getReplaceString` (or the calling code in match.ts) needs to know the offset of the target match within the surrounding text, so it can either (a) set `lastIndex` to skip to the correct match, or (b) the caller can prepend only the content needed for the lookahead rather than the entire line prefix. The simplest fix is to add an optional `matchIndex` parameter to `getReplaceString` that sets `lastIndex` before exec, and have the match.ts caller pass the match's start column within the full preview lines.
