# Issue #208126 — [linked rename] HTML doesn't rename or auto-fill non-ASCII tags

[https://github.com/microsoft/vscode/issues/208126](https://github.com/microsoft/vscode/issues/208126)

## Summary

Still reproduces on main. The root cause is in the vscode-html-languageservice package's scanner, which uses a regex that only matches ASCII word characters for tag names, truncating non-ASCII characters like 'é'.

## Change summary

HTML auto-close tag and linked editing fail for tags containing non-ASCII characters (e.g. `<cliché>` produces `</clich>`).

## Best solution

Fix the `nextElementName` regex in `vscode-html-languageservice` (scanner) to support Unicode characters in tag names — either use the `u` flag with Unicode property escapes (`\p{L}`) or widen the character class. The fix belongs upstream in https://github.com/microsoft/vscode-html-languageservice.

## Evidence

- **root cause in scanner:** extensions/html-language-features/server/node_modules/vscode-html-languageservice/lib/esm/parser/htmlScanner.js line 146: `nextElementName()` uses regex `/^[_:\w][_:\w-.\d]*/` — `\w` is ASCII-only in JS, so 'é' is not matched and the tag name is truncated to 'clich'.
- **upstream package:** vscode-html-languageservice v5.3.3, maintained at https://github.com/microsoft/vscode-html-languageservice
- **linked editing depends on parser:** findLinkedEditingRanges (htmlLinkedEditing.js) uses `node.tag` and `tagLength` from the parser; doTagComplete uses `node.tag` for auto-close. Both inherit the truncated name.
- **HTML spec allows non-ASCII:** Reporter links https://html.spec.whatwg.org/multipage/custom-elements.html#valid-custom-element-name which permits many non-ASCII code points in custom element names.
