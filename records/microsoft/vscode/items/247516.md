# Issue #247516 — font is not reflected when updated in vscode setting 

[https://github.com/microsoft/vscode/issues/247516](https://github.com/microsoft/vscode/issues/247516)

## Summary

Real remaining snap bug: the `XDG_DATA_HOME` override in the snap launcher prevents fonts installed in `~/.local/share/fonts` from loading. An open PR (https://github.com/microsoft/vscode/pull/292808) by deepak1556 removes that override and is the proper fix, but it has not merged yet.

## Change summary

Custom fonts installed in `~/.local/share/fonts` are not picked up by the VS Code snap package due to the snap launcher overriding `XDG_DATA_HOME`.

## Best solution

Land https://github.com/microsoft/vscode/pull/292808 which reverts the `XDG_DATA_HOME` override in `resources/linux/snap/electron-launch`, eliminating the root cause for this and related snap issues (terminal history, trash).

## Evidence

- **root cause in source:** `resources/linux/snap/electron-launch` overrides `XDG_DATA_HOME` to `$SNAP_USER_DATA/.local/share`, breaking fontconfig's access to the user's real `~/.local/share/fonts` directory.
- **prior fix incomplete:** Commit b62b4161a0b (shipped in 1.96.0) added a symlink workaround but it does not fully resolve the issue — the reporter and a confirmer both reproduce on 1.99.3 which includes that fix.
- **proper fix in open PR:** https://github.com/microsoft/vscode/pull/292808 (branch `robo/revert_xdg_data_home_overriding`, commit f89ca3c152d) removes the `XDG_DATA_HOME` override entirely, which is the real root cause. Not yet merged to main.
- **user confirmation:** Comment 4 by ljcoogan confirms: fonts in `/usr/share/fonts` work, fonts in `~/.local/share/fonts` do not — trivially reproducible by moving fonts between the two locations.
