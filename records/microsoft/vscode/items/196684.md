# Issue #196684 — Specific inputs cause replacing single instance in Search View to skip selection past the next instance

[https://github.com/microsoft/vscode/issues/196684](https://github.com/microsoft/vscode/issues/196684)

## Summary

Still open on main: the replace-single-match logic in `getElementToFocusAfterRemoved` computes the next focus element before the replace occurs and tree refresh, so when the replacement text changes character offsets, the pre-computed navigation target can land on the wrong match. Milestone is 'On Deck' — keeping open per scheduling policy.

## Change summary

Replacing a single search match in the Search View skips the next match and selects the one after it, depending on the replacement text length.

## Best solution

Fix `getElementToFocusAfterRemoved` in `searchActionsRemoveReplace.ts` to account for the fact that the tree navigation is computed before `batchReplace` mutates the document. The next-focus computation should either happen after the replace and tree refresh, or the navigator should account for offset changes from the replacement. The fix is localized to `performReplace` and `getElementToFocusAfterRemoved`.
