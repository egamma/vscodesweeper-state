# Issue #292383 — Race with afterDelay save on editor close, especially with git commit

[https://github.com/microsoft/vscode/issues/292383](https://github.com/microsoft/vscode/issues/292383)

## Summary

Valid race condition: when auto-save delay ≤1000ms, VS Code's `hasShortAutoSaveDelay` causes `isSaving()` to return true unconditionally, suppressing dirty warnings and allowing editors to close before the actual save completes — so external consumers (git commit, terminal reads) see stale file content. The defective path is traceable on current main.

## Change summary

With auto-save `afterDelay` ≤1000ms, closing an editor or committing git can race with the pending auto-save, causing git to commit with an incomplete message or external processes to read stale file content.

## Best solution

The git extension's `smartCommit` flow should explicitly save the commit-message document (and any dirty staged files) before invoking `repository.commit()`, regardless of auto-save mode — similar to what `commitMessageAccept` already does at line 2684 (`commitMsgDocument.save()`). Separately, the core auto-save system could force an immediate flush on editor close when `hasShortAutoSaveDelay` is true, rather than merely pretending the save already happened. The fix likely spans `extensions/git/src/commands.ts` (save before commit) and `src/vs/workbench/contrib/files/browser/editors/fileEditorInput.ts` (the `isSaving()` override at line 323).

## Evidence

- **isSaving() override:** src/vs/workbench/contrib/files/browser/editors/fileEditorInput.ts line 323: `if (this.filesConfigurationService.hasShortAutoSaveDelay(this)) { return true; }` — unconditionally reports the editor as 'saving' when auto-save delay ≤1000ms, suppressing dirty indicators and close confirmations.
- **dirty indicator suppression:** src/vs/workbench/contrib/files/common/dirtyFilesIndicator.ts line 43 and src/vs/workbench/contrib/files/browser/views/openEditorsView.ts line 491 both skip dirty-state updates when `hasShortAutoSaveDelay` is true, matching the reporter's observation about no '1 unsaved' note at ≤1000ms.
- **git promptToSaveFilesBeforeCommit:** extensions/git/src/commands.ts line 2384-2386: the save-before-commit prompt filters on `d.isDirty`, but the dirty indicator is suppressed for short-delay auto-save, so this guard may not fire.
- **1000ms threshold:** src/vs/workbench/services/filesConfiguration/common/filesConfigurationService.ts line 125,448-454: DEFAULT_AUTO_SAVE_DELAY is 1000ms; delays ≤ this value map to AFTER_SHORT_DELAY mode, confirming the reporter's observed behavior difference at the 1-second boundary.
