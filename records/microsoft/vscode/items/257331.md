# Issue #257331 — xdg-open freezes on KDE when run from vscode terminal

[https://github.com/microsoft/vscode/issues/257331](https://github.com/microsoft/vscode/issues/257331)

## Summary

Valid snap packaging bug: the electron-launch script prepends snap-internal paths to XDG_DATA_DIRS, which leaks into terminal child processes and causes xdg-open to freeze on KDE. Existing snap env restoration logic in environmentMainService.ts appears insufficient. Related to the broader env leak tracked in #293910 but this issue provides a concrete, actionable reproduction.

## Change summary

Running xdg-open from the VS Code integrated terminal freezes on KDE/Kubuntu when VS Code is installed via snap, because snap-modified XDG_DATA_DIRS leaks into terminal child processes.

## Best solution

Fix the snap environment variable restoration so that XDG_DATA_DIRS (and other snap-modified variables) are properly cleaned before spawning terminal shell processes. The electron-launch script (resources/linux/snap/electron-launch) saves originals via _VSCODE_SNAP_ORIG suffixes, and environmentMainService.ts has unsetSnapExportedVariables() — but the restoration doesn't fully propagate to terminal child processes. This is the same root cause as https://github.com/microsoft/vscode/issues/293910; fixing either fixes both.

## Evidence

- **snap env modification:** resources/linux/snap/electron-launch lines 114-117 prepend snap directories to XDG_DATA_DIRS; line 98 saves the original via copy_env_variable as XDG_DATA_DIRS_VSCODE_SNAP_ORIG.
- **existing restoration code:** src/vs/platform/environment/electron-main/environmentMainService.ts line 80 has unsetSnapExportedVariables() that restores _VSCODE_SNAP_ORIG values, but this only modifies the main process env — it may not propagate correctly to terminal shells.
- **related issue:** https://github.com/microsoft/vscode/issues/293910 reports the same root cause more broadly: all snap-modified environment variables leak into launched programs.
