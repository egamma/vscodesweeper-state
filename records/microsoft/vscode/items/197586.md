# Issue #197586 — [remote] Cannot open VS Code extension URIs in Remote instances

[https://github.com/microsoft/vscode/issues/197586](https://github.com/microsoft/vscode/issues/197586)

## Summary

This appears fixed on current main by the remote CLI openExternal URI parsing change; `vscode://...` is no longer treated as a filesystem path before being sent to the workbench opener.

## Change summary

Remote terminal `$BROWSER` should open extension-owned `vscode://...` URIs instead of creating a blank file named after the URI.

## Best solution

Close as completed and keep the current remote CLI behavior from commit 450a3b337f8d592b8b619e23adfd80545d8e6267, which recognizes protocol-shaped URIs before falling back to path conversion.

  > Thanks for the report. Current main still has that behavior in `src/vs/server/node/server.cli.ts`, so I think this can be closed as completed.

## Evidence

- **fix commit:** Commit 450a3b337f8d592b8b619e23adfd80545d8e6267 (https://github.com/microsoft/vscode/pull/234380) changed src/vs/server/node/server.cli.ts openInBrowser from only recognizing http/https/file URLs to recognizing any /^[a-z-]+:\/\/.+/ URI, specifically describing protocol activation in a remote workspace being treated as a path and malformed.
- **current code:** Current main still has the broadened URI detection in src/vs/server/node/server.cli.ts:396, so a value like vscode://fabiospampinato.vscode-debug-launcher/... is pushed through url.parse(...) rather than pathToURI(...).
- **opener path:** src/vs/workbench/api/node/extHostCLIServer.ts:157-164 skips only file: URIs and sends non-file URI strings to _remoteCLI.openExternal; src/vs/workbench/api/browser/mainThreadCLICommands.ts:30-32 opens those strings via IOpenerService with openExternal and allowTunneling.
- **issue discussion:** A maintainer asked whether the change had been tested, and the reporter replied that they had tested it successfully, supporting that the intended behavior was validated.
