# Issue #169266 — Context for compressed Explorer folders always points to child

[https://github.com/microsoft/vscode/issues/169266](https://github.com/microsoft/vscode/issues/169266)

## Summary

Still reproduces on main: when a compressed folder node (parent/child) is focused or right-clicked, `setContextKeys()` sets the `resource` context key to `stat.resource` — the deepest child in the compressed chain — without consulting the compressed navigation controller's `current` position. The `when`-clause `resource` context is therefore always the child path, making resource-based `when` clauses unreliable for compressed folders.

## Change summary

The `resource` context key for compressed explorer folders always points to the deepest child, not the segment the user actually clicked or focused.

## Best solution

In `setContextKeys()` (explorerView.ts ~line 614), check the compressed navigation controller's `current` to resolve the focused segment's resource, rather than using the raw tree element's resource. This aligns `resourceContext` with the compressed-focus position, just as `onContextMenu` already does for the menu `arg` (line 665).
