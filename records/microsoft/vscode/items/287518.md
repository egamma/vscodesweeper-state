# Issue #287518 — VS Code crashes with Open File in a Snap installation due to missing `'user-home-symbolic-ltr` icon

[https://github.com/microsoft/vscode/issues/287518](https://github.com/microsoft/vscode/issues/287518)

## Summary

Real crash in the snap package caused by missing pixbuf loaders and icon themes. The snap's bundled GTK can't find icons or decode the fallback PNG, triggering a GTK assertion abort. Active fix work exists on a feature branch (snap base upgrade to core22 with librsvg2-common) but hasn't merged to main yet — keeping open.

## Change summary

VS Code snap crashes with a GTK assertion failure when opening native file dialogs because the snap environment lacks proper pixbuf loaders and icon theme access.

## Best solution

Land the snap ARM64/core22 branch (`dev/dmitriv/build-snap-arm64`) which migrates the snap base from core20 to core22 and adds `librsvg2-common` as a stage-package — this provides the missing pixbuf PNG/SVG loaders that prevent the GTK crash. The in-flight branch at commit 723e30fc639 already contains this fix.

## Evidence

- **root cause in snapcraft.yaml:** Current main snapcraft.yaml (resources/linux/snap/snapcraft.yaml) uses `base: core20` and strips icons (`-usr/share/icons` in prime), but does not include `librsvg2-common` — so GTK's pixbuf loaders can't decode PNG icons, causing the assertion in gtkiconhelper.c:494.
- **fix in progress on branch:** Branch `dev/dmitriv/build-snap-arm64` (commit 6e0f5d3577e, 2026-02-28) migrates to core22 and adds `librsvg2-common` as a stage-package, which provides the missing loaders.
- **prior report:** https://github.com/microsoft/vscode/issues/235236 reported the same GTK pixbuf crash in the snap; maintainer deepak1556 identified the fix as upgrading the snap core base.
- **electron-launch script:** resources/linux/snap/electron-launch sets GDK_PIXBUF_MODULE_FILE and GDK_PIXBUF_MODULEDIR but the core20 base doesn't include enough pixbuf loaders to handle the icon formats used by host GTK themes.
