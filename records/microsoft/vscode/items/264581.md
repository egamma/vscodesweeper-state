# Issue #264581 — Modifying the location of any test (TestController API) causes all other test in the file to snap back to their original locations ignoring unsaved changes

[https://github.com/microsoft/vscode/issues/264581](https://github.com/microsoft/vscode/issues/264581)

## Summary

The behavior is by-design per connor4312's explanation, but DanTup raised a valid unresolved counter-argument about whether resetting tracked positions for ALL tests in a file (rather than only the updated one) is necessary. The discussion ended without resolution — keeping open for a maintainer decision on whether the granularity should change.

## Change summary

Updating a single test item's range via the TestController API causes all other tests in the same file to lose their tracked decoration positions and snap back to their originally-published locations, even when the file is dirty.

## Best solution

Consider changing the `DocumentSynced` diff to reset tracked positions only for the specific test item whose range was updated, rather than for all items in the file. The logic lives in `src/vs/workbench/contrib/testing/browser/testingDecorations.ts` (line ~315) and `src/vs/workbench/contrib/testing/common/testItemCollection.ts` (the `documentSynced` call at line 434). Alternatively, provide an explicit API for extensions to signal full-file sync vs single-item update.
