# Issue #183198 — While draging to create mutipile-cursors the start point is where the cursor used to be rather than the position where I clicked.

[https://github.com/microsoft/vscode/issues/183198](https://github.com/microsoft/vscode/issues/183198)

## Summary

Keep open and route to the editor area. Current main still appears to anchor Alt/Option+Shift column selection from the existing cursor/selection state rather than from the mouse-down point, so the reported behavior is not already fixed.

## Change summary

Alt/Option+Shift drag for creating multiple cursors should start the column selection at the mouse-down position, matching middle-mouse drag behavior, and the report also notes cursor alignment problems with full-width characters.

## Best solution

Editor ownership should decide whether Alt/Option+Shift drag should initialize fresh column-selection state from the mouse-down position in the editor mouse/column selection path, or whether this is intended behavior and should be closed explicitly as designed. The full-width alignment concern may need a separate focused issue if it is independent of the drag anchor behavior.

## Evidence

- **anchor selection logic:** src/vs/editor/browser/coreCommands.ts uses prevColumnSelectData.fromViewLineNumber/fromViewVisualColumn when doColumnSelect is true, rather than the current mouse position as the starting point.
- **default previous state:** src/vs/editor/common/cursor/cursor.ts getCursorColumnSelectData falls back to the primary cursor selection start and position when no real column-selection data exists, which matches the reporter's observation that the drag starts from the old cursor.
