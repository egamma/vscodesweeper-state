# Issue #324612 — Main process crashes with SQLITE_BUSY on new APPLICATION_SHARED storage (.vscode-shared\sharedStorage\state.vscdb) when two instances run concurrently

[https://github.com/microsoft/vscode/issues/324612](https://github.com/microsoft/vscode/issues/324612)

## Summary

Confirmed real bug via source inspection: ApplicationSharedStorageMain constructs SQLiteStorageDatabase without a busyTimeout, and transaction() has no SQLITE_BUSY retry logic, so concurrent processes writing to the shared state.vscdb will crash the main process. No fix has been merged. The reporter's root-cause analysis is accurate.

## Change summary

VS Code main process crashes with an unhandled SQLITE_BUSY error when two instances concurrently write to the APPLICATION_SHARED storage database.

## Best solution

Pass a busyTimeout option when constructing SQLiteStorageDatabase in ApplicationSharedStorageMain.doCreate() (src/vs/platform/storage/electron-main/storageMain.ts line 378), and add SQLITE_BUSY retry/backoff logic to the transaction() method in src/vs/base/parts/storage/node/storage.ts (line 413) — at minimum for the shared-storage scope. Additionally, ensure SQLITE_BUSY errors from transaction() are handled gracefully rather than escalating to an uncaught exception that kills the process.

## Evidence

- **missing busyTimeout:** src/vs/platform/storage/electron-main/storageMain.ts lines 378-380: ApplicationSharedStorageMain.doCreate() creates SQLiteStorageDatabase with only logging options — no busyTimeout is passed, so PRAGMA busy_timeout is never set for this multi-process-shared DB.
- **no transaction retry:** src/vs/base/parts/storage/node/storage.ts lines 413-431: the transaction() method calls handleSQLiteError and rejects immediately on any error, including SQLITE_BUSY — no retry or backoff logic exists for ongoing writes.
- **busyTimeout infrastructure exists:** src/vs/base/parts/storage/node/storage.ts lines 28-32, 348-349: the SQLiteStorageDatabase constructor accepts a busyTimeout option and correctly sets PRAGMA busy_timeout when provided — it's just not used by the shared storage scope.
- **design intent:** APPLICATION_SHARED storage was added in PR https://github.com/microsoft/vscode/pull/311317 explicitly for cross-app (multi-process) state sharing, making SQLITE_BUSY contention a foreseeable scenario.
- **no fix on main:** git log shows no commits to these files since the feature landed (717e853871c) other than the unrelated refactor f645d405cbb; the bug is present on current main.
