# Issue #326765 — High CPU usage (infinite loop) When Incorrect Proxy URL is Set

[https://github.com/microsoft/vscode/issues/326765](https://github.com/microsoft/vscode/issues/326765)

## Summary

Real bug: an incorrectly configured HTTP proxy URL pointing at a SOCKS proxy causes Electron's Chromium network service to retry connections indefinitely without backoff, consuming 100% CPU. VS Code's own nodeRequest retry logic (3 attempts with backoff) is properly bounded — the infinite loop is in the Chromium/Electron network layer. VS Code could mitigate by validating proxy URLs or rate-limiting reconnection at its own proxy-resolution layer.

## Change summary

Setting http.proxy to an HTTP URL that actually points at a SOCKS proxy causes infinite reconnection attempts with 100% CPU in the Chromium network service process.

## Best solution

VS Code could mitigate this at the proxy configuration layer — either validate that the configured proxy responds correctly before passing it to Electron, or add circuit-breaker logic in the proxy resolver. The root cause is in Chromium's network service retry behavior and could also be filed upstream against Electron/Chromium. Area-wise this belongs in the proxy/request platform code (src/vs/platform/request/).

## Evidence

- **Chromium network service:** The strace shows node.mojom.NodeService (Electron's Chromium network process) sending HTTP CONNECT to a SOCKS port, getting connection closed, and immediately retrying without limit.
- **proxy.ts validation gap:** src/vs/platform/request/node/proxy.ts line 36 checks for http/https protocol but only for the node-side proxy agent — the Electron/Chromium side receives the raw proxy URL without similar validation.
