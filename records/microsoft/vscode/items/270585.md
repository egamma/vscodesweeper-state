# Issue #270585 — Task commands retrieved from a command won't be recognized as absolute

[https://github.com/microsoft/vscode/issues/270585](https://github.com/microsoft/vscode/issues/270585)

## Summary

Confirmed bug on current main: for process tasks on Windows, `_resolveAndFindExecutable` calls `resolveAsync` which cannot resolve `${command:...}` variables (they require interactive resolution via `resolveWithInteraction`). The unresolved variable string fails the `path.isAbsolute` check, so cwd gets prepended.

## Change summary

Process tasks using `${command:...}` as the command have the workspace folder prepended because the command variable isn't resolved before the absolute-path check.

## Best solution

In `_resolveAndFindExecutable`, the resolved command value (after interactive variable resolution has populated the variables map) should be used instead of re-resolving from `resolveAsync` which cannot handle command variables. The fix likely involves passing the already-resolved command value into the executable-finding logic rather than re-resolving it. The ext host path at `src/vs/workbench/api/node/extHostTask.ts:174` has the same issue.

- **variable resolver behavior:** src/vs/workbench/services/configurationResolver/common/variableResolver.ts lines 194-195 and 335-343: `resolveFromMap` returns the raw match when `commandValueMapping` is undefined, which is always the case for `resolveAsync`.
- **ext host same issue:** src/vs/workbench/api/node/extHostTask.ts line 174: `resolver.resolveAsync(ws, toResolve.process.name)` also won't resolve `${command:...}` variables for the same reason.
