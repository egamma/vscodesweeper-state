# Issue #323245 — Clicking editor in split view scrolls far away

[https://github.com/microsoft/vscode/issues/323245](https://github.com/microsoft/vscode/issues/323245)

## Summary

Valid bug still present on main: the markdown preview's scroll-sync path fires `revealLine` back to the editor when the user clicks the editor after scrolling the preview, causing the editor to jump. An open PR (#323250) proposes a fix but has not merged.

## Change summary

Clicking back into the editor from a scrolled markdown split preview causes the editor to jump to the preview's scroll position and spuriously select text.

## Best solution

Review and land (or iterate on) the open PR https://github.com/microsoft/vscode/pull/323250 which adds timing guards to suppress the stale `revealLine` message posted during the editor-selection-change → preview-scroll feedback loop.

## Evidence

- **source trace:** In extensions/markdown-language-features/src/preview/preview.ts, `#onDidScrollPreview` (line 356) unconditionally calls `scrollEditorToLine` when `scrollEditorWithPreview` is enabled — there is no guard to suppress scroll messages that were triggered by an editor selection change echoing back through the webview.
- **feedback loop path:** Line 784: `onDidChangeTextEditorSelection` posts `onDidChangeTextEditorSelection` message to the webview → webview scrolls to that line → webview emits a scroll event → posts `revealLine` back → `#onDidScrollPreview` calls `scrollEditorToLine` at the stale preview position.
- **open PR:** https://github.com/microsoft/vscode/pull/323250 (open, not merged) directly targets this issue with `Fixes #323245` and adds timing guards on both the webview and extension-host sides.
