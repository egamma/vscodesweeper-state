# Issue #284088 — Automatically clean up (close or delete) worktree after background agent changes are applied

[https://github.com/microsoft/vscode/issues/284088](https://github.com/microsoft/vscode/issues/284088)

## Summary

Team-authored feature request (MEMBER). Worktree cleanup on archive is now implemented (`cleanupWorktreeOnArchive` in `chatSessionWorktreeServiceImpl.ts`, landed in PR #316561), but the gap remains: the 'Apply Changes to Parent Repository' action merges the branch without triggering any cleanup or archive, so worktrees linger after the user's primary workflow completes. The request for automatic cleanup after apply, and the community suggestions (cleanup button, auto-prune setting, `copilot-` prefix), are still unaddressed.

## Change summary

Automatically clean up (close or delete) the worktree directory after a background agent session's changes have been applied to the parent repository.

## Best solution

After the 'Apply Changes to Parent Repository' action successfully merges (`applyChangesToParentRepo.ts`), offer to archive the session (which would trigger the existing `cleanupWorktreeOnArchive` path), or add a direct post-apply cleanup step. The community suggestions — a 'Clean up' button next to Apply, an auto-prune-after-N-days setting, and a `copilot-` worktree name prefix — are worth evaluating. The plumbing for worktree deletion already exists in the git extension and the Copilot worktree service; the missing piece is wiring it into the apply and session-lifecycle flows.
