# Issue #229530 — Semantic tokens types do not override "standard token type" and "font style"

[https://github.com/microsoft/vscode/issues/229530](https://github.com/microsoft/vscode/issues/229530)

## Summary

Keep open: current main still preserves TextMate standard token type metadata under semantic highlighting, and there is no concrete fixing commit for this report. The standard-token-type part may be intentional, but the font-style/rendering or inspector mismatch remains actionable for the grammar/semantic-tokens owners.

## Change summary

Semantic tokens are expected to correct a TextMate grammar token classified as a comment, but the token still reports Comment and remains italicized in the inspector/rendering.

## Best solution

Decide in the editor tokenization/theme path whether semantic token types should ever reset TextMate standard token type/font-style metadata; if not, update the token inspector in src/vs/workbench/contrib/codeEditor/browser/inspectEditorTokens/inspectEditorTokens.ts so it accurately explains that TextMate style is still effective unless a semantic token rule explicitly clears it.

## Evidence

- **issue reproduction:** The report includes a minimal extension grammar for TypeScript that scopes foo as comment, with semantic tokens still available, and says the token remains standard token type Comment and italic under Solarized Light with extensions disabled.
- **current merge behavior:** src/vs/editor/common/tokens/sparseTokensStore.ts:176-184 builds the semantic override mask only from SEMANTIC_USE_ITALIC/BOLD/UNDERLINE/STRIKETHROUGH/FOREGROUND/BACKGROUND; it does not include TOKEN_TYPE_MASK, so TextMate standard token type metadata is preserved.
- **semantic style flags:** src/vs/editor/common/services/semanticTokensProviderStyling.ts:65-83 only sets semantic override bits for font styles and foreground when the semantic token style explicitly defines those properties.
- **inspector behavior:** src/vs/workbench/contrib/codeEditor/browser/inspectEditorTokens/inspectEditorTokens.ts:307-311 renders standard token type from tmMetadata, and lines 474-485 only prefer semantic font-style labels when the semantic value is truthy, so an explicit semantic false does not visibly suppress a TextMate style in the inspector.
- **no fix found:** git log --grep for issue 229530, the issue title, and semantic font-style wording found no fixing commit in the relevant token styling or token inspector paths on current main b1de8f3c232f.
