# Issue #272420 — MCP: programmatically registered MCP server don't get roots set from client

[https://github.com/microsoft/vscode/issues/272420](https://github.com/microsoft/vscode/issues/272420)

## Summary

Confirmed on current main: extension-registered MCP servers never receive workspace roots because the ext-host serialization path omits the `roots` field entirely, unlike mcp.json-configured servers which get workspace folder roots set automatically.

## Change summary

MCP servers registered programmatically via the extension API never receive workspace roots from the client, while servers configured via mcp.json do.

## Best solution

When serializing extension-provided MCP server definitions in `extHostMcp.ts`, default `roots` to workspace folders (matching the behavior of `workspaceMcpDiscoveryAdapter.ts` and `installedMcpServersDiscovery.ts`). Optionally expose a `roots` property on `McpStdioServerDefinition`/`McpHttpServerDefinition` in the extension API so extensions can override the default.

- **Reason:** MCP server integration is part of the agent-native editor direction; consistent roots behavior across configuration methods is a basic correctness requirement for that infrastructure.

- **mcp.json servers get roots:** src/vs/workbench/contrib/mcp/common/discovery/installedMcpServersDiscovery.ts:109 sets `roots: [workspaceFolder.uri]`; workspaceMcpDiscoveryAdapter.ts:72 does the same.
- **server uses roots when provided:** src/vs/workbench/contrib/mcp/common/mcpServer.ts:516-543 sends roots to the MCP connection, defaulting to workspace folders only when `explicitRoots` is undefined — but the ext-host path never sets `roots` on the serialized definition, so it passes `undefined` which triggers workspace-folder fallback at the McpServer level.
- **commenter confirmation:** Comment from elyskov on 2026-04-04 confirms the issue persists on VS Code 1.114.0 — programmatic servers show 'No valid root directories provided by client'.
- **no API surface for roots:** The vscode.d.ts McpStdioServerDefinition class has no `roots` property, so extensions cannot specify custom roots.
