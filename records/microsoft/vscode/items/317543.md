# Issue #317543 — chat view incorrectly highights non-matched custom prompt name

[https://github.com/microsoft/vscode/issues/317543](https://github.com/microsoft/vscode/issues/317543)

## Summary

Confirmed bug, still present on main: the chat viewer highlights any syntactically-valid `/foo` slash prompt text even when no prompt with that exact (case-sensitive) name exists, because the parser falls back to a regex-only check (`isValidSlashCommandName`) rather than verifying the prompt actually resolves. Maintainer aeschli confirmed the behavior and the expected fix.

## Change summary

Chat viewer highlights a `/foo` slash prompt as matched even when the actual prompt name is `Foo` (case-sensitive mismatch), misleading the user into thinking the prompt was included.

## Best solution

The parser's fallback at `chatRequestParser.ts:272` should verify the command against the known prompt set (e.g. `hasPromptSlashCommand`) instead of using the purely syntactic `isValidSlashCommandName`, so unresolved prompt names are not emitted as `ChatRequestSlashPromptPart` and are not highlighted in the chat viewer.
