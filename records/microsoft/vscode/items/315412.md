# Issue #315412 — Copilot Chat does not recover from transient outbound failures (host suspend / EAI_AGAIN); silent token refresh leaves UI stuck until   window reload

[https://github.com/microsoft/vscode/issues/315412](https://github.com/microsoft/vscode/issues/315412)

## Summary

Valid bug: the Copilot extension's silent background fetches (auth refresh, model list) lack retry-with-backoff for transient DNS/connection errors after host resume, and `EAI_AGAIN` is not in the existing `canRetryOnceNetworkError` error-code list. The chat-request path has a single retry via `disconnectAll`, but the background token/model paths do not use it. Routing to the copilot area for triage.

## Change summary

After host suspend/resume, Copilot Chat enters a permanently stuck error state because silent background fetches (auth token refresh, model list) fail on transient network errors and have no retry logic.

## Best solution

Add `EAI_AGAIN` to `canRetryOnceNetworkError` in `extensions/copilot/src/platform/networking/common/networking.ts`, and wrap the silent auth-refresh and `fetchKnownModelList` call sites in retry-with-backoff so transient DNS/connection failures on resume don't leave the extension permanently stuck. The `disconnectAll` + single-retry pattern already exists for chat requests — extend it (or a proper backoff) to background fetches in the token manager and model-list paths.
