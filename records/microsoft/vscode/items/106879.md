# Issue #106879 — macOS: Crash when using `fs.watch` for network drives

[https://github.com/microsoft/vscode/issues/106879](https://github.com/microsoft/vscode/issues/106879)

## Summary

The underlying libuv/kqueue crash in fs.watch() on disconnected network filesystems remains unfixed upstream. VS Code's workaround (refusing to watch /Volumes/) is still in place on current main but is too broad (blocks external SSDs and local APFS volumes) and too narrow (misses non-/Volumes mount points). The issue is correctly labeled as upstream and should stay open as the canonical tracker for this class of crash.

## Change summary

macOS crashes (abort in libuv kqueue) when an SMB/network share disconnects while VS Code is watching files via fs.watch().

## Best solution

The real fix is upstream in libuv (the kqueue.c abort should be a graceful error, not a process kill). VS Code's /Volumes/ heuristic in `src/vs/platform/files/node/watcher/nodejs/nodejsWatcherLib.ts` needs refinement — either detect actual network filesystems via statfs/getfsstat rather than path prefix, or fall back to fs.watchFile (polling) for detected network mounts. The open issue https://github.com/microsoft/vscode/issues/300045 (crash on startup when installed on external drive) is a direct consequence of the too-broad workaround and should be considered alongside any improvement here.
