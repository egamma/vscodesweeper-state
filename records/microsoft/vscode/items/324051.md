The root cause lives in Electron/Chromium, not VS Code's own code. Routing to the appropriate area for upstream tracking.

## Evidence

- **upstream root cause:** When a Chromium-based window crosses a DPI boundary, the compositor re-rasterizes at the new scale. During this transition the old-scale content is briefly letterboxed inside the new-scale frame, exposing the native window background as a white border. This is a Chromium rendering pipeline behavior, not a VS Code issue.
- **VS Code background color handling:** src/vs/platform/windows/electron-main/windows.ts sets backgroundColor on BrowserWindow creation from the theme, but this does not control the compositor's behavior during DPI transitions.
- **similar prior issue:** https://github.com/microsoft/vscode/issues/85655 (white line at top in 4K) was closed with labels upstream + electron, confirming this class of rendering artifact is owned by Electron/Chromium.
- **environment:** Reporter is on Windows 10.0.26200, Electron 42.5.0, dual monitor with different resolutions — classic DPI-mismatch scenario.
