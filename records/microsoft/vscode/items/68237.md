# Issue #68237 — Search with non-standard encodings not supported

[https://github.com/microsoft/vscode/issues/68237](https://github.com/microsoft/vscode/issues/68237)

## Summary

Genuine upstream limitation: ripgrep (via Rust's encoding_rs) does not support legacy codepage encodings like cp437, cp850, or cp852. VS Code passes the encoding to ripgrep's --encoding flag, which rejects it. The issue is correctly labeled upstream and feature-request, has 31 upvotes (well above the 20-upvote threshold), and remains valid — no fix has landed on main.

## Change summary

Workspace text search fails with "Unknown encoding" for non-WHATWG encodings (cp437, cp850, cp852, etc.) because ripgrep does not support them.

## Best solution

The real fix lives in ripgrep's upstream encoding support (encoding_rs only covers the WHATWG Encoding Standard, which excludes DOS codepages). A VS Code-side mitigation could transcode files through iconv-lite before feeding them to the search pipeline, but that would be a significant architectural change in the search service. Keep open as upstream until ripgrep gains broader encoding support or VS Code implements a fallback transcoding path in `src/vs/workbench/services/search/node/ripgrepTextSearchEngine.ts`.
