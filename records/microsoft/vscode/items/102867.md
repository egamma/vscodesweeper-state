# Issue #102867 — CodeLens messes up page scrolling via cursorPageUp / cursorPageDown

[https://github.com/microsoft/vscode/issues/102867](https://github.com/microsoft/vscode/issues/102867)

## Summary

Keep open: the issue is already in Backlog with editor-core/code-lens routing, and current main still appears to use a line-height based page size for cursorPageUp/cursorPageDown while CodeLens is rendered as view zones. There is not enough evidence for a close label or a completed-fix close.

## Change summary

Make cursorPageUp/cursorPageDown page movement account for CodeLens vertical space so PageUp/PageDown aligns visible viewport boundaries consistently.

## Best solution

Editor cursor page movement should either account for view-zone height when computing page movement or document that cursor paging is model-line based while scrollPageUp/scrollPageDown is the viewport-oriented command. The likely owner is editor-core with CodeLens/view zone interaction in src/vs/editor/browser/coreCommands.ts and src/vs/editor/contrib/codelens/browser/codelensWidget.ts.

## Evidence

- **current labels:** The issue is open with feature-request, editor-core, and code-lens labels, and is parked in the Backlog milestone rather than a release milestone.
- **cursor paging implementation:** src/vs/editor/browser/coreCommands.ts maps cursorPageUp/cursorPageDown to CursorMove Unit.WrappedLine with Constants.PAGE_SIZE_MARKER, which is replaced by viewModel.cursorConfig.pageSize.
- **page size computation:** src/vs/editor/common/cursorCommon.ts computes pageSize as Math.floor(layoutInfo.height / lineHeight) - 2, which does not visibly account for CodeLens view-zone height.
- **CodeLens layout:** src/vs/editor/contrib/codelens/browser/codelensWidget.ts creates CodeLensViewZone instances with heightInPx and adds them through viewZoneChangeAccessor.addZone, so CodeLens consumes vertical viewport space outside normal text-line height.
