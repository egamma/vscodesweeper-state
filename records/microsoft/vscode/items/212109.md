# Issue #212109 — Parentheses `()` and square brackets `[]` in file names can break terminal links (eg. next.js dynamic file paths)

[https://github.com/microsoft/vscode/issues/212109](https://github.com/microsoft/vscode/issues/212109)

## Summary

Still reproduces on main: the terminal link regex excludes parentheses from valid path characters, so paths mixing `()` and `[]` (common in Next.js routing) aren't detected as a single link. An open PR (#304992) exists but hasn't merged.

## Change summary

File paths containing parentheses and/or square brackets (e.g. Next.js dynamic routes like `./src/(hello)/[world]/file.tsx`) are split into multiple link segments instead of being detected as one clickable terminal link.

## Best solution

Land or iterate on the open PR https://github.com/microsoft/vscode/pull/304992 which addresses parentheses in terminal link paths. The fix belongs in `terminalLinkParsing.ts` — the `ExcludedPathCharactersClause` regex constants and the `linkWithSuffixPathCharacters` regex need to allow `()` and `[]` within path segments while still disambiguating them from line:col suffix delimiters.

## Evidence

- **root cause in source:** `ExcludedPathCharactersClause` at line 345 of `src/vs/workbench/contrib/terminalContrib/links/browser/terminalLinkParsing.ts` explicitly excludes `()` from valid path characters, and `ExcludedStartPathCharactersClause` (line 346) also excludes `[]`. The suffix regex (line 125) uses `[\[\(]` as line:col delimiters, creating ambiguity with brackets in path names.
- **partial workaround exists:** Lines 318–333 add extra results starting after opening brackets, but this doesn't produce a single unified link for paths with mixed bracket types.
- **open PR:** https://github.com/microsoft/vscode/pull/304992 ('fix: allow parentheses in terminal link paths') is open and unmerged.
