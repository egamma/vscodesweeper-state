# Issue #324845 — Fuzzy search pattern doesn't match words if it's corresponding parts start not after a delimiter/from a capital letter

[https://github.com/microsoft/vscode/issues/324845](https://github.com/microsoft/vscode/issues/324845)

## Summary

The fuzzy scorer intentionally requires the first matched character to land at a word boundary (capital letter, separator, or post-separator position). Matching `fetch` mid-word in `autofetch` is explicitly excluded by design to prevent low-quality noise matches. An existing test (issue #23332) asserts this exact pattern returns no match.

## Change summary

Fuzzy autocomplete doesn't match `fetch` in `git.autofetch` because the `f` is mid-word without a word-boundary signal.

## Best solution

Keep current behavior — the strong-first-match requirement is a deliberate quality filter for the suggestion list. `git.autoFetch` (camelCase) matches because `F` is at a word boundary; `git.autofetch` does not because `f` is mid-word. This prevents noisy results and is tested at `src/vs/base/test/common/filters.test.ts` line 301.

  > Thanks for the detailed report. This is intentional behavior in the fuzzy scorer: the first matched character must land at a "word boundary" — a capital letter (camelCase), a separator (`.`, `_`), or immediately after one — to avoid returning low-quality noise matches.
  > 
  > In your example:
  > - `git.autoFetch` — the `F` is uppercase → word boundary → matches ✓
  > - `git.pruneOnFetch` — the `F` is uppercase → word boundary → matches ✓
  > - `git.autofetch` — the `f` is lowercase mid-word → not a word boundary → filtered out ✓
  > 
  > This is the same design choice tested for issue #23332. The setting name `git.autofetch` itself doesn't follow camelCase conventions (unlike `git.autoFetch`), which is what makes it harder to discover — but the scorer is working as intended.
  > 
  > Closing as by-design. If this significantly impacts discoverability of specific settings, a separate feature request to relax the scorer specifically for settings completions could be considered.
