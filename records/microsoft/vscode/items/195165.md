# Issue #195165 — Quick search is showing results without full match

[https://github.com/microsoft/vscode/issues/195165](https://github.com/microsoft/vscode/issues/195165)

## Summary

Race condition in Quick Search is still present on main: the shared SearchModelImpl and its searchResult are reused across successive searches, and fast typing can cause stale async results from a prior query to appear alongside current results. No fixing commit found.

## Change summary

Quick Search shows partial-match results from a previous query when typing fast, mixing stale results with current ones.

## Best solution

Fix the race condition in `textSearchQuickAccess.ts` by ensuring that async results from a superseded search query are discarded before they reach the picker. The `_getPicks` method reuses `this.searchModel.searchResult` across invocations; stale async completions from an earlier `doSearch` call can still feed into `_getPicksFromMatches`. A debounce was added (commit 4c742de8cf1) but the core race — old async results arriving after a new search starts — remains. The fix likely involves tagging each search invocation and ignoring results from a mismatched generation, or clearing results more aggressively on new input.

  > Race condition in Quick Search (src/vs/workbench/contrib/search/browser/quickTextSearch/textSearchQuickAccess.ts): when the user types fast, `_getPicks` is called multiple times in rapid succession. Each call invokes `doSearch`, which starts an async search on the shared `this.searchModel`. The async results callback (`allMatches.asyncResults.then(...)` at line ~385) reads `this.searchModel.searchResult.matches(false)` — but by the time it resolves, a newer search may have started, and `searchResult` contains results for a different query. This causes stale partial matches to appear in the picker. The fix: introduce a generation counter (e.g. `private _searchGeneration = 0`) that is incremented at the start of each `_getPicks` call. Capture the current generation before starting the search, and in the async callback, check if the generation still matches — if not, return an empty array instead of stale results. Do NOT restructure the search model or the picker base class. Validate with a unit test that simulates two overlapping `_getPicks` calls with different patterns and asserts the first call's async results are discarded.
