# Issue #124910 — Git - stage commands are not queued

[https://github.com/microsoft/vscode/issues/124910](https://github.com/microsoft/vscode/issues/124910)

## Summary

Still reproduces on main: the git `run()` method in `repository.ts` does not serialize write operations, so two concurrent `stageSelectedChanges` calls overlap — the second call's diff context can become stale after the first completes and rewrites the index, causing it to silently fail or produce incorrect results.

## Change summary

Rapidly staging multiple selected ranges in sequence drops the second staging because git commands are not queued/serialized.

## Best solution

Serialize write operations (at least `Stage`) through the `run()` method in `extensions/git/src/repository.ts` — either via a `Sequentializer` or a `Limiter(1)` for non-read-only operations — so that concurrent stage calls are queued rather than overlapping. The `retryRun` backoff for `RepositoryIsLocked` partially addresses this at the git-lock level but not at the index-coherence level.

## Evidence

- **no serialization in run():** `extensions/git/src/repository.ts` line 2702: the `run()` method starts operations immediately without waiting for prior write operations to complete; multiple concurrent calls overlap freely.
- **stageSelectedChanges captures diff at call time:** `extensions/git/src/commands.ts` line 1770–1800: `stageSelectedChanges` reads `workingTreeDiffInformation` and `textEditor.selections` at invocation time, then calls `_stageChanges` which applies a patch against the current index — if a prior stage is in-flight, the diff information is stale.
- **retryRun only handles lock errors:** `extensions/git/src/repository.ts` line 2826: `retryRun` retries on `RepositoryIsLocked` with backoff but does not serialize operations at a higher level to ensure index coherence.
- **commenter confirmation:** Comment by @mikedecr (2022-05-06) confirms staging multiple chunks leads to 'essentially no result' even when the IDE is otherwise responsive.
