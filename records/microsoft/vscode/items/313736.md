# Issue #313736 — Call Stack visual selection can remain on previous thread after per-thread step, despite activeStackItem changing

[https://github.com/microsoft/vscode/issues/313736](https://github.com/microsoft/vscode/issues/313736)

## Summary

Plausible race condition in the Call Stack view: when a per-thread step fires `onDidFocusStackFrame` while `onCallStackChangeScheduler` is still pending, the selection update is deferred via the `selectionNeedsUpdate` flag, but the deferred `updateTreeSelection()` can race with further `onDidChangeCallStack` events (e.g. from `wholeCallStack` resolution), potentially leaving the visual selection on the previous thread. The reported DAP sequence is correct; the bug appears to live in the view's event coordination.

## Change summary

After a per-thread step in a multi-threaded debug session, the Call Stack view's visual selection (bold row) can remain on the previously-focused thread even though `activeStackItem` and the debug view model correctly move to the stepped thread.

## Best solution

Audit the interaction between `onCallStackChangeScheduler`, the `onFocusChange` handler, and `selectionNeedsUpdate` in `src/vs/workbench/contrib/debug/browser/callStackView.ts` (lines 345-370, 178-224). The flag-based deferral is fragile when the async `RunOnceScheduler` callback (which `await`s `tree.updateChildren()`) is not concurrency-safe — a re-scheduled scheduler run can clear the flag before the deferred selection fires. Consider either coalescing both signals into a single scheduler, or ensuring `updateTreeSelection()` always runs after every `updateChildren()` when a focus change has occurred.

## Evidence

- **race condition in callStackView.ts:** Lines 365-367: when `onCallStackChangeScheduler.isScheduled()` is true, the `onFocusChange` handler sets `selectionNeedsUpdate = true` and returns without updating selection. The scheduler callback (line 178) is async — it `await`s `tree.updateChildren()` at line 204, during which `isScheduled()` is false and new `onDidChangeCallStack` events can re-schedule it. The `selectionNeedsUpdate` flag is reset at line 221 before the deferred `updateTreeSelection()` runs, so a concurrent re-schedule can cause the flag to be cleared prematurely.
- **focus logic in debugSession.ts:** Lines 1382-1388: after `topCallStack` resolves, `focus()` calls `debugService.focusStackFrame(undefined, thread)`, which resolves the top stack frame (line 1511-1512 of `getStackFrameThreadAndSessionToFocus`) and fires `onDidFocusStackFrame`. This happens while the 50ms `onCallStackChangeScheduler` is still pending from the earlier `onDidChangeCallStack` triggered by `refreshTopOfCallstack`.
- **DAP sequence is correct:** The reporter's DAP trace (next→continued→stopped with threadId=3, preserveFocusHint=false, allThreadsContinued=false) is spec-compliant. The debug adapter's behavior is not the cause.
- **no fix on main:** git log --since=2025-01-01 on callStackView.ts shows no changes addressing this focus/selection race. The `selectionNeedsUpdate` mechanism has not been modified.
