# Issue #303979 — Reveal Active File in Explorer does nothing — TreeError [DebugRepl] Tree input not set

[https://github.com/microsoft/vscode/issues/303979](https://github.com/microsoft/vscode/issues/303979)

## Summary

Confirmed bug on current main: a typo in uriIdentityService.ts (line 63 compares a variable to itself) makes the casing-change handler dead code, and the workspace TernarySearchTree is never rebuilt when a remote filesystem provider registers with different casing rules. This causes getWorkspaceFolder() to return null for some roots in remote multi-root workspaces, breaking 'Reveal Active File in Explorer' and breadcrumbs. The original DebugRepl tree error was separately fixed in commit 23ba1f38512, but the underlying workspace casing bug remains.

## Change summary

'Reveal Active File in Explorer View' silently fails for some workspace roots in remote multi-root workspaces because the workspace folder map uses stale path-casing assumptions.

## Best solution

Fix the typo on line 63 of src/vs/platform/uriIdentity/common/uriIdentityService.ts (`oldIgnorePathCasingValue` instead of `newIgnorePathCasingValue`), and ensure the Workspace class's TernarySearchTree is rebuilt when the filesystem provider's path-casing capability changes — likely by firing an event from UriIdentityService that Workspace listens to in order to re-insert its folders.
