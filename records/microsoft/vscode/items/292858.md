# Issue #292858 — Colour picker ranges are not tracked correctly on document edits, resulting in corrupt edits

[https://github.com/microsoft/vscode/issues/292858](https://github.com/microsoft/vscode/issues/292858)

## Summary

Confirmed on current main: ColorDetector stores the original provider-returned range in _colorDatas but never updates it when the editor tracks decoration movements, so the color picker edits a stale range after intervening document changes.

## Change summary

When a document is edited while a color provider request is pending, the color picker uses the stale original range instead of the tracked decoration range, corrupting the document.

## Best solution

In `getColorData` (or in the hover participant's `_computeAsync`), use the decoration's tracked range (`d.range`) rather than the stored `colorInfo.range` from the original provider response. This ensures the range passed to `createColorHover` reflects edits made since the last provider response.
