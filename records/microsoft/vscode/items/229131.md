# Issue #229131 — When I use the search function, I get stuck

[https://github.com/microsoft/vscode/issues/229131](https://github.com/microsoft/vscode/issues/229131)

## Summary

The reporter's lag during Find in Files was caused by ripgrep's default thread behavior consuming excessive CPU. The existing `search.ripgrep.maxThreads` setting (default 0 = automatic) is intentionally designed this way — a maintainer explicitly explained the trade-off in comment 11. The reporter confirmed in comment 12 that adjusting the setting resolved their issue.

## Change summary

Find in Files (Ctrl+Shift+F) causes system-wide lag because ripgrep spawns too many threads on certain hardware.

## Best solution

Close as designed. The `search.ripgrep.maxThreads` setting already exists to let users control this trade-off. Changing the default is a separate product decision that should be tracked as a feature request if desired — comment 10 argues for a non-zero default, which could be filed as a distinct issue.

  > Thanks for the report and the follow-up. The lag you experienced is caused by ripgrep using many threads by default, which on some systems saturates CPU and causes noticeable stalls. This is controlled by the existing `search.ripgrep.maxThreads` setting — setting it to a non-zero value (e.g. `1` or `4`) caps the thread count and resolves the lag, as you confirmed.
  > 
  > The current default of `0` (automatic) is intentional — it lets ripgrep choose its thread count, which works well on most configurations but can be too aggressive on others. Since the setting exists and addresses this, closing as designed.
  >
