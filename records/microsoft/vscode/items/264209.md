# Issue #264209 — [Bug] Git Worktree menu does not appear in command palette with git.detectWorktrees enabled over SSH

[https://github.com/microsoft/vscode/issues/264209](https://github.com/microsoft/vscode/issues/264209)

## Summary

Still reproduces on main: when opening a worktree directory directly (especially over SSH), worktree detection skips discovery of the parent repo and sibling worktrees, so worktree commands never appear. An open PR (#309321) addresses this — keeping open for that PR to land.

## Change summary

Git Worktree commands do not appear in the command palette when opening a worktree directory over SSH with `git.detectWorktrees` enabled.

## Best solution

Land the open PR https://github.com/microsoft/vscode/pull/309321 which adds worktree discovery when opening a worktree directory directly, or implement equivalent logic in `extensions/git/src/model.ts` to resolve the parent repository from a worktree's `.git` file and discover sibling worktrees.

## Evidence

- **source trace:** In `extensions/git/src/model.ts` lines 806-828, `checkForWorktrees()` skips detection when `repository.kind === 'worktree'` (line 812-815). When a user opens a worktree directory directly, the repository is discovered as kind 'worktree', so sibling worktrees and the parent repo are never scanned — worktree commands never register.
- **open PR:** https://github.com/microsoft/vscode/pull/309321 ('git: discover worktrees when opening a worktree directory') is open and directly references this issue with fixing intent.
- **relative path fix:** Commit a99073234c6 (2026-06-15) fixed relative git worktree path parsing in `extensions/git/src/git.ts`, but did not address the discovery-when-opening-a-worktree gap.
- **community confirmation:** A second user confirmed the same behavior on Windows 11 + Ubuntu 24.04 via SSH with git v2.43.
