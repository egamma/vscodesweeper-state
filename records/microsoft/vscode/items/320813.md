# Issue #320813 — GitHub sign-in lockout: FetcherService treats 401 as terminal with no engine fallback when the transport drops the Authorization header

[https://github.com/microsoft/vscode/issues/320813](https://github.com/microsoft/vscode/issues/320813)

## Summary

The defect is confirmed in current source: `shouldNotRetry` in `extensions/github-authentication/src/node/fetch.ts` treats all 401s as terminal, preventing engine fallback even when the response body indicates the Authorization header was never received (a transport-level failure, not a credential failure). The issue is well-substantiated and still present on main.

## Change summary

FetcherService treats HTTP 401 as a terminal status that short-circuits the multi-engine fallback, so a transport that drops the Authorization header causes a permanent sign-in lockout with no recovery path other than a cold restart.

## Best solution

Refine the `shouldNotRetry` logic in `extensions/github-authentication/src/node/fetch.ts` to distinguish transport-level header loss (GitHub responds 'Requires authentication') from genuine credential failures (GitHub responds 'Bad credentials'). When the 401 body indicates credentials were absent rather than invalid, allow fallback to the next fetch engine.
