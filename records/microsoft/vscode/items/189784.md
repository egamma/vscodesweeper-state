# Issue #189784 — [Accessibility] problems.decorations.enabled setting must equally benefit screen reader users

[https://github.com/microsoft/vscode/issues/189784](https://github.com/microsoft/vscode/issues/189784)

## Summary

Still reproduces on main. The merged PR #189856 set an aria-label on the iconLabel DOM node, but the tree container's aria-label (what screen readers actually read) still comes from `getAriaLabel()` which returns only `element.name` — decoration info (error/warning counts) never reaches the screen reader. The 'On Deck' milestone also enforces keep-open.

## Change summary

Screen reader users cannot hear problem decoration info (error/warning counts) on files and folders in the Explorer view because the tree row's aria-label lacks that information.

## Best solution

Update `ExplorerDelegate.getAriaLabel()` in `src/vs/workbench/contrib/files/browser/views/explorerViewer.ts` (or the tree's aria-label pipeline) to include decoration data (error/warning counts) from the decoration service, so the container element read by screen readers conveys the same problem info sighted users see. The challenge noted in the discussion is that decorations update independently of tree re-renders — an observable-based `getAriaLabel` (already supported by `abstractTree.ts` line 1857) could solve this without full tree re-renders.
