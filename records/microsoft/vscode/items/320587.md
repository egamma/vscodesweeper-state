# Issue #320587 — Git scheme URI fails to locate correct repo after splitting submodule

[https://github.com/microsoft/vscode/issues/320587](https://github.com/microsoft/vscode/issues/320587)

## Summary

Legitimate bug: `getOpenRepository` resolves repos by path containment only, so when a nested independent repo (non-submodule) shares a path prefix with its former parent, git-scheme URIs for historical refs resolve to the wrong repository. No fix found on main.

## Change summary

Git-scheme URIs fail to locate the correct repository when a directory has been split into a standalone nested repo (non-submodule) because `getOpenRepository` only checks path ancestry and submodule exclusion, not ref validity.

## Best solution

Add ref-aware repository resolution for git-scheme URIs in `extensions/git/src/model.ts`. The cleanest approach is likely option 3 (add an optional `repoPath` to `GitUriOptions`) or a new async variant that validates ref membership via `git ls-files` — the design choice between sync/async tradeoffs needs maintainer input.
