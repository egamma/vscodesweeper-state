# Issue #321834 — MCP OAuth: VS Code does not evict a stale dynamically-registered client / expired token on HTTP 401 (remote server behind APISIX + Keycloak)

[https://github.com/microsoft/vscode/issues/321834](https://github.com/microsoft/vscode/issues/321834)

## Summary

The 401 retry with forceNewRegistration (commit 50e2e985b5c, Oct 2025) already exists on main and should cover this scenario, but the reporter reproduces it on Insiders 1.125.0 (June 2026) with a specific APISIX+Keycloak gateway setup. There may be a remaining edge case — possibly the SSE reconnect loop or a timing issue where the retry path doesn't fully clear the stale token. Worth routing to the authentication/MCP area for investigation.

## Change summary

VS Code does not automatically evict a stale dynamically-registered OAuth client or expired token when an MCP server returns HTTP 401, requiring manual cache clearing.

## Best solution

Investigate whether the `_fetchWithAuthRetry` → `forceNewRegistration: true` path (extHostMcp.ts:830) actually fires in this APISIX+Keycloak scenario. The reporter's logs suggest the retry may not trigger (or the re-registration may silently fail). Check whether the SSE GET reconnect loop properly surfaces 401s through `_fetchWithAuthRetry`, and whether `$getTokenFromServerMetadata` with forceNewRegistration actually evicts and re-creates a working session for pre-provisioned DCR clients.
