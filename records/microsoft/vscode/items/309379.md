# Issue #309379 — `remote.extensionKind` ignored from workspace/.vscode/settings.json and devcontainer Machine settings (only works from user settings)

[https://github.com/microsoft/vscode/issues/309379](https://github.com/microsoft/vscode/issues/309379)

## Summary

Confirmed in source: `remote.extensionKind` is read via `configurationService.getValue()` during `initializeRunningLocation()` before remote/workspace configuration sources are loaded, and the result is cached without invalidation — so only user settings take effect. Well-analyzed, real bug with clear code path.

## Change summary

`remote.extensionKind` from workspace settings and devcontainer Machine settings is ignored because extension kind resolution occurs before those configuration sources are available.

## Best solution

Either ensure remote configuration sources are loaded before `initializeRunningLocation()` calls `getValue('remote.extensionKind')`, or invalidate `_configuredExtensionKindsMap` on configuration changes and re-evaluate extension running locations when workspace/machine settings arrive. The fix touches `src/vs/workbench/services/extensions/common/extensionManifestPropertiesService.ts` (cache invalidation) and potentially `src/vs/workbench/services/extensions/common/abstractExtensionService.ts` (sequencing).
