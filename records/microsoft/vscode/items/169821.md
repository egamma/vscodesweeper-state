# Issue #169821 — VS Code is still adding `/d /c` to task commands

[https://github.com/microsoft/vscode/issues/169821](https://github.com/microsoft/vscode/issues/169821)

## Summary

Still reproduces on current main: when the automation profile specifies a non-cmd shell (e.g. msys2_shell.cmd, nu.exe with explicit args), the task system still appends `/d /c` because `shellSpecified` is only set from the task-level shell config, not from the automation profile. A community fix (PR #251201) was merged but then reverted (commit 3b5110391df). The team is actively investigating (anthonykim1 asked for testing on insiders 2025-07-30).

## Change summary

Task commands get unwanted `/d /c` arguments appended when using a non-cmd automation profile on Windows, making non-cmd shells like MSYS2, nushell, or pwsh unusable as automation profiles.

## Best solution

The `shellSpecified` logic in `terminalTaskSystem.ts` (~line 1251) needs to account for automation-profile-provided args: when the automation profile already supplies run-command args, skip appending default cmd.exe args. The fix should live in `src/vs/workbench/contrib/tasks/browser/terminalTaskSystem.ts` around lines 1251–1302. A previous attempt (PR #251201) was reverted due to regressions — the next attempt needs more conservative logic that only suppresses defaults when the profile explicitly provides args.
