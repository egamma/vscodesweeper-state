# Issue #307646 — [IME][Chromium 147] deleteContentBackward fired before compositionstart deletes text during CJK input

[https://github.com/microsoft/vscode/issues/307646](https://github.com/microsoft/vscode/issues/307646)

## Summary

This is an upstream Chromium 147 regression where the browser fires a spurious deleteContentBackward before compositionstart during CJK IME input. VS Code's editor does not explicitly guard against this new event ordering. The fix belongs in Chromium, though VS Code may need a workaround.

## Change summary

CJK IME input deletes the character before the cursor when composition starts, due to Chromium 147 firing deleteContentBackward before compositionstart.

## Best solution

File/track an upstream Chromium bug for the incorrect deleteContentBackward-before-compositionstart ordering in Chromium 147. In parallel, VS Code may need a defensive workaround in the edit context input handling (e.g., suppressing deleteContentBackward events immediately preceding a compositionstart) if the Chromium fix is slow to ship.

## Evidence

- **upstream event ordering:** The reporter's event trace clearly shows Chromium 147 fires [beforeinput deleteContentBackward] → [input deleteContentBackward] → [compositionstart], while Chrome 146 fires [compositionstart] directly. This is a Chromium-level change in IME event dispatching.
- **VS Code editor code:** src/vs/editor/browser/controller/editContext/native/nativeEditContext.ts handles beforeinput (line 157) only for insertParagraph/insertLineBreak; deleteContentBackward events flow through unguarded. The compositionstart handler (line 221) updates composition state, but by then the deletion has already occurred.
- **browser version specificity:** Only reproduces on Chromium 147 (Edge 147 Beta), not Chrome 146 stable — confirms this is a browser regression, not a VS Code change.
