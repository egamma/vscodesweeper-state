# Issue #322337 — Git: `isRepositoryHuge` flag set during transient clone/checkout permanently freezes stale git status with no recovery short of restart

[https://github.com/microsoft/vscode/issues/322337](https://github.com/microsoft/vscode/issues/322337)

## Summary

The core concern is partially valid: the `isRepositoryHuge` guard in `onFileChange()` blocks automatic file-watcher-triggered refreshes once the limit is hit, and a transient clone spike can trigger this permanently. However, the reporter's claim that manual Refresh is also blocked appears incorrect — `refresh()` calls `getStatus()` which re-evaluates and can reset the flag. The real bug is that auto-refresh has no recovery path without manual intervention after a transient spike.

## Change summary

A transient `git status` spike during re-clone permanently sets `isRepositoryHuge`, blocking automatic file-watcher-based git status refreshes with no self-recovery until manual refresh or window reload.

## Best solution

Add a recovery mechanism in `onFileChange()` so that `isRepositoryHuge` is periodically re-evaluated (e.g., after a cooldown timer, or on the next file-watcher event after a delay), rather than permanently blocking auto-refresh. Alternatively, when `isRepositoryHuge` is set, schedule a single deferred re-check of `git status` to see if the transient state has resolved. The fix would be localized to `extensions/git/src/repository.ts` around the `onFileChange` method and `isRepositoryHuge` flag handling.
