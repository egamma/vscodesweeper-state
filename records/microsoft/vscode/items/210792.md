# Issue #210792 — Persistent error : Could not find pty on pty host

[https://github.com/microsoft/vscode/issues/210792](https://github.com/microsoft/vscode/issues/210792)

## Summary

Still a valid open bug — the `_throwIfNoPty` code path in ptyService.ts (line 672) is unchanged, and the underlying race condition where operations arrive for already-disposed ptys during Remote SSH sessions has not been addressed. Multiple users across different versions continue to report it. Normally `reproductionStatus: unclear` would mandate `needs-info`, but the `confirmation-pending` label already serves that purpose and the team is aware; keeping open for the area owner.

## Change summary

Remote SSH sessions persistently error with 'Could not find pty on pty host' when operations (updateTitle, updateIcon) target a pty ID that has already been removed from the pty map, causing disconnects and extension activation failures.

## Best solution

The root cause is a lifecycle mismatch: the pty is removed from `PtyService._ptys` before all pending RPC calls (updateTitle, updateIcon, etc.) complete. A fix in `src/vs/platform/terminal/node/ptyService.ts` should either guard `_throwIfNoPty` callers against already-disposed ptys (downgrade to a warning instead of throwing) or ensure the pty entry stays in the map until all in-flight operations drain. The `confirmation-pending` label should be refreshed — the issue has 19 upvotes and multiple independent confirmations spanning versions 1.88–1.95+.
