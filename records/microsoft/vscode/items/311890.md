# Issue #311890 — state.vscdb stores deleted model / cannot clean custom models

[https://github.com/microsoft/vscode/issues/311890](https://github.com/microsoft/vscode/issues/311890)

## Summary

Valid bug: deleting a BYOK provider group (e.g. Ollama) via Manage Models does not clean up the stale model entries cached in the model picker's application-scoped storage (`CachedLanguageModelsKey` in `state.vscdb`). The `mergeModelsWithCache` eviction logic depends on the vendor having re-resolved with zero live models, but when the vendor extension is still contributed and its provider hasn't re-resolved after the group deletion, stale cached models survive and keep appearing in the Auto dropdown. Confirmed on 1.117 by a second reporter.

## Change summary

Deleted custom BYOK models (e.g. Ollama) still appear in the model picker dropdown because their cached metadata in `state.vscdb` is not cleaned up when the provider group is removed.

## Best solution

When `removeLanguageModelsProviderGroup` completes and the vendor's live model list changes, the cached model list stored under the `CachedLanguageModelsKey` storage key should be actively purged of models belonging to deleted groups. The fix likely belongs in `chatInputPart.ts` (the `getModels` method around line 1880) or in the `mergeModelsWithCache` logic in `chatModelSelectionLogic.ts`, ensuring that after a group deletion the cache is re-persisted without the stale entries.
