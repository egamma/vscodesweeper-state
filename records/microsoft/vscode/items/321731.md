# Issue #321731 — vsce-sign crashes with SIGILL verifying Pylance on Ubuntu 26.04 aarch64 in Lima on Apple M4

[https://github.com/microsoft/vscode/issues/321731](https://github.com/microsoft/vscode/issues/321731)

## Summary

The SIGILL crash is in the @vscode/vsce-sign pre-compiled native binary, not VS Code's own code. VS Code's extensionSignatureVerificationService.ts merely loads the module; the fix belongs in the vsce-sign package's native build (likely incorrect ARM instruction selection when Ubuntu 26.04's kernel exposes SME/SME2 CPU flags in Lima on Apple M4).

## Change summary

Extension signature verification crashes with SIGILL on Ubuntu 26.04 aarch64 in Lima on Apple M4 due to the vsce-sign binary hitting an illegal ARM instruction.

## Best solution

File and resolve this in the @vscode/vsce-sign repository — the native binary or its crypto dependency likely needs to avoid selecting SME instructions when they are advertised by the kernel but not fully emulated by Lima/QEMU. The VS Code side could improve error handling to surface a clearer message when the child process exits with signal 4 (SIGILL), but the real fix is upstream in vsce-sign's build.

## Evidence

- **upstream binary:** src/vs/platform/extensionManagement/node/extensionSignatureVerificationService.ts loads '@vscode/vsce-sign' as a dynamic module; the crash (exit code 132 = SIGILL) occurs inside that native binary, not in VS Code's TypeScript.
- **platform-specific:** Fails only on Ubuntu 26.04 (kernel 7.0 exposing SME/SME2 CPU flags) under Lima on Apple M4; same binary works on Ubuntu 24.04 (kernel 6.8) on the same host — pointing to a CPU feature detection issue in the native code.
- **workaround exists:** Setting remote.downloadExtensionsLocally: true avoids the crash by skipping remote-side signature verification.
