# Issue #306246 — Reisizing is slow on Wayland / COSMIC tiled layouts

[https://github.com/microsoft/vscode/issues/306246](https://github.com/microsoft/vscode/issues/306246)

## Summary

Legitimate resize performance report on Wayland/COSMIC tiling compositor with a performance trace showing heavy JS execution (~94% of frame time) during resize events, dominated by getBoundingClientRect calls. Needs routing to layout and linux areas for investigation.

## Change summary

Resizing VS Code in a Wayland tiled layout causes visible stutter due to expensive JS layout recalculations on every resize event.

## Best solution

Investigate whether VS Code's workbench layout code (particularly getBoundingClientRect-heavy paths during resize) can be throttled or batched more aggressively. The trace points at `workbench.desktop.main.js` resize event handlers taking 9–14ms per frame. The fix likely lives in `src/vs/workbench/browser/layout.ts` or the resize observer infrastructure in `src/vs/base/browser/dom.ts`. May also have an upstream Electron/Wayland component — worth checking if Electron's resize event frequency on Wayland compositors is configurable.
