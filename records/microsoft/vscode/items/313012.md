# Issue #313012 — vscode-server cli 'code' tool does not report an error when called outside terminal

[https://github.com/microsoft/vscode/issues/313012](https://github.com/microsoft/vscode/issues/313012)

## Summary

Keep open and route as a real remote/server CLI bug: current main still prints the reported message and returns normally when neither the VS Code terminal pipe nor WSL client command environment is present.

## Change summary

The remote server `code --status` CLI reports misuse outside WSL or a VS Code terminal but exits with status 0, making scripts unable to detect failure.

## Best solution

Change the unsupported-environment path in `src/vs/server/node/server.cli.ts` to return a nonzero process exit code, and add focused CLI coverage for the no-`VSCODE_IPC_HOOK_CLI`/no-`VSCODE_CLIENT_COMMAND` case.

## Evidence

- **current code:** `src/vs/server/node/server.cli.ts` lines 92-95 still handle missing `VSCODE_IPC_HOOK_CLI` and `VSCODE_CLIENT_COMMAND` by printing `Command is only available in WSL or inside a Visual Studio Code terminal.` and then `return`ing from `main`, which resolves successfully.
- **exit behavior:** The entry point calls `main(...).then(null, err => { console.error(...) })`; the unsupported-environment `return` does not throw or call `process.exit(1)`, so Node exits with status 0.
- **wrapper behavior:** `resources/server/bin/remote-cli/code-linux.sh` and `code-darwin.sh` invoke `node out/server-cli.js ...` as the final command, so the shell wrapper exposes the Node process exit status.
- **no later fix found:** `git log --since='2026-04-28' -- src/vs/server/node/server.cli.ts resources/server/bin/remote-cli/...` shows no changes after the issue was filed that would address this path.
