# Issue #179146 — VSCode extension API's - some cases when "explorer/context" does not solve "resource" correctly in "when" clause

[https://github.com/microsoft/vscode/issues/179146](https://github.com/microsoft/vscode/issues/179146)

## Summary

Valid bug confirmed by a second user: explorer context menu `when` clauses using `resourceExtname` fail to match after a non-explorer view (e.g. Settings) takes focus. The explorer's `onContextMenu` handler does call `setContextKeys(stat)` before showing the menu, so the root cause is not immediately obvious from source inspection — it may be a subtle context-key-service scoping or focus-sequencing issue. No fix found in git history; keeping open.

## Change summary

Extension-contributed explorer context menu items gated on `resourceExtname` (or similar `resource*` context keys) disappear when the user right-clicks a file without first left-clicking the explorer after another view (e.g. Settings) had focus.

## Best solution

Investigate the context key service scope chain used when `contextMenuService.showContextMenu` evaluates `when` clauses — the `ResourceContextKey` is set on the explorer view's scoped CKS, and the tree's CKS (passed as `contextKeyService` to the menu) should inherit it, but something in the focus/scope resolution path appears to break this. Likely files: `src/vs/workbench/contrib/files/browser/views/explorerView.ts` (context menu handler + `setContextKeys`), `src/vs/workbench/common/contextkeys.ts` (`ResourceContextKey`), and possibly `src/vs/platform/contextview/browser/contextMenuService.ts` (CKS resolution in `showContextMenu`).
