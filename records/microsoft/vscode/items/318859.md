# Issue #318859 — Don’t collapse 3rd-party stack frames (or at least not as aggressively)

[https://github.com/microsoft/vscode/issues/318859](https://github.com/microsoft/vscode/issues/318859)

## Summary

Valid feature request for a setting to disable or reduce auto-collapsing of deemphasized/subtle stack frames in the call stack view. One sub-request (remembering expansion state across stops) was fixed in April 2025 via commit 4763ea75d68, but the primary ask — a setting to opt out of collapsing entirely — remains unaddressed on current main.

## Change summary

Request a setting to disable automatic collapsing of 3rd-party (deemphasized/subtle) stack frames in the debug call stack, plus smarter heuristics about when collapsing is useful.

## Best solution

Add a debug setting (e.g. `debug.callStack.collapseDeemphasizedFrames`) in `src/vs/workbench/contrib/debug/browser/callStackView.ts` that controls whether deemphasized/subtle frames are auto-collapsed, defaulting to the current behavior. The reporter's second bullet (expansion state lost across stops) was already fixed in https://github.com/microsoft/vscode/issues/241661 — the reporter may not be aware of this fix (shipped in VS Code 1.102+). A comment noting the partial fix would help.

## Evidence

- **original collapsing behavior:** Introduced in commit 72250a87006 (March 2024, PR https://github.com/microsoft/vscode/issues/209078), which made 'subtle' frames behave like 'deemphasized' per DAP spec.
- **re-collapse fix shipped:** Commit 4763ea75d68 (April 2025, PR https://github.com/microsoft/vscode/pull/245739) fixed https://github.com/microsoft/vscode/issues/241661 — expanded deemphasized frames no longer re-collapse on the next stop, by switching from a cleared array to a WeakSet.
- **no opt-out setting exists:** Current main has no configuration to disable auto-collapsing of deemphasized frames. The collapsing logic in callStackView.ts (lines 991–1016) and isFrameDeemphasized in debug.ts (line 577) are unconditional.
- **KEEP_SUBTLE_FRAME_AT_TOP:** debugModel.ts line 596 already avoids skipping the top frame for certain stop reasons (breakpoint, step, function breakpoint), partially addressing the reporter's 'never collapse the breakpoint frame' suggestion, but only for the top frame selection — the call stack view still collapses them visually.
