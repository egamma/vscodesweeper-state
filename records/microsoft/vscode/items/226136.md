# Issue #226136 — Git - wrong handling of .gitignore that contains the folder name

[https://github.com/microsoft/vscode/issues/226136](https://github.com/microsoft/vscode/issues/226136)

## Summary

Confirmed bug in IgnoreFile constructor: the trailing-slash normalization mutates only the local parameter, not the parameter property, causing .gitignore patterns to match the parent directory itself. An open PR (https://github.com/microsoft/vscode/pull/313021) already addresses this — keeping open pending that PR's review and merge.

## Change summary

When a .gitignore file inside a folder contains an entry matching the folder's own name, VS Code incorrectly ignores the entire folder instead of only files matching that name within it.

## Best solution

Land https://github.com/microsoft/vscode/pull/313021, which fixes the parameter-property normalization bug in `src/vs/workbench/services/search/common/ignoreFile.ts` so `this.location` receives the trailing-slash-normalized path. Review the PR for correctness and test coverage.

## Evidence

- **source trace:** In `src/vs/workbench/services/search/common/ignoreFile.ts` line 15, `location` is a parameter property (`private readonly location: string`), so `this.location` is assigned before the constructor body. Line 22 `location += '/'` only mutates the local copy; `this.location` on line 24 still lacks the trailing slash.
- **consequence:** Without the trailing slash, `'/test'.startsWith('/test')` is true, so the matcher tests the parent directory itself against glob `**/test`, incorrectly marking the whole folder as ignored.
