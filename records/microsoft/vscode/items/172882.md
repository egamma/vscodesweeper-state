# Issue #172882 — Replace in files doesn't work as expected with match containing newline

[https://github.com/microsoft/vscode/issues/172882](https://github.com/microsoft/vscode/issues/172882)

## Summary

Still reproduces on main: when a regex lookahead references content on lines beyond the match, the replacement falls back to the literal pattern string (e.g. `$1`) because ripgrep's preview text doesn't include enough context for the JS regex to re-match and perform capture-group substitution.

## Change summary

Replace in Files outputs the literal `$1` instead of the captured group value when the regex match includes a lookahead spanning across lines.

## Best solution

The fallback at line 121 of `match.ts` (`return searchModel.replacePattern.pattern`) returns the raw pattern without substitution. A fix should either (a) fetch additional file content beyond ripgrep's preview lines so the lookahead can succeed on re-match, or (b) implement manual `$N` capture-group substitution in the fallback path without relying on `String.replace`. Both approaches touch `src/vs/workbench/contrib/search/browser/searchTreeModel/match.ts` and potentially `src/vs/workbench/services/search/common/replace.ts`.

## Evidence

- **source trace:** In `src/vs/workbench/contrib/search/browser/searchTreeModel/match.ts` lines 83–122, the `replaceString` getter tries 4 variations of the preview text against `getReplaceString`. When all fail (lookahead context missing), it falls through to line 121: `return searchModel.replacePattern.pattern`, which returns the literal `$1`.
- **root cause:** `_fullPreviewLines` comes from `rawMatch.previewText` (ripgrep's `data.lines` via `ripgrepTextSearchEngine.ts` line 303). Ripgrep reports only the line(s) containing the actual match, not lines referenced solely by a lookahead. So `fullMatchText(true)` still lacks the `}` needed by the lookahead `(?=[\S\s\n]*?\})`.
