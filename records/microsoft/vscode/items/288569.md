# Issue #288569 — Disabling Search on Type still allows search results to be updated when editing files containing results

[https://github.com/microsoft/vscode/issues/288569](https://github.com/microsoft/vscode/issues/288569)

## Summary

The "Search on Type" setting (`search.searchOnType`) controls whether typing in the search input box automatically triggers a new search — it does not govern live updating of already-displayed results when editing matched files. The live result updating is a separate, intentional feature implemented via `FileMatchImpl.bindModel()`, which subscribes to `onDidChangeContent` on matched file models regardless of the searchOnType setting. This is as-designed.

## Change summary

Disabling 'Search on Type' does not prevent search results from updating live when editing files that already appear in results.

## Best solution

Close as as-designed. The `search.searchOnType` setting only controls the search-input-to-query pipeline. If there is user demand for a separate setting to disable live result updating in matched files, that would be a distinct feature request.

## Evidence

- **searchOnType scope:** The setting description reads 'Search all files as you type' and the code at src/vs/workbench/contrib/search/browser/searchView.ts:1592 gates it: `if (options.triggeredOnType && !this.searchConfig.searchOnType) { return; }` — it only controls whether input-box typing triggers a new search query.
- **live update mechanism:** src/vs/workbench/contrib/search/browser/searchTreeModel/fileMatch.ts:135-143 — `bindModel()` subscribes to `model.onDidChangeContent()` and schedules `updateMatchesForModel()` on a 250ms debounce. This fires for any matched file regardless of the searchOnType setting.
- **separate features:** These are architecturally independent: searchOnType controls the search-input → query pipeline in searchView.ts, while live result updating operates in the model layer via fileMatch.ts. Disabling one was never intended to disable the other.
