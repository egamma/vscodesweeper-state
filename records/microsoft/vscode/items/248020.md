# Issue #248020 — Missing dependency: libX11-xcb

[https://github.com/microsoft/vscode/issues/248020](https://github.com/microsoft/vscode/issues/248020)

## Summary

Valid packaging gap: `libX11-xcb` is not declared as a dependency in the RPM (or Debian) package metadata. The dep-lists on current main still lack it — `build/linux/rpm/dep-lists.ts` lists `libX11.so.6` and `libxcb.so.1` but not `libX11-xcb.so.1`. On minimal RHEL 8 installs where it is a separate package and not transitively pulled in, VS Code exits silently.

## Change summary

VS Code's Linux RPM package does not declare a dependency on `libX11-xcb`, causing a silent launch failure on minimal RHEL installations where the library is not installed transitively.

## Best solution

Add `libX11-xcb.so.1()(64bit)` to `additionalDeps` in `build/linux/rpm/dep-lists.ts` (and the appropriate entry for `armv7hl`/`aarch64`). On the Debian side, consider adding `libx11-xcb1` to `build/linux/debian/dep-lists.ts`. The `additionalDeps` list exists exactly for runtime dependencies that the automated `find-requires` scanner misses (e.g. dlopen'd libs).

- **dependency scanner limitation:** `build/linux/rpm/calculate-deps.ts` uses `/usr/lib/rpm/find-requires` which reads ELF NEEDED entries. If Electron/Chromium loads `libX11-xcb.so.1` via dlopen at runtime, the scanner misses it — this is what the `additionalDeps` array is designed to cover.
