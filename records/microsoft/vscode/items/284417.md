# Issue #284417 — Copilot attempts writing files in windows format on linux remote host via ssh

[https://github.com/microsoft/vscode/issues/284417](https://github.com/microsoft/vscode/issues/284417)

## Summary

The path corruption bug in the Copilot extension's PromptPathRepresentationService is partially fixed (resolveFilePath direction, commit bafbf12c59b) but the core defect remains: getFilePath() still uses uri.fsPath for vscode-remote URIs (producing Windows backslashes on Windows clients), and tools call resolveFilePath without passing the remote scheme, defaulting to Schemas.file. Multiple fix PRs were closed, indicating the team is aware but hasn't landed a complete fix.

## Change summary

Copilot agent file tools fail in Remote-SSH workspaces from Windows because remote Linux paths are formatted with Windows backslashes, causing 'outside workspace' errors and silent write failures.

## Best solution

Fix getFilePath() in extensions/copilot/src/platform/prompts/common/promptPathRepresentationService.ts to use uri.path (not uri.fsPath) for vscode-remote scheme URIs, and ensure tools pass the correct predominantScheme when calling resolveFilePath in remote contexts. The multiple closed PRs (#311244, #311250, #311351, #311353, vscode-copilot-chat#5075) suggest this needs careful design to handle the round-trip correctly across all platforms.

## Evidence

- **defect in source:** extensions/copilot/src/platform/prompts/common/promptPathRepresentationService.ts line 52-62: getFilePath() returns uri.fsPath for vscodeRemote scheme, which on Windows produces backslash paths for remote Linux URIs.
- **tools missing scheme:** All tools (createFileTool, insertEditTool, etc.) call resolveFilePath() without passing predominantScheme, defaulting to Schemas.file — resolved URIs get file:// scheme instead of vscode-remote://, causing 'outside workspace' checks to fail.
- **partial fix:** Commit bafbf12c59b (2026-02-06) by Connor Peet added handling in resolveFilePath for windows-formatted unix paths and drive letter rectification, but didn't change getFilePath() or the tool callers.
- **closed fix attempts:** PRs https://github.com/microsoft/vscode/pull/311244, https://github.com/microsoft/vscode/pull/311250, https://github.com/microsoft/vscode/pull/311351, https://github.com/microsoft/vscode/pull/311353 all attempted to fix getFilePath path corruption but were closed without merge.
- **community confirmation:** Multiple users confirmed the issue in comments (Remote-WSL, Remote-SSH to Linux VMs from Windows hosts).
