# Issue #194568 — File nesting breaks when moving files

[https://github.com/microsoft/vscode/issues/194568](https://github.com/microsoft/vscode/issues/194568)

## Summary

Still reproduces on main: moving a nested file clears the parent's `nestedChildren` array via `removeChild`, but the tree does not recalculate nesting for the remaining siblings, so they visually disappear until reload. An open community PR exists but has not been merged.

## Change summary

Moving a nested file to a subdirectory causes remaining nested siblings to disappear from the explorer until VS Code is reloaded.

## Best solution

Fix `removeChild` in `src/vs/workbench/contrib/files/common/explorerModel.ts` to properly recalculate nesting for the parent after removing a nested child, rather than simply clearing `nestedChildren` to `undefined`. The open PR https://github.com/microsoft/vscode/pull/198645 targets the right area but has stalled — a maintainer should review or supersede it with a proper fix that triggers nesting recalculation after a move.

## Evidence

- **bug in source:** src/vs/workbench/contrib/files/common/explorerModel.ts line 403-406: `removeChild` sets `this.nestedChildren = undefined`, which clears ALL nested children rather than just removing the moved one. The tree then serves the stale empty state on the fast path at line 316-317 because `nestedChildren` is now `undefined` and no re-nesting is triggered.
- **move method:** Line 422-428: `move()` calls `this.nestedParent?.removeChild(this)` then `this._parent?.removeChild(this)`, both of which nuke `nestedChildren` on the parent without recalculating.
- **open PR:** https://github.com/microsoft/vscode/pull/198645 is an open community PR targeting this issue but has not been reviewed or merged since Nov 2023.
- **no fix on main:** git log on explorerModel.ts since Oct 2023 shows no nesting-related or move-related fix commits.
