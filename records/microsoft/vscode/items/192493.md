# Issue #192493 — Slow startup time on 1.82.0

[https://github.com/microsoft/vscode/issues/192493](https://github.com/microsoft/vscode/issues/192493)

## Summary

A well-investigated Linux startup regression from 1.82.0 with active team involvement (deepak1556). The storage-related half was resolved via the fix for #192742, but the remaining `window.loadUrl()` slowdown appears to be in Electron/Chromium's renderer process creation path and the investigation stalled because tracing itself masks the problem. Keeping open — the team member owns the investigation and several related issues remain open.

## Change summary

VS Code 1.82.0 on Linux shows ~4x slower startup than 1.81.1, with the remaining bottleneck in the Electron renderer process creation step (`window.loadUrl() => begin to require(workbench.desktop.main.js)`).

## Best solution

The remaining slowdown lives in Electron/Chromium's renderer startup path. deepak1556 should determine whether this was resolved by subsequent Electron upgrades (VS Code has moved through several Electron versions since 1.82). If it persists, file an upstream Electron/Chromium bug. Consider consolidating with https://github.com/microsoft/vscode/issues/193948 and https://github.com/microsoft/vscode/issues/196771 which report the same Linux startup regression from the same version.

- **investigation stalled:** deepak1556 (MEMBER) confirmed the trace shows no slowdown because tracing itself adds ~3s+ to startup, masking the issue (comments 15, 27). The heisenbug nature blocked further diagnosis.

- **Chromium root cause:** deepak1556's analysis (comment 19) pointed to the Chromium browser process not starting navigation for ~3s, with idle time before navigation — suggesting an I/O block or GPU process launch delay in the Electron/Chromium layer.
