# Issue #206854 — Debug console is very slow with a single long line

[https://github.com/microsoft/vscode/issues/206854](https://github.com/microsoft/vscode/issues/206854)

## Summary

Still reproducible on main: when debug output uses carriage return (`\r`) without newline, `appendOutputToRepl` concatenates every output event into a single unbounded `ReplOutputElement`, eventually freezing the UI. No fix has landed.

## Change summary

Debug console freezes when a script uses carriage return (`\r`) to overwrite the current line, because output accumulates in a single unbounded REPL element.

## Best solution

Handle `\r` in `appendOutputToRepl` as a line-overwrite (replace the last line segment instead of appending), and/or cap the maximum length of a single `ReplOutputElement` value. The fix is localized to `src/vs/workbench/contrib/debug/common/replModel.ts` around the incomplete-line concatenation logic (lines ~314–337).

## Evidence

- **unbounded concatenation:** In src/vs/workbench/contrib/debug/common/replModel.ts lines 314-337, when the previous element doesn't end with `\n` or `\r\n`, new output is concatenated without any length limit. With `\r`-terminated output (e.g. `print(..., end="\r")`), this grows a single string indefinitely.
- **no \r handling:** The code never interprets `\r` as a line-overwrite; it only checks for `\n` and `\r\n` as line terminators. Standard terminal behavior would replace the current line on `\r`.
- **maximumLines limit is per-element:** The `maximumLines` cap at line 480 limits the number of REPL elements, not the length of a single element's value string, so it doesn't help here.
- **async queue still serializes:** The `outputQueue` in debugSession.ts line 1167 serializes all output events through an async Queue, adding overhead per event as noted by maintainer roblourens in https://github.com/microsoft/vscode/issues/206854#issuecomment-2026903485.
