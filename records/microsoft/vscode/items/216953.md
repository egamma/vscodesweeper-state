# Issue #216953 — Fallback on "legacy" diff algorithm if "advanced" is too slow

[https://github.com/microsoft/vscode/issues/216953](https://github.com/microsoft/vscode/issues/216953)

## Summary

Still valid on main: when the advanced diff algorithm times out (`quitEarly: true`), the result is returned as-is with no fallback to legacy. The user sees no diff for large files. The recently adopted `@vscode/diff` external computer also has a TODO noting `maxComputationTimeMs` is not yet supported, so the timeout/fallback gap remains.

## Change summary

When the advanced diff algorithm times out on large files, VS Code shows no diff instead of falling back to the legacy algorithm.

## Best solution

In `WorkerBasedDocumentDiffProvider.computeDiff` (diffProviderFactoryService.ts), detect when `result.quitEarly` is true and the algorithm is 'advanced'/'advanced-external'/'advanced-wasm', then retry with the legacy algorithm before returning the empty result. This keeps the benefit of the advanced algorithm when it succeeds while providing a usable diff when it doesn't.
