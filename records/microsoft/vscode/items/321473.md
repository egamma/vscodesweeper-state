# Issue #321473 — Drag-and-drop from remote -> desktop creates a file with remote file URI

[https://github.com/microsoft/vscode/issues/321473](https://github.com/microsoft/vscode/issues/321473)

## Summary

Team-authored bug report: dragging a file from a remote explorer to the desktop creates an 'internet location' file instead of blocking the operation. The code in `fillEditorsDragData` correctly skips `DOWNLOAD_URL` for non-`file:` schemes, but still sets `DataTransfers.TEXT` with the remote URI label, which macOS interprets as a web location. Keeping open for maintainer judgment.

## Change summary

Dragging a file from a remote VS Code window to the desktop creates an internet location (.webloc) file containing the remote URI instead of blocking the drag entirely.

## Best solution

In `fillEditorsDragData` (src/vs/workbench/browser/dnd.ts) or in `onDragStart` (src/vs/workbench/contrib/files/browser/views/explorerViewer.ts), detect when all resources are remote (non-`file:` scheme) and either suppress the `DataTransfers.TEXT` data for remote URIs or block the external drag entirely, consistent with the Finder drag behavior that already blocks it.

## Evidence

- **drag data source:** src/vs/workbench/browser/dnd.ts lines 241-255: `fillEditorsDragData` sets `DataTransfers.TEXT` with the resource label for all resources, but only sets `DataTransfers.DOWNLOAD_URL` when the scheme is `file:`. For remote files, the text data (the remote URI path) gets interpreted by macOS as a URL, creating a .webloc file.
- **file filter in explorer:** src/vs/workbench/contrib/files/browser/views/explorerViewer.ts line 1776: the explorer's `onDragStart` only sets `CodeDataTransfers.FILES` for `Schemas.file`, but does not prevent the drag or suppress the text transfer for remote resources.
