# Issue #211163 — Expanding a verbose hover can end up with a non expandable hover

[https://github.com/microsoft/vscode/issues/211163](https://github.com/microsoft/vscode/issues/211163)

## Summary

The issue describes a UX gap in the verbose hover expansion API: if an expanded hover doesn't declare canDecreaseVerbosity, the user is stuck. No fix has been committed; the discussion about whether VS Code or extensions should own the collapse history is unresolved.

## Change summary

Expanding a verbose hover to one that lacks canDecreaseVerbosity leaves the user unable to collapse back to the previous hover.

## Best solution

Either VS Code should maintain a hover history stack (so collapsing always reverts to the prior state regardless of the new hover's flags), or the API contract should require extensions that provide expandable hovers to always set canDecreaseVerbosity on expanded results. The discussion in the comments raises valid consistency concerns with caching — a maintainer decision is needed on which component owns the invariant.
