## Summary

The git extension can enter a high-frequency loop when workspace state references a stale folder path after an external rename, and the .git folder is then deleted. The file system watcher keeps firing, triggering repeated openRepository calls that fail with ENOENT, with no circuit breaker to stop retries. Real bug, still present on main.

## Change summary

After renaming a workspace folder externally and deleting its .git folder, the built-in Git extension enters an infinite loop running `git rev-parse --show-toplevel` and logging ENOENT errors.

## Best solution

Add a failure cache or retry limit in the `openRepository` / `eventuallyScanPossibleGitRepositories` path so that repeatedly failing paths (ENOENT from `getRepositoryExact` or failed `getRepositoryRoot`) are suppressed after a small number of consecutive failures.
