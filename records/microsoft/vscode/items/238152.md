# Issue #238152 — Linux Rocky 9.4, new instances of code open in same (gnome) workspace

[https://github.com/microsoft/vscode/issues/238152](https://github.com/microsoft/vscode/issues/238152)

## Summary

Confirmed by two users on RHEL 9/Rocky 9.4 with GNOME — new VS Code windows pull focus to the first instance's GNOME workspace instead of opening in the current workspace. A previous fix (b9947f2a379) guarded `app.focus({ steal: true })` to macOS-only, and that guard is still in place on current main, so the regression likely comes from a different focus path or from Electron/GNOME interaction changes. Routing to window management and Linux area owners.

## Change summary

On Linux with GNOME workspaces, opening a new VS Code instance switches the user to the GNOME workspace of the first VS Code instance instead of staying in the current workspace.

## Best solution

Investigate the focus calls in `src/vs/platform/windows/electron-main/windowsMainService.ts` — the many `.focus()` calls (default `FocusMode.Transfer`) when opening/reusing windows may trigger GNOME workspace switches. The fix may involve suppressing or deferring focus when the target window is on a different virtual desktop, or filing an Electron upstream issue if `BrowserWindow.focus()` itself causes cross-workspace activation on modern GNOME/Wayland.
