# Issue #307349 — Chat session history entry silently unresponsive with no recovery path when session fails to deserialize

[https://github.com/microsoft/vscode/issues/307349](https://github.com/microsoft/vscode/issues/307349)

## Summary

The described behavior is confirmed in the source: `readSessionFromLocation` returns `undefined` on parse failure (line 692) with only a log-level error and telemetry event, no user-visible notification or partial recovery. The requests for partial recovery, user-facing error surfacing, and in-UI deletion of broken entries are valid improvements but represent new functionality, not a regression from defined behavior.

## Change summary

When a chat session file is malformed or corrupt, the session appears in history but silently fails to load with no user-visible error, no partial recovery, and no way to delete the broken entry.

## Best solution

Add a user-visible notification or inline error state when `readSession` returns `undefined` for a session that exists in the index, and expose a 'Delete Session' action on broken entries. Partial recovery of sealed requests is a stretch goal. The work belongs in `src/vs/workbench/contrib/chat/common/model/chatSessionStore.ts` (detection) and the chat history UI (surfacing). Related to #307348 (specific crash trigger) and #296890 (write corruption).
