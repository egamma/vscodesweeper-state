# Issue #312745 — VSCode ignores "chat.tools.terminal.terminalProfile.windows" setting initially

[https://github.com/microsoft/vscode/issues/312745](https://github.com/microsoft/vscode/issues/312745)

## Summary

Plausible race condition: the model description for the run_in_terminal tool is computed asynchronously at startup via `createRunInTerminalToolData`, and a chat request issued immediately after restart may hit a window where the tool is not yet registered with the correct shell type from the custom profile. Already carries `confirmation-pending` — keeping open for team investigation.

## Change summary

When `chat.tools.terminal.terminalProfile.windows` is set to WSL, the first Chat terminal command after restart uses PowerShell syntax; subsequent commands correctly use Linux syntax.

## Best solution

Investigate the race between `ChatAgentToolsContribution._registerRunInTerminalTool()` (async registration at `WorkbenchPhase.AfterRestored`) and the first chat request. A possible fix is to ensure the tool is not available to the model until the async `createRunInTerminalToolData` resolves, or to emit an updated tool description once the profile is resolved. The relevant code is in `src/vs/workbench/contrib/terminalContrib/chatAgentTools/browser/terminal.chatAgentTools.contribution.ts`.

## Evidence

- **profile reading:** `getCopilotProfile()` in `runInTerminalTool.ts:3429` correctly checks the custom profile setting first, and `wsl.exe` would fall through to `createBashModelDescription` — so once resolved, the description is correct.
- **re-registration gap:** The `onDidChangeConfiguration` listener (lines 150-167) does not include `TerminalChatAgentToolsSettingId.TerminalProfileWindows`, so changing the profile after startup would not trigger a re-registration either — though the reporter's scenario is about restart, not live change.
