# Issue #152639 — Duplicate entries in edit breakpoint and related submenus

[https://github.com/microsoft/vscode/issues/152639](https://github.com/microsoft/vscode/issues/152639)

## Summary

Still reproducible on main: when a line breakpoint and an inline breakpoint resolve to the same verified column, the dedup logic doesn't catch the overlap because `setBreakpointSessionData` never calls `sortAndDeDup()`, and the initial dedup runs before verification data arrives (comparing `undefined` vs a specific column number).

## Change summary

Adding an inline breakpoint at the same position where a line breakpoint already resolved produces duplicate entries in the breakpoint context menus (Remove, Edit, Enable/Disable).

## Best solution

Either call `sortAndDeDup()` in `setBreakpointSessionData` after verified positions arrive, or guard `addBreakpoints` to no-op when a resolved breakpoint already occupies the target position (as connor4312 suggested in comment 4). The fix lives in `src/vs/workbench/contrib/debug/common/debugModel.ts`.

## Evidence

- **dedup key mismatch:** sortAndDeDup (debugModel.ts:1940) uses `bp.column` which returns the verified column via the getter. But it only runs in `addBreakpoints` and `updateBreakpoints` — before session/verification data is available — so a line breakpoint (column=undefined) and an inline breakpoint (column=28) produce different keys and survive dedup.
- **missing dedup call:** setBreakpointSessionData (debugModel.ts:1830-1885) updates verified positions from the debug adapter but does NOT call sortAndDeDup(), so post-verification duplicates are never removed.
- **context menu impact:** breakpointEditorContribution.ts:432-452 builds submenus from all breakpoints on the line; when duplicates exist in the model, the submenus show duplicate entries.
