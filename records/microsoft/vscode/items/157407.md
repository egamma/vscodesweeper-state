# Issue #157407 — Neo2 Layout Keybinding wrong key used

[https://github.com/microsoft/vscode/issues/157407](https://github.com/microsoft/vscode/issues/157407)

## Summary

Current main still has the same product tradeoff: `keyboard.dispatch=keyCode` dispatches only on the browser `keyCode`, while `code` dispatch does not broadly honor Neo2's virtual navigation-layer keys. This should stay open in the keyboard-layout/Linux area rather than be closed as fixed or duplicate.

## Change summary

Neo2 on Linux cannot bind the physical X and Ö positions separately while also keeping the Neo navigation layer working.

## Best solution

Investigate in `src/vs/workbench/services/keybinding/common/macLinuxKeyboardMapper.ts` and the Linux native keyboard layout path whether Neo2 navigation-layer virtual keys such as Escape can be handled in code dispatch, or whether the limitation must be documented as Chromium/native-keymap input data loss.

## Evidence

- **issue context:** Reporter provided concrete Linux/X11 Neo2 event data: Ö and x both arrive with keyCode=88 under Chromium, while Neo navigation Escape arrives as code=KeyZ, key=Esc, keyCode=27.
- **current keyCode path:** `src/vs/workbench/services/keybinding/electron-browser/nativeKeyboardLayout.ts` still returns `FallbackKeyboardMapper` when `keyboard.dispatch` is `keyCode`; `src/vs/workbench/services/keybinding/common/fallbackKeyboardMapper.ts` builds the dispatch chord from `keyboardEvent.keyCode` only, so two keys with keyCode=88 remain indistinguishable in that mode.
- **current code path:** `src/vs/workbench/services/keybinding/common/macLinuxKeyboardMapper.ts` resolves normal keyboard events from `keyboardEvent.code`, with keyCode-based overrides for arrows, Delete, Insert, Home, End, PageUp/PageDown, Backspace, and numpad keys; Escape is not in that override list.
- **history checked:** Commit e0baf3af8ebecc6dd55405d0a86beab870a8bc05 fixed browser keyboard layout detection fallback behavior, but it only touched `src/vs/workbench/services/keybinding/browser/keyboardLayoutService.ts` and does not address the Electron/Linux `keyCode` collision described here.
- **triage state:** The issue is author NONE, has no verification/candidate/important/security label, and is in the Backlog milestone, so no hard keep-open rule applies; the reason to keep it open is that the reported gap still appears actionable and unproven fixed.
