# Issue #164895 — GitHub authentication fails when running Code as admin

[https://github.com/microsoft/vscode/issues/164895](https://github.com/microsoft/vscode/issues/164895)

## Summary

Still valid on main: the GitHub authentication extension has no admin-detection logic — the URI handler flow is attempted even when VS Code is elevated (which will always fail on Windows), with no upfront warning. The `preferDeviceCodeFlow` setting (added in 2025) offers a manual workaround but doesn't address the core ask of proactive detection.

## Change summary

GitHub authentication via URI handler fails silently when VS Code runs as administrator on Windows, with no upfront warning or automatic fallback to device code flow.

## Best solution

Have the GitHub authentication extension check `INativeHostService.isAdmin()` (or an equivalent environment signal) and either skip the URI handler flow automatically or warn the user before attempting it, defaulting to device code flow when elevated. The fix would touch `extensions/github-authentication/src/flows.ts` (the `getFlows` function) and possibly `extensions/github-authentication/src/githubServer.ts`.

- **URI handler path unguarded:** extensions/github-authentication/src/flows.ts `getFlows()` does not check whether VS Code is elevated; the URI handler flow (which cannot receive callbacks from a non-elevated browser on Windows) is still attempted first.
- **workaround exists:** Commit 55cdeb57712 (Oct 2025) added `github-authentication.preferDeviceCodeFlow` setting, and the flow system has `retryFallbacks: true` so device code is offered after the first failure — but neither is proactive.
