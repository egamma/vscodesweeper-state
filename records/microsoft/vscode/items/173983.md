# Issue #173983 — Selecting text highlighting in large files silently drops highlighting of occurrences

[https://github.com/microsoft/vscode/issues/173983](https://github.com/microsoft/vscode/issues/173983)

## Summary

Keep this open as an editor highlighting feature request: the 1000-ish cap is intentional for performance, but the reported product gap is that selection highlights disappear without feedback and may omit matches in the visible area.

## Change summary

Improve selection occurrence highlighting in large files so capped results are either communicated to the user or prioritized more usefully around the current selection/viewport.

## Best solution

In `src/vs/editor/contrib/multicursor/browser/multicursor.ts`, decide whether selection highlighting should request a higher/explicit match limit, prioritize visible/current-region matches, or surface a lightweight capped-results indication while preserving the performance guard documented by the editor owner.

- **Proposed comment:**

  > Thanks for the report and follow-up context. The current implementation still goes through `TextModel.findMatches` without an explicit limit from the selection highlighter, so it inherits the default capped result count. Given the performance concern, I would keep this tracked as an editor-highlight feature request: either make capped selection highlights visible to the user, or change the highlighter to prioritize matches around the current selection/viewport without regressing large-file performance.

## Evidence

- **maintainer comment:** A VS Code member explained on 2023-02-25 that selection highlights are limited to 1000 for performance because they are implicit and can easily match very common text.
- **current code:** `src/vs/editor/contrib/multicursor/browser/multicursor.ts:833-836` still calls `model.findMatches(...)` for selection highlights without an explicit `limitResultCount`.
- **current code:** `src/vs/editor/common/model/textModel.ts:125` defines `LIMIT_FIND_COUNT = 999`, and `TextModel.findMatches` uses that as the default at `src/vs/editor/common/model/textModel.ts:1178`.
- **current behavior gap:** The selection highlighter filters and decorates the returned matches in `src/vs/editor/contrib/multicursor/browser/multicursor.ts:1010-1040`; there is no evidence in this path of user feedback when the default match cap truncates results.
- **classification:** The issue is currently labeled `feature-request` and `editor-highlight`, which matches treating the remaining work as a design/product improvement rather than a high-confidence close as completed or duplicate.
