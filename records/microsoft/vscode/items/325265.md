# Issue #325265 — Webview resources fail silently with ERR_INSUFFICIENT_RESOURCES when a webview loads thousands of files (Linux renderer keeps default 1024 nofile soft limit)

[https://github.com/microsoft/vscode/issues/325265](https://github.com/microsoft/vscode/issues/325265)

## Summary

Well-documented, independently confirmed bug: on Linux with default nofile soft limit (1024), webview resource loading exhausts file descriptors in the renderer process, causing silent ERR_INSUFFICIENT_RESOURCES failures. No fix or throttling exists on current main. The loadResource method in webviewElement.ts fires all resource requests concurrently without any concurrency limit.

## Change summary

Webview resources fail silently with ERR_INSUFFICIENT_RESOURCES on Linux when an extension webview loads thousands of files concurrently, because the renderer process inherits the desktop session's default 1024 nofile soft limit.

## Best solution

Add concurrency throttling to the webview resource loading path in `src/vs/workbench/contrib/webview/browser/webviewElement.ts` (the `load-resource` event handler at line ~285 fires `loadResource` without any limiter). A `Limiter` from `src/vs/base/common/async.ts` capping concurrent in-flight loads would prevent fd exhaustion regardless of the OS limit. Additionally, surfacing the error to the extension/user rather than silently failing would help diagnosability. Raising the renderer's soft fd limit is a complementary but Electron/Chromium-level change.
