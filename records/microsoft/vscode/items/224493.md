# Issue #224493 — Word wrapping computations are inaccurate

[https://github.com/microsoft/vscode/issues/224493](https://github.com/microsoft/vscode/issues/224493)

## Summary

The reported behavior matches the documented default wrapping strategy: VS Code defaults to a fast simple algorithm that assumes equal-width glyphs, while the existing advanced strategy delegates wrapping to the browser for variable-width and complex glyph cases.

## Change summary

Word wrap can wrap too early when proportional fonts, emoji, CJK, or other non-uniform glyph widths are used.

## Best solution

Keep the current default/simple wrapping behavior for performance, and direct users who need pixel-accurate wrapping with variable-width glyphs to set `editor.wrappingStrategy` to `advanced`.

- **Proposed comment:**

  > Thanks for the report. This matches the current design of word wrapping: the default `editor.wrappingStrategy: "simple"` is a fast algorithm that assumes equal-width glyphs, so it can be conservative with proportional fonts, emoji, CJK, and other glyphs with non-uniform widths. For pixel-accurate wrapping, please use `editor.wrappingStrategy: "advanced"`, which delegates the wrapping computation to the browser but can be slower on large files.

## Evidence

- **existing setting:** src/vs/editor/common/config/editorOptions.ts defines `editor.wrappingStrategy` with default `simple`; its description says `simple` assumes all characters are the same width, while `advanced` delegates wrapping point computation to the browser and works correctly in all cases but is slower.
- **implementation routing:** src/vs/editor/common/viewModel/viewModelLines.ts selects `domLineBreaksComputerFactory` only when `wrappingStrategy === 'advanced'`; otherwise it uses the monospace/simple line-break computer.
- **viewport column computation:** src/vs/editor/common/config/editorOptions.ts computes the viewport wrapping column from `contentWidth / typicalHalfwidthCharacterWidth`, which explains the conservative wrapping the reporter sees under the simple strategy.
- **issue state:** The reporter confirmed the problem remained in VS Code 1.92, but there is no later evidence of a regression beyond the documented simple-vs-advanced strategy tradeoff.
