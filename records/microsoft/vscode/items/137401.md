# Issue #137401 — "Keybinding" shown is misleading for Ctrl-modified keyboard layouts (like Dvorak+QWERTY)

[https://github.com/microsoft/vscode/issues/137401](https://github.com/microsoft/vscode/issues/137401)

## Summary

Keep open: the current Windows keybinding path still has a plausible mismatch between the persisted keybinding string and the displayed UI label for non-WYSIWYG layouts. I did not find a concrete fixing commit or canonical duplicate, and the issue is in the Backlog rather than a release milestone.

## Change summary

Ctrl-modified Dvorak+QWERTY layouts can show keybinding labels/search results using the Dvorak character while storing and dispatching the QWERTY keybinding.

## Best solution

Investigate the Windows keyboard mapper and keybindings editor display/search paths so Ctrl-modified layouts use a consistent label for display, matching dispatch and user settings. The likely area is `src/vs/workbench/services/keybinding/common/windowsKeyboardMapper.ts` plus the keybindings editor model/widget.

## Evidence

- **current status:** Issue https://github.com/microsoft/vscode/issues/137401 is open with labels `bug`, `windows`, and `keyboard-layout`, has no comments, and is assigned to the Backlog holding milestone.
- **display vs stored label:** `src/vs/workbench/contrib/preferences/browser/keybindingWidgets.ts:246-267` renders the resolved keybinding through `KeybindingLabel.set(...)` but fires/stores `getUserSettingsLabel()`, so UI and persisted text can diverge for non-WYSIWYG layouts.
- **Windows mapper behavior:** `src/vs/workbench/services/keybinding/common/windowsKeyboardMapper.ts:390-417` derives UI/ARIA labels from the keyboard-layout label map while user settings labels come from `KeyCodeUtils`; keyboard events are resolved as `KeyCodeChord`s from `keyboardEvent.keyCode`.
- **search behavior:** `src/vs/workbench/services/preferences/browser/keybindingsEditorModel.ts:150-153` and `:201-205` split searches such as `ctrl+j` into keybinding words, while `:373-378` treats user settings label, aria label, and UI label as separate exact-match candidates.
- **related existing special case:** `src/vs/workbench/services/keybinding/electron-browser/nativeKeyboardLayout.ts:89-94` has a macOS-only special case for `DVORAK-QWERTYCMD`, but Windows still routes to `WindowsKeyboardMapper` at `:80-81`.
