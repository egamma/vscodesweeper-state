# Issue #251076 — File protocol links in editor do not open in WSL

[https://github.com/microsoft/vscode/issues/251076](https://github.com/microsoft/vscode/issues/251076)

## Summary

Keep open: the report is specific and a maintainer reproduced the same behavior through the editor's default link detection path, so there is not enough evidence for a close label. Current main still shows test failure text rendered through an embedded editor with generic editor links, while terminal-style result output uses a separate terminal link path.

## Change summary

file:// links shown in inline/test-result failure messages open as Windows host paths instead of WSL remote files.

## Best solution

Make file URI link opening remote-aware for editor-rendered test failure messages, likely either in the editor link contribution or in the testing PlainTextMessagePeek path, while preserving the terminal result output behavior that already resolves remote file links.

## Evidence

- **maintainer repro:** Connor Peet commented on 2025-06-10 that this is the default link detection logic in the editor and that putting a file URI in an editor reproduces the same behavior.
- **test result editor path:** src/vs/workbench/contrib/testing/browser/testResultsView/testResultsOutput.ts:80 enables editor links, and PlainTextMessagePeek creates an EmbeddedCodeEditorWidget/CodeEditorWidget for non-output string test messages at lines 324-370.
- **generic editor link opening:** src/vs/editor/contrib/links/browser/links.ts:229-253 resolves detected links and passes them to openerService.open; the local special handling only covers relative file URIs, not WSL/remote authority rewriting for absolute file:// paths.
- **terminal contrast:** The reporter notes that terminal links and Show Result Output work correctly; current code supports that distinction because TerminalMessagePeek renders output through a detached terminal at src/vs/workbench/contrib/testing/browser/testResultsView/testResultsOutput.ts:420-545, and terminalLinkResolver rewrites file URIs to vscode-remote when remoteAuthority is present at src/vs/workbench/contrib/terminalContrib/links/browser/terminalLinkResolver.ts:27-33.
- **related scope:** The issue body links terminal issues https://github.com/microsoft/vscode/issues/211307 and https://github.com/microsoft/vscode/issues/207239 as similar, but explicitly says this remaining failure is in the Test Result tab/inline result window rather than the terminal.
