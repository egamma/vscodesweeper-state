# Issue #326546 — NVDA Stops Reading After Opening a File or Folder from Within VSCode, or Closing the Dialog

[https://github.com/microsoft/vscode/issues/326546](https://github.com/microsoft/vscode/issues/326546)

## Summary

Well-documented NVDA accessibility regression after opening/closing native file dialogs, with detailed IAccessible2 error logs and a UIA workaround. The previous report (#238520) was auto-closed without resolution. The root cause likely lives in Chromium's IAccessible2 accessibility tree management, making this an upstream candidate, but it needs maintainer investigation to confirm ownership. Keeping open for the accessibility team.

## Change summary

NVDA screen reader loses all IAccessible2 communication with VS Code after a native file/folder dialog is opened and closed, rendering the editor unusable to screen reader users until restart.

## Best solution

The accessibility team should investigate whether VS Code's native dialog flow (in `src/vs/platform/dialogs/electron-main/dialogMainService.ts`) or Electron's Chromium layer is responsible for the IAccessible2 tree corruption. If it's Chromium/Electron, file an upstream bug and track it here. The reporter's finding that UIA mode works is a strong diagnostic clue — it may also point to a viable mitigation (e.g. enabling UIA by default on Windows when a screen reader is detected).
