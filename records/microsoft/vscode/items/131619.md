# Issue #131619 — Editor resolver: untyped `editorGroupFinder#findGroup` causes issues

[https://github.com/microsoft/vscode/issues/131619](https://github.com/microsoft/vscode/issues/131619)

## Summary

The underlying debt remains: `editorResolverService.resolveUntypedInputAndGroup` still calls `findGroup` with an untyped editor, and `isActive`/`isOpened` still call `matches` on those untyped inputs. The code has evolved (modal groups, aux windows, locked groups) but the core design concern — calling `findGroup` before the editor type is resolved — persists. Author is a CONTRIBUTOR (bpasero, a core maintainer), and the issue sits in Backlog.

## Change summary

Refactor `findGroup` so it is not called with untyped editor inputs before the editor resolver determines the concrete editor type, avoiding incorrect `matches`/`contains` checks.

## Best solution

Restructure the editor resolver flow so `findGroup` receives a typed `EditorInputWithOptions` rather than an `IUntypedEditorInput`. This likely requires the resolver to determine the editor type first, then find the group — the reverse of the current order. The main files are `src/vs/workbench/services/editor/browser/editorResolverService.ts` and `src/vs/workbench/services/editor/common/editorGroupFinder.ts`.
