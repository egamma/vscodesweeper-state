# Issue #148041 — Detect when our `vscode://` UriHandler isn't registered on the OS and inform the user

[https://github.com/microsoft/vscode/issues/148041](https://github.com/microsoft/vscode/issues/148041)

## Summary

Team-authored debt issue requesting detection of unregistered vscode:// protocol handlers. The feature has not been implemented on current main — no call to `isDefaultProtocolClient` or equivalent detection exists. Kept open per team-author policy and because the gap is real.

## Change summary

Detect when the vscode:// URI handler is not registered with the OS (e.g. zip/portable builds) and inform the user or let extensions handle it gracefully.

## Best solution

Add an `isDefaultProtocolClient` check in `src/vs/platform/url/electron-main/electronUrlListener.ts` (or a new sibling service) and surface a notification or expose a flag via the URL service so extensions like GitHub Auth can fall back gracefully when the protocol handler is missing. The portable-mode skip added in commit aa19df565fd partially addresses one scenario but does not inform the user.
