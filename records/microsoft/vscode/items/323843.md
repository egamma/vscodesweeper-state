# Issue #323843 — Updated contributions are not available when restarting extensions

[https://github.com/microsoft/vscode/issues/323843](https://github.com/microsoft/vscode/issues/323843)

## Summary

Valid bug: when an extension updates and changes its contribution points (e.g. debugger type), 'Restart Extensions' stops and restarts extension hosts with a delta mechanism, but if the extension was previously activated (`canRemoveExtension` returns false for activated extensions in the non-stop path), the old contribution remains cached. The full stop-then-delta path in `startExtensionHosts` should handle this, but contribution points like debuggers that register via `_doHandleExtensionPoints` may not fully re-process changed contributions without a window reload.

## Change summary

Updated extension contributions (e.g. new debugger types) are not recognized after 'Restart Extensions' — only a full VS Code restart picks them up.

## Best solution

Investigate whether `_doHandleExtensionPoints` in `abstractExtensionService.ts` correctly re-processes changed contributions when an extension is removed-then-added via the delta path during `startExtensionHosts`. The debugger extension point handler in `src/vs/workbench/contrib/debug/` should fully clear and rebuild its contribution state on delta changes. If certain contribution points fundamentally require a reload, the runtime state logic in `getRuntimeState` should return `ReloadWindow` instead of `RestartExtensions` for those cases.
