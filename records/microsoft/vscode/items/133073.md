# Issue #133073 — Actions are unclickable in Modal dialog when opened through a CodeLens (and all mouse event on the system disabled)

[https://github.com/microsoft/vscode/issues/133073](https://github.com/microsoft/vscode/issues/133073)

## Summary

Confirmed Linux-specific bug where native modal dialogs triggered from a CodeLens click cause an X11 pointer grab that disables all mouse input system-wide. Root cause is likely in Electron/Chromium's X11 pointer-grab handling when a native GTK dialog opens during a synthetic click event. Already labeled `confirmed` and `help wanted`; the workaround (`window.dialogStyle: custom`) remains valid on current main since native dialogs are still the default on desktop Linux.

## Change summary

Opening a native modal dialog from a CodeLens command on Linux causes an X11 pointer grab that freezes all mouse input system-wide until Escape is pressed.

## Best solution

Root-cause is in Electron's X11 pointer-grab lifecycle when a native GTK dialog is spawned from a programmatic click context (CodeLens). The best end state is either (1) an upstream Electron fix that releases the pointer grab before showing the native dialog, or (2) VS Code-side mitigation — e.g. deferring the `showWarningMessage` call out of the CodeLens click handler via a microtask/timeout to break the grab chain (similar to the 500ms delay workaround noted in the last comment). A related report exists at https://github.com/microsoft/vscode/issues/289111 describing the same X11 grab symptom in a broader context.
