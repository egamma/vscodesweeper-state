# Issue #245607 — In fish, can't make shell integration and starship both work at the same time

[https://github.com/microsoft/vscode/issues/245607](https://github.com/microsoft/vscode/issues/245607)

## Summary

Still reproduces on main: the fish shell integration script has no starship-compatibility logic — unlike the bash script (which detects `STARSHIP_SESSION_KEY` and can coexist), the fish script's `__init_vscode_shell_integration` unconditionally rewrites `fish_prompt` and `fish_mode_prompt`, so whichever of VS Code SI or starship initializes second clobbers the other's hooks. No fix has landed.

## Change summary

In fish shell, VS Code shell integration and starship cannot both work simultaneously because each replaces the other's prompt hooks regardless of initialization order.

## Best solution

The fish shell integration script needs starship-aware wrapping logic similar to what exists in the bash script: detect `STARSHIP_SESSION_KEY`, preserve `$status`/`$pipestatus` across escape-sequence emissions, and ensure the prompt-wrapping in `__init_vscode_shell_integration` composes with starship's prompt replacement rather than clobbering it. The fix belongs in `src/vs/workbench/contrib/terminal/common/scripts/shellIntegration.fish`. Issue https://github.com/microsoft/vscode/issues/319058 covers the closely related `$status` clobbering aspect and could be addressed together.
