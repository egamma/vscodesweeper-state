# Issue #170529 — Search with regular expression ignores match whole word

[https://github.com/microsoft/vscode/issues/170529](https://github.com/microsoft/vscode/issues/170529)

## Summary

Still reproduces on main: when both regex and whole-word are enabled, the `createRegExp` helper decides whether to append `\b` by testing the literal last character of the pattern string, not considering what it matches as a regex metacharacter. For `fa.`, the `.` is a non-word character so `\b` is omitted from the end, producing `\bfa.` instead of `\bfa.\b`, which over-matches.

## Change summary

Search with regex + 'Match Whole Word' fails to add a trailing `\b` when the regex pattern ends with a metacharacter like `.`, causing over-matching.

## Best solution

Fix the `\b`-appending logic in `createRegExp` (src/vs/base/common/strings.ts, lines 209–216) so that when `isRegex` is true, it always wraps the pattern in `\b…\b` (or uses a more sophisticated boundary strategy) rather than inspecting literal characters of the regex source. The ripgrep path in ripgrepTextSearchEngine.ts (line 494–497) and the local file search path both go through `createRegExp`, so fixing strings.ts should cover both. Issue https://github.com/microsoft/vscode/issues/175794 describes the same root cause and should be consolidated.

  > The bug is in `createRegExp` in src/vs/base/common/strings.ts (lines 209–216). When `isRegex` is true and `wholeWord` is true, the function decides whether to prepend/append `\b` by testing the literal first/last characters of the regex pattern string with `!/\B/.test(char)`. For a regex like `fa.`, the last character is `.` (a non-word char), so `\B` matches it and the trailing `\b` is skipped, producing `\bfa.` instead of `\bfa.\b`. This causes over-matching: 'face', 'factory' etc. all match when only 3-letter words like 'fas' should. Fix: when `isRegex` is true, always add `\b` boundaries unconditionally (or use a smarter heuristic that recognizes metacharacters). The non-regex path (where the pattern is escaped) can keep the current char-inspection logic since escaped patterns have no metacharacters.
