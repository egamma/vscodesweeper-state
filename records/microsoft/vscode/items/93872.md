# Issue #93872 — [rename on type] feedback incorrect after Rename

[https://github.com/microsoft/vscode/issues/93872](https://github.com/microsoft/vscode/issues/93872)

## Summary

Team-authored bug (MEMBER) — hard keep-open. The linked editing decorations go out of sync after an F2 Rename because the rename edits are broken up by the edit minifier, causing tracked ranges to cover incorrect regions. The issue appears to remain on current main: the `onDidChangeModelContent` handler in `linkedEditing.ts` does not distinguish rename-originated edits from user typing, so the decoration ranges can desync when the edit structure doesn't match the decoration boundaries.

## Change summary

After performing F2 Rename on a tag in linked editing mode, the synced-region decoration no longer covers the full renamed text.

## Best solution

When a rename (or any non-user-typing bulk edit) lands, the linked editing contribution should clear and re-query ranges from the provider rather than trying to incrementally sync decorations that were distorted by minified edits. The fix likely belongs in `src/vs/editor/contrib/linkedEditing/browser/linkedEditing.ts` around the `onDidChangeModelContent` handler (lines 158–169), possibly by detecting edits that originate from rename and forcing a full `updateRanges()` instead of `triggerRangeSync`.
