Legitimate feature request to auto-set +x on WSL server shell scripts during update. No fix found on main; maintainer asked for details and got a response. Keeping open as it's in Backlog with no resolution.

## Change summary

VS Code server upgrade fails in WSL when wsl.conf uses fmask=111, because shell scripts lose their executable bit and require manual chmod.

## Best solution

The server update/install flow (likely in the remote server bootstrap or the CLI update path) should chmod +x the .sh scripts in the server bin directory after extracting/updating. This belongs in the WSL area and may involve changes to server extraction logic or a post-install hook.
