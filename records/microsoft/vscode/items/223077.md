# Issue #223077 — Make the `TextDocument.version` properly monotonically increasing during the whole application lifecycle

[https://github.com/microsoft/vscode/issues/223077](https://github.com/microsoft/vscode/issues/223077)

## Summary

Keep open as an active API/editor design discussion: the current implementation still allows the document version history to be lost when closed-model state is not retained, and a maintainer explicitly reopened the issue for further discussion.

## Change summary

Guarantee that `TextDocument.version` for a URI never resets during the VS Code application lifecycle, even after close/reopen and disposed-model eviction.

## Best solution

Have the editor/API owners decide whether `TextDocument.version` should remain scoped to a live/restored text model or become a lifecycle-wide per-URI identity. If accepted, the likely core change belongs around `src/vs/editor/common/services/modelService.ts` and `src/vs/editor/common/model/textModel.ts`, with API documentation updated to state the close/reopen semantics.

## Evidence

- **current code:** `src/vs/editor/common/model/textModel.ts` initializes each new `TextModel` with `_versionId = 1`, so a newly created model can reset unless prior disposed-model data is restored.
- **current retention behavior:** `src/vs/editor/common/services/modelService.ts` stores `DisposedModelInfo` with `versionId`, but the cache is bounded by `ModelService.MAX_MEMORY_FOR_CLOSED_FILES_UNDO_STACK = 20 * 1024 * 1024` and `_ensureDisposedModelsHeapSize` removes old entries.
- **current restore behavior:** `src/vs/editor/common/services/modelService.ts` only calls `_overwriteVersionId(disposedModelData.versionId)` when a matching disposed model is found and the SHA1 matches, so the requested lifecycle-wide guarantee is not implemented on current `main`.
- **existing test scope:** `src/vs/editor/test/common/services/modelService.test.ts` covers maintaining version id for the same resource and same content while the disposed model is retained, but not the proposed forever-per-URI lifecycle guarantee.
- **discussion state:** Maintainer hediet first argued this may be out of scope, then commented "Reopening for further discussion"; the issue currently carries `under-discussion`, so it should not be closed by cleanup.
- **related work:** The issue body links the related prior request https://github.com/microsoft/vscode/issues/160072 and implementation PR https://github.com/microsoft/vscode/pull/223079, but the local `main` history does not show this proposal as merged.
