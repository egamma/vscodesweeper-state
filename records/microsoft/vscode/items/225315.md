# Issue #225315 — Search Editor Progress Bar Animates Endlessly After Search Completed (intermittent)

[https://github.com/microsoft/vscode/issues/225315](https://github.com/microsoft/vscode/issues/225315)

## Summary

Still reproduces on current main: the original fix (PR #225549) guards `triggerSearch` but not the deferred `doRunSearch` callback — a race between the delayer and an input change to an invalid regex leaves the progress bar spinning indefinitely.

## Change summary

Search Editor progress bar animates endlessly when transitioning from a valid regex to an invalid one while search-on-type is active.

## Best solution

Add a validity re-check at the top of `doRunSearch()` in `src/vs/workbench/contrib/searchEditor/browser/searchEditor.ts` (or cancel the `runSearchDelayer` when `triggerSearch` detects invalid input) so that a deferred search callback that fires after the input became invalid exits before starting the progress bar.

The race is in `src/vs/workbench/contrib/searchEditor/browser/searchEditor.ts`: `triggerSearch` (line ~500) checks `this.queryEditorWidget.searchInput?.inputBox.isInputValid()` and schedules `doRunSearch` via `this.runSearchDelayer.trigger(callback, delay)` (line ~505). If the user changes the input to an invalid regex before the delayer fires, `doRunSearch` (line ~540) runs without re-checking validity. When `queryBuilder.text()` at line ~591 throws for the invalid regex, `doRunSearch` returns at line ~594 — but `searchOperation.start(500)` (line ~597) and `ongoingOperations++` (line ~598) have NOT been reached yet in this path, so the progress bar issue is actually from a different path: the `searchModel.search(query)` call may also fail in a way that doesn't trigger the `finally` block. The safest fix is to add the same validity check (`if (!(this.queryEditorWidget.searchInput?.inputBox.isInputValid())) { return; }`) at the very beginning of `doRunSearch()`, before any search work begins. This ensures the deferred callback is a no-op when the input has become invalid. Do NOT restructure the delayer or search pipeline. Add a unit test that verifies `doRunSearch` returns early without starting the progress operation when the input is invalid.
