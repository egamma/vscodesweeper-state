# Issue #248978 — hangs on startup if kwallet is disabled in linux kde plasma desktop

[https://github.com/microsoft/vscode/issues/248978](https://github.com/microsoft/vscode/issues/248978)

## Summary

Verified duplicate of #248664, which tracks the same KDE/kwallet startup hang. Both report the same Chromium-level blocking D-Bus call (`Bus::SendWithReplyAndBlock` to kwalletd6) that freezes the main process for ~25s when kwallet is disabled. The root cause is upstream in Chromium (https://bugs.kde.org/show_bug.cgi?id=504014).

## Change summary

VS Code hangs for ~50s on startup on KDE Plasma when kwallet is disabled, because Chromium's synchronous D-Bus call to kwalletd6 blocks the main process.

## Best solution

Consolidate on https://github.com/microsoft/vscode/issues/248664 which already carries `perf`/`perf-startup` labels and links to the upstream Chromium bug (https://bugs.kde.org/show_bug.cgi?id=504014). The fix lives in Electron/Chromium — either making the kwallet probe async or adding a shorter timeout. VS Code's workaround is `--password-store=basic`.

## Evidence

- **canonical issue:** https://github.com/microsoft/vscode/issues/248664 is open, filed around the same time (May 2025), already triaged with `perf`/`perf-startup` labels, and a commenter confirmed the upstream Chromium bug.
- **upstream bug:** Chromium bug https://bugs.kde.org/show_bug.cgi?id=504014 tracks the synchronous D-Bus call in `bus.cc` / `kwallet_dbus.cc` that blocks the main thread.
- **code path:** VS Code calls Electron's `safeStorage.isEncryptionAvailable()` in `src/vs/platform/encryption/electron-main/encryptionMainService.ts:67`, which triggers the blocking Chromium kwallet probe. VS Code has no async alternative.
- **still reproduces:** Comment from Sep 2025 confirms the issue persists in VS Code 1.104.0; no fix has landed in VS Code or Electron for this.
