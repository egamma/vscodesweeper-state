# Issue #269591 — Unicode Highlight, allowedCharacters not working for some supplementary-plane characters (U+1D448, U+1D462, U+1D703)

[https://github.com/microsoft/vscode/issues/269591](https://github.com/microsoft/vscode/issues/269591)

## Summary

Confirmed bug on current main: the Unicode highlighter regex lacks the `u` flag, causing supplementary-plane characters at end-of-line to be matched as individual surrogates rather than full code points, bypassing the allowedCharacters check.

## Change summary

Supplementary-plane characters (U+10000+) in `editor.unicodeHighlight.allowedCharacters` are still highlighted when they appear as the last character on a line.

## Best solution

Add the `u` flag to the regex on line 23 of `src/vs/editor/common/services/unicodeTextModelHighlighter.ts` (and possibly the candidate-based regex on line 25), and fix the off-by-one guard at line 57 (`endIndex + 1 < lineLength` → `endIndex < lineLength`). This would make the regex iterate by code point, eliminating the surrogate-pair workaround issues.

## Evidence

- **missing u flag:** src/vs/editor/common/services/unicodeTextModelHighlighter.ts line 23: regex `[^\t\n\r\x20-\x7E]` uses 'g' but not 'u', so it matches individual UTF-16 surrogates for supplementary-plane chars.
- **off-by-one guard:** Line 57: `endIndex + 1 < lineLength` prevents extending the match when the supplementary char is at end-of-line (endIndex=1, lineLength=2 → 2<2 is false). First surrogate match stays un-extended.
- **allowedCodePoints bypass:** The un-extended first match yields substring with codePointAt(0) = 0xD835 (the high surrogate), not 0x1D448 (the actual character), so the allowedCodePoints check fails.
- **no fix on main:** git log shows no relevant changes since ESM merge (6b924c51528); the bug persists on current main.
