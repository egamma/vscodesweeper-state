# Issue #193603 — [wsl] Can't run `code .` when offline and VS Code got updated

[https://github.com/microsoft/vscode/issues/193603](https://github.com/microsoft/vscode/issues/193603)

## Summary

The issue describes the legacy WSL shell script destructively removing the VS Code Server before downloading its replacement, breaking offline use. The current architecture on main has been redesigned with a download cache (LRU, keeps 5 entries) and fallback CLI discovery, but it's unclear whether the old shell-script code path (`code .` from WSL) still exhibits this behavior for users on current stable releases.

## Change summary

Running `code .` in WSL while offline after a VS Code update removes the working server installation before attempting (and failing) to download the new one, leaving the user unable to use VS Code in WSL until connectivity is restored.

## Best solution

Verify whether the current WSL connection path (agentHost + CLI download cache) still exhibits destructive-removal-before-download. If the old shell script path is still reachable for users who haven't migrated, add a connectivity pre-check or adopt the same cache/fallback pattern. The WSL area owner should confirm the current user experience.
