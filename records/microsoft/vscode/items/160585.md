# Issue #160585 — [html] Extensions using htmlLanguageParticipants don't activate vscode.html-language-features

[https://github.com/microsoft/vscode/issues/160585](https://github.com/microsoft/vscode/issues/160585)

## Summary

Still an open gap on main: there is no mechanism to auto-activate vscode.html-language-features when a third-party extension contributes htmlLanguageParticipants for a custom language. The activation events are hardcoded to html and handlebars only.

## Change summary

Extensions contributing htmlLanguageParticipants for custom languages (e.g. jinja-html) don't trigger activation of vscode.html-language-features, requiring workarounds.

## Best solution

Add a mechanism so that when any installed extension contributes htmlLanguageParticipants for a languageId, the html-language-features extension is implicitly activated on that language. This likely requires changes in the extension host activation logic (src/vs/workbench/api/common/extHostExtensionService.ts or similar) to scan htmlLanguageParticipants contributions and synthesize onLanguage activation events for html-language-features.
