## Summary

Plausible bug in the shell integration `executeCommand().read()` stream when commands contain emoji on Windows/PowerShell, but the report lacks the diagnostic detail needed to confirm the root cause — requesting more info.

## Change summary

The `read()` async iterable returned by `shellIntegration.executeCommand()` yields no data and never completes when the command string contains emoji characters on Windows/PowerShell.

## Best solution

Diagnostic logs from the terminal output channel would narrow this down.

- **potential mismatch site:** In `src/vs/workbench/api/common/extHostTerminalShellIntegration.ts:279-302`, `startShellExecution` matches the shell-reported command line against pending executions by exact string equality. If ConPTY or PowerShell mangles the emoji in the reported command line, the match fails and a new execution object is created — the original `read()` stream never receives data.
