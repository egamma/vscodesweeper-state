# Issue #299402 — confusing line termination status for single line files without a terminator

[https://github.com/microsoft/vscode/issues/299402](https://github.com/microsoft/vscode/issues/299402)

## Summary

The status bar shows the model's actual EOL sequence. For files with no line terminators, VS Code uses the `files.eol` configuration (defaulting to LF on macOS/Linux) — this is intentional. EditorConfig integration is provided by a marketplace extension, not VS Code core.

## Change summary

Status bar shows LF for single-line files without terminators even when .editorconfig specifies CRLF for that file type.

## Best solution

This is working as designed. For files with zero line endings, VS Code picks the EOL from the `files.eol` setting (which defaults to `\n` on macOS/Linux or `auto`). The EditorConfig marketplace extension is responsible for overriding this per-resource. If that override isn't working, the issue belongs on the EditorConfig extension repo. The user can also set `"files.eol": "\r\n"` in workspace settings with a language/resource override.

  > Thanks for the report. This is actually working as designed in VS Code core.
  > 
  > When a file contains no line terminators (a single line with no EOL), VS Code cannot detect the line ending style from the file content, so it falls back to the configured `files.eol` setting. On macOS/Linux this defaults to LF.
  > 
  > EditorConfig integration is provided by a [marketplace extension](https://marketplace.visualstudio.com/items?itemName=EditorConfig.EditorConfig), which is responsible for overriding `files.eol` on a per-resource basis. If the EditorConfig extension isn't applying CRLF for your `.cmd` files, that would be best reported on the [EditorConfig extension repo](https://github.com/editorconfig/editorconfig-vscode).
  > 
  > As a workaround, you can set `"files.eol": "\r\n"` in your workspace settings with a language override for batch files.
