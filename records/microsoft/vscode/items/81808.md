# Issue #81808 — [css] css.lint.hexColorLength seems to be now irrelevant

[https://github.com/microsoft/vscode/issues/81808](https://github.com/microsoft/vscode/issues/81808)

## Summary

Still reproduces on main: the CSS parser's _parseHexColor regex only matches valid-length hex values (3/4/6/8 digits), so invalid lengths never produce a HexColorValue node and the hexColorLength lint rule never fires. The fix belongs in the upstream vscode-css-languageservice package.

## Change summary

The css.lint.hexColorLength setting never fires because the CSS parser rejects invalid-length hex colors before the linter can check them.

## Best solution

In the upstream vscode-css-languageservice, broaden _parseHexColor to accept any #[0-9A-Fa-f]+ token as a HexColorValue node (as @aeschli suggested), then let visitHexColorValue validate the length. Once fixed upstream, bump the dependency in extensions/css-language-features/server/package.json.
