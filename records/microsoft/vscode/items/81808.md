# Issue #81808 — [css] css.lint.hexColorLength seems to be now irrelevant

[https://github.com/microsoft/vscode/issues/81808](https://github.com/microsoft/vscode/issues/81808)

## Summary

Still reproducible on current main — the root cause is in the upstream vscode-css-languageservice parser, which rejects invalid-length hex strings before the lint rule can fire. No fix has landed.

## Change summary

The `css.lint.hexColorLength` setting never fires because the CSS parser's strict regex rejects malformed hex colors before the lint rule can evaluate them.

## Best solution

Fix `_parseHexColor` in `microsoft/vscode-css-languageservice` (src/parser/cssParser.ts) to accept any `#[0-9A-Fa-f]+` as a HexColorValue node, then let the lint rule in src/services/lint.ts flag invalid lengths. The VS Code repo just needs to bump the dependency version afterward.

## Evidence

- **parser regex:** vscode-css-languageservice src/parser/cssParser.ts: `_parseHexColor` uses `/^#([A-Fa-f0-9]{3}|[A-Fa-f0-9]{4}|[A-Fa-f0-9]{6}|[A-Fa-f0-9]{8})$/g` — a 7-digit hex never matches, so no HexColorValue node is created.
- **lint rule exists:** vscode-css-languageservice src/services/lint.ts has the check: if length !== 9/7/5/4, addEntry(node, Rules.HexColorLength). But it only runs on HexColorValue nodes, which the parser won't create for invalid lengths.
- **maintainer confirmation:** aeschli (MEMBER) confirmed in comment #3 that the parser should create a HexColor for all `#[0-9A-Fa-f]+` values.
- **no upstream fix:** Current vscode-css-languageservice main still has the strict regex; no commit addressing this was found.
