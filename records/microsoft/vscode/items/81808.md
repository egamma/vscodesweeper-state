# Issue #81808 — [css] css.lint.hexColorLength seems to be now irrelevant

[https://github.com/microsoft/vscode/issues/81808](https://github.com/microsoft/vscode/issues/81808)

## Summary

Still reproduces: the upstream vscode-css-languageservice parser rejects invalid-length hex values at the scanner level (producing a property-value-expected error), so the hexColorLength lint rule never fires. Team member aeschli acknowledged the fix path in 2019 but no commit addresses it.

## Change summary

The `css.lint.hexColorLength` setting has no effect because the CSS parser fails to parse invalid hex colors as HexColor nodes, preventing the lint rule from ever triggering.

## Best solution

Fix the CSS scanner in microsoft/vscode-css-languageservice to accept `#[0-9A-Fa-f]+` as a HexColor token regardless of length, then let the lint rule in `src/services/lint.ts` flag invalid lengths. Consumers of HexColor in completions/hover must tolerate invalid lengths gracefully.
