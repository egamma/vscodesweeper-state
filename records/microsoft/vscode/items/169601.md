# Issue #169601 — merge-conflict extension cannot handler '<<<<<<<<' conflict tag

[https://github.com/microsoft/vscode/issues/169601](https://github.com/microsoft/vscode/issues/169601)

## Summary

Still reproduces on main: the merge-conflict parser's splitter check uses exact equality (`line.text === '======='`) which rejects extended conflict markers (8+ characters) that git produces for rename/delete conflicts and nested-conflict scenarios. The start and end markers use `startsWith` and would match, but the splitter fails, so the entire conflict block is silently dropped.

## Change summary

The merge-conflict extension fails to detect conflict blocks when git produces extended markers (e.g. 8 angle brackets/equals signs) for rename/delete or nested conflicts.

## Best solution

Make the splitter check context-aware: when the start marker has N>7 characters, accept a splitter line of exactly N equals signs (or, more simply, accept a line matching `/^={7,}$/`). This must not regress #95040, where a line of `========` inside markdown was falsely matched — the key difference is that #95040's false positive occurred outside a conflict block, while this bug is about lines already inside one (after a matched `<<<<<<<<` header). Scoping the relaxed match to only fire when `currentConflict` is non-null and the start header itself was extended would satisfy both constraints.
