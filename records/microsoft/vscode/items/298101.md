# Issue #298101 — ${workspaceFolderBasename} does not resolve in projectPath

[https://github.com/microsoft/vscode/issues/298101](https://github.com/microsoft/vscode/issues/298101)

## Summary

This is caused by the C# Dev Kit extension, not VS Code core. VS Code core correctly substitutes variables (including `${workspaceFolderBasename}`) in all config fields after `resolveDebugConfiguration` runs. The `"type": "dotnet"` debugger is contributed by C# Dev Kit, which likely consumes `projectPath` during its `resolveDebugConfiguration` handler — before variable substitution occurs — causing the unresolved variable.

## Change summary

${workspaceFolderBasename} is not resolved in the `projectPath` field when using the C# Dev Kit `dotnet` debug type.

## Best solution

Report to the C# Dev Kit extension (dotnet/vscode-csharp). The extension's `resolveDebugConfiguration` should not interpret `projectPath` before VS Code's variable substitution pass, or it should call VS Code's variable resolution API itself.

  > Thanks for the report. After tracing through VS Code's debug launch flow, the variable substitution for `${workspaceFolderBasename}` is handled by VS Code core and applies to **all** fields of the launch configuration — including `projectPath`. However, the `"type": "dotnet"` debugger is contributed by the C# Dev Kit extension, and its `resolveDebugConfiguration` handler runs *before* VS Code performs variable substitution (see `debugService.ts` line 507–511). This means the extension is likely consuming `projectPath` with the raw variable string before substitution occurs.
