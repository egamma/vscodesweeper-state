# Issue #317305 — git.stage moves focus to parent section header instead of next sibling in SCM tree

[https://github.com/microsoft/vscode/issues/317305](https://github.com/microsoft/vscode/issues/317305)

## Summary

Valid request for better focus management after git.stage removes an item from the SCM tree. The tree's default splice behavior moves focus to the parent node; no sibling-preserving mechanism or list.focusNextSibling command exists on current main.

## Change summary

After git.stage removes a file from the SCM tree, focus jumps to the parent group header instead of advancing to the next sibling file, breaking keyboard-driven stage-and-advance workflows.

## Best solution

Either have the SCM tree's resource removal logic preserve focus on the next sibling (in `src/vs/workbench/contrib/scm/browser/`), or expose a `list.focusNextSibling` command in the tree widget layer (`src/vs/base/browser/ui/tree/`) that chord bindings can use. The former is more robust for the specific staging workflow.
