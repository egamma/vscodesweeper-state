# Issue #300045 — VS Code crash on startup when installed on external drive on Mac

[https://github.com/microsoft/vscode/issues/300045](https://github.com/microsoft/vscode/issues/300045)

## Summary

Real crash on current main: portable mode on macOS external drives with non-HFS+/APFS filesystems (e.g. exFAT) fails because `createStaticIPCHandle` places the Unix socket inside `userDataPath` on the external drive, whose filesystem doesn't support Unix sockets (ENOTSUP). The `/Volumes/` file-watcher block is a secondary concern tracked by #106879. No fix found on main.

## Change summary

VS Code crashes on startup in portable mode on macOS when installed on an external drive whose filesystem does not support Unix domain sockets.

## Best solution

On macOS, when `userDataPath` is under `/Volumes/` (portable mode on a removable drive), fall back to placing the IPC socket in a tmpdir or XDG_RUNTIME_DIR equivalent (e.g. `os.tmpdir()`) instead of the user-data directory, similar to how Linux already prefers `XDG_RUNTIME_DIR` in `createStaticIPCHandle`. The `handleStartupDataDirError` should also handle `ENOTSUP` to show a meaningful dialog instead of crashing silently. Likely files: `src/vs/base/parts/ipc/node/ipc.net.ts` (socket path logic) and `src/vs/code/electron-main/main.ts` (error handling).

## Evidence

- **crash path in source:** In `src/vs/base/parts/ipc/node/ipc.net.ts` line 944, on macOS (`process.platform === 'darwin'`), the socket is placed inside `directoryPath` (the user-data path). Unlike Linux, macOS never uses `XDG_RUNTIME_DIR`. When the user-data path is on an exFAT/FAT32 external drive, socket creation fails with ENOTSUP.
- **unhandled error code:** In `src/vs/code/electron-main/main.ts` line 366, the catch block only handles `EADDRINUSE`; `handleStartupDataDirError` (line 481) only handles `EACCES`/`EPERM`. ENOTSUP is thrown unhandled, killing the process.
- **reporter log:** The verbose log shows `Error: listen ENOTSUP: operation not supported on socket /Volumes/X10 Pro/Applications/VS Code/code-portable-data/user-data/1.11-main.sock` followed immediately by `Lifecycle#kill()`.
- **secondary: /Volumes/ watcher:** The `/Volumes/` check in `src/vs/platform/files/node/watcher/nodejs/nodejsWatcherLib.ts` line 197 is a known upstream workaround tracked by https://github.com/microsoft/vscode/issues/106879 (open, upstream).
- **no existing fix:** git log on ipc.net.ts and main.ts shows no ENOTSUP handling or macOS socket-path fallback added since the issue was filed.
