# Issue #200262 — Deleted collapsed folders remain visible until explorer refresh

[https://github.com/microsoft/vscode/issues/200262](https://github.com/microsoft/vscode/issues/200262)

## Summary

Still reproducible on main: the async data tree's collapsed-node optimization (asyncDataTree.ts:1111-1116) can prevent the explorer view from reflecting a deletion when the parent's only child was collapsed. The attempted fix PR #200570 was closed without merge, and no subsequent commit addresses this.

## Change summary

Deleting a collapsed folder whose parent contains no other visible children leaves the deleted folder visible in the explorer until a manual refresh.

## Best solution

Fix in the tree widget layer (src/vs/base/browser/ui/tree/asyncDataTree.ts) or in the explorer's deletion handler (src/vs/workbench/contrib/files/browser/explorerService.ts) to ensure that when a child is removed from the model and a view refresh is triggered on the parent, the tree actually removes the stale node even when the deleted child was collapsed.

## Evidence

- **collapsed-skip logic:** src/vs/base/browser/ui/tree/asyncDataTree.ts lines 1111-1116: when a tree node is collapsed, refreshSubTree marks it stale and sets children to empty without fully refreshing, which can prevent deletion propagation.
- **explorer delete handler:** src/vs/workbench/contrib/files/browser/explorerService.ts lines 422-445: on DELETE, model removes the child then calls view.refresh(false, parent), which delegates to tree.updateChildren — the collapsed-skip path may interfere.
- **closed PR:** https://github.com/microsoft/vscode/pull/200570 attempted a fix by refreshing from the nearest uncollapsed parent but was closed without merge.
- **no subsequent fix:** git log since 2024-01-01 on both asyncDataTree.ts and explorerService.ts shows no commit addressing this scenario.
