# Issue #284939 — Ctrl + PageDown and Ctrl + PageUp did not work as expected, Chrome shortcut was invoked.

[https://github.com/microsoft/vscode/issues/284939](https://github.com/microsoft/vscode/issues/284939)

## Summary

Ctrl+PageDown/PageUp are browser-reserved shortcuts for switching browser tabs — the browser intercepts them before VS Code Web can handle them. This is a well-known platform limitation, not a VS Code bug. The same class of issue was closed as *as-designed in https://github.com/microsoft/vscode/issues/137767.

## Change summary

Ctrl+PageDown and Ctrl+PageUp switch browser tabs instead of VS Code editor tabs when using VS Code in a browser.

## Best solution

This is inherent browser behavior that VS Code Web cannot override. Workarounds: install as a PWA (More Tools > Create Shortcut), use fullscreen mode, or rebind the action to a non-browser-reserved shortcut. No code change in VS Code can fix this.

  > Thanks for the report. Ctrl+PageDown and Ctrl+PageUp are browser-reserved shortcuts — Chrome intercepts them to switch browser tabs before VS Code Web gets a chance to handle them. This is a browser platform limitation, not something VS Code can override.
  > 
  > Workarounds:
  > - **Install as a PWA**: in Chrome, go to ⋮ > More Tools > Create Shortcut (check "Open as window") — this gives VS Code its own window where browser shortcuts don't apply.
  > - **Use fullscreen mode** (F11) — some browsers release reserved shortcuts in fullscreen.
  > - **Rebind the action**: assign a different, non-browser-reserved keybinding to "Open Next Editor" / "Open Previous Editor" in VS Code's keyboard shortcuts.
  > 
  > This is the same class of issue as https://github.com/microsoft/vscode/issues/137767. Closing as by design — the browser owns these shortcuts.

## Evidence

- **browser limitation:** Ctrl+PageDown and Ctrl+PageUp are Chrome's built-in shortcuts for switching browser tabs; they are intercepted by the browser before reaching the web page.
- **keybinding registered:** VS Code does bind Ctrl+PageDown/PageUp to 'workbench.action.nextEditor'/'workbench.action.previousEditor' in src/vs/workbench/browser/parts/editor/editorActions.ts — the binding exists but the browser eats the keypress first.
