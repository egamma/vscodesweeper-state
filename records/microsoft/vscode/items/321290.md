# Issue #321290 — Chat session marked empty and hidden while Copilot transcript/edit state remain intact

[https://github.com/microsoft/vscode/issues/321290](https://github.com/microsoft/vscode/issues/321290)

## Summary

Team-authored issue reporting chat session data loss — the session index marks a session as empty while transcript and edit-state artifacts prove it was non-empty. Recent fixes (e266aa425b9 for large-session clone crashes, 5e9baab3da4 for ObjectMutationLog corruption on failed writes) address closely related persistence bugs, but this specific split-brain scenario (session file overwritten to a 3-line stub mid-session) may still be open. Keeping open per team-member policy.

## Change summary

A long Copilot Chat agent session disappears from the history list because the chatSessions JSONL file is overwritten to an empty stub and the index is marked isEmpty:true, even though the Copilot transcript and edit-state files remain intact.

## Best solution

Investigate the write path in `chatSessionStore.ts` (around line 866 where `isEmpty` is derived from `session.getRequests().length === 0`) and the ObjectMutationLog persistence flow. The root cause is likely that the in-memory ChatModel loses its requests (e.g. during an interrupted tool call or rehydration failure) and then a save overwrites the previously-rich JSONL with an empty snapshot. A defensive guard — refusing to overwrite a non-empty session file with an empty one, or cross-checking the transcript — would prevent the data loss. The recent commit e266aa425b9 may partially address this for large sessions, but a general guard is still needed.
