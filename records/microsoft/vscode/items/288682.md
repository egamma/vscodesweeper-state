# Issue #288682 — Terminal rendering corruption on macOS Apple Silicon - GPU acceleration workaround ineffective

[https://github.com/microsoft/vscode/issues/288682](https://github.com/microsoft/vscode/issues/288682)

## Summary

Terminal rendering corruption on macOS Apple Silicon with WebGL GPU acceleration — multiple independent confirmations across M1/M4 and different macOS versions, but the root cause is unclear (could be xterm.js WebGL addon, Chromium/Electron GPU compositing, or macOS Metal driver). The issue is intermittent and not yet triaged or area-labeled; routing to terminal area.

## Change summary

Terminal display becomes intermittently corrupted/garbled on macOS Apple Silicon when using GPU-accelerated (WebGL) rendering, and setting gpuAcceleration to 'auto' does not help as it does on Windows.

## Best solution

The terminal team should investigate whether the WebGL context-loss handler in `xtermTerminal.ts` covers partial corruption (not just full context loss), and whether macOS Apple Silicon needs a platform-specific fallback policy in the `gpuAcceleration: 'auto'` path. If the corruption is in xterm.js's WebGL addon or Chromium's Metal backend, it should be filed upstream with a reproduction and tracked as `upstream`. One commenter confirmed that `gpuAcceleration: 'off'` resolves it — the team could consider detecting repeated corruption and auto-falling back on macOS.

## Evidence

- **multiple confirmations:** At least 6 independent reporters confirm the issue across M1 (macOS 26.3), M4 (macOS 15.4, 26.5.1), spanning VS Code 1.108–1.127.
- **workaround:** Comment 6 by cfloressuazo confirms gpuAcceleration: 'off' fixes it on macOS; 'auto' does not help unlike on Windows.
- **context loss handler exists:** src/vs/workbench/contrib/terminal/browser/xterm/xtermTerminal.ts:869-871 handles onContextLoss by disposing the WebGL renderer, but intermittent corruption may not trigger a full context loss event.
- **auto fallback logic:** XtermTerminal._suggestedRendererType only gets set to 'dom' on WebGL load failure (line 885), not on intermittent rendering corruption — the auto-recovery path doesn't cover this scenario.
- **predecessor issue:** https://github.com/microsoft/vscode/issues/284827 was a related terminal rendering issue closed via info-needed timeout, not a confirmed fix.
