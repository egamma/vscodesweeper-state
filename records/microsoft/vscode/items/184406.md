# Issue #184406 — editor.renderFinalNewline is misleading

[https://github.com/microsoft/vscode/issues/184406](https://github.com/microsoft/vscode/issues/184406)

## Summary

Keep open as an editor-core polish/product decision. Current main still exposes `editor.renderFinalNewline` and its `off`/`dimmed` values for the final empty line's line number, but there is no concrete duplicate, fixing commit, or clear policy basis for closing the naming/behavior request.

## Change summary

The reporter wants `editor.renderFinalNewline` renamed or changed so `off` hides the final empty newline line itself rather than only affecting the final line number.

## Best solution

Editor-core maintainers should decide whether to keep the existing public setting for compatibility or clarify/replace it with a better-named line-number setting. Any actual change would touch the editor option schema/API plus `src/vs/editor/browser/viewParts/lineNumbers/lineNumbers.ts`, where the final empty line number is rendered.

## Evidence

- **current setting schema:** `src/vs/editor/common/config/editorOptions.ts` still registers `editor.renderFinalNewline` with values `'off' | 'on' | 'dimmed'` and describes it as "Render last line number when the file ends with a newline."
- **current implementation:** `src/vs/editor/browser/viewParts/lineNumbers/lineNumbers.ts` reads `EditorOption.renderFinalNewline`; for the final empty line, `off` clears `renderLineNumber` and `dimmed` adds `dimmed-line-number`, so the setting affects line-number rendering rather than removing the final view line.
- **public API/history:** Commit `607528b742a4548a18cc20dd2c0dd6be44854a44` renamed the older `displayBlankLastLineNumber` option to `renderFinalNewline`, and `src/vs/monaco.d.ts` still exposes `renderFinalNewline`, so renaming it now would be an API/settings compatibility decision.
