# Issue #300863 — Opening a workspace within a directory whose name has changed results in Vscode recreating the directory under the old name

[https://github.com/microsoft/vscode/issues/300863](https://github.com/microsoft/vscode/issues/300863)

## Summary

The described behavior is by design: VS Code restores sessions at stored absolute paths, and the file service creates parent directories on save via `mkdirp` (see `src/vs/platform/files/common/fileService.ts:400`). The user's ask — a warning when a parent directory no longer exists — is a feature request. The reporter was asked to upgrade from 1.109.5 to the latest build 128 days ago with no response.

## Change summary

Warn the user when VS Code restores a workspace whose parent directory has been renamed or no longer exists, to prevent silent recreation of the old directory structure on save.

## Best solution

If pursued, this would belong in the workspace/session restore flow — detect that the workspace folder no longer exists before restoring editors and hot-exit backups, and prompt the user. However, this is a niche scenario and the current behavior (restore at stored paths, create directories on write) is intentional. The reporter should first confirm this reproduces on the latest build.
