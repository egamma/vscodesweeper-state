# Issue #169899 — Investigate performance of `menubar.updateOverflowAction`

[https://github.com/microsoft/vscode/issues/169899](https://github.com/microsoft/vscode/issues/169899)

## Summary

Still relevant on main: `updateOverflowAction` reads `container.offsetWidth` and per-element `buttonElement.offsetWidth` causing forced synchronous layouts during resize. The `update()` path schedules it via `scheduleAtNextAnimationFrame` (line 632), but the method itself still triggers layout thrashing, and `showMenubar()` calls it directly without scheduling.

## Change summary

The `menubar.updateOverflowAction` method forces multiple synchronous layouts via `offsetWidth` reads during window resize, degrading resize performance.

## Best solution

Adopt the measure/modify pattern inside `updateOverflowAction` (batch all `offsetWidth` reads before any style writes), and ensure all call sites go through the existing `scheduleAtNextAnimationFrame` gate. The `showMenubar()` direct call at line 658 should also be scheduled. File: `src/vs/base/browser/ui/menu/menubar.ts`.
