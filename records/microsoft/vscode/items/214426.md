# Issue #214426 — Hanged during folder rename when a file was being edited - Should show proper error message

[https://github.com/microsoft/vscode/issues/214426](https://github.com/microsoft/vscode/issues/214426)

## Summary

The issue is valid and traceable in the current source: `renameWithRetry` in `src/vs/base/node/pfs.ts` retries EACCES/EPERM/EBUSY errors for up to 60 seconds on Windows before surfacing the error, causing the perceived hang when a locked file blocks a folder rename. The abort-early guard (line 543–555) only checks whether the *target* is a non-file — it does not detect a locked file inside the *source* folder, so the full 60-second retry loop runs before the user sees the error.

## Change summary

Renaming a folder in the explorer hangs for up to 60 seconds on Windows when a file inside the folder is locked by another program, instead of showing a prompt error quickly.

## Best solution

Improve the abort heuristic in `renameWithRetry` (src/vs/base/node/pfs.ts) for folder renames: when the source is a directory and the first rename attempt fails with EACCES/EPERM/EBUSY, abort the retry loop immediately (or after a much shorter timeout) and surface the error promptly. The diskFileSystemProvider rename error handler (line 689) already rewrites EBUSY into a user-friendly message, so the main gap is the 60-second silent retry.
