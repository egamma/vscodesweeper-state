# Issue #165956 — About the filter behavior when entering Japanese into showQuickPick extension api.

[https://github.com/microsoft/vscode/issues/165956](https://github.com/microsoft/vscode/issues/165956)

## Summary

The bug describes inconsistent fuzzy-filter behavior for QuickPick items containing Japanese characters. The core fuzzy matching code (`matchesFuzzy` in `src/vs/base/common/filters.ts`) has no special CJK handling, and the QuickPick filter path (`quickInputTreeFilter.ts`) was refactored to use ObjectTree but still delegates to the same fuzzy matcher. The defect is plausible on current main but hasn't been confirmed on a modern build. Keeping open as a real, unresolved quick-pick filtering edge case.

## Change summary

QuickPick filtering behaves inconsistently when Japanese characters are entered — one item with a Japanese filename is filtered out while an ASCII-only item remains, when both should behave the same.

## Best solution

Investigate the fuzzy matching path in `src/vs/base/common/filters.ts` (`matchesFuzzy` / `fuzzyContiguousFilter`) and `src/vs/platform/quickinput/browser/tree/quickInputTreeFilter.ts` to ensure CJK/non-ASCII characters are handled consistently when matching against items containing mixed scripts. The fix would likely be in how `convertSimple2RegExpPattern` or the camelCase/contiguous matchers handle multi-byte characters.
