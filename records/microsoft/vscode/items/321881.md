# Issue #321881 — VS Code can become stuck reopening the Agent login window and prevent access to the normal editor

[https://github.com/microsoft/vscode/issues/321881](https://github.com/microsoft/vscode/issues/321881)

## Summary

Still reproduces on main: after commit e342779306f allowed agents window restoration, closing the normal editor first leaves only the agents workspace in the window state, so on restart only the agents login window appears with no way to reach the editor.

## Change summary

VS Code can get stuck reopening only the Agent login window on restart, preventing access to the normal editor.

## Best solution

The window restore logic in `windowsMainService.ts` should ensure that when the agents window is the only window being restored, a normal editor window is also opened (or provide a clear escape hatch from the agents window). The fix likely belongs in `doGetPathsFromLastSession` or around the `ensureAgentsWindow` flow — either by always opening an empty editor window alongside a sole agents-window restore, or by detecting the locked-out state and offering recovery.

## Vision fit

- **Fit:** aligned
- **Reason:** The agents window is core to the current agent-experience priorities, but a flow that locks users out of the editor violates the 'lightweight and fast' value and damages trust in the product.
- **Evidence:** Values: 'Out of the box, VS Code is a text editor: lightweight and fast.' Users must always be able to reach the editor.

- **restore logic:** In `src/vs/platform/windows/electron-main/windowsMainService.ts` lines 1008–1061, `doGetPathsFromLastSession` restores the last active window's workspace. If the agents window was the last closed, only its workspace is restored.
