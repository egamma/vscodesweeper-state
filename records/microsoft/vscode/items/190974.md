# Issue #190974 — env.shell API should not be the empty string on start up

[https://github.com/microsoft/vscode/issues/190974](https://github.com/microsoft/vscode/issues/190974)

## Summary

Still reproduces on main: `env.shell` returns empty string at startup because `_defaultProfile` is undefined until `$acceptDefaultProfile` is called. The `onDidChangeShell` event (PR #192886) shipped as a workaround but the core ask — a non-empty value on startup — remains unfixed. Author noted the fix is harder than expected due to profile resolution depending on workspace context.

## Change summary

The `vscode.env.shell` API returns an empty string at startup because the default terminal profile hasn't been resolved yet when extensions first access it.

## Best solution

Cache the resolved default profile path across sessions (e.g. in the main process or persisted storage) so that `env.shell` can return an optimistic value immediately, firing `onDidChangeShell` only if the cached value turns out stale. The fix likely touches `src/vs/workbench/api/common/extHostTerminalService.ts` (initial `_defaultProfile` value) and the main-side profile resolution in `src/vs/workbench/contrib/terminal/browser/terminalProfileResolverService.ts`.
