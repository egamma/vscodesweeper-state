# Issue #170708 — Cannot paste full file paths after copying file from file explorer on Linux

[https://github.com/microsoft/vscode/issues/170708](https://github.com/microsoft/vscode/issues/170708)

## Summary

Still reproduces on current main. The root cause is that Electron/Chromium on Linux does not convert the system clipboard's `text/uri-list` (set by file managers like Nautilus/Dolphin) into the browser's DataTransfer API during paste events — only `text/plain` (containing the filename) is exposed. VS Code's paste infrastructure correctly handles `text/uri-list` when present, but never receives it from the platform layer.

## Change summary

Pasting a file copied from a Linux file manager into VS Code's editor inserts only the filename instead of the full file path.

## Best solution

The ideal fix is upstream in Chromium/Electron: the paste event's DataTransfer should include `text/uri-list` when the system clipboard contains file URIs. A VS Code-side workaround could use Electron's native `clipboard.read('text/uri-list')` API (or `clipboard.readBuffer`) to directly read the system clipboard's uri-list and inject it into the DataTransfer before paste providers run — this would live in `src/vs/editor/contrib/dropOrPasteInto/browser/copyPasteController.ts` around the `readResources` path. Issue https://github.com/microsoft/vscode/issues/320375 is a more recent duplicate with detailed root-cause analysis that should be consolidated.

## Evidence

- **root cause:** src/vs/editor/browser/dataTransfer.ts toExternalVSDataTransfer() correctly builds a uri-list from files if present, but Linux file managers' clipboard content (text/uri-list) is not exposed by Chromium's paste DataTransfer — only text/plain (filename) arrives.
