# Issue #307203 — chat.hookFilesLocations: rejects UNC/absolute paths inconsistently vs chat.pluginLocations + hooks inside agent plugins should be auto-discovered

[https://github.com/microsoft/vscode/issues/307203](https://github.com/microsoft/vscode/issues/307203)

## Summary

The path validation difference is intentional: `chat.pluginLocations` is MACHINE-scoped (not synced) so it accepts absolute paths, while `chat.hookFilesLocations` uses sync-safe relative-only validation. The feature request (auto-discover hooks from plugins) is already implemented, as connor4312 confirmed. The remaining discussion about workspace-level pluginLocations is tracked separately.

## Change summary

User wants `chat.hookFilesLocations` to accept UNC/absolute paths like `chat.pluginLocations` does, and wants hooks auto-discovered from plugins.

## Best solution

The hook auto-discovery from plugins already works. The path restriction is intentional for settings-sync portability (aeschli's explanation). Users on WSL should use the Remote - WSL extension so paths resolve locally. The workspace-level pluginLocations request is tracked by https://github.com/microsoft/vscode/issues/303853.

  > Thanks for the detailed report. To address both parts:
  > 
  > 1. **Hook auto-discovery from plugins** — this already works. As Connor mentioned, hooks defined inside a registered plugin directory are discovered automatically. If you have a concrete case where this isn't happening, please share the plugin structure.
  > 
  > 2. **Path validation difference** — this is intentional. `chat.pluginLocations` is scoped to `MACHINE` (not synced across devices), so it can safely accept absolute/UNC paths. `chat.hookFilesLocations` uses relative-only validation to remain portable across settings sync, remotes, and devcontainers. For WSL scenarios, the recommended approach is to use the [Remote - WSL extension](https://marketplace.visualstudio.com/items?itemName=ms-vscode-remote.remote-wsl) so VS Code runs directly in the Linux filesystem — no UNC paths needed.
  > 
  > Since hook discovery from plugins already works (eliminating the need for manual `hookFilesLocations` entries in most cases), and the path restriction is a deliberate design choice for sync safety, I'm closing this as by-design.
