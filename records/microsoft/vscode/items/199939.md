# Issue #199939 — `createSession` passes in a set of scopes and the session returned shouldn't need to match those scopes

[https://github.com/microsoft/vscode/issues/199939](https://github.com/microsoft/vscode/issues/199939)

## Summary

Still reproduces on main: when an auth provider's createSession returns a session with scopes different from those requested, the sign-in badge persists because updateNewSessionRequests uses scopesMatch (exact equality). A team member confirmed this should be fixed.

## Change summary

The sign-in request badge does not dismiss when createSession returns a session whose scopes differ from the originally requested scopes.

## Best solution

In mainThreadAuthentication.ts around line 510, after createSession completes, clear the sign-in request for the *requested* scopes (scopeListOrRequest) rather than relying on the returned session's scopes to match. This likely means adding a variant of updateNewSessionRequests that accepts the requested scope key directly, or passing the requested scopes alongside the session.
