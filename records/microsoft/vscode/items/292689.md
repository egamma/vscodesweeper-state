# Issue #292689 — Setting output channel default verbosity has issues with extension channels

[https://github.com/microsoft/vscode/issues/292689](https://github.com/microsoft/vscode/issues/292689)

## Summary

Team-authored bug report — keeping open. The issue appears to be a race or ordering problem between how the argv-persisted default log levels are read on startup and when extension output channels register their loggers. The default level may not yet be loaded from argv when extension loggers initialize.

## Change summary

Setting a default output verbosity for extension channels does not persist correctly across window reloads or full restarts.

## Best solution

Investigate the initialization order in `DefaultLogLevelsService` and how extension host loggers pick up their initial log level. The argv file write in `setDefaultLogLevel` appears correct, but on reload the extension loggers may register before `_parseLogLevelsFromArgv` is consulted, or the file-change listener may fire too late. The fix likely belongs in `src/vs/workbench/services/log/common/defaultLogLevels.ts` (ensuring defaults are read synchronously on init) and possibly in the extension host logger registration path.

## Evidence

- **persistence mechanism:** Default log levels are persisted to argv.json via `_writeLogLevelsToArgv` in src/vs/workbench/services/log/common/defaultLogLevels.ts:136-144. On reload, they are re-read asynchronously via `_parseLogLevelsFromArgv` triggered by a file-change event (line 60-64).
- **race condition:** The initial `_defaultLogLevels` (line 56-59) only uses env-level values (`_getDefaultLogLevelFromEnv`/`_getExtensionsDefaultLogLevelsFromEnv`), not argv — the argv file is only consulted reactively via onDidFilesChange, meaning extension loggers that register before the file-change fires will get the wrong default.
