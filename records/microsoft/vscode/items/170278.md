# Issue #170278 — Alt+NumPadKey with num lock will still write the alt code character on top of performing the action

[https://github.com/microsoft/vscode/issues/170278](https://github.com/microsoft/vscode/issues/170278)

## Summary

This is a real but niche bug involving Windows alt-code character generation when Alt+NumPad keys are pressed with numlock off. No fix has been committed on main. The issue sits at the intersection of Windows OS alt-code behavior and VS Code's editor input handling, possibly requiring Electron/Chromium-level cooperation.

## Change summary

Pressing Alt+NumPad key (with numlock off, so the key acts as an arrow) correctly triggers the bound command but also inserts a spurious alt-code character.

## Best solution

The editor input layer (textAreaEditContextInput or nativeEditContext) should suppress character insertion when a keydown with Alt+NumPad was already dispatched as a keybinding. This may require tracking whether the keybinding service consumed the event and preventing the subsequent character input. Likely involves coordination between `src/vs/editor/browser/controller/editContext/` and `src/vs/workbench/services/keybinding/`. However, this may partly be an upstream Chromium/Electron behavior — needs investigation.
